  - id: why-isolates
    title: "Why the event loop isn't enough"
    summary: |
      We covered the event loop in the Futures episode. Dart is single-threaded — one piece of code runs at a time. The event loop keeps the UI responsive by scheduling work in small chunks.
      <br><br>
      But what happens when we need to run something that can't be split into small chunks? Parsing a 10 MB JSON file. Encrypting a large blob of data. Processing image pixels. These operations are <strong>CPU-bound</strong> — they hog the processor for hundreds of milliseconds with no natural pause points.
      <br><br>
      <pre><code>// This blocks the event loop — no frame renders until it's done
      List&lt;int&gt; fibonacci(int n) {
        if (n &lt;= 1) return [0, 1].take(n + 1).toList();
        var result = [0, 1];
        for (var i = 2; i &lt;= n; i++) {
          result.add(result[i - 1] + result[i - 2]);
        }
        return result;
      }

      void main() {
        var huge = fibonacci(50000000);  // blocks for seconds
        print('Done');
      }</code></pre>
      While <code>fibonacci</code> runs, no UI events are processed. The screen freezes. Users think the app has crashed. Async/await won't help — that's for <strong>I/O-bound</strong> work where we're waiting on external systems. For CPU-bound work, we need a second processor.
      <br><br>

      <strong>The mental model:</strong> Futures handle waiting. Isolates handle computing. Use the right tool for the shape of the problem.

  - id: what-is-isolate
    title: "What is an isolate?"
    summary: |
      An isolate is Dart's unit of concurrency. Each isolate has its own memory heap, its own event loop, and its own thread of execution. Multiple isolates can run truly in parallel on multi-core CPUs.
      <br><br>
      The name comes from the key design choice: <strong>isolates share no memory</strong>. Zero. None. They communicate only by passing messages — copying data or transferring ownership.

      <br><br>
      Why is this a feature, not a limitation? Because shared memory is the source of every concurrency nightmare: race conditions, deadlocks, torn reads, heisenbugs that only appear under load. By disallowing shared state, Dart eliminates entire categories of bugs at the language level. We can't accidentally mutate the same object from two threads — there are no shared objects.
      <br><br>
      <pre><code>// In the main isolate
      var counter = 0;

      // In a background isolate — this is a COPY
      // Incrementing it has no effect on the main isolate's counter</code></pre>
      <strong>What can be sent between isolates?</strong>
      <br><br>
      • Primitive values: int, double, bool, String, null
      <br>• Lists and Maps of sendable types
      <br>• Typed data (Uint8List, etc.)
      <br>• SendPort and ReceivePort (for bidirectional communication)
      <br>• Objects marked with special annotations in some cases
      <br><br>
      Non-sendable things include closures that capture mutable state, most custom objects, and anything with a connection to native resources (files, sockets, database handles).

  - id: spawn-and-compute
    title: "Spawning isolates — the modern API"
    summary: |
      Dart 2.19 introduced <code>Isolate.run()</code>, which is the simplest way to offload work. Give it a function and an argument, get back a Future with the result.
      <br><br>
      <pre><code>import 'dart:isolate';

      Future&lt;List&lt;int&gt;&gt; computeFibonacci(int n) async {
        // Runs in a separate isolate
        return await Isolate.run(() =&gt; _fibonacci(n));
      }

      List&lt;int&gt; _fibonacci(int n) {
        if (n &lt;= 1) return [0, 1].take(n + 1).toList();
        var result = [0, 1];
        for (var i = 2; i &lt;= n; i++) {
          result.add(result[i - 1] + result[i - 2]);
        }
        return result;
      }

      void main() async {
        print('Starting computation...');
        var result = await computeFibonacci(40);
        print('Done: \${result.length} numbers');
      }</code></pre>
      <code>Isolate.run()</code> handles all the plumbing: spawning, message passing, and cleanup. The closure runs in a fresh isolate, and the result is sent back to the caller.
      <br><br>
      <strong>The top-level function rule.</strong> The function passed to <code>Isolate.run()</code> must be a top-level function or a static method — not an instance method or a closure that captures local state. This is because the function needs to be serialised and sent to the new isolate.
      <br><br>
      <pre><code>// Works — top-level function
      int _square(int x) =&gt; x * x;
      var result = await Isolate.run(() =&gt; _square(42));

      // Works — static method
      class Math {
        static int cube(int x) =&gt; x * x * x;
      }
      var result = await Isolate.run(() =&gt; Math.cube(3));

      // FAILS — closure capturing local variable
      var multiplier = 10;
      var result = await Isolate.run(() =&gt; 5 * multiplier);  // Error!</code></pre>
      <strong>For Flutter: compute().</strong> Flutter wraps this further with the <code>compute()</code> helper from <code>foundation.dart</code>. It's the same idea but slightly more ergonomic for Flutter apps.
      <br><br>
      <pre><code>import 'package:flutter/foundation.dart';

      Future&lt;String&gt; parseJson(String rawJson) async {
        return await compute(_doParse, rawJson);
      }

      String _doParse(String json) {
        // Heavy parsing work here
        return jsonDecode(json).toString();
      }</code></pre>

      Here's the full pattern for a long-lived worker isolate:
      <br><br>
      <pre><code>import 'dart:isolate';

      void main() async {
        // Create a port to receive messages from the worker
        final receivePort = ReceivePort();

        // Spawn the worker, passing our SendPort
        await Isolate.spawn(
          _workerMain,
          receivePort.sendPort,
        );

        // First message from worker is its SendPort
        final workerSendPort = await receivePort.first as SendPort;

        // Now we can send tasks and receive results
        final responsePort = ReceivePort();
        workerSendPort.send(['fibonacci', 40, responsePort.sendPort]);

        final result = await responsePort.first;
        print('Result: \$result');
      }

      void _workerMain(SendPort mainSendPort) {
        final workerReceivePort = ReceivePort();

        // Send our port back to main so it can send us tasks
        mainSendPort.send(workerReceivePort.sendPort);

        // Listen for tasks
        workerReceivePort.listen((message) {
          final task = message[0] as String;
          final arg = message[1] as int;
          final replyPort = message[2] as SendPort;

          if (task == 'fibonacci') {
            final result = _fibonacci(arg);
            replyPort.send(result);
          }
        });
      }</code></pre>
      The handshake pattern: main creates a ReceivePort, spawns the worker with that port's SendPort, the worker creates its own ReceivePort and sends its SendPort back. Now both sides can talk.

  - id: isolate-lifecycle
    title: "Lifecycle — spawn, communicate, terminate"
    summary: |
      Isolates have a simple lifecycle: they're spawned, they run, they can be killed or they can exit naturally when their event loop empties.
      <br><br>
      <strong>Spawning.</strong> We've seen <code>Isolate.run()</code> and <code>Isolate.spawn()</code>. There's also the older <code>Isolate.spawnUri()</code> which loads code from a separate file — rarely needed in practice.
      <br><br>
      <pre><code>// Simple: run a function, get result, isolate exits
      var result = await Isolate.run(() =&gt; heavyWork());

      // Long-lived: spawn and keep reference
      final isolate = await Isolate.spawn(workerMain, sendPort);</code></pre>
      <strong>Termination.</strong> An isolate exits when its event loop has no more work and no open ports. We can also force termination:
      <br><br>
      <pre><code>// From outside: kill the isolate
      isolate.kill(priority: Isolate.immediate);

      // From inside: exit with a value
      Isolate.exit(sendPort, result);</code></pre>
      <code>Isolate.exit()</code> is more efficient than sending a message and returning — it transfers the data without copying.
      <br><br>
      <strong>Error handling.</strong> Errors in an isolate don't crash the main isolate. By default they're silently swallowed. To catch them:
      <br><br>
      <pre><code>final isolate = await Isolate.spawn(
        workerMain,
        sendPort,
        onError: errorPort.sendPort,  // errors sent here
        onExit: exitPort.sendPort,    // notified when isolate exits
      );

      errorPort.listen((error) {
        print('Worker error: \$error');
      });</code></pre>
      <strong>Debugging tip:</strong> Isolate errors can be hard to trace because stack traces don't cross isolate boundaries. Always add logging at the entry point of your worker function.

  - id: when-to-use
    title: "When to use isolates"
    summary: |
      Isolates have overhead. Spawning takes a few milliseconds. Message passing involves copying (or transferring) data. Don't use them for trivial work.
      <br><br>
      <strong>Good candidates for isolates:</strong>
      <br><br>
      • Parsing large JSON (>100KB typically)
      <br>• Image processing (resizing, filtering, compression)
      <br>• Cryptographic operations (hashing, encryption)
      <br>• Complex calculations (pathfinding, physics simulations)
      <br>• Database serialisation/deserialisation
      <br>• Any computation that takes >16ms (one frame at 60fps)
      <br><br>
      <strong>Bad candidates for isolates:</strong>
      <br><br>
      • Small JSON parsing (overhead exceeds savings)
      <br>• Simple arithmetic
      <br>• I/O-bound work (use Futures instead)
      <br>• Anything requiring UI access

  - id: flutter-patterns
    title: "Practical patterns for Flutter"
    summary: |
      In Flutter apps, the most common isolate pattern is <code>compute()</code> for one-off heavy work.
      <br><br>
      <pre><code>import 'package:flutter/foundation.dart';
      import 'dart:convert';

      class UserListScreen extends StatefulWidget {
        @override
        _UserListScreenState createState() =&gt; _UserListScreenState();
      }

      class _UserListScreenState extends State&lt;UserListScreen&gt; {
        List&lt;User&gt;? users;
        bool loading = true;

        @override
        void initState() {
          super.initState();
          _loadUsers();
        }

        Future&lt;void&gt; _loadUsers() async {
          final response = await http.get(Uri.parse('https://api.example.com/users'));

          // Heavy JSON parsing in isolate — UI stays responsive
          final parsed = await compute(_parseUsers, response.body);

          setState(() {
            users = parsed;
            loading = false;
          });
        }

        // Must be top-level or static
        static List&lt;User&gt; _parseUsers(String jsonString) {
          final List&lt;dynamic&gt; data = jsonDecode(jsonString);
          return data.map((json) =&gt; User.fromJson(json)).toList();
        }
      }</code></pre>
      <strong>Isolate pools for repeated work.</strong> If we're doing many isolate tasks, spawning a new isolate each time is wasteful. The <code>IsolatePool</code> from <code>package:isolate</code> or similar patterns keep workers alive:
      <br><br>
      <pre><code>// Conceptual pattern — pool implementation varies
      final pool = IsolatePool(concurrency: 4);

      Future&lt;List&lt;ProcessedImage&gt;&gt; processImages(List&lt;Uint8List&gt; images) async {
        final futures = images.map((img) =&gt; pool.run(_processImage, img));
        return Future.wait(futures);
      }</code></pre>
      <strong>Typed data transfer.</strong> For binary data like images, use <code>TransferableTypedData</code> or pass <code>Uint8List</code> directly. These can be transferred without copying when using <code>Isolate.exit()</code>:
      <br><br>
      <pre><code>// Efficient: data is transferred, not copied
      void _processImage(SendPort resultPort) {
        final processed = Uint8List(1000000);
        // ... fill processed ...
        Isolate.exit(resultPort, processed);  // zero-copy transfer
      }</code></pre>
      <strong>State management with isolates.</strong> Keep isolate results in your state management (Provider, Riverpod, Bloc). Don't let isolate logic leak into widgets. The widget asks for data; the repository decides whether to use an isolate.
      <br><br>
      <pre><code>class UserRepository {
        Future&lt;List&lt;User&gt;&gt; fetchUsers() async {
          final response = await _api.getUsers();
          // Decision to use isolate is encapsulated here
          if (response.body.length &gt; 100000) {
            return compute(_parseUsers, response.body);
          }
          return _parseUsers(response.body);
        }
      }</code></pre>
      The widget doesn't know or care that an isolate was used. That's the right separation of concerns.
