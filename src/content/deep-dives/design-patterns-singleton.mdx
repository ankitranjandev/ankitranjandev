  - id: the-problem-exactly-once
    title: "The problem — some things should exist exactly once"
    summary: |
      Some objects are meant to be the only one of their kind. A logger writing to a single file. An app config loaded once at startup. A database handle holding one connection pool. An HTTP client reusing one set of sockets. Spin up a second of any of these and you get two log buffers fighting over a file, two copies of config drifting out of sync, two connection pools where you budgeted for one.
      <br><br>
      So the question is simple. How do you guarantee that everyone in the app shares the exact same instance?
      <br><br>
      The naive answer is "just be careful — only call the constructor once". But an app has hundreds of call sites, and nothing stops the next one from typing <code>Logger()</code> again.
      <br><br>
      <pre><code>class Logger {
        final List&lt;String&gt; _buffer = [];
        void log(String msg) =&gt; _buffer.add(msg);
      }

      final a = Logger();
      final b = Logger();        // a second, separate logger

      a.log('saved');
      print(b._buffer.length);   // 0 — b never saw it</code></pre>
      Two objects, two buffers, and a bug that only shows up when one part of the app logs and another reads. We need the type itself to enforce the rule, not the discipline of whoever calls it.

      <br><br>
      That's the singleton's whole job: make the right-hand side true. Every <code>Logger()</code> in the codebase resolves to one object. The pattern is famous, it's everywhere, and — fair warning — it's also one of the most argued-over patterns there is. We'll build it first, then be honest about why.

  - id: the-dart-idiom
    title: "The Dart idiom — private constructor, static instance, factory gate"
    summary: |
      Most languages reach for a static <code>getInstance()</code> method. Dart has something cleaner. Because Dart lets a constructor be a <code>factory</code>, you can intercept <code>Logger()</code> itself and hand back a stored object. The call site looks like an ordinary constructor; behind it, there's a gate.
      <br><br>
      Three pieces make it work.
      <br>• <strong>A private constructor</strong> — <code>Logger._()</code>. The underscore makes it library-private, so no outside code can ever call <code>new</code> and mint a fresh object.
      <br>• <strong>A static final instance</strong> — built exactly once and held on the class.
      <br>• <strong>A factory constructor</strong> — the public <code>Logger()</code>, which returns that stored instance instead of building anything.
      <br><br>
      <pre><code>class Logger {
        // 1. private constructor — only this library can call it
        Logger._();

        // 2. the one and only instance, built once
        static final Logger _instance = Logger._();

        // 3. the public gate — always returns the same object
        factory Logger() =&gt; _instance;

        final List&lt;String&gt; _buffer = [];
        void log(String msg) =&gt; _buffer.add(msg);
      }</code></pre>

      <br><br>
      Now the call site can't break the rule even if it tries.
      <br><br>
      <pre><code>final a = Logger();
      final b = Logger();

      a.log('saved');
      print(b._buffer.length);     // 1 — same buffer
      print(identical(a, b));      // true — same object</code></pre>
      <code>a</code> and <code>b</code> are not two loggers that happen to agree. They are one logger with two names. That's the guarantee the type now enforces, and it took three short lines to get it.
      <br><br>
      You'll also see the same shape written with a <code>static</code> getter — <code>static Logger get instance =&gt; _instance</code>, called as <code>Logger.instance</code>. It's the same idea; the factory version just keeps the familiar <code>Logger()</code> call site.

  - id: lazy-vs-eager
    title: "Lazy vs eager — when does the instance actually get built?"
    summary: |
      Here's a detail Dart hands you for free. That <code>static final</code> field is not built when the program starts. It's built the first time something reads it — and never again. Dart static fields are <em>lazily initialised</em>.
      <br><br>
      Why care? Because the instance might be expensive. A database handle opens a connection. A config loader reads a file. If you eagerly built every singleton at startup, your app would pay for things it might never use. Lazy means you pay only on first touch.
      <br><br>
      <pre><code>class Database {
        Database._() {
          print('opening connection...');   // the expensive bit
        }
        static final Database _instance = Database._();
        factory Database() =&gt; _instance;
      }

      void main() {
        print('app started');
        final db = Database();    // 'opening connection...' prints HERE
        Database();               // nothing — already built
      }
      // app started
      // opening connection...</code></pre>
      Notice the order. <code>'app started'</code> prints first. The connection only opens when <code>Database()</code> is first called, not when <code>main</code> begins. The second call builds nothing — it walks straight to the stored instance.
      <br><br>
      <strong>Want eager instead?</strong> If you need the instance ready before anything asks for it — say, to fail fast on a bad config at launch — just touch it on purpose during startup.
      <br><br>
      <pre><code>void main() {
        Database();         // force the lazy field now
        runApp(MyApp());
      }</code></pre>
      Either way the construction runs once. Lazy is the default and usually what you want; eager is a deliberate nudge for the rare case where you'd rather crash at launch than mid-request.

  - id: service-locator
    title: "The service locator — the grown-up global"
    summary: |
      Hand-writing the private-constructor dance for every shared object gets old. And it bakes a hard decision into each class: the class itself decides it's a singleton. Sometimes you'd rather decide that from outside — register one shared instance in one place, and let the rest of the app look it up.
      <br><br>
      That's a <strong>service locator</strong>. In Flutter the popular one is the <code>get_it</code> package. You register your services once at startup, then resolve them anywhere by type.
      <br><br>
      <pre><code>final getIt = GetIt.instance;

      void setupLocator() {
        // one shared instance, created lazily on first lookup
        getIt.registerLazySingleton&lt;Api&gt;(() =&gt; HttpApi());
        getIt.registerLazySingleton&lt;Logger&gt;(() =&gt; FileLogger());
      }

      // anywhere in the app
      final api = getIt&lt;Api&gt;();        // same Api every time
      final logger = getIt&lt;Logger&gt;();  // same Logger every time</code></pre>
      Look at what changed. The class no longer forces itself to be a singleton — <code>HttpApi</code> is an ordinary class with an ordinary constructor. The <em>locator</em> decides there's one shared instance, and it registers against the <code>Api</code> interface rather than the concrete type. That last part matters more than it looks, and we'll come back to it.
      <br><br>
      <code>get_it</code> gives you a few registration modes worth knowing.
      <br>• <code>registerLazySingleton</code> — one instance, built on first lookup. The common choice.
      <br>• <code>registerSingleton</code> — one instance, built eagerly at registration.
      <br>• <code>registerFactory</code> — a fresh instance on every lookup. Not a singleton at all — the same container, opposite behaviour.
      <br><br>
      It's a tidy upgrade on a bare global variable: typed, central, and swappable in one place. But make no mistake about what it is. A service locator is still global access with better manners. Everything that reaches into <code>getIt</code> still depends on a global — it's just spelled more politely. Which brings us to the part nobody likes.

  - id: the-dangers
    title: "The dangers — global state, hidden dependencies, and tests"
    summary: |
      A singleton is a global variable wearing a class as a disguise. Everything you've ever heard about why globals are dangerous applies here, in full.
      <br><br>
      <strong>Global mutable state.</strong> One shared, writable object that any code can reach means any code can change it. A value set in one corner of the app surfaces as a surprise in another. With no single owner, you can't reason locally — you have to hold the whole app in your head to know what state the singleton is in right now.
      <br><br>
      <strong>Hidden dependencies.</strong> Look at this constructor and tell me what it needs to run.
      <br><br>
      <pre><code>class CheckoutController {
        CheckoutController();   // depends on... nothing?

        void submit(Cart cart) {
          Api().post('/checkout', cart);   // surprise: it needs Api
          Logger().log('checkout done');   // and Logger
        }
      }</code></pre>
      The signature lies. It claims no dependencies, then reaches out to two globals inside a method. Nothing in the type tells you this class can't work without a live <code>Api</code>. The dependencies are real, but invisible — you only find them by reading every line.
      <br><br>
      <strong>The testability problem.</strong> This is the one that bites hardest. You want to test <code>submit</code> without hitting the network. But <code>Api()</code> is hard-wired inside the method — there is no seam, no parameter, no way to slip a fake in. The test is forced to use the real thing.

      And there's a second sting in the tail: state leaks <em>between</em> tests. Because the singleton lives for the whole process, whatever test one writes into it is still there when test two runs. Your suite passes or fails depending on the order it ran in — the worst kind of flake. None of this means singletons are forbidden. It means you've taken on a real cost, and you should know you're paying it.

  - id: singleton-vs-injection
    title: "Singleton vs dependency injection — why Flutter passes things in"
    summary: |
      Every danger in the last topic has the same root: the dependency is fetched from a global instead of handed in. Flip that one decision and the problems dissolve. That flip has a name — <strong>dependency injection</strong> — and it's the idiom Flutter leans on.
      <br><br>
      Injection just means a class receives what it needs from outside, rather than reaching out to grab it. Compare the two shapes directly.
      <br><br>
      <pre><code>// Singleton: reaches out for the global
      class Checkout {
        void submit() =&gt; Api().post('/checkout');   // grabs it
      }

      // Injection: takes it in
      class Checkout {
        Checkout(this.api);
        final Api api;
        void submit() =&gt; api.post('/checkout');      // uses what it was given
      }</code></pre>
      The injected version has an honest signature. You cannot construct a <code>Checkout</code> without handing it an <code>Api</code>, so the dependency is visible, the test can pass a fake, and two checkouts can use two different APIs if you ever need that. Same behaviour, none of the hidden coupling.

      Flutter has machinery built for exactly this flow. The framework's <code>InheritedWidget</code> lets a value placed high in the tree be read by any descendant below it — and packages like <code>Provider</code> and <code>Riverpod</code> wrap that into a clean API. You put one instance near the top, and every widget under it reads the same object. You still get "one shared instance", which is what you wanted from the singleton — but it flows <em>down the tree</em> instead of sitting in a global, so a test can wrap the widget in a provider holding a fake.
      <br><br>
      <pre><code>// One shared Api, placed above the app
      Provider&lt;Api&gt;(
        create: (_) =&gt; HttpApi(),
        child: MyApp(),
      );

      // any descendant reads the same instance
      final api = context.read&lt;Api&gt;();</code></pre>
      Same single instance, opposite plumbing. That's why "prefer injection over a singleton" is the standard Flutter advice — not because shared state is bad, but because how it reaches the code that uses it makes all the difference.

  - id: when-singletons-are-fine
    title: "When a singleton is genuinely fine, and how to keep it testable"
    summary: |
      After all that, it would be easy to swear off singletons entirely. Don't. There are cases where one is the right call, and the goal isn't to ban the pattern — it's to use it without inheriting its worst traits.
      <br><br>
      A singleton earns its place when the thing it wraps is naturally one-of-a-kind and mostly stateless. A logger. A read-only config loaded once at boot. A wrapper over a resource the operating system already gives you exactly one of. The trouble was never "one instance" — it was global mutable state and hidden coupling. Remove those two, and a singleton is just a convenient shared object.
      <br><br>
      The trick is to <strong>program to an interface and inject the instance</strong>. Define the capability as an interface. Let the concrete class be a singleton if you like — but have your code depend on the interface, and receive it as a parameter.
      <br><br>
      <pre><code>// the contract code depends on
      abstract interface class Logger {
        void log(String msg);
      }

      // the concrete singleton — fine to keep one instance
      class FileLogger implements Logger {
        FileLogger._();
        static final FileLogger _instance = FileLogger._();
        factory FileLogger() =&gt; _instance;

        @override
        void log(String msg) =&gt; print('[file] \$msg');
      }

      // code depends on the INTERFACE and takes it in
      class Checkout {
        Checkout(this.logger);
        final Logger logger;
        void submit() =&gt; logger.log('checkout done');
      }</code></pre>
      In production you wire the real one: <code>Checkout(FileLogger())</code>. The single shared file logger flows in. In a test you hand over a fake that records calls instead of writing a file.
      <br><br>
      <pre><code>class FakeLogger implements Logger {
        final messages = &lt;String&gt;[];
        @override
        void log(String msg) =&gt; messages.add(msg);
      }

      test('checkout logs once', () {
        final logger = FakeLogger();
        Checkout(logger).submit();
        expect(logger.messages, ['checkout done']);   // no file, no global
      });</code></pre>
      That's the whole reconciliation. Keep the single instance where it helps, but never let your code <em>reach</em> for it — depend on an interface and inject the instance. You get the shared object without the global, and the seam for tests stays open. The singleton stops being a trap and goes back to being what it should have been all along: just one object, shared on purpose.
