  - id: the-problem-with-inheritance
    title: "The problem with single inheritance"
    summary: |
      We saw in the previous episode that a class can only extend one other class. Dog extends Animal. That's it. One parent.
      <br><br>
      But what if Dog also needs to be Swimmable? And Trainable? And Serializable?
      <br><br>
      <pre><code>class Animal {
        void breathe() => print('breathing');
      }

      class Swimmable {
        void swim() => print('swimming');
      }

      class Dog extends Animal, Swimmable {  // ERROR!
        // Dart doesn't allow multiple inheritance
      }</code></pre>
      We could use interfaces, but then we have to reimplement every method. If Swimmable has 10 methods, every swimming creature must write those 10 methods from scratch. That's not reuse — that's duplication.

  - id: what-is-a-mixin
    title: "What is a mixin?"
    summary: |
      A mixin is a class that provides methods for other classes to use — but it's not meant to be instantiated on its own.
      <br><br>
      <pre><code>mixin Swimmable {
        void swim() => print('swimming');

        void dive() => print('diving deep');
      }

      mixin Trainable {
        bool trained = false;

        void train() {
          trained = true;
          print('training complete');
        }
      }

      class Dog extends Animal with Swimmable, Trainable {
        Dog(String name) : super(name);
      }

      var dog = Dog('Buddy');
      dog.swim();    // swimming
      dog.train();   // training complete
      dog.breathe(); // breathing (from Animal)</code></pre>
      The keyword is <code>with</code>. A class can extend one parent and mix in any number of mixins. The dog gets behaviour from Animal, Swimmable, and Trainable — all without code duplication.
      <br><br>
      <strong>Key insight:</strong> Mixins are about sharing implementation, not just interfaces. When Dog mixes in Swimmable, it gets the actual <code>swim()</code> code. It doesn't have to write it.

  - id: mixin-vs-class
    title: "Mixin vs class — when to use which"
    summary: |
      Before Dart 3, you could use a class as a mixin with the <code>mixin class</code> declaration. But this blurred the line between classes and mixins. Modern Dart encourages a cleaner separation.
      <br><br>
      <strong>Use a mixin when:</strong>
      <br>• You're defining reusable behaviour (not identity)
      <br>• The behaviour doesn't need its own constructor
      <br>• Multiple unrelated classes need this behaviour
      <br><br>
      <strong>Use a class when:</strong>
      <br>• You're defining a concrete thing (a User, a File, a Button)
      <br>• You need constructors and initialisers
      <br>• Inheritance makes semantic sense (a Dog is-a Animal)
      <br><br>
      <pre><code>// Good: mixin for behaviour
      mixin Loggable {
        void log(String message) => print('[\$runtimeType] \$message');
      }

      // Good: class for identity
      class User with Loggable {
        final String name;
        User(this.name);
      }

      // Bad: class pretending to be reusable behaviour
      class LoggingCapability {  // should be a mixin
        void log(String msg) => print(msg);
      }</code></pre>
      <strong>The practical test:</strong> If you'd never write <code>var x = MyThing()</code> to create an instance, it should probably be a mixin.

  - id: mixin-on
    title: "Restricting mixins with 'on'"
    summary: |
      Sometimes a mixin only makes sense on certain types of classes. A mixin that accesses <code>this.name</code> only works if the class has a <code>name</code> field.
      <br><br>
      The <code>on</code> keyword restricts which classes can use a mixin.
      <br><br>
      <pre><code>class Named {
        final String name;
        Named(this.name);
      }

      mixin Greeter on Named {
        void greet() => print('Hello, I am \$name');
      }

      class Person extends Named with Greeter {
        Person(String name) : super(name);
      }

      class Robot with Greeter {  // ERROR!
        // Robot doesn't extend Named
      }</code></pre>
      The mixin Greeter can only be applied to classes that extend (or implement) Named. This gives us compile-time safety — if the mixin needs certain members, the <code>on</code> clause guarantees they exist.
      <br><br>
      <pre><code>mixin Persistable on Model {
        Future&lt;void&gt; save() async {
          // 'id' and 'toJson()' come from Model
          await database.save(id, toJson());
        }
      }

      abstract class Model {
        int get id;
        Map&lt;String, dynamic&gt; toJson();
      }

      class User extends Model with Persistable {
        @override
        int get id => _id;

        @override
        Map&lt;String, dynamic&gt; toJson() => {'name': name};
      }</code></pre>
      <strong>Multiple constraints:</strong> A mixin can require multiple supertypes.
      <br><br>
      <pre><code>mixin Auditable on Named, Timestamped {
        void audit() {
          print('\$name modified at \$lastModified');
        }
      }</code></pre>

  - id: mixin-order
    title: "Linearisation — the order of mixins matters"
    summary: |
      When multiple mixins define the same method, which one wins? Dart uses <strong>linearisation</strong> — the rightmost mixin takes precedence.
      <br><br>
      <pre><code>mixin A {
        void greet() => print('A');
      }

      mixin B {
        void greet() => print('B');
      }

      class Test with A, B {}

      void main() {
        Test().greet();   // prints: B
      }</code></pre>
      B comes after A, so B's <code>greet()</code> wins. The order is: first the superclass, then mixins left to right. Later entries override earlier ones.

      <br><br>
      <strong>Calling super in mixins.</strong> A mixin can call <code>super.method()</code> to invoke the previous version in the chain.
      <br><br>
      <pre><code>mixin Logger {
        void log(String msg) => print('LOG: \$msg');
      }

      mixin TimestampLogger on Object {
        void log(String msg) {
          var time = DateTime.now();
          super.log('[\$time] \$msg');  // calls previous log()
        }
      }

      class App with Logger, TimestampLogger {}

      App().log('started');
      // prints: LOG: [2026-05-24 10:30:00] started</code></pre>
      TimestampLogger calls <code>super.log()</code>, which resolves to Logger's <code>log()</code> method. This is called the <strong>super chain</strong>.

  - id: real-world-mixins
    title: "Real-world mixin patterns"
    summary: |
      Mixins shine when you have cross-cutting concerns — behaviours that span multiple unrelated classes.
      <br><br>
      <strong>Pattern 1: Diagnostics and debugging</strong>
      <br><br>
      <pre><code>mixin DiagnosticableMixin {
        String get debugLabel => '\$runtimeType#\$hashCode';

        void debugPrint(String message) {
          print('[\$debugLabel] \$message');
        }

        Map&lt;String, dynamic&gt; debugInfo() => {
          'type': '\$runtimeType',
          'hashCode': hashCode,
        };
      }</code></pre>
      <strong>Pattern 2: State management helpers</strong>
      <br><br>
      <pre><code>mixin ChangeNotifierMixin {
        final _listeners = &lt;void Function()&gt;[];

        void addListener(void Function() listener) {
          _listeners.add(listener);
        }

        void notifyListeners() {
          for (var listener in _listeners) {
            listener();
          }
        }
      }

      class Counter with ChangeNotifierMixin {
        int _count = 0;

        int get count => _count;

        void increment() {
          _count++;
          notifyListeners();
        }
      }</code></pre>
      <strong>Pattern 3: Validation</strong>
      <br><br>
      <pre><code>mixin Validatable {
        List&lt;String&gt; validate();

        bool get isValid => validate().isEmpty;

        void assertValid() {
          var errors = validate();
          if (errors.isNotEmpty) {
            throw ValidationError(errors);
          }
        }
      }

      class User with Validatable {
        final String email;
        final int age;

        User(this.email, this.age);

        @override
        List&lt;String&gt; validate() {
          var errors = &lt;String&gt;[];
          if (!email.contains('@')) errors.add('Invalid email');
          if (age < 0) errors.add('Age cannot be negative');
          return errors;
        }
      }</code></pre>
      These patterns work because the mixin provides a template — some behaviour is implemented, some is left for the class to define.
