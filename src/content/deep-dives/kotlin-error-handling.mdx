  - id: no-checked-exceptions
    title: "The checked-exceptions decision"
    summary: |
      Java has two kinds of exceptions. <strong>Unchecked</strong> ones (like <code>NullPointerException</code>) can be thrown anywhere. <strong>Checked</strong> ones (like <code>IOException</code>) must be declared in a method's <code>throws</code> clause and handled by every caller, or the code will not compile. Checked exceptions were a bold 1990s experiment in forcing programmers to confront failure. Kotlin looked at three decades of evidence and decided they had failed.
      <br><br>
      <strong>In Kotlin, every exception is unchecked.</strong> There is no <code>throws</code> clause, and the compiler never forces you to catch anything. This was a deliberate, somewhat controversial choice, and the Kotlin team has explained their reasoning at length.
      <br><br>
      The problem with checked exceptions, as observed in large Java codebases, is that they do not scale. In small examples they look responsible. At scale they produce two antipatterns. The first is the empty <code>catch</code> block — developers, forced to handle an exception they cannot meaningfully recover from right there, simply swallow it: <code>catch (e: IOException) { }</code>. That is strictly worse than not catching at all, because it hides the failure. The second is "exception pollution" — a low-level <code>throws IOException</code> ripples up through every intermediate method signature, coupling layers that should not know about each other.
      <br><br>
      <pre><code>// Java forces this; Kotlin does not&#10;// The infamous swallowed exception:&#10;try {&#10;    readFile()&#10;} catch (e: IOException) {&#10;    // do nothing — the failure vanishes silently&#10;}</code></pre>
      Notably, C# made the same call years earlier, and influential voices — including Java luminaries — came to agree the feature was a net negative. Kotlin's stance is that error handling is a real responsibility, but the <em>compiler</em> mandating where it happens does more harm than good. You are trusted to handle errors at the layer where you can actually do something useful about them.

  - id: try-and-throw-as-expressions
    title: "try and throw are expressions"
    summary: |
      In Kotlin, almost everything is an <strong>expression</strong> — it produces a value. This applies to <code>try</code> and even <code>throw</code>, which unlocks concise patterns impossible in Java.
      <br><br>
      A <code>try/catch</code> evaluates to a value: the value of the <code>try</code> block if it succeeds, or the value of the matching <code>catch</code> block if it throws.
      <br><br>
      <pre><code>val number: Int = try {&#10;    input.toInt()&#10;} catch (e: NumberFormatException) {&#10;    0          // the value of the whole try/catch when parsing fails&#10;}</code></pre>
      Even <code>throw</code> is an expression — and its type is <code>Nothing</code>, the bottom type from the Type System episode. Because <code>Nothing</code> is a subtype of every type, a <code>throw</code> can appear anywhere a value is expected, which is exactly what makes the Elvis-throw idiom type-check:
      <br><br>
      <pre><code>val name = user.name ?: throw IllegalArgumentException("name required")&#10;// throw has type Nothing, so the whole expression's type is the&#10;// non-null type of user.name — name is a non-null String here.</code></pre>
      This is a recurring theme in Kotlin's design: by making control-flow constructs into expressions and giving "never returns" a real type (<code>Nothing</code>), the language lets error handling compose naturally with everything else, instead of being a special statement-only corner of the syntax.

  - id: preconditions
    title: "Preconditions: require, check, and error"
    summary: |
      Most exceptions you throw deliberately fall into two buckets: "the caller gave me bad arguments" and "my object is in a state where this should not happen." Kotlin's standard library provides three precondition functions that make these intentions explicit and concise.
      <br><br>
      <pre><code>fun withdraw(amount: Int) {&#10;    require(amount &gt; 0) { "amount must be positive, was $amount" }&#10;    check(balance &gt;= amount) { "insufficient funds" }&#10;    balance -= amount&#10;}</code></pre>
      <strong><code>require</code></strong> validates arguments and throws <code>IllegalArgumentException</code> if the condition is false — use it at the top of a function to reject bad input (a "fail fast" guard). <strong><code>check</code></strong> validates state and throws <code>IllegalStateException</code> — use it when an operation is only valid in certain states. <strong><code>error(message)</code></strong> simply throws <code>IllegalStateException</code> unconditionally, and because it returns <code>Nothing</code>, it works in expression position (a common <code>else -&gt; error("unreachable")</code> in a <code>when</code>).
      <br><br>
      The message is a lambda, not a string — and that lambda is only evaluated when the check actually fails. So you can build a rich, interpolated error message without paying its cost on the (overwhelmingly common) success path. There is also <code>requireNotNull</code> and <code>checkNotNull</code>, which both validate and smart-cast the value to non-null, folding a null check and an error into one line.

  - id: runcatching-and-result
    title: "Functional errors: runCatching and Result"
    summary: |
      Sometimes you want to treat a failure as a <em>value</em> to pass around and transform, rather than a control-flow jump to catch. Kotlin's standard library offers <code>Result&lt;T&gt;</code> — a type that holds either a success value or a failure (a <code>Throwable</code>) — and <code>runCatching</code>, which executes a block and wraps the outcome in a <code>Result</code>.

      <br><br>
      <pre><code>val result: Result&lt;Config&gt; = runCatching { loadConfig() }&#10;&#10;val config = result&#10;    .map { it.validated() }       // transform the success value&#10;    .getOrElse { Config.DEFAULT } // supply a fallback on failure&#10;&#10;// Or handle both branches explicitly:&#10;result.fold(&#10;    onSuccess = { render(it) },&#10;    onFailure = { log(it) },&#10;)</code></pre>
      This style shines for pipelines where you want to keep flowing and decide how to handle failure at the end — <code>getOrNull</code>, <code>getOrDefault</code>, <code>recover</code>, and <code>mapCatching</code> give you a small algebra of outcomes.
      <br><br>
      <strong>The coroutine caveat.</strong> There is one trap that ties back to the Concurrency episode: a blanket <code>runCatching</code> (or <code>catch (e: Exception)</code>) inside a coroutine will also catch <code>CancellationException</code> — the very exception that implements cancellation — and silently break it. In coroutine code, either avoid swallowing everything or explicitly rethrow <code>CancellationException</code>. <code>Result</code> is a sharp tool; like the <code>!!</code> operator, it is excellent when used deliberately and dangerous when used to paper over failures.

  - id: resources-and-cleanup
    title: "Cleanup: finally and use"
    summary: |
      When something must run whether or not an exception is thrown — releasing a lock, closing a connection — Kotlin has the familiar <code>finally</code> block. But for the common case of "open a resource, use it, always close it," there is something cleaner.
      <br><br>
      Java introduced try-with-resources in Java 7 with special syntax. Kotlin needed no new syntax — it expresses the same idea as an ordinary extension function, <code>use</code>, available on anything that implements <code>Closeable</code>/<code>AutoCloseable</code>:
      <br><br>
      <pre><code>// 'use' closes the resource automatically, even if the block throws&#10;File("data.txt").bufferedReader().use { reader -&gt;&#10;    println(reader.readText())&#10;}   // reader.close() is guaranteed here</code></pre>
      The <code>use</code> function takes a lambda (the Functions episode again), runs it, and closes the resource in a <code>finally</code> internally — propagating your block's exception if there was one, or any close exception otherwise. It is a small but telling example of Kotlin's philosophy: where Java added a language feature, Kotlin reached for a library function built on lambdas and extensions. The same building blocks, composed differently, often remove the need for new syntax entirely.
