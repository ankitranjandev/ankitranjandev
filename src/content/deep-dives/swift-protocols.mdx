summary: |
  Protocols are Swift's way of describing <strong>what a type can do</strong> rather than <strong>what a type is</strong>. A protocol lists requirements; a type conforms by satisfying them. Unlike a base class, a protocol carries no stored state and a type may conform to many.<br><br>
  <pre><code>protocol Drawable {&#10;    var area: Double { get }&#10;    func draw()&#10;}&#10;&#10;extension Drawable {&#10;    func describe() { print("area is \(area)") }  // default impl&#10;}</code></pre>
  Two keywords decide how a protocol is used as a type. <code>any P</code> is a boxed <strong>existential</strong>: it can hold any conforming value and dispatches through a witness table at runtime. <code>some P</code> is an <strong>opaque type</strong>: one hidden concrete type the caller cannot name, with no box. Roughly: <strong>any</strong> hides many possible types behind a box; <strong>some</strong> hides one real type behind a curtain.
topics:
  - id: protocols-are-contracts
    title: "Protocols declare requirements, not inheritance"
    summary: |
      Let's start with the core idea. A protocol is a contract. It lists the methods, properties, initialisers, and subscripts that a conforming type must provide. It says nothing about how they work, only that they must exist.
      <br><br>
      <pre><code>protocol Shape {&#10;    var area: Double { get }&#10;    func draw()&#10;}&#10;&#10;struct Circle: Shape {&#10;    let radius: Double&#10;    var area: Double { .pi * radius * radius }&#10;    func draw() { print("drawing a circle") }&#10;}</code></pre>
      The colon in <code>struct Circle: Shape</code> declares conformance. In Swift, conformance is always explicit. A type does not accidentally satisfy a protocol because its method names happen to line up, the way Go interfaces work. You opt in by name, and the compiler then checks that every requirement is met.
      <br><br>
      So how is this different from inheriting a base class? A protocol is not a base class. It has no stored properties of its own, no designated initialiser to chain through, and no single-parent rule. A class can inherit from exactly one superclass, but any type — struct, enum, class, or actor — can conform to as many protocols as it likes.

  - id: protocol-extensions-default-impls
    title: "Protocol extensions and default implementations"
    summary: |
      A contract that only lists requirements is useful, but it gets repetitive. If every conforming type has to write the same helper method, we have not saved much. Swift 2, back in 2015, fixed this with <strong>protocol extensions</strong>.
      <br><br>
      A protocol extension lets the protocol itself ship behaviour. Conforming types then get those methods for free, without writing a line.
      <br><br>
      <pre><code>protocol Greeter {&#10;    func hello() -> String&#10;}&#10;&#10;extension Greeter {&#10;    func greetTwice() {&#10;        print(hello())&#10;        print(hello())&#10;    }&#10;}&#10;&#10;struct Robot: Greeter {&#10;    func hello() -> String { "beep" }&#10;}&#10;&#10;Robot().greetTwice()  // beep, then beep — for free</code></pre>
      We can even provide a default for a requirement, so a type may skip implementing it entirely and fall back on the protocol's version. This is how the standard library hands you dozens of methods on <code>Collection</code> from only a handful of requirements.
      <br><br>
      Now here is the subtlety that trips up everyone at least once. Inside a protocol extension, only members that are <strong>declared as requirements in the protocol</strong> are dynamically dispatched. A member added purely in the extension — never listed in the protocol body — is statically dispatched, chosen by the variable's compile-time type.

  - id: protocol-oriented-programming
    title: "Protocol-Oriented Programming — start with a protocol"
    summary: |
      In June 2015, at WWDC, Dave Abrahams gave a talk that gave the whole approach a name: "Protocol-Oriented Programming in Swift." Fans still call it the "Crusty" talk, after the grizzled old engineer Abrahams role-played to voice every objection a die-hard object-oriented programmer might raise. The mantra that came out of it was short and memorable: <strong>start with a protocol, not a class.</strong>
      <br><br>
      Why would we want to? Classes carry baggage. They are reference types, so two variables can point at the same instance and mutate it from under each other. Inheritance forces a single lineage, and a deep base class accumulates state that every subclass drags along. Abrahams' argument was that much of what we reach inheritance for — sharing behaviour, modelling capability — is better expressed as a protocol with a default implementation, adopted by a value type.
      <br><br>
      <pre><code>protocol Animatable {&#10;    var duration: Double { get }&#10;}&#10;&#10;extension Animatable {&#10;    func animate() { print("animating over \(duration)s") }&#10;}&#10;&#10;struct Fade: Animatable { let duration = 0.3 }&#10;struct Slide: Animatable { let duration = 0.5 }</code></pre>
      Here <code>Fade</code> and <code>Slide</code> are structs. They are copied, not shared, so there is no spooky aliasing. They gain <code>animate()</code> from the protocol extension without inheriting from anything. Add a new capability later and you write another small protocol rather than reshaping a class tree.
      <br><br>
      POP is not a rule that bans classes. Reference semantics are exactly right for shared, long-lived objects like a network session or a view controller. The point is the default: reach for a protocol and a value type first, and add a class only when you genuinely need identity and shared mutable state. That single shift in starting point is what Abrahams was selling, and it stuck.

  - id: associated-types-where-clauses
    title: "Associated types and where clauses"
    summary: |
      Some protocols cannot name their types up front. Take a container: it has elements, but it should not care whether they are <code>Int</code>, <code>String</code>, or anything else. Swift expresses this with an <strong>associated type</strong>, a placeholder the conforming type fills in.
      <br><br>
      <pre><code>protocol Container {&#10;    associatedtype Element&#10;    var count: Int { get }&#10;    mutating func append(_ item: Element)&#10;    subscript(i: Int) -> Element { get }&#10;}&#10;&#10;struct Stack&lt;T&gt;: Container {&#10;    private var items: [T] = []&#10;    var count: Int { items.count }&#10;    mutating func append(_ item: T) { items.append(item) }&#10;    subscript(i: Int) -> T { items[i] }&#10;    // Element is inferred to be T&#10;}</code></pre>
      This is exactly the design behind the standard library's <code>Sequence</code> and <code>Collection</code>, where the associated type is named <code>Element</code>. Every <code>map</code>, <code>filter</code>, and <code>reduce</code> you have ever written rides on it.
      <br><br>
      When we need to constrain those associated types, we use a <code>where</code> clause. It lets a generic function demand that a container's elements be, say, <code>Equatable</code>, or that two containers share the same element type.
      <br><br>
      <pre><code>func allEqual&lt;C: Container&gt;(_ c: C) -> Bool&#10;    where C.Element: Equatable {&#10;    // ...&#10;}</code></pre>
      For a long time, writing <code>Container&lt;Int&gt;</code> as a shorthand was impossible — you had to spell out the <code>where</code> clause every time. That changed with <strong>primary associated types</strong> in SE-0346 (Swift 5.7). By marking the associated type as primary, <code>protocol Container&lt;Element&gt;</code>, we can now write <code>some Container&lt;Int&gt;</code> or <code>any Collection&lt;Int&gt;</code> and constrain the element directly in angle brackets. It reads like a generic, and it removed a great deal of ceremony.

  - id: existentials-any
    title: "Existentials — any P and the box"
    summary: |
      Suppose we want a single variable that can hold any shape — a circle today, a square tomorrow. We cannot use the protocol as a normal generic constraint there, because we want one storage slot that swaps its concrete type at runtime. That is an <strong>existential</strong>, and in modern Swift we spell it <code>any Shape</code>.
      <br><br>
      The <code>any</code> keyword became explicit in SE-0335 (Swift 5.6). Before that, you wrote a bare protocol name as a type and the box was invisible, which hid a real cost. The keyword makes the cost legible.
      <br><br>
      <pre><code>let shapes: [any Shape] = [Circle(radius: 1), Square(side: 2)]&#10;for shape in shapes {&#10;    shape.draw()   // dispatched through the witness table&#10;}</code></pre>
      What actually happens? The value is wrapped in a box. The box holds the payload — stored inline if small, or a pointer to the heap if large — alongside a pointer to the type's <strong>protocol witness table</strong>. When you call a requirement, the runtime follows that pointer, looks up the method's slot, and jumps to the concrete implementation.

      That extra hop, plus the boxing, is the price of flexibility. It also explains an old frustration. A protocol with <code>Self</code> requirements or associated types — like <code>Equatable</code>, whose <code>==</code> needs two values of the <em>same</em> concrete type — could not be used as an existential at all. The compiler had no way to guarantee the boxed types matched. Swift has since relaxed much of this, and primary associated types let you write constrained existentials such as <code>any Collection&lt;Int&gt;</code>, but the underlying reason for the old limitation was the box's lost type identity.

  - id: opaque-types-some
    title: "Opaque types — some P, reverse generics"
    summary: |
      The existential box is flexible but costly, and sometimes we do not need the flexibility. If a function always returns the same concrete type, we just do not want to spell its name out — perhaps because it is an unpronounceable nest of generics. That is what <code>some P</code> is for. It arrived in SE-0244 (Swift 5.1).
      <br><br>
      An opaque type is sometimes called <strong>reverse generics</strong>. With an ordinary generic, the caller picks the type and the function adapts. With <code>some P</code>, the function picks one concrete type and hides it; the caller knows only that it conforms to <code>P</code>.

      Because the compiler knows the one true type, there is no box, no extra pointer, and dispatch can stay static. Crucially, <strong>identity is preserved</strong>: two values returned from a <code>some Shape</code> function are guaranteed to be the same underlying type, so the compiler will let you compare them if that type is <code>Equatable</code>. An <code>any Shape</code> gives you no such promise.
      <br><br>
      This is precisely why SwiftUI writes <code>var body: some View</code>. A view's body is a deeply nested generic — something like <code>VStack&lt;TupleView&lt;(Text, Button&lt;Text&gt;)&gt;&gt;</code> — that nobody wants to type or even see. Returning <code>some View</code> lets you hide that monstrous type while keeping it fully concrete, so SwiftUI's diffing engine can rely on the stable identity to update the screen efficiently. No boxing, full performance, and a clean signature.

  - id: putting-it-together
    title: "Putting it together — conformances and any vs some"
    summary: |
      A few features tie the whole picture together. <strong>Retroactive conformance</strong> lets you make a type you do not own conform to a protocol you do not own. Want <code>Int</code> to conform to your own <code>Describable</code> protocol? Write <code>extension Int: Describable { ... }</code> in your module and it is done. Use it judiciously — if two modules both retroactively conform the same type to the same protocol, you get a clash.
      <br><br>
      <strong>Conditional conformance</strong>, from SE-0143, lets a generic type conform only when its parameters do. An <code>Array</code> is <code>Equatable</code> only when its <code>Element</code> is <code>Equatable</code>, and the standard library expresses that as <code>extension Array: Equatable where Element: Equatable</code>. You get conformance exactly when it makes sense, and not a moment sooner.
      <br><br>
      And much of the time you write no conformance code at all. The compiler can <strong>synthesise</strong> <code>Equatable</code>, <code>Hashable</code>, and <code>Codable</code> for you. Declare the conformance, and as long as every stored property already conforms, Swift writes <code>==</code>, <code>hash(into:)</code>, and the coding logic itself.
      <br><br>
      <pre><code>struct Point: Equatable, Hashable, Codable {&#10;    let x: Int&#10;    let y: Int&#10;    // no method bodies needed — all synthesised&#10;}</code></pre>
      So when do we reach for <code>any</code> and when for <code>some</code>? Here is the short version. Use <code>some P</code> when there is one underlying type you simply want to hide — a return value, a SwiftUI body, anything where you want full performance and preserved identity. Use <code>any P</code> when you genuinely need a heterogeneous bag — an array that holds different conforming types at once, or a value whose concrete type is decided at runtime. Put plainly: <strong>some</strong> hides one real type behind a curtain; <strong>any</strong> hides many possible types inside a box. Default to <code>some</code>, and pay for <code>any</code> only when the flexibility earns its keep.
