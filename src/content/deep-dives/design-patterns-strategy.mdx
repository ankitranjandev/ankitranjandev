  - id: the-problem-many-ways
    title: "The problem — one job, many ways to do it"
    summary: |
      Picture a checkout. You can pay by card, by PayPal, by UPI. The job is always the same — take the money — but the <em>how</em> changes every time. So where does that how live?
      <br><br>
      The first instinct is one big method with a branch per option.
      <br><br>
      <pre><code>void checkout(String method, int amount) {
        if (method == 'card') {
          // charge the card...
        } else if (method == 'paypal') {
          // redirect to PayPal...
        } else if (method == 'upi') {
          // open the UPI app...
        }
        // and one more branch every single time
      }</code></pre>
      It works. Right up until the fifth payment method, and the sixth, and the tax rule that applies to only two of them. The method swells, every edit risks the branches you didn't touch, and testing one path means dragging the whole thing along.

      <br><br>
      The second instinct is a subclass per option — <code>CardCheckout</code>, <code>PayPalCheckout</code>, <code>UpiCheckout</code>. Cleaner at first. But now the behaviour is welded to the class hierarchy, and you can't switch it on a live object. The user changes their mind on the payment screen? You can't swap one subclass for another mid-flight.
      <br><br>
      Both instincts share the same flaw: the thing that varies is tangled into the thing that doesn't. Let's pull them apart.

  - id: pull-the-behaviour-out
    title: "The pattern — pull the behaviour into its own object"
    summary: |
      Here's the move. Take the part that varies — the payment behaviour — and lift it into its own object. The checkout stops knowing <em>how</em> to pay. It just holds something that does, and calls it.
      <br><br>
      There are two roles, and they map onto the two halves of the problem.
      <br><br>
      <strong>The context</strong> is the part that stays the same — the checkout flow. It holds a strategy and delegates to it.
      <br><br>
      <strong>The strategy</strong> is the part that varies — a tiny interface with one job, and a concrete class for each way of doing it.
      <br><br>
      <pre><code>// The strategy: an interface with one job
      abstract interface class PaymentStrategy {
        void pay(int amount);
      }

      class CardPayment implements PaymentStrategy {
        @override
        void pay(int amount) =&gt; print('Charging card: \$amount');
      }

      class UpiPayment implements PaymentStrategy {
        @override
        void pay(int amount) =&gt; print('Opening UPI for: \$amount');
      }

      // The context: holds a strategy, never the details
      class Checkout {
        final PaymentStrategy strategy;
        Checkout(this.strategy);

        void complete(int amount) =&gt; strategy.pay(amount);
      }</code></pre>

      Look at what <code>Checkout</code> depends on: the interface, never the concrete classes. That one detail is the whole payoff. You can add a tenth payment method tomorrow, and <code>Checkout</code> doesn't change a line.
      <br><br>
      <pre><code>var order = Checkout(CardPayment());
      order.complete(500);                  // Charging card: 500

      // user switched at the last second — just swap the object
      order = Checkout(UpiPayment());
      order.complete(500);                  // Opening UPI for: 500</code></pre>
      Same context, different strategy, different behaviour. No branches, no subclassing the checkout.

  - id: a-strategy-is-often-a-function
    title: "In Dart, a strategy is often just a function"
    summary: |
      Most of the time, though, a full interface is more ceremony than you need. A strategy is behaviour, and behaviour is a function — and in Dart functions are first-class. We leaned on that a few episodes back. This is where it pays off again.
      <br><br>
      Take sorting. <code>List.sort</code> has no idea how <em>you</em> want things ordered. So you hand it the rule.
      <br><br>
      <pre><code>final numbers = [3, 1, 2];

      numbers.sort((a, b) =&gt; a - b);   // ascending  → [1, 2, 3]
      numbers.sort((a, b) =&gt; b - a);   // descending → [3, 2, 1]

      final words = ['bb', 'a', 'ccc'];
      words.sort((a, b) =&gt; a.length - b.length);   // → ['a', 'bb', 'ccc']</code></pre>

      <br><br>
      And here's the thing — you've been passing strategies for this whole series without calling them that. <code>where</code> takes a predicate strategy. <code>map</code> takes a transform strategy. <code>fold</code> takes a combine strategy. The functional toolkit from the collections episode <em>is</em> the strategy pattern, sold without the label.

  - id: class-based-strategies
    title: "Class-based strategies — when behaviour carries state"
    summary: |
      So when do you reach for a whole class instead of a one-line function? The moment the strategy needs to <em>remember</em> something — configuration, dependencies, a name you'll reuse across the app.
      <br><br>
      A lambda is perfect for a throwaway rule. But a "10% bulk discount" strategy has state: the percentage. A class holds that cleanly.
      <br><br>
      <pre><code>abstract interface class PricingStrategy {
        int priceFor(int subtotal);
      }

      class FullPrice implements PricingStrategy {
        @override
        int priceFor(int subtotal) =&gt; subtotal;
      }

      class BulkDiscount implements PricingStrategy {
        final int percent;        // the state a lambda couldn't hold
        BulkDiscount(this.percent);

        @override
        int priceFor(int subtotal) =&gt; subtotal - (subtotal * percent ~/ 100);
      }</code></pre>
      Now the discount is a thing you can construct, configure, store, and pass around — <code>BulkDiscount(10)</code> versus <code>BulkDiscount(25)</code>. Try squeezing that into an anonymous function and you'll be capturing variables and losing the name.
      <br><br>
      <strong>Rule of thumb:</strong> stateless and short, reach for a function; stateful, configurable, or reused, reach for a class. Both are the exact same pattern. You're only choosing how much structure the behaviour has earned.

  - id: composition-over-inheritance
    title: "The real lesson — composition over inheritance"
    summary: |
      Strategy is one face of a principle you'll hear forever in Flutter circles: <strong>favour composition over inheritance.</strong> This pattern is what that slogan actually looks like in code.
      <br><br>
      Inheritance answers "what kind of thing is this?" by making a subclass. That's fine when behaviour varies on one axis. But it rarely does. Payment method <em>and</em> pricing rule <em>and</em> currency — model each axis with subclasses and the count multiplies. Three payments times two pricings is six classes, and the next axis multiplies all six again.

      Composition asks a different question: "what parts is this made of?" One <code>Checkout</code>, holding a payment strategy and a pricing strategy, each chosen on its own.       Flutter is built on this from the ground up. You don't subclass <code>Container</code> to get a padded, bordered, rounded box. You <em>wrap</em> a <code>Padding</code> around a <code>DecoratedBox</code> around your child. Widgets are parts you assemble, not types you extend. Strategy is the same instinct, pointed at behaviour instead of layout.

  - id: strategy-all-over-flutter
    title: "Strategy is threaded through all of Flutter"
    summary: |
      Once the shape clicks, you start spotting strategies everywhere in the framework — including in widgets you already use every day.
      <br>• <code>ListView.builder</code> takes an <code>itemBuilder</code> — a strategy for <em>how to build each row</em>.
      <br>• <code>showDialog</code> takes a <code>builder</code> — a strategy for <em>what to put on screen</em>.
      <br>• A scrollable takes <code>ScrollPhysics</code> — <code>BouncingScrollPhysics</code> for the iOS feel, <code>ClampingScrollPhysics</code> for Android. Swap the object, swap the behaviour.
      <br>• <code>sort</code> takes a <code>Comparator</code>. An animation takes a <code>Curve</code>. A text field takes a list of <code>TextInputFormatter</code>s.
      <br><br>
      <pre><code>ListView.builder(
        physics: const BouncingScrollPhysics(),   // scroll strategy
        itemCount: messages.length,
        itemBuilder: (context, i) =&gt; MessageTile(messages[i]),  // build strategy
      )</code></pre>
      None of these are subclasses of the widget that uses them. They're behaviours handed in from outside. That's exactly why one <code>ListView</code> can render a chat, a feed, or a settings page — you give it a different building strategy each time, and the list itself never changes.

  - id: choosing-and-neighbours
    title: "Choosing a form, and how Strategy relates to its neighbours"
    summary: |
      So how do you reach for this day to day? Start with the smallest thing that works and grow only when you must.
      <br>• A one-off rule → a lambda, written inline.
      <br>• A rule you reuse → a named function, passed as a tear-off.
      <br>• Behaviour with state or config → a class implementing a small interface.
      <br><br>
      All three are the strategy pattern. You're only sliding along a scale of how much structure the behaviour has earned. Most Flutter code lives at the top of that list and never says the word "strategy" out loud.
      <br><br>
      <strong>Strategy versus the Observer we just met.</strong> Observer is about <em>who gets told</em> when something changes. Strategy is about <em>which behaviour runs</em> when something happens. Different questions — and you'll often use both on one screen: an observer notifies the UI that the cart changed, and a strategy decides how the checkout button behaves.
      <br><br>
      <strong>Strategy and first-class functions.</strong> In a language without them, this pattern needs the full interface-and-classes machinery every time. Dart hands you most of it for free, which is why so much Dart code passes a function and never names the pattern at all.
      <br><br>
      And that's the quiet truth of design patterns in Dart: half of them are already in your hands as functions. Naming them just tells you when you're holding one.
