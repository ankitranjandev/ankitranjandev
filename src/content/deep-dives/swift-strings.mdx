  - id: string-is-a-collection-of-character
    title: "A String is a value type and a Collection of Character"
    summary: |
      Most languages treat a string as an array of bytes, or an array of code units, and let the programmer worry about what a "character" really means. Swift made a different, opinionated choice. A <code>String</code> is a value type — copying one gives you an independent copy, with copy-on-write keeping that cheap — and it is a <code>Collection</code> whose element type is <code>Character</code>.
      <br><br>
      So when we iterate a Swift string, we do not get bytes. We do not get code units. We get <code>Character</code> values, one per visible character, in order.
      <br><br>
      <pre><code>let greeting = "Cafe&#769;"&#10;for c in greeting {&#10;    print(c)   // C, a, f, e&#769;  &#8212; four Characters&#10;}&#10;print(type(of: greeting.first!))   // Character</code></pre>
      This is the thread that runs through the whole episode. Swift decided that the natural unit of a string is the thing a human points at and calls "a character", and it built the entire API around that decision. Almost everything that surprises newcomers — why <code>count</code> is slow, why you cannot write <code>s[2]</code>, why two differently encoded strings can still be equal — falls out of this one commitment.
      <br><br>
      The contrast with Dart is sharp. In the Dart series we saw that a Dart <code>String</code> is a sequence of UTF-16 code units, so iterating or measuring it exposes the encoding. Swift hides the encoding behind a model built around human perception. Both are defensible. They simply optimise for different things, and the rest of this episode is about that fork in the road.

  - id: character-is-a-grapheme-cluster
    title: "Character = extended grapheme cluster"
    summary: |
      So what exactly is a <code>Character</code>? It is what the Unicode standard calls an <strong>extended grapheme cluster</strong> — the smallest sequence of Unicode scalar values that a reader perceives as one character. The crucial word is <em>sequence</em>. A single <code>Character</code> may be built from several scalars working together.
      <br><br>
      Take the letter "&#233;". It can be a single scalar, U+00E9 (LATIN SMALL LETTER E WITH ACUTE). Or it can be two scalars: a plain "e" (U+0065) followed by a combining acute accent (U+0301) that floats on top of the previous letter. Both spellings render identically, and Swift treats each as exactly one <code>Character</code>.
      <br><br>
      It gets richer. A flag emoji such as the flag of Japan is two <em>regional indicator</em> scalars (a regional "J" plus a regional "P"). A family emoji is several full emoji glued together with invisible Zero-Width Joiners.

      Swift even updates its notion of grapheme clusters as the Unicode standard evolves, which is why the same code can count newer emoji correctly after a runtime update. The unit is human perception, and human perception is exactly what Unicode keeps refining.

  - id: count-is-on-and-string-index
    title: "Why count is O(n) and there is no Int subscripting"
    summary: |
      Here is the price of that decision. Because grapheme clusters are <strong>variable width</strong> — one cluster might be a single byte, the next might be eight — Swift cannot know where the fifth character lives without walking past the first four. There is no formula that turns "character number 5" into a memory offset.

      <br>
      So two things follow. First, <code>count</code> is <strong>O(n)</strong>: to count the characters, Swift must walk the whole string and segment it into clusters. Second, <code>String</code> is deliberately <strong>not</strong> subscriptable by <code>Int</code>. You cannot write <code>s[2]</code>. Instead you index with a <code>String.Index</code>, which represents a real position reached by walking from a known one.
      <br><br>
      <pre><code>let s = "ca&#233;&#127919;"&#10;// let third = s[2]            // compile error&#10;let i = s.index(s.startIndex, offsetBy: 2)&#10;print(s[i])                    // "&#233;"&#10;print(s.count)                 // 4  &#8212; computed in O(n)</code></pre>
      This feels hostile at first, especially coming from languages where <code>s[2]</code> is free. But it is honest. An <code>Int</code> subscript on a Unicode string is a lie in any language that offers it cheaply, because the integer indexes code units, not characters. Swift refuses to hand us a fast answer to the wrong question. Dart, by contrast, gives us O(1) indexing precisely because it indexes UTF-16 code units — which is why <code>"&#127919;".length</code> is 2 in Dart, while <code>"&#127919;".count</code> is 1 in Swift. One number is fast; the other is correct.

  - id: the-four-views
    title: "The four views of a string"
    summary: |
      Swift does not pretend the encoding does not exist. It just refuses to make it the default. When we genuinely need bytes or code units — for a network protocol, a file format, a C API — we ask for the appropriate <em>view</em>. The same string offers four of them, each a <code>Collection</code> at a different granularity.
      <br><br>
      <strong>1.</strong> The <code>Character</code> collection — the string itself, one grapheme cluster per element.<br>
      <strong>2.</strong> <code>unicodeScalars</code> — the individual Unicode scalar values (code points, excluding surrogates).<br>
      <strong>3.</strong> <code>utf16</code> — UTF-16 code units, the same granularity Dart's <code>.length</code> exposes.<br>
      <strong>4.</strong> <code>utf8</code> — UTF-8 code units, which are simply the bytes.
      <br><br>
      For plain ASCII all four counts agree. For anything interesting they diverge, and the divergence is the whole point.

  - id: canonical-equivalence
    title: "Canonical equivalence — when different bytes are equal"
    summary: |
      Now for the feature that most languages get quietly wrong. Recall the two spellings of "&#233;": the single scalar U+00E9, and the pair "e" + U+0301. They look identical. Are they equal?

  - id: utf8-backing-and-the-string-revolution
    title: "UTF-8 backing and the String revolution"
    summary: |
      What lives underneath all this? Since Swift 5, the native storage for a <code>String</code> is <strong>UTF-8</strong>. This was a substantial piece of engineering known informally as the "String revolution", driven on the standard-library side by Michael Ilseman and the Swift team, and it shipped in 2019.
      <br><br>
      It was not always so. Earlier Swift could back a string with UTF-16, largely so that bridging to <code>NSString</code> on Apple platforms — which is UTF-16 internally — could be cheap. But the centre of gravity for Swift shifted. Server-side Swift, Linux, and the web all speak UTF-8. The web is overwhelmingly UTF-8. Most text in the wild, especially ASCII-heavy content like JSON, HTML, and source code, is far more compact in UTF-8 than in UTF-16. So UTF-8 won on memory and on interoperability, and Swift moved its native representation accordingly.
      <br><br>
      <pre><code>let s = "hello"&#10;// Backed by UTF-8 in contiguous memory.&#10;s.withContiguousStorageIfAvailable { _ in /* fast path */ }&#10;Array(s.utf8)   // [104, 101, 108, 108, 111]</code></pre>
      There is a second optimisation worth knowing: the <strong>small-string optimisation</strong>. A short string (on a 64-bit platform, up to fifteen UTF-8 bytes) is stored inline inside the <code>String</code> value itself, with no heap allocation at all. Names, keys, short identifiers, and the like never touch the heap. Only when a string grows past that inline budget does Swift allocate backing storage and switch on copy-on-write. For the countless tiny strings a real program juggles, this is a quiet but large win.

  - id: practical-takeaways
    title: "Practical takeaways and trivia"
    summary: |
      Let's pull it together into a working mental model.
      <br><br>
      <strong>Reach for the right view.</strong> For anything a person reads — truncating a label, counting characters, reversing text — work with <code>Character</code> values and <code>count</code>. For protocols, hashing, and storage, use <code>utf8</code>. Use <code>utf16</code> mainly when interoperating with APIs that speak code units, such as ranges handed to you by Cocoa text systems. Reach for <code>unicodeScalars</code> when you care about code points specifically, for example classifying or normalising.
      <br><br>
      <strong>Bridging to NSString.</strong> On Apple platforms <code>String</code> bridges to <code>NSString</code>, but their world views differ: <code>NSString</code>'s <code>length</code> and integer indices are UTF-16 code units, exactly like Dart's <code>.length</code>. When you take an <code>NSRange</code> from a UIKit or AppKit text component and want to apply it to a Swift <code>String</code>, convert through the <code>utf16</code> view and <code>String.Index</code> so you do not slice through a character.
      <br><br>
      <pre><code>// Wrong mental model: "char at integer position".&#10;// Right model: index by walking, measure by the right view.&#10;let s = "Re&#769;sume&#769;"&#10;print(s.count)               // 6 Characters&#10;print(s.unicodeScalars.count)// 8 scalars&#10;print(s.utf8.count)          // 10 bytes</code></pre>
      <strong>The cross-language picture.</strong> Where Swift bakes grapheme-cluster correctness into the core type, other languages bolt it on. Dart needs the <code>package:characters</code> library to count grapheme clusters; without it, <code>"&#128512;".length</code> is 2 because Dart counts UTF-16 code units. JavaScript has the same UTF-16 default and reaches for <code>Intl.Segmenter</code>. Python 3 counts Unicode scalars, so <code>len("&#128512;")</code> is 1 there but <code>len("&#127471;&#127477;")</code> is 2 — closer to Swift's scalar view than to its character view. Swift is unusual in making the human-perceived character the default unit of the language's own string type.
      <br><br>
      One last contrast to close the loop with the Dart episode. In Dart, <code>"&#127919;".length == 2</code> because the string is UTF-16 and the dart emoji is a surrogate pair. In Swift, <code>"&#127919;".count == 1</code> because the string is a collection of grapheme clusters. Neither is wrong. Dart optimises for a fast, predictable code-unit model and asks you to opt in to correctness; Swift optimises for correctness and asks you to opt in to speed. Knowing which model you are standing in is the whole game.
