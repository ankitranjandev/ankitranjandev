  - id: from-compositors
    title: "From compositors to strings"
    summary: |
      We now know how to deal with numbers. But what about my last chat with my friend? How do we store that?
      <br><br>
      In the 19th-century printing industry, <a href="https://en.wikipedia.org/wiki/Compositor_(typesetting)">compositors</a> used to typeset words and characters for printing, letter by letter, in metal type. They used the word <strong>string</strong> to describe a length of typeset text — and a compositor's pay was measured by the length of the strings they had typeset. In Dart, strings work in a remarkably similar way: a sequence of characters with a measurable length, that we can take apart and reassemble.
      <br><br>
      Let's start with the basics.
      <br><br>
      <pre><code>var str1 = 'Hello Dart';
      String str2 = 'Hello again';</code></pre>
      Both forms work. <code>String</code> is the explicit type. <code>var</code> lets Dart infer it. Most idiomatic Dart uses <code>var</code> for local variables.
      <br><br>
      <strong>Single quotes and double quotes are equivalent</strong> in Dart — unlike some languages. Pick whichever has fewer quotes inside the string:
      <br><br>
      <pre><code>var a = 'I said "hello"';
      var b = "It's a string";</code></pre>
      <strong>Triple quotes give us multi-line strings:</strong>
      <br><br>
      <pre><code>var sql = '''
        SELECT name, age
        FROM users
        WHERE active = true
      ''';</code></pre>
      <strong>The <code>r</code> prefix gives us a raw string</strong> — backslashes are not escape characters, and dollar signs are not interpolation. Useful for regex patterns and Windows paths:
      <br><br>
      <pre><code>var pattern = r'\d+\.\d+';   // literally  \d+\.\d+
      var path    = r'C:\Users';   // literally  C:\Users</code></pre>
      That covers the surface. The interesting parts are how Dart stores these objects, what we can do with them, and the surprising ways text behaves when we look closely. Let's go deeper.

  - id: concatenation
    title: "Three ways to glue strings together"
    summary: |
      Dart gives us three flavours of string joining, and each has its place.
      <br><br>
      <strong>1. Literal concatenation.</strong> Two adjacent string literals are automatically joined at compile time. No operator needed:
      <br><br>
      <pre><code>var greeting = 'Hello '
                     'Dart';   // 'Hello Dart'</code></pre>
      This is great for breaking long literal strings across multiple lines without using <code>+</code> at runtime.
      <br><br>
      <strong>2. The <code>+</code> operator.</strong> The classic concatenation:
      <br><br>
      <pre><code>var first = 'Ankit';
      var last  = 'Ranjan';
      var name  = 'Hello ' + first + ' ' + last;</code></pre>
      Works, but it gets noisy fast. And under the hood, every <code>+</code> creates a new string.
      <br><br>
      <strong>3. String interpolation.</strong> The idiomatic Dart way:
      <br><br>
      <pre><code>var name = 'Hello $first $last';</code></pre>
      Inside a string, <code>$identifier</code> inserts that variable's value. For more complex expressions, wrap them in <code>${}</code>:
      <br><br>
      <pre><code>var price = 9.99;
      var line = 'Total: \$${price.toStringAsFixed(2)}';   // 'Total: $9.99'</code></pre>
      Note the escape — we used <code>\$</code> to write a literal dollar sign.
      <br><br>
      <strong>A small exercise — the compositor's pay.</strong> Compositors were paid by the length of the text they set. Let's write that function:
      <br><br>
      <pre><code>double calculatePay(String text, double ratePerCharacter) {
        return text.length * ratePerCharacter;
      }

      void main() {
        var line = 'The quick brown fox';
        print(calculatePay(line, 0.05));   // 0.95
      }</code></pre>
      Innocent enough. We'll come back to this function in a few topics — and discover that <code>length</code> has a surprise in store for us.

  - id: immutability
    title: "Strings are immutable — and StringBuffer fills the gap"
    summary: |
      Here is a fact about Dart strings that quietly powers everything else in this episode: <strong>strings are immutable</strong>.
      <br><br>
      Once a <code>String</code> object exists, its characters cannot change. We cannot reach in and overwrite a letter. Every method that looks like it modifies a string — <code>toUpperCase</code>, <code>replaceAll</code>, <code>trim</code>, <code>substring</code> — actually returns a brand new string and leaves the original untouched.
      <br><br>
      <pre><code>var s = 'hello';
      s.toUpperCase();   // returns 'HELLO' — but s is still 'hello'
      s = s.toUpperCase();   // now s is 'HELLO'</code></pre>
      Immutability feels restrictive, but it is the foundation for several powerful optimizations:
      <br><br>
      • <strong>Interning</strong> — if a string can't change, two variables can safely share one object. (We'll get to this in the next topic.)
      <br>
      • <strong>Hash code caching</strong> — once a string's hash code is computed, it can be cached forever. This is why <code>Map&lt;String, V&gt;</code> is so fast.
      <br>
      • <strong>Safe sharing across isolates</strong> — Dart's concurrency model can share immutable strings between isolates without copying.
      <br>
      • <strong>Fast comparison shortcuts</strong> — for interned strings, equality can be a pointer comparison instead of a character-by-character scan.
      <br><br>
      But there is one case where immutability hurts: building a string piece by piece in a loop.
      <br><br>
      <pre><code>// The naive way — and it's badly slow for big loops.
      var result = '';
      for (var word in words) {
        result += word + ' ';   // every += creates a new string
      }</code></pre>
      Each <code>+=</code> allocates a new string, copies the previous content, and appends the new piece. For <code>n</code> words, that's roughly <code>n²/2</code> character copies. For a thousand words, half a million copies. For ten thousand, fifty million. This is the classic O(n²) trap.
      <br><br>
      The fix is <strong>StringBuffer</strong> — a mutable, growing buffer designed exactly for this case:
      <br><br>
      <pre><code>var buffer = StringBuffer();
      for (var word in words) {
        buffer.write(word);
        buffer.write(' ');
      }
      var result = buffer.toString();</code></pre>

      Notice the parallel to Episode 2's booleans — <code>true</code> and <code>false</code> are also canonical instances. The same trick, applied to strings.
      <br><br>
      <strong>What gets interned, and what doesn't.</strong> String literals appearing in our source code are reliably interned. Constants computed at compile time (via <code>const</code>) are interned. But strings built dynamically at runtime — through <code>+</code>, <code>$interpolation</code>, <code>StringBuffer</code>, or any computation — generally aren't.
      <br><br>
      <pre><code>var str1 = 'Hello Dart';
      var str2 = 'Hello Dart';
      var dyn  = 'Hello ' + ('Dart');

      print(identical(str1, str2));   // true  — both literals, interned
      print(identical(str1, dyn));    // false — dyn was built at runtime</code></pre>
      <strong>A note for the curious.</strong> Unlike Java's <code>String.intern()</code>, Dart does not expose a public method to force a runtime-built string into the pool. The canonicalisation rules are defined by the language specification and largely handled by the compiler. Modern Dart consistently canonicalises string literals across <code>dart</code>, <code>dart compile js</code>, and <code>dart compile exe</code>, but the behaviour for runtime concatenation is implementation-dependent and shouldn't be relied on.
      <br><br>
      <strong>The practical takeaway.</strong> For string equality, always use <code>==</code> (which compares content). Use <code>identical()</code> only when we specifically need to know whether two references point to the same object — for example, when reasoning about memory or debugging. Never write equality logic that assumes string interning.

  - id: internal-storage
    title: "Inside the VM — how strings are actually stored"
    summary: |
      So far we have talked about strings as a sequence of characters. Inside the Dart VM, they are more concretely a sequence of <strong>code units</strong>, stored in one of two compact representations.
      <br><br>
      • <strong>OneByteString</strong> — used when every character in the string fits in 8 bits (Latin-1 range, U+0000 to U+00FF). One byte per character. This covers all ASCII text plus the accented characters of Western European languages.
      <br>
      • <strong>TwoByteString</strong> — used when any character is above Latin-1. Two bytes per character (UTF-16 encoding).
      <br><br>
      The VM picks automatically. Most English text lives in OneByteString — 50% smaller than it would be in TwoByteString. The moment we add a single character outside Latin-1, the whole string promotes to TwoByteString.
      <br><br>
      <pre><code>var ascii   = 'Hello';       // OneByteString — 5 bytes of content
      var french  = 'Café';        // OneByteString — 4 bytes (é fits in Latin-1)
      var chinese = '你好';        // TwoByteString — 4 bytes (Chinese needs 16-bit chars)
      var mixed   = 'Hello 你好';  // TwoByteString — entire string upgraded</code></pre>
      The internal class is an implementation detail we don't reference directly in code. But it matters for two reasons:
      <br><br>
      • <strong>Memory:</strong> a million ASCII strings of length 100 are about 100 MB in OneByteString and 200 MB in TwoByteString. The choice doubles or halves our memory footprint.
      <br>
      • <strong>Performance:</strong> operations like equality, hashing, and iteration are faster on OneByteString because the VM doesn't have to deal with surrogate pairs.
      <br><br>
      <strong>The historical reason for two byte representations.</strong> Dart was designed to compile to JavaScript, which has native UTF-16 strings. To make the boundary cheap, Dart picked the same internal encoding. OneByteString is an optimization layered on top, transparent to user code.
      <br><br>
      We won't normally think about this distinction. But it explains why some surprising behaviours — like an emoji counting as length 2 — make sense when we look at the underlying bytes.

      <br><br>
      The smiley emoji is Unicode code point U+1F600. That value is bigger than 16 bits can hold (0xFFFF). UTF-16 splits it into a <strong>surrogate pair</strong> — two 16-bit values that together encode the one code point. <code>String.length</code> counts those individual 16-bit code units, not the underlying characters.
      <br><br>
      In Dart, when we work with text, we have to choose between three different views.
      <br><br>
      <strong>1. Code units</strong> — the raw 16-bit UTF-16 values. This is what <code>String.length</code> and <code>String[]</code> give us. Useful for low-level binary work, often wrong for human-facing counts.
      <br><br>
      <strong>2. Code points (runes)</strong> — the actual Unicode scalar values, after combining surrogate pairs. Use <code>String.runes</code>:
      <br><br>
      <pre><code>var text = 'Dart 💙';
      print(text.length);              // 7  — counts code units
      print(text.runes.length);        // 6  — counts code points

      for (var rune in text.runes) {
        print('$rune  →  ${String.fromCharCode(rune)}');
      }
      // 68  →  D
      // 97  →  a
      // 114 →  r
      // 116 →  t
      // 32  →  (space)
      // 128153 → 💙</code></pre>
      We can also write Unicode escapes inline:
      <br><br>
      <pre><code>var heart  = '\u2764';      // ❤   (code point U+2764, fits in one code unit)
      var smiley = '\u{1F600}';   // 😀  (code point U+1F600, needs the braces)</code></pre>
      <strong>3. Grapheme clusters (characters)</strong> — what humans actually perceive as a "character." For most cases this matches code points, but some characters are composed of multiple code points joined together. The family emoji <code>👨‍👩‍👧</code> is three person emoji joined by zero-width joiners — five code points, but humans see it as one character.
      <br><br>
      For this we need the <code>characters</code> package (officially maintained by the Dart team):
      <br><br>
      <pre><code>import 'package:characters/characters.dart';

      var emoji = '👨‍👩‍👧';
      print(emoji.length);                // 8   (code units)
      print(emoji.runes.length);          // 5   (code points)
      print(emoji.characters.length);     // 1   (one grapheme cluster — correct!)</code></pre>
      <strong>Which one do we want?</strong> It depends on what we're counting.
      <br><br>
      • Counting bytes for storage or network → code units (or convert to UTF-8 first)
      <br>
      • Counting Unicode characters (e.g., for code analysis) → runes
      <br>
      • Counting what a human would call "characters" (for UI display, validation, truncation) → <code>characters</code>
      <br><br>
      For our compositor function: if we're paying for "characters as humans see them," we should be using <code>text.characters.length</code>. If we're literally paying for the cost of storing the typeset, we might actually want code units (some encodings are smaller, some are larger). The right answer depends on what "length" means in the context — and now we can pick deliberately.

  - id: utf8-and-the-outside-world
    title: "UTF-8 and the outside world"
    summary: |
      Inside Dart, strings are UTF-16. Outside Dart — in files, network sockets, JSON, HTTP responses, databases — the world is overwhelmingly <strong>UTF-8</strong>.
      <br><br>
      UTF-8 has a few advantages that have made it the de facto standard for text on disk and over the wire.
      <br><br>
      • <strong>ASCII compatibility</strong> — ASCII characters are encoded as single bytes, identical to ASCII. So plain English text in UTF-8 is half the size of UTF-16.
      <br>
      • <strong>Variable width</strong> — common characters use 1 byte, less common ones use 2, 3, or 4. The most-used characters are smallest.
      <br>
      • <strong>Self-synchronising</strong> — if a byte stream gets truncated, we can find the next character boundary easily.
      <br><br>
      Dart's <code>dart:convert</code> library handles the conversion both ways.
      <br><br>
      <pre><code>import 'dart:convert';

      var message = 'Hello 😀';

      // Dart String → UTF-8 bytes (for writing to a file or network)
      List&lt;int&gt; bytes = utf8.encode(message);
      print(bytes);
      // [72, 101, 108, 108, 111, 32, 240, 159, 152, 128]
      // 'H', 'e', 'l', 'l', 'o', ' ', and 4 bytes for the smiley

      // UTF-8 bytes → Dart String (when reading)
      String decoded = utf8.decode(bytes);
      print(decoded);   // 'Hello 😀'</code></pre>
      Notice the asymmetry: the emoji is <strong>2 code units</strong> in Dart's internal UTF-16, but <strong>4 bytes</strong> in UTF-8 on the wire. ASCII characters are <strong>2 bytes</strong> internally but <strong>1 byte</strong> on the wire. Conversion is not free, but it is almost always cheap compared to the I/O it enables.
      <br><br>
      <code>dart:convert</code> also exposes <code>latin1</code> and <code>ascii</code> codecs for the rarer cases where we know the data uses those encodings. And there's <code>jsonEncode</code> / <code>jsonDecode</code> for JSON, which internally handle the UTF-8 conversion for us.
      <br><br>
      <strong>The substring trap.</strong> One more performance corner worth knowing. Dart's <code>substring</code> is implemented as a straightforward copy — it allocates a new string and copies the bytes. This is O(n) in the length of the substring, not O(1) as we might assume.
      <br><br>
      <pre><code>// Building substrings in a loop is quadratic for the same reason as += was.
      for (var i = 0; i &lt; text.length; i++) {
        process(text.substring(0, i));   // O(i) work each iteration
      }</code></pre>
      Combined with the surrogate pair behaviour, <code>substring</code> has a second trap: if our indices land in the middle of a surrogate pair, we get a malformed string with an unpaired surrogate. For text containing emoji or other supplementary characters, prefer <code>characters</code> for slicing:
      <br><br>
      <pre><code>import 'package:characters/characters.dart';

      var text = 'Hi 😀!';
      print(text.substring(0, 4));          // 'Hi \uD83D' — broken!
      print(text.characters.take(4).toString());  // 'Hi 😀'  — correct</code></pre>
      <br><br>
      That wraps up strings. We have learnt how to represent transistor states with booleans, how to count with integers and doubles, and now how to store every piece of textual information — chat messages, file contents, names, anything humans want to write down. That's a complete foundation for representing data in our programs. In the next episode we'll move on to <strong>collections</strong> — lists, maps, and sets — the building blocks for organising all this data into something useful.
