  - id: the-problem-observers-solve
    title: "The problem — being told instead of asking"
    summary: |
      One value changes, and a dozen things need to know. The counter goes up, and the label must redraw, the badge must update, the analytics must fire. So how do all those things find out?
      <br><br>
      The naive answer is to keep asking. You loop, you check, you compare against the last value you saw. "Changed yet? Changed yet?" That's called <strong>polling</strong>, and it's wasteful. Most of the time the answer is no, and every one of those checks was work done for nothing.

      The observer pattern flips it around. Instead of everyone asking, the thing that owns the data does the telling. You register your interest once, then you wait. When the value actually changes, you get a call. Nothing in between.
      <br><br>
      That's the whole idea — pull becomes push. And it turns out to be one of the most important patterns in app development. Get this one and Flutter state management stops being mysterious.

  - id: subject-and-observers
    title: "The shape of the pattern — subject and observers"
    summary: |
      There are only two roles here. Get them straight and the rest falls into place.
      <br><br>
      <strong>The subject</strong> owns the state and keeps a list of everyone who cares about it.
      <br><br>
      <strong>The observers</strong> are those interested parties. Each one registers with the subject, and from then on the subject calls it back whenever something changes.

      Look at the arrows. They all point one way — out from the subject. The subject holds references to its observers, but the observers don't know about each other. One rebuilds a widget, one writes a log, one saves to disk, and none of them has any idea the others exist.
      <br><br>
      <strong>That's the whole point: loose coupling.</strong> The subject depends on nothing but a list of callbacks. Want a fourth observer next week? Add it. You never touch the subject. The thing that changed and the things that react to it have been pulled apart, and that separation is what keeps a big app from turning into a tangle.

  - id: building-it-from-scratch
    title: "Building one from scratch"
    summary: |
      The fastest way to trust a pattern is to build the smallest version of it yourself. So let's write an observable counter by hand. You need three things: a list to hold the observers, a way to add to it, and a way to fire.
      <br><br>
      <pre><code>typedef Listener = void Function();

      class Counter {
        int _value = 0;
        final List&lt;Listener&gt; _listeners = [];

        int get value =&gt; _value;

        void addListener(Listener fn) =&gt; _listeners.add(fn);
        void removeListener(Listener fn) =&gt; _listeners.remove(fn);

        void increment() {
          _value++;
          _notify();
        }

        void _notify() {
          for (final fn in _listeners) {
            fn();
          }
        }
      }</code></pre>
      That's the entire pattern. A list of callbacks, and a loop that runs them. Nothing more.
      <br><br>
      Notice the observer is just a function — a <code>void Function()</code>. We met first-class functions a few episodes back, and this is where they pay off. The subject doesn't need an interface or a base class to call into. A plain function is enough.
      <br><br>
      Using it looks like this:
      <br><br>
      <pre><code>final counter = Counter();

      counter.addListener(() =&gt; print('Label sees ${counter.value}'));
      counter.addListener(() =&gt; print('Badge sees ${counter.value}'));

      counter.increment();
      // Label sees 1
      // Badge sees 1</code></pre>
      One call to <code>increment</code>, two reactions. Nobody asked whether the value changed — they were told. And if this shape looks familiar, it should. It's almost exactly what Flutter ships in its core library. Let's go meet it.

  - id: change-notifier
    title: "ChangeNotifier — the built-in subject"
    summary: |
      Good news: you don't have to write that counter by hand. Flutter already did. <code>ChangeNotifier</code> lives in <code>package:flutter/foundation.dart</code>, and it's the same idea with the rough edges filed off. It keeps the listener list, hands you <code>addListener</code> and <code>removeListener</code>, and gives you <code>notifyListeners()</code> to fire.
      <br><br>
      <pre><code>import 'package:flutter/foundation.dart';

      class CartModel extends ChangeNotifier {
        final List&lt;String&gt; _items = [];

        List&lt;String&gt; get items =&gt; List.unmodifiable(_items);

        void add(String item) {
          _items.add(item);
          notifyListeners();   // tell everyone the cart changed
        }
      }</code></pre>
      You extend <code>ChangeNotifier</code>, mutate your own state, and call <code>notifyListeners()</code> when something worth announcing happens. That's the contract. The subject decides <em>when</em> a change matters — adding an item counts, reading the list doesn't.
      <br><br>
      On the other side, a widget listens. You rarely call <code>addListener</code> yourself in UI code. Instead you hand the notifier to a builder that does it for you:
      <br><br>
      <pre><code>ListenableBuilder(
        listenable: cart,
        builder: (context, child) =&gt; Text('${cart.items.length} items'),
      )</code></pre>
      Every time the cart calls <code>notifyListeners()</code>, the builder reruns and the text updates. The widget asked for nothing. It was told, and it redrew. That's the observer pattern wearing a Flutter costume.

  - id: valuenotifier-and-streams
    title: "ValueNotifier and Streams — the same idea, narrower and wider"
    summary: |
      Once you can spot the pattern, you start seeing it everywhere in Dart. <code>ChangeNotifier</code> is the general case. It has two cousins, one on each side.
      <br><br>
      <strong>ValueNotifier&lt;T&gt;</strong> is the narrow one — a subject that wraps exactly one value. You assign to <code>.value</code>, and it notifies for you. But only if the new value is actually different by <code>==</code>.
      <br><br>
      <pre><code>final count = ValueNotifier&lt;int&gt;(0);
      count.value = 1;   // notifies
      count.value = 1;   // no change, no notification</code></pre>
      <strong>Stream</strong> is the wide one — built for a <em>sequence</em> of values arriving over time, with proper support for async, errors, and completion. You met Streams in the async section. Now you can name what they really are: the observer pattern for data that keeps coming.

  - id: the-listener-leak
    title: "The leak — listeners that outlive their observers"
    summary: |
      Here's the one rule that bites everyone at least once. When an observer registers, the subject now holds a reference to it. That reference is real, and the garbage collector respects it. So what happens when the observer's job is done but it never unregisters?
      <br><br>
      It can't be collected. The widget left the screen, but the notifier is still holding its callback — and through that callback, the whole <code>State</code> object stays in memory. Do this on a long-lived or global notifier and you've got a leak that grows every single time the screen opens.

  - id: backbone-of-state-management
    title: "Why this is the backbone of Flutter state management"
    summary: |
      Step back, and the whole landscape snaps into focus. Every Flutter state-management approach is the observer pattern with different ergonomics bolted on top. Every single one.
      <br><br>
      <strong>setState</strong> is the pattern at its smallest — the widget's <code>Element</code> is the observer, and calling <code>setState</code> marks it dirty so it rebuilds.
      <br><br>
      <strong>Provider</strong> wraps a <code>ChangeNotifier</code> and uses an <code>InheritedWidget</code> to deliver it down the tree, so descendants can observe it without being handed the object by hand.
      <br><br>
      <strong>Riverpod and Bloc</strong> lean on <code>Stream</code>-shaped notifications under the surface.
      <br><br>
      Different names, different conveniences, one engine.
      <br><br>
      <pre><code>// The same loop, every time:
      // 1. something observes a subject
      // 2. the subject's state changes
      // 3. the subject notifies
      // 4. the observer rebuilds</code></pre>
      So when you sit down to pick a state-management library, you're not choosing a new idea. You're choosing how much ceremony you want around an idea you already understand. That's the payoff of learning the pattern before the packages — every one of them suddenly reads like a variation on a theme you've already seen. The step into <code>Provider</code> from here is a short one.
