  - id: the-problem-with-new
    title: "The problem — a constructor is a hard wire"
    summary: |
      Call <code>Circle()</code> and you get a <code>Circle</code>. Always. That's the deal with a normal constructor: you name a concrete class, and you're wired straight to it. No room to cache, no room to validate, no room to hand back something smarter.
      <br><br>
      Most of the time that's fine. But construction isn't always so simple. Sometimes building an object needs <em>logic</em>.
      <br>• You want to reuse an instance instead of allocating a new one every time.
      <br>• You want to reject bad inputs before an object even exists.
      <br>• You want to decide <em>which</em> concrete type to build based on what you're handed.
      <br><br>
      A plain constructor can't do any of that. It runs, it fills the fields, it returns a brand new instance of exactly that class. Full stop.
      <br><br>
      <pre><code>// We want one shared logger per name.
      // A normal constructor can't give us that —
      // every call builds a fresh object.
      class Logger {
        final String name;
        Logger(this.name);
      }

      final a = Logger('net');
      final b = Logger('net');
      print(identical(a, b));   // false — two separate objects</code></pre>
      And there's a deeper version of the same problem. The caller has to know the exact class name. Write <code>Circle()</code> everywhere and you've scattered that concrete dependency across the codebase. Want to swap in a cached circle, or pick a shape from a string? You'd have to touch every call site. The constructor leaves you no seam to slip logic into.

  - id: the-factory-constructor
    title: "Dart's factory constructor — a constructor that can think"
    summary: |
      A normal constructor is <em>generative</em>: it generates a fresh instance and you have no say in that. Mark it <code>factory</code> instead and the rules change. A factory constructor doesn't have to create anything new. It just has to <em>return an instance of the class</em> — and it can get that instance however it likes.
      <br><br>
      Here's the syntax. Same call site, a body that decides.
      <br><br>
      <pre><code>class Logger {
        final String name;

        // The real, private constructor that actually builds.
        Logger._internal(this.name);

        // The factory: called like Logger('net'), but it
        // returns from a cache instead of always allocating.
        static final _cache = &lt;String, Logger&gt;{};

        factory Logger(String name) {
          return _cache.putIfAbsent(name, () =&gt; Logger._internal(name));
        }
      }

      final a = Logger('net');
      final b = Logger('net');
      print(identical(a, b));   // true — same shared object</code></pre>
      Two things make a factory constructor different, and both matter.
      <br><br>
      <strong>It returns, it doesn't initialise.</strong> A generative constructor has no <code>return</code> — it fills <code>this</code> and you're done. A factory constructor has a body with an explicit <code>return</code>, and there is <em>no</em> <code>this</code>. It can't touch instance fields, because at the point it runs there's no instance yet. That's the trade: you lose <code>this</code>, you gain a decision.
      <br><br>
      <strong>It can return a subtype.</strong> The only constraint is that what you return is the class or something that <em>is-a</em> the class. So a factory on <code>Shape</code> can hand back a <code>Circle</code> or a <code>Square</code>. That's the seam the plain constructor never gave us — and it's the next topic.

  - id: factory-method-pick-the-type
    title: "Factory method — pick the concrete type at runtime"
    summary: |
      Now the payoff. Because a factory can return a subtype, you can put one in front of a family of classes and let it choose which one to build — at runtime, from whatever input you're handed.
      <br><br>
      Say you decode shapes from a config string. The caller asks for a <code>Shape</code> and shouldn't care which concrete class shows up.
      <br><br>
      <pre><code>abstract interface class Shape {
        double get area;

        // One door. The right concrete type walks out.
        factory Shape(String kind, double size) {
          return switch (kind) {
            'circle' =&gt; Circle(size),
            'square' =&gt; Square(size),
            _ =&gt; throw ArgumentError('Unknown shape: \$kind'),
          };
        }
      }

      class Circle implements Shape {
        final double r;
        Circle(this.r);
        @override
        double get area =&gt; 3.14159 * r * r;
      }

      class Square implements Shape {
        final double side;
        Square(this.side);
        @override
        double get area =&gt; side * side;
      }</code></pre>

      Look at the call site. <code>Shape('circle', 10)</code> reads exactly like a constructor, yet it hands back a <code>Circle</code>. The caller depends on <code>Shape</code> and nothing else.
      <br><br>
      <pre><code>final s = Shape('circle', 10);
      print(s.area);            // 314.159
      print(s is Circle);       // true — but the caller didn't ask for one</code></pre>
      This is the classic "factory method" use: choosing an implementation behind a shared interface. Platform-specific code lives here too — a <code>Storage()</code> factory returning <code>WebStorage</code> on the web and <code>FileStorage</code> on mobile, with every caller blissfully unaware which one it got.

  - id: factories-all-over-flutter
    title: "Factories are everywhere in Flutter"
    summary: |
      Once you know the shape, you'll see factories threaded through the framework and through every app you write. Some are named the pattern; most just quietly use it.
      <br><br>
      <strong>The .of(context) idiom.</strong> <code>Theme.of(context)</code>, <code>MediaQuery.of(context)</code>, <code>Navigator.of(context)</code> — these are factory-method-style lookups. <code>Theme.of</code> is a static method that walks up the widget tree and returns the nearest inherited <code>ThemeData</code>. You ask for theme data; the method finds and hands back the right instance. You never construct it yourself.
      <br><br>
      <pre><code>// You don't build ThemeData. You ask for it,
      // and the framework returns the inherited one.
      final theme = Theme.of(context);
      final color = theme.colorScheme.primary;</code></pre>
      <strong>The .fromJson factory.</strong> Every model class you parse from an API uses this convention — a factory constructor that takes a decoded <code>Map</code> and returns a typed object.
      <br><br>
      <pre><code>class User {
        final int id;
        final String name;
        User._(this.id, this.name);

        factory User.fromJson(Map&lt;String, dynamic&gt; json) {
          return User._(json['id'] as int, json['name'] as String);
        }
      }</code></pre>

  - id: factories-and-const-caching
    title: "Factories, const, and caching"
    summary: |
      Caching is where factory constructors earn their keep. If two callers ask for the same thing, why build it twice? The factory holds a store and hands back what it already made.
      <br><br>
      <pre><code>class Currency {
        final String code;
        const Currency._(this.code);

        static final _pool = &lt;String, Currency&gt;{};

        factory Currency(String code) {
          return _pool.putIfAbsent(code, () =&gt; Currency._(code));
        }
      }

      final usd1 = Currency('USD');
      final usd2 = Currency('USD');
      print(identical(usd1, usd2));   // true — one object, shared</code></pre>
      Now the rule you have to keep straight. <strong>A factory constructor cannot itself be <code>const</code>.</strong> A <code>const</code> constructor has to be a compile-time recipe the compiler can evaluate and canonicalise. A factory runs arbitrary code at runtime — a cache lookup, a switch, a throw. The two ideas don't mix, and Dart won't let you write <code>const factory</code>.
      <br><br>
      But here's the nuance that trips people up: a factory <em>can return</em> an instance that was built by a <code>const</code> constructor. So you still get canonicalisation — just chosen at runtime rather than guaranteed at compile time.
      <br><br>
      <pre><code>class Glyph {
        final int code;
        const Glyph._(this.code);     // const, canonicalised

        // Not const itself, but it returns canonicalised
        // const instances. Two 'star' lookups share one object.
        factory Glyph(String name) =&gt; switch (name) {
          'star' =&gt; const Glyph._(0x2605),
          'heart' =&gt; const Glyph._(0x2665),
          _ =&gt; throw ArgumentError(name),
        };
      }</code></pre>
      So the boundary is sharp. The factory is runtime logic; the things it returns can still be compile-time constants. You lose <code>const</code> on the door and keep it on what walks through.

  - id: abstract-factory
    title: "Abstract Factory — families of related objects"
    summary: |
      A single factory picks one object. Sometimes you need to pick a whole <em>family</em> of objects that are meant to work together — and that's the Abstract Factory pattern.
      <br><br>
      The classic Flutter example is platform styling. On iOS you want a <code>CupertinoApp</code>, a sliding navigation bar, a rounded switch — a matching set. On Android you want the Material equivalents. You don't want to mix one from each; a Cupertino switch inside a Material app looks wrong. So you choose the <em>factory</em> once, and every widget it produces belongs to the same family.

      In code, an abstract factory is an interface whose methods each build one member of the family. Concrete factories implement it per platform.
      <br><br>
      <pre><code>abstract interface class WidgetKit {
        Widget button(String label, VoidCallback onTap);
        Widget toggle(bool value, ValueChanged&lt;bool&gt; onChanged);
      }

      class MaterialKit implements WidgetKit {
        @override
        Widget button(String label, VoidCallback onTap) =&gt;
            ElevatedButton(onPressed: onTap, child: Text(label));
        @override
        Widget toggle(bool v, ValueChanged&lt;bool&gt; onChanged) =&gt;
            Switch(value: v, onChanged: onChanged);
      }

      // CupertinoKit implements the same interface with
      // CupertinoButton and CupertinoSwitch.

      // Choose the family once.
      final WidgetKit kit = Platform.isIOS ? CupertinoKit() : MaterialKit();
      kit.button('Save', save);   // the right button, guaranteed to match</code></pre>
      The whole screen now leans on <code>WidgetKit</code>, never on a concrete widget. Swap the kit and the entire UI re-skins, every part still matching its siblings.

  - id: when-to-reach-for-a-factory
    title: "When to use a factory — and when not to"
    summary: |
      A factory is construction with logic bolted on. That logic has a cost, so reach for it only when a plain constructor genuinely can't do the job.
      <br><br>
      <strong>Use a factory when:</strong>
      <br>• You need to return a cached or shared instance instead of always allocating.
      <br>• You need to choose a concrete subtype at runtime — from a string, a platform, a config.
      <br>• Construction needs validation or transformation, like parsing a <code>Map</code> in <code>fromJson</code>.
      <br>• You want a named, intention-revealing way to build, such as <code>Colour.fromHex('#9333ea')</code>.
      <br><br>
      <strong>Stick with a plain constructor when:</strong>
      <br>• You just need to set fields. A factory here is pure ceremony.
      <br>• You want <code>const</code>. A factory can't be const, so a value type you'd like canonicalised at compile time should keep a const generative constructor.
      <br>• Subclasses must call <code>super</code>. A factory isn't part of the initialisation chain — a subclass can't invoke a factory constructor with <code>super</code>, so a base class meant to be extended usually needs a generative constructor.
      <br><br>
      <strong>The trade-offs, plainly.</strong> A factory buys you decoupling and a seam for caching or substitution. It costs you <code>this</code>, costs you <code>const</code>, and adds a layer the reader has to step through to see what's really built. The decoupling is real and worth paying for when the construction is doing something. When it isn't, you're wrapping a one-line job in a keyword for no reason.
      <br><br>
      And notice the through-line back to Strategy. Strategy decouples <em>which behaviour runs</em>; Factory decouples <em>which object gets built</em>. Both push a concrete decision behind an interface so the caller stays ignorant — and an abstract factory is really just a strategy whose job happens to be construction.
