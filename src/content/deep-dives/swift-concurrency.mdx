summary: |
  <strong>async/await and actors</strong> are Swift's answer to two old miseries: unreadable callback code and unreproducible data races.<br><br>
  <code>async</code> marks a function that can suspend; <code>await</code> marks the point where it does. Suspension frees the thread rather than blocking it, so a handful of threads serve thousands of tasks.<br><br>
  <pre><code>// Before: nested completion handlers&#10;fetchUser { user in&#10;  fetchPosts(user) { posts in&#10;    render(posts)&#10;  }&#10;}&#10;&#10;// After: flat, await-ed, throwing&#10;let user  = try await fetchUser()&#10;let posts = try await fetchPosts(user)&#10;render(posts)</code></pre>
  An <strong>actor</strong> is a reference type that serialises access to its own mutable state, so two tasks can never touch it at once. Combined with <code>Sendable</code> and the Swift 6 language mode, many data races become things the compiler simply refuses to build.
topics:
  - id: before-async-await
    title: "Life before async/await"
    summary: |
      Let's start with the pain, because every good language feature is an answer to a specific pain. Before Swift 5.5, asynchronous work meant <strong>completion handlers</strong> — you passed a closure, and the framework called it back when the work finished.
      <br><br>
      <pre><code>func loadProfile(completion: @escaping (Profile?, Error?) -&gt; Void) {&#10;    fetchUser { user, error in&#10;        guard let user = user else { completion(nil, error); return }&#10;        fetchPosts(for: user) { posts, error in&#10;            guard let posts = posts else { completion(nil, error); return }&#10;            fetchAvatar(for: user) { image, error in&#10;                completion(Profile(user, posts, image), nil)&#10;            }&#10;        }&#10;    }&#10;}</code></pre>
      Three dependent calls, and already the code marches off to the right. People christened this the <strong>pyramid of doom</strong>. And the deeper problem is not the indentation — it's the error handling. Every callback has its own error path, and forgetting one is silent. There's no compiler nagging you to handle the failure, because as far as the type system is concerned a closure that never gets called is perfectly fine.

      <br><br>
      Underneath all of this sat <strong>Grand Central Dispatch</strong>, the dispatch-queue library Apple shipped in 2009 with Mac OS X 10.6 Snow Leopard. GCD gave us a clean way to push work onto background queues and hop back to the main queue for UI updates.
      <br><br>
      <pre><code>DispatchQueue.global().async {&#10;    let data = expensiveWork()&#10;    DispatchQueue.main.async {&#10;        self.label.text = data   // hop back for UI&#10;    }&#10;}</code></pre>
      GCD was a genuine leap forward in its day, but it was a library bolted onto the language, not part of it. The compiler couldn't see the control flow. It couldn't tell that the inner closure runs on a different queue, couldn't warn you about a captured variable being mutated from two queues, couldn't guarantee you eventually called <code>completion</code> exactly once. We needed the language itself to understand asynchrony. So in 2021, it learned.

  - id: async-await-basics
    title: "async/await — code that reads top to bottom"
    summary: |
      Swift's answer arrived as <strong>SE-0296</strong> and shipped in <strong>Swift 5.5</strong> at WWDC 2021. The idea is small and the payoff is enormous. You mark a function <code>async</code> if it might need to wait, and at each waiting point you write <code>await</code>.
      <br><br>
      <pre><code>func fetchUser() async throws -&gt; User { ... }&#10;&#10;func loadProfile() async throws -&gt; Profile {&#10;    let user  = try await fetchUser()&#10;    let posts = try await fetchPosts(for: user)&#10;    let image = try await fetchAvatar(for: user)&#10;    return Profile(user, posts, image)&#10;}</code></pre>
      That's the whole pyramid, flattened. It reads top to bottom like ordinary synchronous code, and errors flow through <code>try</code> / <code>throws</code> exactly the way they do everywhere else in Swift. One error path, and the compiler insists you account for it.
      <br><br>
      So what does <code>await</code> actually do? It marks a <strong>suspension point</strong>. When execution reaches an <code>await</code>, the function may pause — and here's the crucial bit — <strong>suspending frees the thread</strong>. It is not a block. The thread is handed back so it can run other work while this function waits. When the awaited operation completes, the function resumes, possibly on a different thread, and carries on from exactly where it left off.
      <br><br>
      <pre><code>// Each await is a place the function can pause and the thread can leave&#10;let a = await stepOne()    // suspend, thread does other work, resume&#10;let b = await stepTwo(a)   // suspend again&#10;return b</code></pre>
      A couple of rules fall out of this. You can only <code>await</code> from inside an <code>async</code> context, just as you can only <code>try</code> meaningfully where errors are handled. And to get from synchronous code into the async world — say from a button handler — you open a <code>Task</code>, which we'll meet properly in a moment. The mental shift is this: <code>await</code> is not "wait here and waste the thread", it's "this is a safe place to step aside".

  - id: structured-concurrency
    title: "Structured concurrency — child tasks that can't escape"
    summary: |
      Sequential <code>await</code>s are easy to read, but sometimes the work is independent and we'd rather run it all at once. Swift's tool for that is <strong>structured concurrency</strong>, and the word "structured" is doing real work. It means concurrent tasks have a tree shape that mirrors your code's block structure: a child task cannot outlive the scope that created it.
      <br><br>
      The lightest tool is <code>async let</code>. It kicks off a child task immediately and binds a promise of the result; you <code>await</code> the binding only when you actually need the value.
      <br><br>
      <pre><code>func loadDashboard() async throws -&gt; Dashboard {&#10;    async let user = fetchUser()      // starts now&#10;    async let feed = fetchFeed()      // starts now, concurrently&#10;    async let ads  = fetchAds()       // starts now, concurrently&#10;    // all three run at once; we suspend here until each is ready&#10;    return Dashboard(try await user, try await feed, try await ads)&#10;}</code></pre>

      <br><br>
      When the number of children isn't known in advance — say you're fetching a list of URLs whose length is decided at runtime — you reach for a <strong>task group</strong> via <code>withTaskGroup</code>. You add as many child tasks as you like, then iterate their results as they arrive.
      <br><br>
      <pre><code>func fetchAll(_ urls: [URL]) async throws -&gt; [Data] {&#10;    try await withThrowingTaskGroup(of: Data.self) { group in&#10;        for url in urls {&#10;            group.addTask { try await download(url) }&#10;        }&#10;        var results: [Data] = []&#10;        for try await data in group {   // results stream in as they finish&#10;            results.append(data)&#10;        }&#10;        return results&#10;    }&#10;}</code></pre>
      The guarantee that makes this safe is the scoping rule: when the <code>withTaskGroup</code> block returns, every child it spawned has finished. Children cannot leak out the back. This is why <strong>cancellation</strong> can be cooperative and tidy. Cancel the parent and the cancellation propagates to every child. But — and this matters — cancellation in Swift is a polite request, not a forced stop. A task keeps running until it checks. So long-running work should ask, either by reading <code>Task.isCancelled</code> or by calling <code>try Task.checkCancellation()</code>, which throws if the task has been cancelled.
      <br><br>
      <pre><code>for item in hugeList {&#10;    try Task.checkCancellation()   // bail out promptly if cancelled&#10;    process(item)&#10;}</code></pre>

  - id: actors
    title: "Actors — serialising access to mutable state"
    summary: |
      async/await tames the timeline. But the moment two tasks run concurrently, we face the oldest concurrency demon of all: the <strong>data race</strong>. Two tasks reading and writing the same mutable variable at the same time, producing torn values and bugs that vanish the instant you attach a debugger. Swift's answer is the <strong>actor</strong>, introduced in <strong>SE-0306</strong>.
      <br><br>
      An actor is a reference type — like a class — but with one defining superpower: it <strong>serialises access to its own mutable state</strong>. Only one task may be executing inside an actor's isolated code at any moment. Everyone else queues. The state is therefore protected without you ever writing a lock.
      <br><br>
      <pre><code>actor BankAccount {&#10;    private var balance = 0&#10;&#10;    func deposit(_ amount: Int) {&#10;        balance += amount        // safe: only one task in here at a time&#10;    }&#10;&#10;    func currentBalance() -&gt; Int { balance }&#10;}</code></pre>

      Because access is serialised, calling into an actor from <em>outside</em> might mean waiting your turn — so those calls are <code>await</code>-ed. This is <strong>actor isolation</strong>: from elsewhere you cross a boundary, and crossing it is a potential suspension point.
      <br><br>
      <pre><code>let account = BankAccount()&#10;await account.deposit(100)               // cross the boundary&#10;let total = await account.currentBalance()   // await your turn&#10;// account.balance                        // error: balance is isolated</code></pre>
      Now the subtlety that trips everyone up: actors are <strong>reentrant</strong>. If an actor method itself contains an <code>await</code>, the actor suspends that call and is free to run <em>another</em> queued call in the meantime. So state you observed before an <code>await</code> may have changed by the time you resume. The lesson: never assume an actor's state is unchanged across a suspension point. Re-check invariants after every <code>await</code> inside actor code, just as you would after yielding anywhere else.

  - id: sendable-and-main-actor
    title: "Sendable, @MainActor, and Swift 6"
    summary: |
      Actors protect the state they own. But what about values that travel <em>between</em> concurrency domains — a struct you hand to a task, a result you send back? Swift needs a way to know which types are safe to cross those boundaries. That marker is the <strong>Sendable</strong> protocol.
      <br><br>
      A <code>Sendable</code> type is one that can be passed between concurrency domains without introducing a data race. Value types made entirely of <code>Sendable</code> parts — most structs and enums — are <code>Sendable</code> automatically. A class is trickier, because two references can point at the same mutable instance; a class is only <code>Sendable</code> if it's immutable, or an actor, or you've taken responsibility yourself with <code>@unchecked Sendable</code>.
      <br><br>
      <pre><code>struct Point: Sendable { var x: Int; var y: Int }   // fine: value type&#10;&#10;final class Counter {            // NOT Sendable: mutable shared state&#10;    var value = 0&#10;}</code></pre>
      The other half of the story is the main thread. UI work must happen there, full stop. Swift expresses that with <strong>@MainActor</strong>, a global actor that pins whatever it annotates to the main thread.
      <br><br>
      <pre><code>@MainActor&#10;final class ViewModel: ObservableObject {&#10;    @Published var title = ""&#10;&#10;    func refresh() async {&#10;        let text = await loadTitle()   // off the main actor&#10;        title = text                   // back on main, safely&#10;    }&#10;}</code></pre>
      For years all of this was advisory. The change came with the <strong>Swift 6 language mode</strong>, released in 2024, which turns on <strong>strict concurrency checking</strong>. Send a non-<code>Sendable</code> value across an actor boundary, touch a UI property off the main actor, share mutable state without protection — and the compiler now refuses to build. Many data races that used to be runtime ghosts became plain old compile errors. It is stricter, occasionally noisier, and a great deal safer.

  - id: continuations
    title: "Bridging the old world with continuations"
    summary: |
      The async world is lovely, but the world is full of older APIs that still speak callbacks — delegate methods, completion handlers, C libraries. We need a bridge from "call me back later" to "await me". That bridge is a <strong>continuation</strong>.
      <br><br>
      <code>withCheckedContinuation</code> (and its throwing sibling <code>withCheckedThrowingContinuation</code>) hands you a continuation object and suspends. You stash that object, call the old API, and when its callback fires you <code>resume</code> the continuation with the result. The suspended <code>await</code> wakes up with that value.
      <br><br>
      <pre><code>func loadImage(named name: String) async throws -&gt; UIImage {&#10;    try await withCheckedThrowingContinuation { continuation in&#10;        legacyLoadImage(name) { image, error in&#10;            if let image {&#10;                continuation.resume(returning: image)&#10;            } else {&#10;                continuation.resume(throwing: error ?? LoadError.unknown)&#10;            }&#10;        }&#10;    }&#10;}</code></pre>
      Now the caller just writes <code>let image = try await loadImage(named: "logo")</code> and never sees the callback at all.
      <br><br>
      Why the word <strong>checked</strong>? Because a continuation carries one ironclad rule: it must be resumed <strong>exactly once</strong>. Resume it zero times and the awaiting task hangs forever — a leak. Resume it twice and you've corrupted the runtime. These are classic callback bugs, easy to write and miserable to find. The checked variants police the contract at runtime: if a checked continuation is resumed more than once it traps, and if it's discarded without ever being resumed it logs a warning. There's a faster <code>withUnsafeContinuation</code> that skips the checks, but reach for it only once you're certain the logic is correct.

      <br><br>
      It's worth contrasting this with Dart, which we covered in the isolates episode, because the two languages chose opposite philosophies. Dart <strong>isolates share no memory whatsoever</strong> — each has its own heap, and they communicate only by passing copies of messages. That design makes data races structurally impossible: there's nothing shared to race over. Swift took the other road. Swift tasks and actors <strong>share memory</strong>, which is fast and convenient, and then it guards that shared memory with actors (serialising access) and <code>Sendable</code> (policing what may cross boundaries). Two reasonable answers to the same hard problem: Dart removes the sharing; Swift keeps it and fences it. Understanding which model you're in tells you exactly what kinds of bug you'll never have to chase.
