  - id: why-sets-exist
    title: "Why Sets exist — the uniqueness guarantee"
    summary: |
      Lists are great for ordered collections, but what if you need to ensure no duplicates? You could loop through a List checking for existing items before adding — but that's O(n) for every insertion. Sets solve this elegantly.
      <br><br>
      A Set is a collection of <strong>unique items</strong> with <strong>O(1) membership testing</strong>. The magic comes from hash tables — instead of scanning every element, Sets compute a hash code and jump directly to where an item would be stored.
      <br><br>
      <pre><code>// List approach — O(n) contains check&#10;var items = &lt;String&gt;[];&#10;if (!items.contains('apple')) {  // O(n)&#10;  items.add('apple');&#10;}&#10;&#10;// Set approach — O(1) automatic deduplication&#10;var items = &lt;String&gt;{};&#10;items.add('apple');  // O(1), duplicates ignored&#10;items.add('apple');  // No effect, still just one apple</code></pre>
      <strong>When to use Set over List:</strong>
      <br>
      - You need uniqueness (tags, visited nodes, seen IDs)<br>
      - You frequently check membership (<code>contains</code>)<br>
      - You perform set operations (union, intersection, difference)<br>
      - Order doesn't matter (or insertion order is fine)

      <br><br>
      <strong>The process:</strong>
      <br>
      1. Compute <code>hashCode</code> for the item<br>
      2. Apply modulo to get a bucket index: <code>hashCode % bucketCount</code><br>
      3. Check/store at that bucket (handle collisions if needed)
      <br><br>
      <strong>Collisions</strong> happen when two different items hash to the same bucket. Dart uses <strong>open addressing</strong> — it probes the next slot until it finds an empty one. This keeps operations O(1) on average.

  - id: hashcode-equality
    title: "Custom objects in Sets — hashCode and equality"
    summary: |
      For Sets to work correctly with custom objects, you must implement <code>hashCode</code> and <code>==</code> properly. The rule: <strong>equal objects must have equal hash codes</strong>.
      <br><br>
      <pre><code>class User {&#10;  final String id;&#10;  final String name;&#10;  &#10;  User(this.id, this.name);&#10;  &#10;  // Two users are equal if they have the same ID&#10;  @override&#10;  bool operator ==(Object other) =>&#10;      other is User && other.id == id;&#10;  &#10;  // Hash code must be consistent with equality&#10;  @override&#10;  int get hashCode => id.hashCode;&#10;}&#10;&#10;void main() {&#10;  var users = &lt;User&gt;{};&#10;  users.add(User('123', 'Alice'));&#10;  users.add(User('123', 'Alice Updated'));  // Same ID!&#10;  &#10;  print(users.length);  // 1 — deduplication works&#10;}</code></pre>
      <strong>Common mistakes:</strong>
      <br>
      - Using mutable fields in <code>hashCode</code> (breaks Set when object changes)<br>
      - Forgetting to override both <code>==</code> and <code>hashCode</code><br>
      - Making <code>hashCode</code> return a constant (kills performance)
      <br><br>
      <strong>Best practice:</strong> Use <code>@immutable</code> classes or <code>package:equatable</code> to generate correct implementations automatically.

      <strong>When to parallelise Set operations:</strong>
      <br>
      - Large set operations (union/intersection/difference on 10k+ items)<br>
      - Complex filtering with expensive predicates<br>
      - Building Sets from large data sources
      <br><br>
      <strong>When NOT to parallelise:</strong>
      <br>
      - Small Sets (overhead exceeds benefit)<br>
      - Simple add/remove/contains (already O(1))
