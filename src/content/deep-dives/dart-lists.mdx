    summary: |
      Before collections existed, programmers managed multiple items with numbered variables: <code>item0</code>, <code>item1</code>, <code>item2</code>. This approach doesn't scale. What if you need 1000 items? How do you iterate? How do you pass them to a function?

      <br><br>
      Lists solve this by providing a <strong>single abstraction</strong> that encapsulates the complexity of managing multiple items. The backing array, capacity management, and memory allocation are all hidden behind a clean interface.
      <br><br>
      This is OOP at its best: <strong>encapsulation</strong> (hiding implementation details), <strong>abstraction</strong> (simple interface), <strong>polymorphism</strong> (<code>List&lt;T&gt;</code> works with any type), and <strong>interface segregation</strong> (Dart's List implements <code>Iterable</code>, <code>EfficientLengthIterable</code>, and more).

      <strong>Where Lists win:</strong> <code>list[i]</code> (O(1)), <code>add()</code> at end (amortized O(1)), iteration.<br>
      <strong>Where Lists lose:</strong> <code>contains()</code> (O(n)), <code>remove(item)</code> (O(n)), <code>insert(0, x)</code> (O(n)).
      <br><br>
      If you're frequently checking membership, consider a Set. If you need fast insertion at both ends, consider a Queue.

      <strong>Typed Data Lists</strong> — use for large numeric arrays (<code>Float64List</code>, <code>Int32List</code>) for better memory efficiency and performance.<br>
      <strong>Set</strong> — use when you need unique items and fast membership testing.<br>
      <strong>LinkedList</strong> — rarely needed; only when you need O(1) splicing in the middle and can sacrifice random access.

  - id: performance-optimizations
    title: "Performance optimizations for Lists"
    summary: |
      When working with large Lists, small optimizations can make a big difference. Here are practical tips.
      <br><br>
      <strong>1. Pre-allocate capacity when you know the size:</strong>
      <pre><code>// Slow: grows multiple times&#10;var list = &lt;int&gt;[];&#10;for (var i = 0; i &lt; 10000; i++) {&#10;  list.add(i);  // triggers resizing&#10;}&#10;&#10;// Fast: no resizing needed&#10;var list = List&lt;int&gt;.filled(10000, 0);&#10;for (var i = 0; i &lt; 10000; i++) {&#10;  list[i] = i;&#10;}</code></pre>
      <strong>2. Use <code>addAll()</code> instead of repeated <code>add()</code>:</strong>
      <pre><code>// Slower&#10;for (var item in otherList) {&#10;  myList.add(item);&#10;}&#10;&#10;// Faster — single operation&#10;myList.addAll(otherList);</code></pre>
      <strong>3. Avoid <code>insert(0, x)</code> in loops — it's O(n²) total:</strong>
      <pre><code>// Terrible: O(n²)&#10;for (var item in items) {&#10;  list.insert(0, item);&#10;}&#10;&#10;// Better: build normally, then reverse&#10;var list = items.toList();&#10;list = list.reversed.toList();</code></pre>
      <strong>4. Use <code>List.generate()</code> for computed values:</strong>
      <pre><code>// Creates a list of squares&#10;var squares = List.generate(100, (i) => i * i);</code></pre>

  - id: parallel-computations
    title: "Lists and parallel computations"
    summary: |
      Dart runs on a single-threaded event loop, but you can use Isolates for true parallel computation with Lists. The key challenge is that Isolates don't share memory.
      <br><br>
      <pre><code>import 'dart:isolate';&#10;&#10;// Heavy computation in a separate isolate&#10;Future&lt;List&lt;int&gt;&gt; processInParallel(List&lt;int&gt; data) async {&#10;  return await Isolate.run(() {&#10;    // This runs in a separate isolate&#10;    return data.map((x) => x * x).toList();&#10;  });&#10;}&#10;&#10;void main() async {&#10;  var data = List.generate(1000000, (i) => i);&#10;  var result = await processInParallel(data);&#10;  print('Processed ${result.length} items');&#10;}</code></pre>
      <strong>When to parallelise:</strong>
      <br>
      - CPU-bound operations on large Lists (>10k items)<br>
      - Complex transformations (encryption, compression, parsing)<br>
      - When you can afford the overhead of copying data to the isolate
      <br><br>
      <strong>When NOT to parallelise:</strong>
      <br>
      - Small Lists (overhead exceeds benefit)<br>
      - Simple operations (<code>map</code>, <code>filter</code> on primitives)<br>
      - When you need to share state (isolates can't share mutable state)

  - id: typed-data-lists
    title: "Typed data Lists — when performance is critical"
    summary: |
      For numerical data, Dart's typed data lists offer significant performance benefits. They store values directly in memory without boxing overhead.
      <br><br>
      <pre><code>import 'dart:typed_data';&#10;&#10;// Standard List — each int is a boxed object&#10;var normalList = &lt;int&gt;[1, 2, 3, 4, 5];&#10;&#10;// Typed List — compact, contiguous memory&#10;var typedList = Int32List.fromList([1, 2, 3, 4, 5]);&#10;&#10;// Float64List for floating-point data&#10;var floats = Float64List(1000);&#10;for (var i = 0; i &lt; 1000; i++) {&#10;  floats[i] = i * 0.1;&#10;}</code></pre>
      <strong>Available typed lists:</strong>
      <br>
      - <code>Int8List</code>, <code>Int16List</code>, <code>Int32List</code>, <code>Int64List</code><br>
      - <code>Uint8List</code>, <code>Uint16List</code>, <code>Uint32List</code>, <code>Uint64List</code><br>
      - <code>Float32List</code>, <code>Float64List</code>
      <br><br>
      <strong>Benefits:</strong> Lower memory usage (no boxing), better cache locality, faster iteration, interop with native code (FFI).
      <br><br>
      <strong>Tradeoffs:</strong> Fixed size, limited to numeric types, slightly less convenient API.

  - id: list-flavours
    title: "List flavours — growable, fixed, unmodifiable, const"
    summary: |
      Dart has four flavours of List, each with different mutability characteristics.
      <br><br>
      <pre><code>// 1. Growable (default) — can add/remove/modify&#10;var growable = [1, 2, 3];&#10;growable.add(4);        // OK&#10;growable[0] = 10;       // OK&#10;&#10;// 2. Fixed-length — can modify, cannot resize&#10;var fixed = List.filled(3, 0);&#10;fixed[0] = 10;          // OK&#10;// fixed.add(4);        // Error!&#10;&#10;// 3. Unmodifiable — cannot modify or resize&#10;var unmod = List.unmodifiable([1, 2, 3]);&#10;// unmod[0] = 10;       // Error!&#10;// unmod.add(4);        // Error!&#10;&#10;// 4. Const — compile-time constant, shared&#10;const constList = [1, 2, 3];&#10;// Identical to all other const [1, 2, 3] literals</code></pre>
      <strong>When to use each:</strong>
      <br>
      - <strong>Growable</strong>: Default choice for most use cases<br>
      - <strong>Fixed-length</strong>: When you know exact size and want to prevent accidental resizing<br>
      - <strong>Unmodifiable</strong>: Defensive copies in APIs, returning internal state safely<br>
      - <strong>Const</strong>: Static configuration data, truly immutable values
