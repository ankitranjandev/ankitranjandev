  - id: escape-hatches
    title: "Three ways to escape the boxing cost"
    summary: |
      We have three reliable ways to avoid the cost of boxing doubles.
      <br><br>
      <strong>1. Float64List for collections.</strong>
      <br><br>
      For lots of doubles, <code>Float64List</code> is the workhorse. Values are packed contiguously, no boxing, no pointers between elements.
      <br><br>
      <pre><code>import 'dart:typed_data';

      // A million doubles, exactly 8 MB, no boxing
      Float64List samples = Float64List(1000000);</code></pre>
      <strong>2. FFI for native interop.</strong>
      <br><br>
      When passing doubles to C code via <code>dart:ffi</code>, the values are unboxed at the boundary. A <code>Float64List</code> can be handed to native code by pointer with zero copying — the same memory is seen from both sides.
      <br><br>
      <strong>3. VM optimization in hot paths.</strong>
      <br><br>
      When the optimizer can prove a value is always a <code>double</code> and never escapes to where its identity matters, it keeps the value in a CPU floating-point register or on the stack. We don't write any special syntax for this — it just happens when the JIT or AOT compiler can prove it is safe.
      <br><br>
      What defeats it:
      <br><br>
      • Storing the value in a <code>List&lt;double&gt;</code> or <code>List&lt;dynamic&gt;</code>
      <br>
      • Passing it to a function with <code>num</code> or <code>Object</code> parameter type
      <br>
      • Returning it from a function whose declared return type isn't <code>double</code>
      <br>
      • Calling <code>identical()</code> on it
      <br><br>
      In well-typed hot loops, the optimizer can be very effective. But we can't rely on it. When boxing cost matters and we want a guarantee, reach for <code>Float64List</code>.

  - id: subnormals
    title: "Subnormal numbers — the slow zone near zero"
    summary: |
      IEEE 754 has a special encoding for values too small to fit the normal floating-point format. We call them <strong>subnormal</strong> numbers (or, historically, denormalized).
      <br><br>
      The normal range of a double runs from about 2.2 × 10⁻³⁰⁸ up to 1.7 × 10³⁰⁸. Below the normal minimum, IEEE 754 reserves a special set of bit patterns to fill the gap — values down to about 5 × 10⁻³²⁴, with steadily decreasing precision as they shrink.
      <br><br>
      This is great in theory. It gives us <strong>gradual underflow</strong>: values approach zero smoothly instead of suddenly snapping to it. Without subnormals, the smallest non-zero double would be 2.2 × 10⁻³⁰⁸, and everything tinier would just become zero.
      <br><br>
      But there is a real-world catch. Many CPUs handle subnormal arithmetic via slow microcode that is 10–100 times slower than normal arithmetic. In audio processing, simulation, and tight numerical loops, this is a known performance trap. A simulation running fine can suddenly slow down when its values drift into the subnormal range.
      <br><br>
      <pre><code>double tiny = 1e-310;
      print(tiny);                  // 1e-310
      print(tiny.isFinite);         // true
      print(tiny == 0);             // false
      // But operations on tiny may run noticeably slower on some hardware.</code></pre>
      The standard fixes (in native code or libraries that expose CPU flags):
      <br><br>
      • Enable "flush to zero" mode so subnormals are silently treated as zero.
      <br>
      • Clamp tiny intermediate values to zero explicitly.
      <br><br>
      Dart itself doesn't expose CPU-flag control — if we are hitting this problem, we'd typically use FFI to set it from native code. But just knowing the slow zone exists is half the battle.

  - id: rounding
    title: "Rounding modes — banker's versus away-from-zero"
    summary: |
      IEEE 754 defines five rounding modes, and Dart's behaviour involves a couple of subtleties worth knowing.
      <br><br>
      When the result of an arithmetic operation can't be represented exactly, the CPU picks the nearest representable value. The default IEEE 754 mode is <strong>round half to even</strong> — also called banker's rounding. When the value is exactly halfway between two representable numbers, the one whose last bit is even wins.
      <br><br>
      This is what every floating-point operation in Dart uses internally. It is deterministic and reproducible: everyone follows the same rule.
      <br><br>
      But Dart's <code>.round()</code> method is a different story. It rounds <strong>half away from zero</strong>:
      <br><br>
      <pre><code>print((0.5).round());   // 1
      print((1.5).round());   // 2
      print((2.5).round());   // 3
      print((-0.5).round());  // -1
      print((-1.5).round());  // -2</code></pre>
      Compare with banker's rounding, which would give 0, 2, 2, 0, -2. They agree on 1.5 → 2 (both "away from zero" and "to even" land on 2), but disagree on 0.5 and 2.5.
      <br><br>
      This is a deliberate Dart language choice. Away-from-zero matches what humans normally mean by "rounding," which is why <code>.round()</code> uses it. Banker's rounding is for arithmetic correctness — it avoids the small statistical bias that always-away-from-zero introduces over many operations.
      <br><br>
      If we need banker's rounding specifically (often required for accounting standards like IEEE 754, financial regulations, or scientific publications), we'll need to implement it ourselves or use a decimal library that supports it.

  - id: bit-level
    title: "Bit-level inspection"
    summary: |
      Sometimes we want to see exactly what bits Dart is using to represent a double. The standard tool is <code>ByteData</code> from the typed_data library we met in Episode 3.
      <br><br>
      <pre><code>import 'dart:typed_data';

      double value = 1.0;
      ByteData bytes = ByteData(8);
      bytes.setFloat64(0, value);

      int rawBits = bytes.getInt64(0);
      print(rawBits.toRadixString(16).padLeft(16, '0'));
      // 3ff0000000000000</code></pre>
      That hex string is the IEEE 754 encoding of <code>1.0</code>:
      <br><br>
      • <strong>Sign:</strong> <code>0</code> (positive)
      <br>
      • <strong>Exponent:</strong> <code>01111111111</code> (1023, the bias — representing 2⁰)
      <br>
      • <strong>Mantissa:</strong> all zeros (with the implicit leading 1, the value is 1.0)
      <br><br>
      We can go the other way too:
      <br><br>
      <pre><code>// Build a double from raw bits
      ByteData bytes = ByteData(8);
      bytes.setInt64(0, 0x4000000000000000);  // 2.0 in IEEE 754
      double value = bytes.getFloat64(0);
      print(value);  // 2.0</code></pre>
      This is occasionally useful for:
      <br><br>
      • Custom binary serialisation formats
      <br>
      • Building hash functions over doubles
      <br>
      • Inspecting NaN payloads (different NaN values have different mantissa bits)
      <br>
      • Sanity-checking our understanding of IEEE 754
      <br><br>
      Once we can see the bits, every quirk we have discussed in these two episodes becomes mechanical to reason about. Boxing? It is the heap object that wraps these 8 bytes. Precision loss? Bits get truncated to fit 52. Subnormals? Specific patterns in the exponent. NaN? Exponent all ones, mantissa non-zero. It is all there in the bytes.
      <br><br>
      That wraps up the floating-point story. In the next main episode we move on to strings, and we will see that text is somehow even messier than numbers.
