  - id: the-problem-too-many-parameters
    title: "The problem — one constructor, too many knobs"
    summary: |
      Some objects are simple. A <code>Point(x, y)</code> needs two numbers and you're done. But some objects have a dozen knobs, half of them optional, and you have to set them in some order. That's where the single constructor starts to hurt.
      <br><br>
      Picture a burger order. Patty, cheese, lettuce, sauce, a size, a bun choice. Cram it all into positional parameters and you get this.
      <br><br>
      <pre><code>final burger = Burger(true, false, true, 2, false, true, 'large', null, true, false);</code></pre>
      What is that fourth argument? Which boolean is the cheese? Swap any two of those booleans and it still compiles — you've shipped a bug that the type system can't see. This is the <strong>telescoping constructor</strong>, and it rots in two directions at once: it's unreadable at the call site, and it grows a new parameter every time the product gains an option.

      <br><br>
      Dart softens the worst of this with named parameters — <code>Burger(patty: true, cheese: false, ...)</code> reads far better than ten bare booleans. That alone solves a lot of cases, and you should reach for it first. But it doesn't help when the object has to be assembled in <em>stages</em>: when you build a piece, run some logic, build another piece, and only at the end produce the finished thing. For that, you separate the assembling from the result.

  - id: separate-assembly-from-result
    title: "The pattern — split the assembly from the finished object"
    summary: |
      Here's the move. Instead of one constructor that takes everything at once, you make a second object whose only job is to <em>collect</em> the parts. You poke at it step by step. When you're done, you ask it for the finished thing.
      <br><br>
      There are two roles, and they're deliberately separate.
      <br><br>
      <strong>The builder</strong> is mutable scratch space. It holds half-finished state and exposes one method per decision.
      <br><br>
      <strong>The product</strong> is the real object — often immutable — that the builder hands back from <code>build()</code>.
      <br><br>
      <pre><code>class Burger {
        final bool patty;
        final bool cheese;
        final String size;
        Burger(this.patty, this.cheese, this.size);
      }

      class BurgerBuilder {
        bool _patty = false;
        bool _cheese = false;
        String _size = 'regular';

        void addPatty() =&gt; _patty = true;
        void addCheese() =&gt; _cheese = true;
        void setSize(String s) =&gt; _size = s;

        Burger build() =&gt; Burger(_patty, _cheese, _size);
      }</code></pre>
      The call site now reads like a list of decisions, not a row of mystery arguments.
      <br><br>
      <pre><code>final b = BurgerBuilder()
        ..addPatty()
        ..addCheese()
        ..setSize('large');

      final burger = b.build();</code></pre>
      Look at what that bought you. The booleans have names again. The order is whatever you want. And the builder can hold <em>logic</em> between steps — validate, branch, loop — which a single constructor call never could. <strong>The trade:</strong> you carry an extra object around. For a two-field class that's overkill; named parameters win. For an object you genuinely assemble in pieces, the builder earns its keep.

  - id: flutter-builder-callbacks
    title: "Flutter's builders are functions that build on demand"
    summary: |
      Now, the classic builder above is a whole class. Flutter rarely makes you write that. Instead it hands the pattern to you as a <em>callback</em> — a function named <code>builder</code> that produces a widget when asked.
      <br><br>
      You've already met these. <code>ListView.builder</code> takes an <code>itemBuilder</code>. <code>showDialog</code> takes a <code>builder</code>. There's even a widget literally called <code>Builder</code>. The shape is always the same: you hand in a function, and Flutter calls it later, when it's ready, to produce the widget.
      <br><br>
      <pre><code>ListView.builder(
        itemCount: messages.length,
        itemBuilder: (context, index) {
          return ListTile(title: Text(messages[index]));
        },
      );</code></pre>

      The type of that callback is <code>IndexedWidgetBuilder</code> — which is just an alias for <code>Widget Function(BuildContext, int)</code>. Flutter passes you the <code>BuildContext</code> and the row index; you return one widget. You're not building the whole list. You're handing the framework a recipe for building <em>one item</em>, and it runs that recipe whenever it needs another item.
      <br><br>
      The bare <code>Builder</code> widget exists for a narrower reason: it gives you a fresh <code>BuildContext</code> located inside the current widget. That matters when something like <code>Scaffold.of(context)</code> needs a context that sits <em>below</em> the <code>Scaffold</code>, not the one above it.
      <br><br>
      <pre><code>Scaffold(
        body: Builder(
          builder: (context) {
            // this context is inside the Scaffold
            return ElevatedButton(
              onPressed: () =&gt; Scaffold.of(context).openDrawer(),
              child: const Text('Open'),
            );
          },
        ),
      );</code></pre>

  - id: builders-defer-work
    title: "Why the callback matters — deferred, lazy building"
    summary: |
      So why does Flutter bother with a callback at all? Why not just hand it a list of widgets? Because a function lets the framework <em>defer the work</em>. It calls your builder only when it actually needs the result — and for a long scrolling list, that's the whole game.
      <br><br>
      Compare the two ways to make a list. The plain constructor takes a finished <code>children</code> list. Every widget in it is built <em>now</em>, before a single pixel hits the screen.
      <br><br>
      <pre><code>// Eager: all 10,000 ListTiles built up front
      ListView(
        children: [
          for (var i = 0; i &lt; 10000; i++)
            ListTile(title: Text('Item \$i')),
        ],
      );</code></pre>
      The builder version takes a function and an <code>itemCount</code>. Flutter calls the function only for the rows near the viewport. Scroll, and it builds the next few; the ones far off-screen never get built until they're needed.
      <br><br>
      <pre><code>// Lazy: only the visible ListTiles are built
      ListView.builder(
        itemCount: 10000,
        itemBuilder: (context, i) =&gt; ListTile(title: Text('Item \$i')),
      );</code></pre>

  - id: cascade-and-fluent-builders
    title: "Cascades and StringBuffer — building without a builder class"
    summary: |
      You saw the <code>..</code> in the burger example and the <code>ListView</code> calls. That's the <strong>cascade operator</strong>, and it's Dart's built-in answer to fluent, step-by-step building. It lets you run a sequence of operations on one object without naming that object over and over.
      <br><br>
      The trick is what <code>..</code> evaluates to. A normal <code>.</code> returns whatever the method returns. A <code>..</code> runs the method but <em>throws away its result and returns the receiver instead</em>. So every step in the chain hands the same object to the next.
      <br><br>
      <pre><code>final paint = Paint()
        ..color = Colors.blue
        ..strokeWidth = 4.0
        ..style = PaintingStyle.stroke;
      // paint is the Paint, not the result of the last assignment</code></pre>
      The classic standard-library builder is <code>StringBuffer</code>. Concatenating strings with <code>+</code> in a loop allocates a fresh string every time — strings are immutable, remember. A <code>StringBuffer</code> is mutable scratch space: you <code>write</code> into it as many times as you like, then call <code>toString()</code> once to produce the finished, immutable string.
      <br><br>
      <pre><code>final sb = StringBuffer()
        ..write('Hi ')
        ..write('there')
        ..write('!');

      final greeting = sb.toString();   // 'Hi there!'</code></pre>

  - id: copywith-immutable-builder
    title: "copyWith — a builder for immutable objects"
    summary: |
      There's one more flavour, and it's the one you'll use most in real Flutter code. Sometimes you don't want a mutable builder at all. You have an immutable object and you want a <em>slightly different</em> immutable object — change one field, keep the rest. That's what <code>copyWith</code> is for.
      <br><br>
      The problem it solves: with a <code>final</code> field, you can't just reassign it. And rebuilding the whole object by hand means restating every field you <em>didn't</em> change, which is noise and a bug waiting to happen the moment you add a field.
      <br><br>
      <pre><code>class Settings {
        final bool darkMode;
        final String language;
        final double fontSize;

        const Settings({
          required this.darkMode,
          required this.language,
          required this.fontSize,
        });

        Settings copyWith({
          bool? darkMode,
          String? language,
          double? fontSize,
        }) {
          return Settings(
            darkMode: darkMode ?? this.darkMode,
            language: language ?? this.language,
            fontSize: fontSize ?? this.fontSize,
          );
        }
      }</code></pre>
      Each parameter is nullable, and <code>?? this.field</code> means "use what you were given, otherwise keep the old value." So you name only what changes.
      <br><br>
      <pre><code>const current = Settings(
        darkMode: false, language: 'en', fontSize: 14.0,
      );

      final updated = current.copyWith(darkMode: true);
      // darkMode flips to true; language and fontSize are untouched</code></pre>
      This is a builder that produces in one step, with no mutable scratch object in between. It's the heart of immutable state management — Bloc, Riverpod, plain <code>setState</code> with immutable models all lean on it. You never mutate the old state; you build a new state that differs in exactly the fields you touched. <strong>One caveat:</strong> the <code>?? this.field</code> trick can't distinguish "leave it alone" from "set it to null." When a field is itself nullable and you genuinely need to null it, plain <code>copyWith</code> can't express that — you'll see code generators like <code>freezed</code> step in to handle it.

  - id: when-to-reach-for-a-builder
    title: "When to reach for a builder, and the thread back to widgets"
    summary: |
      So when does a builder earn its place over a plain constructor? Walk down this list and stop at the first one that fits.
      <br>• Two or three fields, all known at once → a plain constructor, named parameters. No builder.
      <br>• Many optional fields, still all known at once → still a constructor with named parameters. Dart makes this clean enough that you don't need more.
      <br>• The object is assembled in <em>stages</em>, with logic between steps → a cascade, or a real builder class.
      <br>• You're producing widgets <em>on demand</em>, lazily → a builder callback like <code>itemBuilder</code>.
      <br>• You have an immutable object and want a tweaked copy → <code>copyWith</code>.
      <br><br>
      Notice these aren't five patterns. They're one idea — separate the act of constructing from the finished result — wearing five outfits sized to how much structure the job has earned. Most of the time Dart's cascade and named parameters cover you, and you never write the word "builder" at all.
      <br><br>
      <strong>Builder versus Strategy.</strong> We met Strategy last time: it's about <em>which behaviour runs</em>. Builder is about <em>how an object gets assembled</em>. They sit close together in Flutter — an <code>itemBuilder</code> is a builder callback, but it's also a strategy for how each row is constructed. The same callback can wear both labels, and that's fine; the labels are there to help you reason, not to box the code in.
      <br><br>
      And here's the thread back to everything Flutter. A widget tree <em>is</em> an object assembled from parts — you nest a <code>Padding</code> around a <code>Column</code> around your tiles, building the bigger thing out of smaller ones. The <code>build</code> method on every widget is a builder: it takes a context and returns the assembled subtree. You've been writing builders since your very first <code>build(BuildContext context)</code>. Naming the pattern just tells you what your hands already knew.
