  - id: the-real-cost-of-threads
    title: "The real cost of threads and callbacks"
    summary: |
      Before we admire the machinery, let's be honest about what it replaced, because every line of this design is an answer to a concrete cost. The old unit of concurrency was the <strong>OS thread</strong>, and a thread is expensive in a way that is easy to forget.
      <br><br>
      When you create a thread, the kernel reserves a stack for it &mdash; on Apple platforms a secondary thread's stack is around 512&nbsp;KB, and the main thread's is larger still. That memory is committed whether the thread is doing real work or sitting idle. Spin up a few hundred threads and you have spent hundreds of megabytes on stacks alone. Worse, switching between threads is a <strong>context switch</strong>: a trap into the kernel, a save and restore of the register file, and a cache that no longer holds anything useful. Each switch costs on the order of microseconds, and they add up fast.

      <br><br>
      Under Grand Central Dispatch this produced a specific failure called <strong>thread explosion</strong>. The pattern was innocent: dispatch a blocking call onto a global queue, and when that work blocks, GCD &mdash; trying to keep the queue making progress &mdash; spins up another thread. Block that one too and it spins up another. A program that thinks it has a handful of background tasks can suddenly have sixty-four live threads, all blocked, all holding stacks, all thrashing the scheduler.
      <br><br>
      <pre><code>// The thread-explosion recipe, circa 2015&#10;for i in 0..&lt;100 {&#10;    DispatchQueue.global().async {&#10;        let data = try? Data(contentsOf: url)  // BLOCKS the thread&#10;        process(data)&#10;    }&#10;}&#10;// GCD keeps minting threads to drain the queue. The box wilts.</code></pre>
      And the callback style that came with it had its own tax. Dependent asynchronous calls nested into the <strong>pyramid of doom</strong>, marching off to the right, and &mdash; the deeper wound &mdash; every callback carried its own error path. Forget to call <code>completion</code> on one branch and nothing complained; as far as the type system was concerned, a closure that is never invoked is perfectly legal. The errors didn't crash. They simply went missing.
      <br><br>
      So what did we actually want? Three things at once: code that reads top to bottom like synchronous code; thousands of in-flight operations without thousands of threads; and a compiler that understands the control flow well enough to catch the mistakes. Threads and callbacks could give us at most one of those. Swift's concurrency model was built to give us all three, and the rest of this article is how.

  - id: coroutine-splitting
    title: "async/await as coroutine splitting"
    summary: |
      Here is the single most important idea in the whole model, and it is a compiler idea, not a runtime one. An <code>async</code> function is not run the way an ordinary function is. The compiler performs a <strong>coroutine transform</strong>: it cuts the function into pieces at every suspension point, and each piece becomes its own <strong>partial function</strong>.
      <br><br>
      Think about what a suspension has to do. The function must be able to stop in the middle, let the thread go off and do other work, and later pick up exactly where it left off &mdash; same local variables, same place in the code. A normal function can't do that, because the moment it returns, its stack frame is gone. So the compiler rewrites the function so that returning and pausing become the same operation. Each <code>await</code> is a place where the current partial function does its work, hands control back to whoever resumed it, and arranges for the <em>next</em> partial function to be called when the awaited result is ready.

      Read that diagram as the literal shape of the compiled code. The function you wrote with three statements becomes three entry points. Part&nbsp;0 runs up to the first <code>await</code> and returns. Part&nbsp;1 is the resume point for that first await; it runs to the second <code>await</code> and returns. Part&nbsp;2 runs the tail. The runtime never "pauses a function" in any mysterious sense &mdash; it simply calls the next partial function when it's time.
      <br><br>
      <pre><code>func load() async {&#10;    let a = work()           // part 0 begins&#10;    let b = await fetch(a)   // part 0 ends here; part 1 resumes here&#10;    let c = await save(b)    // part 1 ends here; part 2 resumes here&#10;    show(c)                  // part 2&#10;}</code></pre>
      Now a subtlety that even experienced developers get wrong: <strong><code>await</code> marks a <em>potential</em> suspension point, not a guaranteed one.</strong> It says "this is a place where the function is <em>allowed</em> to suspend." If the awaited value is already available &mdash; an actor that happens to be free, a value that's been cached &mdash; the runtime may sail straight through without ever giving up the thread. So <code>await</code> is not a synonym for "slow" or "yields here, always." It is a marker that tells you, and the compiler, "across this point, assume nothing about thread or state continuity." That single reframing prevents a whole class of bugs we'll meet when we get to actors.

  - id: async-frames-on-the-heap
    title: "Async frames live on the heap"
    summary: |
      If a function can be cut into partial functions that run at different times, possibly on different threads, then its local variables can't live on the ordinary C call stack. The call stack is strictly last-in-first-out: a frame exists only between a call and its return. But an async function <em>returns</em> at every suspension point and then runs again later. Its state has to outlive those returns. So Swift puts it somewhere that does outlive them: the <strong>heap</strong>.
      <br><br>
      When an async function needs to keep state across a suspension, the runtime allocates an <strong>async frame</strong> on the heap. It holds the live locals, a pointer to which partial function should run on resume, and a link to the caller's async frame. That chain of linked frames forms what's called the <strong>async stack</strong> &mdash; logically a stack, but built from heap allocations rather than a contiguous block of memory.
      <br><br>
      <pre><code>// Roughly what the runtime keeps per suspended async call:&#10;struct AsyncFrame {&#10;    resumePointer: () -&gt; Void   // which partial function runs next&#10;    savedLocals:   ...          // the values that must survive the await&#10;    callerFrame:   AsyncFrame?  // link up the async stack&#10;}</code></pre>
      The consequence is the headline feature of the entire model: <strong>a suspended task occupies no thread at all.</strong> When <code>load()</code> hits <code>await fetch(a)</code> and suspends, its state is sitting safely in a heap frame, and the thread that was running it is completely free. It is not parked, not sleeping, not blocked &mdash; it has genuinely left and is running something else. Resuming the task later means allocating nothing new of significance; the runtime just calls the resume pointer with the saved frame.
      <br><br>
      This is why "a few threads serving thousands of tasks" is not marketing. A blocked thread holds a half-megabyte stack hostage; a suspended Swift task holds a few dozen bytes of heap. There's a real optimisation here too: if an async function can prove it never actually suspends, the compiler can sometimes avoid the heap allocation entirely. But the model you should carry in your head is simple &mdash; <strong>the C stack is for synchronous frames; the heap async stack is for suspendable ones</strong>, and only the latter can survive an <code>await</code>.

  - id: tasks
    title: "Tasks — the unit of asynchronous work"
    summary: |
      A partial function has to be <em>driven</em> by something, and that something is a <strong>Task</strong>. A task is the runtime's unit of asynchronous work: it owns an async stack, carries a <strong>priority</strong>, holds a cancellation flag, and threads through task-local values. Every piece of async code runs inside some task. There is no async without a task underneath it.
      <br><br>
      To get from the synchronous world into the async one, you create a task explicitly. <code>Task { }</code> creates an <strong>unstructured</strong> task that <em>inherits</em> its context &mdash; the current actor, the priority, the task-local values &mdash; from where you wrote it. <code>Task.detached { }</code> deliberately inherits none of that; it starts from a clean slate, off any actor, at default priority. Reach for detached only when you genuinely want to escape the surrounding context, which is rarer than people think.
      <br><br>
      <pre><code>@MainActor&#10;func onTap() {&#10;    Task {                     // inherits @MainActor + priority&#10;        let data = await load()&#10;        self.label = data      // already back on the main actor&#10;    }&#10;    Task.detached {            // inherits NOTHING; not on the main actor&#10;        await warmCaches()&#10;    }&#10;}</code></pre>
      Most concurrency, though, should be <strong>structured</strong>, which means tasks form a <strong>tree</strong>. A parent task spawns children, and the children's lifetimes nest inside the parent's scope &mdash; a child cannot outlive the scope that created it. The lightest way to spawn a child is <code>async let</code>: it starts a child task <strong>eagerly</strong>, the instant the binding is created, and you <code>await</code> the binding only at the point where you actually need the value.
      <br><br>
      <pre><code>func loadDashboard() async throws -&gt; Dashboard {&#10;    async let user = fetchUser()   // child starts NOW&#10;    async let feed = fetchFeed()   // this child starts NOW too, concurrently&#10;    // ... do other work while both run ...&#10;    return Dashboard(try await user, try await feed)  // await at point of use&#10;}</code></pre>
      That word "eagerly" matters. <code>async let user = fetchUser()</code> does not wait politely to be awaited; the child is off and running immediately, in parallel with the next line. The <code>await</code> later is just where the parent collects the result and, if the child threw, where the error surfaces. This tree shape is what makes the next two ideas &mdash; task groups and cancellation &mdash; both possible and tidy.

  - id: task-groups-and-cancellation
    title: "Task groups and cooperative cancellation"
    summary: |
      <code>async let</code> is perfect when you know at compile time how many children you want. When the count is dynamic &mdash; download this list of URLs, whatever length it turns out to be &mdash; you reach for a <strong>task group</strong> via <code>withTaskGroup</code> or, when the work can throw, <code>withThrowingTaskGroup</code>. You add as many child tasks as you like, then drain their results as they finish.

      The scoping rule is the safety net: when the <code>withTaskGroup</code> block returns, every child it spawned has finished. Children cannot leak out the back. That guarantee is precisely what lets <strong>cancellation</strong> be clean. Cancel the parent and the cancellation flag is set on every descendant, flowing <strong>down</strong> the tree, never up.
      <br><br>
      But here is the rule people forget: <strong>cancellation is cooperative, not pre-emptive.</strong> Cancelling a task does not stop it. It does not unwind a stack or raise an exception out of nowhere. It sets a boolean. The task keeps running, line after line, until <em>it</em> chooses to look. Code that wants to be cancellable must therefore ask, by reading <code>Task.isCancelled</code> or by calling <code>try Task.checkCancellation()</code>, which throws <code>CancellationError</code> if the flag is set.
      <br><br>
      <pre><code>for item in hugeList {&#10;    try Task.checkCancellation()   // bail out promptly, throwing if cancelled&#10;    process(item)&#10;}&#10;// A tight loop that never checks will run to completion even after cancel().</code></pre>
      This is a deliberate trade. Pre-emptive cancellation &mdash; killing a task mid-instruction &mdash; would leave half-mutated state and broken invariants, the same reason killing threads is dangerous. Cooperative cancellation hands the task control over <em>where</em> it's safe to stop. The cost is that you must actually check, especially in long loops and around expensive work. A well-behaved async API checks for you at its own suspension points, so awaiting a cancellation-aware function often propagates cancellation without you writing a thing.

  - id: the-cooperative-pool
    title: "The cooperative thread pool"
    summary: |
      We've said a suspended task holds no thread. So which threads run the partial functions? A small, fixed set of them: the <strong>cooperative thread pool</strong>. Its width is bounded &mdash; roughly the number of CPU cores &mdash; and that bound is the whole point. The pool offers a <strong>forward-progress guarantee</strong>: because tasks only ever step aside at <code>await</code> (handing the thread straight back) rather than blocking, a fixed number of threads is always enough to keep making progress. There's no reason to mint a sixty-fourth thread, because no thread is ever stuck.
      <br><br>
      That guarantee rests entirely on one promise from your code: <strong>never block a cooperative thread.</strong> Suspending is fine &mdash; it frees the thread. Blocking is catastrophic &mdash; it holds the thread hostage while doing nothing. A blocking call is a synchronous wait that the runtime can't see: <code>DispatchSemaphore.wait()</code>, a synchronous network read, <code>Thread.sleep</code>, a long <code>while</code> loop with no <code>await</code> in it. The thread can't be reclaimed, and with only a few threads in the pool, blocking enough of them grinds everything to a halt.

The pool's threads are a shared resource with a strict contract, and honouring that contract is what keeps a handful of threads serving an ocean of tasks.

  - id: executors-and-jobs
    title: "Executors and jobs"
    summary: |
      We now have the two halves: partial functions on one side, a pool of threads on the other. The thing that connects them is the <strong>executor</strong>, and the currency it deals in is the <strong>job</strong>. The executor's one responsibility is to decide where and when its jobs run.

      There are two flavours you'll meet constantly. A <strong>SerialExecutor</strong> is the opposite promise: it still borrows threads from the pool, but it guarantees that only <em>one</em> of its jobs runs at a time. That serial guarantee is the mechanism behind actors, as we're about to see &mdash; an actor is essentially state plus a serial executor that protects it.
      <br><br>
      Since Swift 5.9 you can supply your own. <strong>SE-0392</strong> opened up <strong>custom executors</strong>: conform a type to <code>SerialExecutor</code>, implement <code>enqueue(_ job:)</code> to place the job wherever you want &mdash; a specific dispatch queue, a dedicated thread, your own run loop &mdash; and hand it to an actor via the <code>unownedExecutor</code> property. The actor's isolation now runs on your machinery.
      <br><br>

      The flip side is <code>assumeIsolated</code>. Sometimes you have synchronous code that you <em>know</em> is already running on a particular actor's executor &mdash; a delegate callback that the framework promises to deliver on the main thread, say &mdash; but the compiler can't prove it. <code>MainActor.assumeIsolated { }</code> lets you assert that fact: if you're right, you get synchronous access to isolated state with no hop; if you're wrong, it traps rather than silently racing. It's the runtime-checked escape hatch for "trust me, we're already here."

  - id: actors
    title: "Actors — state behind a serial executor"
    summary: |
      Now the pieces snap together. An <strong>actor</strong> is a reference type, like a class, but with one structural difference: it owns a <strong>SerialExecutor</strong>, and all of its isolated state can only be touched by jobs running on that executor. Because a serial executor runs one job at a time, only one task is ever inside the actor's isolated code at any instant. Two tasks therefore <em>cannot</em> race on the actor's state &mdash; not because of a lock you wrote, but because the executor refuses to run two of its jobs concurrently.
      <br><br>
      <pre><code>actor BankAccount {&#10;    private var balance = 0&#10;    func deposit(_ amount: Int) { balance += amount }   // isolated&#10;    func currentBalance() -&gt; Int { balance }            // isolated&#10;}</code></pre>
      Calling into an actor from outside means your job has to be run by <em>that actor's</em> executor, not whichever executor you happen to be on. So the runtime performs an <strong>executor hop</strong>: it suspends your task, enqueues a job on the actor's serial executor, and resumes you there. That hop is a suspension point, and that is the real reason cross-actor calls are <code>await</code>-ed. The <code>await</code> isn't decoration; it's the place where you might have to wait your turn on the actor's queue.

  - id: actor-reentrancy
    title: "Actor reentrancy — the hazard across await"
    summary: |
      An actor runs one job at a time. So what happens when a job <em>itself</em> hits an <code>await</code> &mdash; say an actor method that calls out to the network &mdash; and suspends? The actor has nothing to do while that job is parked. Should it sit idle until the job resumes? Swift's answer is no: actors are <strong>reentrant</strong>. While one job is suspended at an <code>await</code>, the actor is free to pick up <em>another</em> queued job and run it.
      <br><br>
      This is a good thing for liveness &mdash; it's what stops actors deadlocking when two of them call each other &mdash; but it has a sharp edge. Any state you observed <em>before</em> an <code>await</code> inside an actor method may have been changed by another job <em>during</em> the suspension. The actor never let two jobs run at literally the same instant, so there's no torn memory; but your method's assumptions can still be quietly invalidated across the gap.

No lock would have saved you, because there was never simultaneous access. The invariant simply didn't survive the suspension.
      <br><br>
      The discipline is a single sentence: <strong>never assume an actor's state is unchanged across an <code>await</code></strong>. Re-check what matters after every suspension, or restructure so the check and the mutation are on the same side of the await with nothing suspendable between them. The safest version of <code>withdraw</code> reads and writes <code>balance</code> with no <code>await</code> in between, doing the network confirmation either entirely before the guard or entirely after the deduction. Reentrancy is a feature; treating it as if it weren't there is the bug.

  - id: global-actors-and-isolation
    title: "Global actors and isolation control"
    summary: |
      Some state isn't owned by an instance you can pass around &mdash; it's global, like everything that touches the UI. UIKit and SwiftUI must be driven from the main thread, full stop. Swift expresses "this belongs to one shared executor" with a <strong>global actor</strong>, and the one you'll use daily is <code>@MainActor</code>. It pins whatever it annotates &mdash; a type, a method, a property &mdash; to the main thread's executor.
      <br><br>
      <pre><code>@MainActor&#10;final class ViewModel {&#10;    var title = ""                 // main-actor isolated&#10;    func refresh() async {&#10;        let text = await loadTitle()   // hops OFF the main actor to load&#10;        title = text                   // hops BACK; safe main-thread write&#10;    }&#10;}</code></pre>
      A global actor is just a singleton actor with a type-level annotation. You can define your own when you have a domain that should be globally serialised &mdash; say all database access &mdash; by marking a type <code>@globalActor</code> and giving it a shared instance.
      <br><br>
      <pre><code>@globalActor&#10;actor DataActor {&#10;    static let shared = DataActor()&#10;}&#10;&#10;@DataActor func writeToDisk(_ rows: [Row]) { ... }   // all such work serialised</code></pre>
      Isolation isn't all-or-nothing, and the fine control is where real code lives. A member marked <code>nonisolated</code> opts out of the actor's isolation &mdash; useful for things that touch no mutable state, like a computed <code>id</code> or a <code>Sendable</code> constant &mdash; so it can be called synchronously from anywhere without a hop.
      <br><br>
      <pre><code>@MainActor&#10;final class Player {&#10;    let id: UUID                       // immutable&#10;    nonisolated var label: String { "Player \(id)" }   // no hop needed&#10;    var score = 0                      // still main-actor isolated&#10;}</code></pre>
      Going the other way, an <strong>isolated parameter</strong> lets a free function run <em>on</em> a given actor's executor. Declaring <code>isolated account: BankAccount</code> means the function is already inside that actor when it runs, so it can touch the actor's state synchronously and callers <code>await</code> the function instead of each access inside it. Between <code>nonisolated</code>, isolated parameters, and global actors, you can describe exactly which executor every piece of code belongs to &mdash; and that precision is what the data-race checker reads.

  - id: sendable-and-data-race-safety
    title: "Sendable and data-race safety"
    summary: |
      Actors guard the state they own. But values <em>cross</em> between concurrency domains all the time &mdash; an argument you pass to <code>group.addTask</code>, a result you return from an actor. The compiler needs to know which types are safe to send across those boundaries, and that's what <strong>Sendable</strong> marks. It's a <em>marker protocol</em>: it has no methods, and conformance is checked entirely at compile time by inspecting the type's shape.
      <br><br>
      The rules are mechanical. A value type whose every stored member is <code>Sendable</code> is itself <code>Sendable</code> &mdash; most structs and enums get it for free. A class is not <code>Sendable</code> unless it's immutable (all <code>let</code>, all-Sendable), or it's an actor, or you take responsibility by hand with <code>@unchecked Sendable</code>. A closure that captures across a boundary must be <code>@Sendable</code>, which forbids it from capturing mutable shared state.
      <br><br>
      <pre><code>struct Point: Sendable { var x, y: Int }     // automatic: value type of Sendables&#10;&#10;final class Counter { var n = 0 }            // NOT Sendable: mutable shared class&#10;&#10;// I promise I serialise access myself (e.g. with an internal lock):&#10;final class Cache: @unchecked Sendable {&#10;    private let lock = NSLock()&#10;    private var store: [String: Data] = [:]&#10;}</code></pre>
      For years all of this was advisory. The <strong>Swift 6 language mode</strong>, shipped in 2024, turns on <strong>strict data-race checking</strong>: send a non-<code>Sendable</code> value across a boundary, touch isolated state from the wrong executor, and the compiler refuses to build. Many data races stopped being runtime ghosts and became plain compile errors.
      <br><br>
      The trouble was that early strict checking was <em>too</em> conservative. It would reject code that was obviously fine &mdash; create a non-<code>Sendable</code> object, hand it to one task, and never touch it again, and the checker would still complain, because it reasoned per-value, not per-region. <strong>Region-based isolation (SE-0414)</strong> fixed this by tracking <em>regions</em> of values that could reference one another. If the compiler can prove a region is <strong>disconnected</strong> &mdash; nothing else still holds a reference &mdash; it lets you send the whole region across a boundary safely, even if the types aren't <code>Sendable</code>. Building on that, <strong>sending parameters (SE-0430)</strong> let a function declare a parameter as <code>sending</code>, meaning ownership is transferred in: the caller gives it up, the callee receives a disconnected region, and no race is possible.
      <br><br>
      <pre><code>// SE-0430: the value is transferred into the task, not shared&#10;func handOff(_ value: sending NonSendableThing) async {&#10;    await someActor.store(value)   // safe: caller relinquished it&#10;}</code></pre>
      Together these turned strict concurrency from a wall of false positives into something workable. The model is now coherent: <code>Sendable</code> says "safe to share freely," and <code>sending</code> plus region analysis says "safe to hand over exclusively." Most real code needs both.

  - id: bridging-and-the-bigger-picture
    title: "Bridging, streams, and the bigger picture"
    summary: |
      The async world is lovely, but the world is full of older code that still speaks callbacks &mdash; delegates, completion handlers, C libraries. The bridge from "call me back" to "await me" is a <strong>continuation</strong>. <code>withCheckedContinuation</code> (and its throwing sibling) hands you a continuation object and suspends the task. You stash the object, call the old API, and when its callback fires you <code>resume</code> the continuation with the result. The suspended <code>await</code> wakes up with that value.

      Why <strong>checked</strong>? Because a continuation carries one ironclad rule: it must be resumed <strong>exactly once</strong>. Resume it zero times and the awaiting task hangs forever &mdash; a leak. Resume it twice and you corrupt the runtime. These are classic callback bugs, easy to write and miserable to find. The checked variants police the contract: resume more than once and they trap; discard one without ever resuming and they log a warning. There's a faster <code>withUnsafeContinuation</code> that skips the checks &mdash; reach for it only once the logic is proven.
      <br><br>
      For values that arrive <em>over time</em> rather than once, the abstraction is <strong>AsyncSequence</strong>, and the easiest way to build one from a callback is <strong>AsyncStream</strong>. You feed it elements with a continuation that yields repeatedly and finishes once, and consumers iterate it with <code>for await</code> &mdash; back-pressure and cancellation handled for you.
      <br><br>
      <pre><code>let ticks = AsyncStream&lt;Int&gt; { continuation in&#10;    let timer = startTimer { continuation.yield($0) }&#10;    continuation.onTermination = { _ in timer.cancel() }&#10;}&#10;for await tick in ticks { print(tick) }   // streams until finished or cancelled</code></pre>
      Step back and the philosophy comes into focus, especially against Dart, which we contrasted in the isolates episode. Dart's <strong>isolates share nothing</strong>: each has its own heap and they communicate only by copying messages, so data races are structurally impossible &mdash; there's nothing shared to race over. Swift took the opposite road. Swift tasks and actors <strong>share memory</strong>, which is fast and convenient, and then fences that shared memory with actors (serialising access through a serial executor) and <code>Sendable</code> plus region isolation (policing what may cross boundaries). Two honest answers to the same hard problem: Dart removes the sharing; Swift keeps it and proves, at compile time, that the sharing is safe. Knowing which model you're in tells you exactly which class of bug you will never have to chase &mdash; and which discipline keeps you honest.
