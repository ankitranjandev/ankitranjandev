  - id: variables-are-boxes
    title: "Variables are labelled boxes"
    summary: |
      A variable is a name attached to a value. Think of it as a labelled box: the label is the variable name, the contents are the value. When we write <code>var name = 'Dart';</code>, we're creating a box labelled "name" and putting the string 'Dart' inside.

      In Dart, every variable has a type. The type describes what kind of values the box can hold. A <code>String</code> variable can only hold strings. An <code>int</code> variable can only hold integers. Try putting a string into an int box, and the compiler will refuse.

      <pre><code>// Creating variables&#10;var name = 'Dart';      // Type inferred as String&#10;String language = 'Dart'; // Type explicitly stated&#10;int count = 42;         // Integer variable&#10;double pi = 3.14159;    // Floating-point variable&#10;bool isReady = true;    // Boolean variable</code></pre>

      The key question is: what can we do with this box once it's created? Can we replace the contents? Can we change them after the program is compiled? This is where keywords like <code>var</code>, <code>final</code>, and <code>const</code> come in.

  - id: var-the-mutable-default
    title: "var — the mutable default"
    summary: |
      The <code>var</code> keyword creates a variable whose value can be changed. It's the most flexible option. Declare it once, reassign it as many times as you want.

      <pre><code>var score = 0;&#10;score = 10;      // OK&#10;score = 100;     // OK&#10;score = 'high';  // ERROR: can't change type</code></pre>

      Notice the last line. With <code>var</code>, we can change the <em>value</em>, but not the <em>type</em>. Once Dart infers that <code>score</code> is an <code>int</code>, it stays an <code>int</code> forever. This is static typing at work — the type is fixed at compile time, even though the value isn't.

      When should we use <code>var</code>? When we genuinely need to reassign the variable. Loop counters, accumulators, state that changes over time. But if we don't need to reassign, there's a better option.

  - id: final-set-once
    title: "final — set once, keep forever"
    summary: |
      A <code>final</code> variable can only be assigned once. After that first assignment, the reference is locked.

      <pre><code>final name = 'Dart';&#10;name = 'Flutter';  // ERROR: can't reassign final variable</code></pre>

      The value of <code>final</code> is computed at runtime. This means we can assign the result of a function call, user input, or any expression that evaluates when the program runs.

      <pre><code>final now = DateTime.now();     // Computed at runtime&#10;final user = fetchCurrentUser(); // From a function&#10;final config = loadConfig();    // Loaded from file</code></pre>

      Using <code>final</code> tells other developers (and ourselves): "This value won't change after it's set." It makes code easier to reason about. We don't have to track whether the variable might have been reassigned somewhere else.

      The Dart style guide recommends <code>final</code> over <code>var</code> when reassignment isn't needed. It's a good default.

      <pre><code>const pi = 3.14159;           // OK: literal value&#10;const doubled = pi * 2;       // OK: computed from const&#10;const now = DateTime.now();   // ERROR: not compile-time</code></pre>

      The third line fails because <code>DateTime.now()</code> returns a different value each time the program runs. The compiler can't know what "now" will be.

      <code>const</code> is powerful because identical <code>const</code> values share memory. Create the same <code>const</code> list in two places, and Dart stores only one copy. This matters for performance and equality checks.

      When should we use explicit types? The Dart style guide says: prefer <code>var</code> for local variables when the type is obvious. Use explicit types for public APIs (function parameters, return types, class fields) where clarity matters more.

      <pre><code>// Local variables: var is fine&#10;var items = fetchItems();&#10;&#10;// Public API: explicit types are clearer&#10;List&lt;Item&gt; fetchItems() { ... }&#10;void addItem(Item item) { ... }</code></pre>

  - id: late-deferred-init
    title: "late — promise to initialise later"
    summary: |
      The <code>late</code> keyword tells Dart: "I'm not assigning this now, but I promise it will be assigned before anyone reads it."

      If we break the promise and read before assignment, Dart throws a <code>LateInitializationError</code> at runtime. It's a contract: we take responsibility for ensuring initialisation happens.

      <pre><code>class UserService {&#10;  late final Database _db;&#10;&#10;  void init(Database db) {&#10;    _db = db;  // Assigned after constructor&#10;  }&#10;&#10;  User getUser(int id) => _db.query('...');&#10;}</code></pre>

      The <code>late final</code> combination is powerful: lazy initialisation that can only happen once. The expensive computation runs only when first accessed.

  - id: dynamic-escape-hatch
    title: "dynamic — the escape hatch"
    summary: |
      Sometimes we genuinely don't know the type. JSON from a server, values from reflection, interop with JavaScript. For these cases, Dart offers <code>dynamic</code> — a way to opt out of static type checking.

      With <code>dynamic</code>, any method call compiles — but might crash at runtime. The alternative is <code>Object</code> (or <code>Object?</code>), which accepts any value but only allows methods that exist on all objects.

      <pre><code>dynamic d = 'hello';&#10;print(d.length);    // OK at compile and runtime&#10;print(d.foo());     // Compiles, crashes at runtime&#10;&#10;Object o = 'hello';&#10;print(o.length);    // Compile error: Object has no length</code></pre>

      Use <code>dynamic</code> sparingly. Every use is a place where the compiler can't help us catch mistakes.

  - id: practical-guidelines
    title: "Practical guidelines"
    summary: |
      So what should we actually use? Here's a decision tree:
      <br><br>
      • <strong>Will the value change?</strong> No → use <code>final</code><br>
      • <strong>Is it a compile-time constant?</strong> Yes → use <code>const</code><br>
      • <strong>Do you need to reassign?</strong> Yes → use <code>var</code><br>
      • <strong>Can't initialise in constructor?</strong> Consider <code>late</code><br>
      • <strong>Don't know the type at all?</strong> Use <code>Object</code> first, <code>dynamic</code> as last resort<br><br>

      <pre><code>// Good defaults&#10;final name = 'Dart';           // Won't change&#10;const pi = 3.14159;            // Compile-time constant&#10;var count = 0;                 // Needs to change&#10;late final config = loadConfig(); // Lazy + immutable</code></pre>

      The goal is to give the compiler (and future readers) as much information as possible. <code>final</code> says "this won't change." <code>const</code> says "this can't change, ever, and is known at compile time." These constraints help us write safer code.

      As a rule of thumb: default to <code>final</code>, reach for <code>const</code> when possible, use <code>var</code> only when reassignment is genuinely needed.
