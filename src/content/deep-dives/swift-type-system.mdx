summary: |
  Generics and protocols look like two separate features, but at runtime they share one mechanism: <strong>witness tables</strong>.<br><br>
  A generic body is compiled once and handed hidden arguments — <strong>type metadata</strong>, a <strong>Value Witness Table</strong> (how to copy, move, and destroy a value of unknown size), and one <strong>Protocol Witness Table</strong> per constraint.<br>
  An <code>any P</code> existential is a container: a 3-word inline buffer, a metadata pointer, and witness pointers, with large values <strong>heap-boxed</strong>.<br>
  A <code>some P</code> opaque type fixes one concrete type with no box and preserved identity.<br>
  <pre><code>func process&lt;T: Hashable&gt;(_ v: T) { ... }&#10;// You write:    process(user)&#10;// Swift passes: user, User.self, T's VWT, T's Hashable PWT</code></pre>
  This episode unifies the generics and protocols stories at the level the optimiser sees: metadata records, the dispatch taxonomy (static, vtable, witness, message), the existential container layout, and the conformance cache that powers <code>as?</code>.
topics:
  - id: types-at-the-centre
    title: "Types at the centre — nominal typing and the compile/run split"
    summary: |
      Let's begin with a claim that sounds grand but pays off all the way down: in Swift, the type is the unit of meaning, and almost every interesting decision the language makes is about types. So before we touch generics or protocols, we need to be precise about what a type even is here.
      <br><br>
      Swift uses <strong>nominal typing</strong>. A type is identified by its declared name, not by the shape of its members. Two structs with byte-for-byte identical layouts are still different types if they have different names, and one will never silently stand in for the other. This is the opposite of <strong>structural typing</strong>, where compatibility is decided by shape — the way TypeScript objects or Go interfaces work. In Swift, conformance and identity are always explicit and by name.
      <br><br>
      <pre><code>struct Metres { let value: Double }&#10;struct Seconds { let value: Double }&#10;&#10;let d = Metres(value: 10)&#10;let t: Seconds = d   // error: Metres is not Seconds,&#10;                     // even though the layout is identical</code></pre>
      That nominal discipline is exactly what lets the type system catch unit mix-ups and a thousand subtler bugs. Now, two special types sit at the very top. <code>Any</code> is the supertype of <em>everything</em> — every value, value type or reference type, is an <code>Any</code>. <code>AnyObject</code> is narrower: the supertype of all <em>class</em> instances only. A struct is an <code>Any</code> but not an <code>AnyObject</code>.
      <br><br>
      <pre><code>let things: [Any] = [42, "hi", Metres(value: 1)]&#10;let objects: [AnyObject] = [NSString(), UIView()]  // classes only</code></pre>
      And here is the split that the rest of this episode keeps returning to. Swift is a statically typed, compiled language, so most type work happens at <strong>compile time</strong>: inference, constraint checking, overload resolution, devirtualisation. But unlike Java, Swift does <em>not</em> erase types. It keeps a <strong>runtime</strong> representation of every type — a metadata record — which is what makes <code>type(of:)</code>, dynamic casts with <code>as?</code>, and unspecialised generics possible at all. Keep that division in mind: some questions are answered by the compiler and vanish; others are answered by metadata that survives into the running program.
      <br><br>
      Some questions vanish at compile time; others are answered by metadata that survives. The moment you call a generic, the compiler quietly supplies the runtime with everything it needs to operate on a type it was not specialised for — which is exactly where we are headed next.
  - id: generics-and-constraints
    title: "Generics — type parameters, constraints, and where clauses"
    summary: |
      Generics are Swift's form of <strong>parametric polymorphism</strong>: one piece of code, parameterised by a type, that works uniformly across every type you plug in while keeping full static safety. We covered the basics in the generics episode, so let's move quickly and aim for the corners that matter at the type-system level.
      <br><br>
      A type parameter is a placeholder bound at the call site. On its own it is nearly useless, because the compiler knows nothing about it, so a <strong>constraint</strong> is how we trade a promise for a capability. Promise that <code>T</code> is <code>Comparable</code>, and in return the compiler lets you write <code>a &lt; b</code> inside the body.
      <br><br>
      <pre><code>func clamp&lt;T: Comparable&gt;(_ x: T, to range: ClosedRange&lt;T&gt;) -&gt; T {&#10;    min(max(x, range.lowerBound), range.upperBound)&#10;}&#10;&#10;clamp(12, to: 0...10)        // 10&#10;clamp("m", to: "a"..."z")     // "m"</code></pre>
      When the requirement is about an <em>associated</em> type rather than the parameter itself, a bare <code>: Protocol</code> can't say it. That is what the <code>where</code> clause is for — it expresses relationships the angle-bracket shorthand cannot.
      <br><br>
      <pre><code>func sum&lt;S: Sequence&gt;(_ s: S) -&gt; Int&#10;    where S.Element == Int {&#10;    s.reduce(0, +)&#10;}&#10;&#10;sum([1, 2, 3])   // 6&#10;sum(1...100)     // 5050, a Range is a Sequence too</code></pre>
      The most quietly powerful corner is <strong>conditional conformance</strong> (SE-0143). It lets a generic type conform to a protocol only when its parameters do. The standard library leans on it constantly: an <code>Array</code> is <code>Equatable</code> exactly when its <code>Element</code> is, <code>Hashable</code> exactly when its <code>Element</code> is, and an <code>Optional</code> the same.
      <br><br>
      <pre><code>extension Array: Equatable where Element: Equatable {&#10;    // == compares element by element&#10;}&#10;&#10;[1, 2, 3] == [1, 2, 3]   // true — Int is Equatable&#10;// [view1] == [view2]    // won't compile if View isn't Equatable</code></pre>
      Conditional conformance is the feature that makes Swift's collections feel like they 'just work' — you never declare that <code>[Int]</code> is comparable, yet it is, precisely when it should be and not a moment sooner. All of this is the compile-time face of generics. The real question, the one this episode exists to answer, is what survives into the running program. What actually happens when a generic runs?
  - id: how-a-generic-runs
    title: "How an unspecialised generic actually runs"
    summary: |
      Here is the heart of it. A Swift generic, by default, is <strong>not</strong> compiled into a separate copy per type the way a C++ template is, and it is <strong>not</strong> erased the way a Java generic is. The compiler emits <em>one</em> shared body, and that body is handed the type information it needs as hidden runtime arguments.
      <br><br>
      There are three things a shared body cannot know about an arbitrary <code>T</code>: how big a <code>T</code> is, how to copy or destroy one, and how <code>T</code> satisfies its constraints. Swift answers each with a table. The first two come from the <strong>type metadata</strong> and its <strong>Value Witness Table</strong>; the third from a <strong>Protocol Witness Table</strong> per constraint.
      <br><br>
      The Value Witness Table (VWT) is the one most developers have never heard of, and it is the load-bearing one. A generic body that copies, moves, or destroys a value of type <code>T</code> cannot emit a fixed-size memcpy — it doesn't know the size. So it calls through the VWT: <em>initialise-with-copy</em>, <em>assign</em>, <em>move</em>, <em>destroy</em>, plus the type's <em>size</em>, <em>stride</em>, and <em>alignment</em>. Every value type carries one.
      <br><br>
      <pre><code>func duplicate&lt;T&gt;(_ x: T) -&gt; (T, T) {&#10;    (x, x)   // copying x calls T's value witness 'initializeWithCopy'&#10;}</code></pre>
      The Protocol Witness Table (PWT) answers the conformance question: for each requirement of a constraint, where is the concrete implementation? One PWT is threaded in per protocol constraint, so a <code>&lt;T: Hashable&gt;</code> gets one table, a <code>&lt;T: Hashable &amp; Codable&gt;</code> gets two.

      <br><br>
      So the call you write, <code>process(user)</code>, becomes — at the ABI level — a call that also receives <code>User.self</code> as metadata, the address of <code>User</code>'s VWT, and the address of <code>User</code>'s <code>Hashable</code> PWT. The shared body uses the VWT to manage the value's memory and the PWT to dispatch <code>hash(into:)</code>. This is why Swift generics keep full type identity at runtime, and also why an unspecialised generic carries a small, real overhead: every value operation and every requirement call is an indirect one through a table. Which raises the obvious question — can we ever get rid of that indirection?
  - id: generic-specialisation
    title: "Specialisation — when the optimiser clones a generic"
    summary: |
      We can, and the optimiser does it routinely. <strong>Generic specialisation</strong> is the compiler cloning a generic body for a specific concrete type, baking the type in and throwing the tables away. The result is indistinguishable from a hand-written function for that type: fixed-size copies instead of VWT calls, direct calls instead of PWT lookups.
      <br><br>
      <pre><code>func add&lt;T: Numeric&gt;(_ a: T, _ b: T) -&gt; T { a + b }&#10;&#10;let n = add(3, 4)   // optimiser can stamp out 'add for Int':&#10;                    // a tight, table-free integer add</code></pre>
      For this to fire, the optimiser must be able to <em>see</em> both the generic's body and the concrete type at the same time. Within a single file that is easy. Across the whole module it needs <strong>whole-module optimisation</strong> (WMO), which compiles the module as one unit so the body and its call sites are visible together. WMO is the single biggest reason a release build of Swift can be dramatically faster than a debug build for generic-heavy code.

      <br><br>
      But module boundaries are a wall. If a generic lives in the standard library and you call it from your app, the optimiser normally cannot see the library's body, so it cannot specialise. That is what <code>@inlinable</code> solves: it publishes a function's body into the module's interface, so client code can inline and specialise across the boundary. Huge swathes of the standard library — <code>map</code>, <code>filter</code>, <code>reduce</code>, the <code>Collection</code> machinery — are marked <code>@inlinable</code> precisely so that your concrete uses compile down to tight, specialised code rather than table-driven generic calls.
      <br><br>
      <pre><code>@inlinable&#10;public func twice&lt;T: Numeric&gt;(_ x: T) -&gt; T { x + x }&#10;// Body is now visible to clients; they can specialise it for Int.</code></pre>
      The trade is the classic one: specialisation buys speed with code size, since each concrete clone is more machine code. Swift's default — one shared body, specialise where it pays — keeps the binary small while letting hot paths reach C++-template speed. You rarely steer this by hand; you simply build for release with WMO and let the optimiser earn its keep.
  - id: protocols-and-the-pwt
    title: "Protocols as requirements — the Protocol Witness Table"
    summary: |
      Now the other half of the type system. A protocol is a set of <strong>requirements</strong> — methods, properties, initialisers, subscripts — that a conforming type must satisfy. The protocols episode covered conformance and protocol-oriented design; here we go straight to the runtime object that makes it all work, the same Protocol Witness Table we met threading through generics.
      <br><br>
      When a type conforms to a protocol, the compiler builds a PWT for that pairing. It has one slot per requirement, and each slot holds a pointer to the type's concrete implementation of that requirement. Conformance, at runtime, <em>is</em> this table.
      <br><br>
      <pre><code>protocol Drawable {&#10;    var area: Double { get }&#10;    func draw()&#10;    func scale(_ factor: Double) -&gt; Self&#10;}&#10;&#10;struct Circle: Drawable {&#10;    var radius: Double&#10;    var area: Double { .pi * radius * radius }&#10;    func draw() { /* ... */ }&#10;    func scale(_ f: Double) -&gt; Circle { Circle(radius: radius * f) }&#10;}&#10;// The compiler builds a 'Drawable PWT for Circle' with 3 slots.</code></pre>

      <br><br>
      When you call a requirement on a value whose type is only known to be <code>Drawable</code> — through an <code>any Drawable</code>, or through a generic <code>&lt;T: Drawable&gt;</code> — the call doesn't know <code>Circle</code> statically. It indexes the requirement's fixed slot in the PWT, follows the pointer, and jumps to <code>Circle.draw</code>. That is <strong>witness-table dispatch</strong>: one indirection, no message-send machinery, no string lookup.
      <br><br>
      Notice the symmetry with the previous topics. The VWT answers 'how big is this value and how do I move it?'; the PWT answers 'how does this value satisfy that contract?'. Generics receive both as hidden arguments; existentials, as we are about to see, package both <em>inside</em> the value itself. Same tables, different delivery. But first, the single most important gotcha in all of protocol-oriented Swift.
  - id: the-dispatch-trap
    title: "Protocol extensions and the dispatch trap"
    summary: |
      Here is the trap that catches every Swift developer at least once, and it falls straight out of how the witness table is built. The rule: <strong>only members declared as requirements in the protocol body are dispatched through the witness table.</strong> A member that exists <em>solely</em> in a protocol extension is not a requirement, has no witness slot, and is therefore <strong>statically dispatched</strong> — resolved by the variable's compile-time type, not the value's dynamic type.
      <br><br>
      <pre><code>protocol Greeter {&#10;    func hello() -&gt; String       // a requirement&#10;}&#10;&#10;extension Greeter {&#10;    func hello() -&gt; String { "default hello" }&#10;    func bye()   -&gt; String { "default bye" }   // extension only!&#10;}&#10;&#10;struct Robot: Greeter {&#10;    func hello() -&gt; String { "beep" }   // overrides requirement&#10;    func bye()   -&gt; String { "boop" }   // shadows extension&#10;}</code></pre>
      Now call both methods through a variable typed as the protocol, and watch the asymmetry.
      <br><br>
      <pre><code>let g: any Greeter = Robot()&#10;g.hello()   // "beep"  — requirement: dynamic, runs Robot's version&#10;g.bye()     // "boop"? NO — "default bye"  — extension only: static!</code></pre>
      <br><br>
      Read that again. <code>g.hello()</code> gives <code>"beep"</code> because <code>hello</code> is a real requirement with a witness slot, so the call dispatches dynamically to <code>Robot</code>'s implementation. But <code>g.bye()</code> gives <code>"default bye"</code>, not <code>"boop"</code>, because <code>bye</code> lives only in the extension. There is no witness slot for it, so the compiler binds it statically against the type it can see — <code>any Greeter</code> — and runs the extension's version. <code>Robot.bye()</code> is never consulted, even though the value really is a <code>Robot</code>.
      <br><br>
      This is not a bug; it is the direct, logical consequence of the table only holding declared requirements. The fix is one line: if you want an extension method to be overridable, <strong>declare it as a requirement</strong> in the protocol body too. Then it earns a witness slot, and the override takes effect. Whenever a protocol method 'mysteriously' ignores a type's own version, this is almost always why.
  - id: existentials-any
    title: "Existentials — any P and the container layout"
    summary: |
      We have seen generics receive their tables as hidden arguments. An <strong>existential</strong> takes the other approach: it packages the value and its tables together into one self-describing container. In modern Swift you spell it <code>any P</code> (the explicit keyword arrived in SE-0335, Swift 5.6, to make the cost legible).
      <br><br>
      An <code>any Shape</code> can hold a <code>Circle</code> today and a <code>Square</code> tomorrow, in the same storage slot, with the concrete type decided at runtime. To make that possible the container must carry everything needed to operate on whatever is inside.
      <br><br>
      <pre><code>let shapes: [any Shape] = [Circle(radius: 1), Square(side: 2)]&#10;for s in shapes {&#10;    s.draw()   // dispatched through the box's witness pointer&#10;}</code></pre>
      What is the layout? The existential container is a fixed five words on a 64-bit platform: a <strong>three-word inline value buffer</strong>, a <strong>type-metadata pointer</strong>, and a <strong>witness-table pointer</strong> (one per protocol the existential requires). If the value fits in three words, it lives inline and there is no allocation. If it is larger, Swift <strong>heap-allocates a box</strong> for the payload and the inline buffer holds a pointer to it.

      That layout explains the costs people attribute vaguely to 'protocols being slow'. A small value in an <code>any</code> is cheap — inline storage, one witness hop per call. A large value pays an allocation on the way in and a pointer chase on every access. And because the metadata pointer travels with the value, the runtime can recover the real type later, which is what makes <code>as?</code> on an existential work.
      <br><br>
      It also explains an old frustration. Because the box <em>erases concrete type identity</em>, the compiler can't prove that two <code>any Equatable</code> values share a type — and <code>==</code> needs them to. That is why protocols with <code>Self</code> or associated-type requirements could not be used as plain existentials for years. We'll see how that wall came down shortly. For now, the takeaway: <code>any</code> buys you a heterogeneous, runtime-flexible value, and you pay for it in indirection and sometimes allocation. When you don't need that flexibility, there is a cheaper option.
  - id: opaque-types-some
    title: "Opaque types — some P, reverse generics"
    summary: |
      That cheaper option is <code>some P</code>, the <strong>opaque type</strong>, introduced in SE-0244 (Swift 5.1). It is best understood as <em>reverse generics</em>. With an ordinary generic, the <em>caller</em> chooses the concrete type and the function adapts. With <code>some P</code>, the <em>callee</em> chooses one concrete type, fixes it, and hides its name — the caller knows only that it conforms to <code>P</code>.
      <br><br>
      <pre><code>func makeShape() -&gt; some Shape {&#10;    Circle(radius: 5)   // always a Circle; caller can't name it&#10;}</code></pre>
      The crucial difference from <code>any</code>: there is exactly <em>one</em> underlying type, known to the compiler, so there is <strong>no box, no metadata-carrying container, and no dynamic dispatch by necessity</strong>. The value is just a <code>Circle</code>; only the name is curtained off. And because the type is fixed, <strong>identity is preserved</strong> — two values returned from a <code>some Shape</code> function are guaranteed to be the same type, so the compiler will let you compare them if that type is <code>Equatable</code>. An <code>any Shape</code> gives no such promise.

      <br><br>
      SE-0341 (Swift 5.7) then extended <code>some</code> to <strong>parameter position</strong>.   - id: associated-types-and-self
    title: "Associated types, Self, and constrained existentials"
    summary: |
      Some protocols can't name their types up front. <code>Container</code> has elements, but it shouldn't care whether they are <code>Int</code> or <code>String</code>. Swift expresses this with an <strong>associated type</strong>: a placeholder the conforming type fills in. And some protocols mention <code>Self</code> — <code>Equatable</code>'s <code>==</code> takes two <code>Self</code> values. Both features were, historically, what kept a protocol from being a plain existential.
      <br><br>
      <pre><code>protocol Container {&#10;    associatedtype Element&#10;    var count: Int { get }&#10;    subscript(i: Int) -&gt; Element { get }&#10;}</code></pre>
      Why couldn't you write <code>let c: Container = ...</code> in older Swift? Because the existential box erases type identity. If you had two <code>Equatable</code> existentials, the compiler couldn't guarantee they boxed the same concrete type, so <code>==</code> — which demands matching <code>Self</code> — had nowhere safe to land. Associated types had the same problem: an existential <code>Container</code> couldn't tell you what its <code>Element</code> was, so you couldn't safely use the values it produced. These were the famous 'Protocol can only be used as a generic constraint' errors.
      <br><br>
      Two proposals dismantled most of that wall. <strong>Primary associated types</strong> (SE-0346, Swift 5.7) let a protocol promote an associated type into angle brackets — <code>protocol Container&lt;Element&gt;</code> — so you can constrain it the way you constrain a generic type's parameter. Building on that, <strong>constrained existentials</strong> (SE-0353) let you write <code>any Collection&lt;Int&gt;</code>: an existential that has erased the <em>concrete</em> type but still pins down the <em>element</em> type.
      <br><br>
      <pre><code>// Now legal and readable:&#10;func describe(_ c: any Collection&lt;Int&gt;) {&#10;    print("a collection of \(c.count) ints")&#10;}&#10;&#10;describe([1, 2, 3])           // Array&lt;Int&gt;&#10;describe(Set([4, 5]))         // Set&lt;Int&gt; — different concrete type&#10;describe((1...10))            // Range&lt;Int&gt;</code></pre>
      This is the payoff of understanding the box. The old restriction was never arbitrary; it was the compiler refusing to let you do something it couldn't prove safe once identity was erased. Pin down enough of the type — the element, via a primary associated type — and the existential becomes usable again, because now there is something concrete to hold onto.
  - id: dispatch-taxonomy
    title: "The dispatch taxonomy — four ways a call finds its code"
    summary: |
      We've now met every dispatch mechanism Swift uses, scattered across topics. Let's collect them into a single taxonomy, because knowing which one a given call uses is the key to reasoning about performance. There are four, and they run from free to fairly expensive.
      <br><br>
      <strong>Static (direct) dispatch</strong> is a plain call to a known address — no lookup at all. It applies to methods on <code>struct</code>s and <code>enum</code>s, anything marked <code>final</code>, global functions, and anything the optimiser has devirtualised. It is also the only kind that can be inlined.
      <br><br>
      <strong>Vtable dispatch</strong> is virtual dispatch for classes. A non-<code>final</code> method of a regular (non-<code>@objc</code>) class is found by indexing the class's method table — one indirection, then the jump. This is what lets a subclass override a method.
      <br><br>
      <strong>Witness-table dispatch</strong> is the protocol mechanism we've studied: index the PWT slot for the requirement, follow the pointer, jump. It applies whenever you call a requirement through <code>any P</code> or through a generic constraint without specialisation.
      <br><br>
      <strong>Message dispatch</strong> is the Objective-C runtime's <code>objc_msgSend</code>: a selector-based lookup with a cache. It applies to <code>@objc dynamic</code> members and to <code>NSObject</code> subclasses' overridable methods. It is the slowest, but also the most dynamic — it's what makes KVO, method swizzling, and many of UIKit's tricks possible.

      How do we steer this? Push calls toward <code>static</code> for speed: mark classes or methods <code>final</code> when you don't need overriding, prefer value types, and build for release so whole-module optimisation can devirtualise vtable and even witness calls when it can prove the concrete type. Pull toward <code>dynamic</code> only when you need the Objective-C runtime's flexibility. The mechanism isn't usually something you pick explicitly — it falls out of how you declare the type — but knowing the mapping lets you predict, and control, the cost of every call you write.
  - id: metadata-and-conformances
    title: "Type metadata, metatypes, and the conformance cache"
    summary: |
      We keep saying Swift 'keeps the type at runtime'. Let's make that concrete. For every type — struct, enum, class, the lot — the compiler emits a <strong>type metadata record</strong>: a runtime data structure describing the type. It holds the type's kind, its name, its layout, a pointer to its VWT, and for classes the vtable and superclass. This record is the runtime's handle on the type.
      <br><br>
      You touch metadata through <strong>metatypes</strong>. <code>T.self</code> gives the metatype value for a statically known type; <code>type(of: x)</code> gives the <em>dynamic</em> metatype of a value, which for an existential or a class can differ from its static type.
      <br><br>
      <pre><code>let m1 = Int.self            // Int.Type, a metatype value&#10;let shape: any Shape = Circle(radius: 1)&#10;let m2 = type(of: shape)    // Circle.Type — the DYNAMIC type,&#10;                            // recovered from the box's metadata ptr&#10;print(m2)                   // Circle</code></pre>
      <br><br>
      Conformances get their own runtime presence. When a type conforms to a protocol, the compiler emits a <strong>protocol conformance record</strong> into a section of the binary that the runtime scans. So at runtime there is, in effect, a registry: 'these (type, protocol) pairs have witness tables, and here they are'. This is what powers the dynamic cast.
      <br><br>
      <pre><code>let things: [Any] = [1, "two", Circle(radius: 3)]&#10;for t in things {&#10;    if let s = t as? Shape {   // runtime: is there a conformance&#10;        s.draw()               // record for type(of: t) : Shape?&#10;    }&#10;}</code></pre>
      When you write <code>t as? Shape</code>, the runtime takes the value's metadata, asks 'does this type conform to <code>Shape</code>?', and answers by consulting those conformance records. Because that lookup would be slow to repeat, the runtime maintains a <strong>conformance cache</strong>: the first query for a given (type, protocol) pair does the scan, caches the resulting witness-table pointer (or its absence), and every later <code>as?</code> on that pair is a fast cache hit. So the seemingly magical <code>as?</code> is just metadata plus a cached table lookup — the same tables we've been tracing all along, now reachable dynamically.
  - id: variance-and-erasure
    title: "Variance and the type-erasure pattern"
    summary: |
      One last corner of the type system that surprises people: <strong>variance</strong>. If <code>Dog</code> is a subtype of <code>Animal</code>, is <code>Box&lt;Dog&gt;</code> a subtype of <code>Box&lt;Animal&gt;</code>? In Swift, for your own generic types, the answer is <strong>no</strong>. User-defined generics are <strong>invariant</strong>: <code>Box&lt;Dog&gt;</code> and <code>Box&lt;Animal&gt;</code> are unrelated types, and neither substitutes for the other.
      <br><br>
      <pre><code>struct Box&lt;T&gt; { var value: T }&#10;let dogBox = Box(value: Dog())&#10;let animalBox: Box&lt;Animal&gt; = dogBox   // error: invariant</code></pre>
      Invariance is the safe default — covariant mutable containers are a classic source of unsoundness. But the standard library special-cases the two places where covariance is genuinely wanted: <code>Array&lt;T&gt;</code> and <code>Optional&lt;T&gt;</code> get built-in <strong>covariant conversions</strong>, so <code>[Dog]</code> can be passed where <code>[Animal]</code> is expected (the conversion bridges them under the hood). Function types vary too, and in two directions at once: <strong>parameters are contravariant, results are covariant</strong> — a function may accept a <em>wider</em> parameter type and return a <em>narrower</em> result and still substitute.

      So how did people work around invariance before <code>some</code> and primary associated types existed? With the <strong>type-erasure pattern</strong>. You wrap an unknown conforming type behind a single concrete type that forwards every call. The standard library ships several — <code>AnySequence</code>, <code>AnyCollection</code>, <code>AnyHashable</code> — and for years you hand-rolled your own <code>AnyView</code>-style wrappers, storing closures that captured the underlying value.
      <br><br>
      <pre><code>// The old way: a hand-written eraser&#10;struct AnyShape: Shape {&#10;    private let _draw: () -&gt; Void&#10;    init&lt;S: Shape&gt;(_ s: S) { _draw = s.draw }&#10;    func draw() { _draw() }&#10;}&#10;&#10;let shapes: [AnyShape] = [AnyShape(Circle(radius: 1))]</code></pre>
      That boilerplate is exactly what modern Swift reduced. <code>some P</code> replaced erasure wherever you just wanted to hide one fixed type, and constrained existentials such as <code>any Collection&lt;Int&gt;</code> replaced it wherever you needed a real heterogeneous container with a known element type — without writing a forwarding wrapper by hand. The arc of the whole type system bends this way: every feature we've covered, from witness tables to opaque types, exists to let us hide a type when we want to while keeping exactly as much of it as we need.
