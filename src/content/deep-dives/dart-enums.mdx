  - id: basic-enums
    title: "Basic enums — simple enumerated types"
    summary: |
      Sometimes a value can only be one of a small, fixed set of options. Days of the week. Directions on a compass. HTTP request methods. We could use strings or integers, but that leads to bugs.
      <br><br>
      <pre><code>// The fragile way — strings
      String status = 'loading';  // typo: 'laoding' compiles fine
      if (status == 'Loading') {  // case mismatch — always false
        // ...
      }</code></pre>
      Strings are flexible, but they're too flexible. The compiler can't catch our typos. Enter enums.
      <br><br>
      <pre><code>enum Status { loading, success, error }

      void main() {
        var current = Status.loading;
        print(current);        // Status.loading
        print(current.name);   // 'loading'
        print(current.index);  // 0
      }</code></pre>

      <br><br>
      Every enum automatically has three things:
      <br><br>
      1. <strong>.index</strong> — the position in the declaration (0, 1, 2...)
      <br>
      2. <strong>.name</strong> — the string representation ('loading', 'success', 'error')
      <br>
      3. <strong>EnumType.values</strong> — a list of all values in declaration order
      <br><br>
      <pre><code>// Iterate all values
      for (var s in Status.values) {
        print('${s.name} is at index ${s.index}');
      }
      // loading is at index 0
      // success is at index 1
      // error is at index 2

      // Parse from string (Dart 2.15+)
      var parsed = Status.values.byName('success');  // Status.success
      var maybe = Status.values.asNameMap()['error']; // Status.error</code></pre>
      With enums, misspelling a value is a compile error. That alone makes them worth using. But basic enums have a limitation — they're just names. What if we want each value to carry data?

  - id: enhanced-enums
    title: "Enhanced enums — adding fields and constructors"
    summary: |
      Before Dart 2.17 (May 2022), enums were limited to bare values. If we wanted to attach data to enum values, we had to use a separate Map or a class hierarchy. Enhanced enums changed everything.
      <br><br>
      <pre><code>enum HttpStatus {
        ok(200, 'OK'),
        created(201, 'Created'),
        badRequest(400, 'Bad Request'),
        notFound(404, 'Not Found'),
        serverError(500, 'Internal Server Error');

        final int code;
        final String message;

        const HttpStatus(this.code, this.message);
      }</code></pre>
      Now each enum value carries its own data. We access it just like any other field.
      <br><br>
      <pre><code>void main() {
        var status = HttpStatus.notFound;
        print(status.code);     // 404
        print(status.message);  // 'Not Found'
        print(status.name);     // 'notFound' (still works)
      }</code></pre>

      <br><br>
      <strong>The rules for enhanced enums:</strong>
      <br><br>
      1. <strong>All fields must be final.</strong> Enum values are constants — they cannot change after creation.
      <br><br>
      2. <strong>Constructors must be const.</strong> Every enum value is a compile-time constant.
      <br><br>
      3. <strong>All values must be declared before any members.</strong> The values come first, then the semicolon, then fields/constructors/methods.
      <br><br>
      4. <strong>Values must call a constructor.</strong> Each value is an instance created by calling one of the enum's constructors.
      <br><br>
      <pre><code>// Named constructors work too
      enum Priority {
        low.withLevel(1),
        medium.withLevel(5),
        high.withLevel(10),
        critical.withLevel(100);

        final int level;
        const Priority.withLevel(this.level);
      }</code></pre>

  - id: enum-methods
    title: "Enum methods and getters"
    summary: |
      Enhanced enums aren't just data holders — they can have methods and computed properties. This lets us encapsulate behaviour right where the data lives.
      <br><br>
      <pre><code>enum Planet {
        mercury(3.7),
        venus(8.87),
        earth(9.81),
        mars(3.72),
        jupiter(24.79),
        saturn(10.44),
        uranus(8.87),
        neptune(11.15);

        final double gravity;  // m/s²
        const Planet(this.gravity);

        // Computed property
        double get weightFactor => gravity / earth.gravity;

        // Method
        double calculateWeight(double earthWeight) {
          return earthWeight * weightFactor;
        }

        // Static method for lookup
        static Planet? fromName(String name) {
          return values.cast&lt;Planet?&gt;().firstWhere(
            (p) => p?.name.toLowerCase() == name.toLowerCase(),
            orElse: () => null,
          );
        }
      }

      void main() {
        var mars = Planet.mars;
        print(mars.gravity);                    // 3.72
        print(mars.weightFactor);               // ~0.38
        print(mars.calculateWeight(70));        // ~26.55 kg

        var found = Planet.fromName('JUPITER');
        print(found?.gravity);                  // 24.79
      }</code></pre>
      The pattern is powerful. Instead of scattering planet-related calculations across utility functions, we put them on the enum itself. The data and its behaviour live together.
      <br><br>
      <pre><code>// Another example: HTTP status categories
      enum HttpStatus {
        ok(200, 'OK'),
        created(201, 'Created'),
        badRequest(400, 'Bad Request'),
        unauthorized(401, 'Unauthorized'),
        notFound(404, 'Not Found'),
        serverError(500, 'Internal Server Error');

        final int code;
        final String message;
        const HttpStatus(this.code, this.message);

        bool get isSuccess => code >= 200 && code < 300;
        bool get isClientError => code >= 400 && code < 500;
        bool get isServerError => code >= 500;

        String toHeader() => '$code $message';
      }

      void main() {
        var status = HttpStatus.notFound;
        print(status.isClientError);  // true
        print(status.toHeader());     // '404 Not Found'
      }</code></pre>
      The methods make the enum self-documenting. We don't need to remember which ranges mean what — the enum tells us.

  - id: enums-implementing-interfaces
    title: "Enums implementing interfaces"
    summary: |
      Enums can implement interfaces. This is how we integrate them into larger type hierarchies. The enum values must satisfy the interface contract.
      <br><br>
      <pre><code>abstract class Describable {
        String describe();
      }

      enum Colour implements Describable {
        red(0xFF0000),
        green(0x00FF00),
        blue(0x0000FF),
        white(0xFFFFFF),
        black(0x000000);

        final int hex;
        const Colour(this.hex);

        @override
        String describe() => '$name (#${hex.toRadixString(16).padLeft(6, '0')})';

        String get hexString => '#${hex.toRadixString(16).padLeft(6, '0').toUpperCase()}';
      }

      void printDescription(Describable item) {
        print(item.describe());
      }

      void main() {
        printDescription(Colour.red);    // 'red (#ff0000)'
        printDescription(Colour.blue);   // 'blue (#0000ff)'
      }</code></pre>
      The <code>implements</code> keyword works the same as with classes. The enum must provide implementations for all abstract members.
      <br><br>
      <strong>Important limitation:</strong> Enums cannot extend other classes. The enum type already extends <code>Enum</code>, and Dart doesn't allow multiple inheritance. But they can implement any number of interfaces.
      <br><br>
      <pre><code>// Multiple interfaces work fine
      abstract class Serialisable {
        Map&lt;String, dynamic&gt; toJson();
      }

      abstract class Comparable&lt;T&gt; {
        int compareTo(T other);
      }

      enum Priority implements Serialisable, Comparable&lt;Priority&gt; {
        low(1),
        medium(5),
        high(10);

        final int level;
        const Priority(this.level);

        @override
        Map&lt;String, dynamic&gt; toJson() => {'name': name, 'level': level};

        @override
        int compareTo(Priority other) => level.compareTo(other.level);
      }

      void main() {
        var sorted = [Priority.high, Priority.low, Priority.medium]..sort();
        print(sorted.map((p) => p.name));  // (low, medium, high)
      }</code></pre>
      By implementing <code>Comparable</code>, our enum values can be sorted. By implementing <code>Serialisable</code>, they can be converted to JSON. The enum participates in the type system like any other class.

  - id: enums-with-generics
    title: "Enums with generics"
    summary: |
      Enhanced enums can be generic. This is useful when each enum value handles a specific type of data.
      <br><br>
      <pre><code>enum JsonType&lt;T&gt; {
        string&lt;String&gt;(),
        number&lt;num&gt;(),
        boolean&lt;bool&gt;(),
        array&lt;List&gt;(),
        object&lt;Map&gt;(),
        nil&lt;Null&gt;();

        Type get dartType => T;

        bool matches(dynamic value) {
          if (this == nil) return value == null;
          return value is T;
        }
      }

      void main() {
        print(JsonType.string.dartType);           // String
        print(JsonType.string.matches('hello'));   // true
        print(JsonType.string.matches(42));        // false
        print(JsonType.nil.matches(null));         // true
      }</code></pre>
      The generic parameter <code>T</code> is specified for each value. This creates a type-safe association between the enum value and a Dart type.
      <br><br>
      <pre><code>// A more practical example: form field types
      enum FieldType&lt;T&gt; {
        text&lt;String&gt;(''),
        number&lt;int&gt;(0),
        decimal&lt;double&gt;(0.0),
        toggle&lt;bool&gt;(false),
        date&lt;DateTime?&gt;(null);

        final T defaultValue;
        const FieldType(this.defaultValue);

        bool isValid(dynamic value) => value is T;
      }

      class FormField&lt;T&gt; {
        final String label;
        final FieldType&lt;T&gt; type;
        T value;

        FormField(this.label, this.type) : value = type.defaultValue;
      }

      void main() {
        var nameField = FormField('Name', FieldType.text);
        nameField.value = 'Alice';  // type-safe: must be String

        var ageField = FormField('Age', FieldType.number);
        ageField.value = 25;        // type-safe: must be int
      }</code></pre>
      Generic enums shine when building type-safe APIs. The enum value carries type information that flows through the rest of the code.

  - id: exhaustive-switching
    title: "Exhaustive switching with enums"
    summary: |
      Enums have always enabled exhaustive switching in Dart. The compiler knows every possible value, so it can verify that a switch covers all cases.
      <br><br>
      <pre><code>enum Direction { north, south, east, west }

      String getArrow(Direction d) {
        return switch (d) {
          Direction.north => '↑',
          Direction.south => '↓',
          Direction.east => '→',
          Direction.west => '←',
        };
        // No default needed — compiler knows this is complete
      }</code></pre>

      <br><br>
      The real power shows when we modify the enum. Add a new value, and the compiler flags every incomplete switch.
      <br><br>
      <pre><code>// Later, we add a new direction
      enum Direction { north, south, east, west, up, down }

      // Now getArrow() fails to compile:
      // Error: The type 'Direction' is not exhaustively matched
      // by the switch cases since it doesn't match 'Direction.up'
      // or 'Direction.down'.</code></pre>
      Compare this to a default case:
      <br><br>
      <pre><code>// With a default, new values are silently ignored
      String getArrow(Direction d) {
        return switch (d) {
          Direction.north => '↑',
          Direction.south => '↓',
          Direction.east => '→',
          Direction.west => '←',
          _ => '?',  // catches up, down — no compiler warning
        };
      }</code></pre>
      The default case hides the problem. We won't know we forgot to handle <code>up</code> and <code>down</code> until runtime. Avoid default cases with enums unless you genuinely want a fallback.
      <br><br>
      <strong>Pattern matching with enhanced enums:</strong>
      <br><br>
      <pre><code>enum HttpStatus {
        ok(200), notFound(404), serverError(500);
        final int code;
        const HttpStatus(this.code);
      }

      String categorise(HttpStatus status) {
        return switch (status) {
          HttpStatus(:var code) when code < 400 => 'Success',
          HttpStatus(:var code) when code < 500 => 'Client Error',
          HttpStatus() => 'Server Error',
        };
      }</code></pre>
      The <code>:var code</code> syntax destructures the enum's field. The <code>when</code> guard adds conditions. This is Dart 3 pattern matching at work.

      <br><br>
      <strong>Use an enum when:</strong>
      <br><br>
      <strong>1. All values have the same fields.</strong> Every enum value has the same structure — just different data.
      <br><br>
      <pre><code>// Good enum: all values have code + message
      enum HttpStatus {
        ok(200, 'OK'),
        notFound(404, 'Not Found');
        final int code;
        final String message;
        const HttpStatus(this.code, this.message);
      }</code></pre>
      <br>
      <strong>2. The set is truly fixed.</strong> Days of the week. Cardinal directions. HTTP methods. These won't change.
      <br><br>
      <strong>3. You need .index or .values.</strong> Enums provide these for free. Useful for serialisation, iteration, dropdowns.
      <br><br>
      <strong>Use a sealed class when:</strong>
      <br><br>
      <strong>1. Variants have different fields.</strong> Success has data; Error has a message and stack trace; Loading has nothing.
      <br><br>
      <pre><code>// Sealed is better: variants have different shapes
      sealed class ApiResult {}
      final class Loading extends ApiResult {}
      final class Success extends ApiResult {
        final Data data;
        Success(this.data);
      }
      final class Error extends ApiResult {
        final String message;
        final StackTrace? trace;
        Error(this.message, [this.trace]);
      }</code></pre>
      <br>
      <strong>2. You need runtime instance creation.</strong> Sealed classes can be instantiated at runtime with different data. Enum values are fixed at compile time.
      <br><br>
      <strong>3. You need inheritance.</strong> Sealed subtypes can extend other classes; enums cannot.
      <br><br>
      <strong>The heuristic:</strong> If every variant looks the same (same fields, just different values), use an enum. If variants have different shapes, use a sealed class.
