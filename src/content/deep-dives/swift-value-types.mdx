  - id: value-vs-reference-semantics
    title: "Value vs reference semantics"
    summary: |
      Here is the single most important question in all of Swift. When we assign one variable to another, or pass a value into a function, does the second name get its own copy, or do both names point at the same thing?
      <br><br>
      The answer depends entirely on whether we are dealing with a <strong>value type</strong> or a <strong>reference type</strong>. A struct is a value type: it is copied on assignment and copied again when passed into a function. A class instance is a reference type: it is shared, so two names can point at one underlying object. Get this distinction wrong and Swift will surprise you in the worst way — by quietly changing data you thought was untouched.
      <br><br>
      Let's run the classic experiment. Change one thing, then ask: did the other change too?
      <br><br>
      <pre><code>struct Point { var x: Int }&#10;&#10;var a = Point(x: 1)&#10;var b = a        // b gets a COPY&#10;b.x = 99&#10;print(a.x)       // 1  — a is untouched&#10;print(b.x)       // 99&#10;&#10;class Box { var x: Int; init(x: Int) { self.x = x } }&#10;&#10;let p = Box(x: 1)&#10;let q = p        // q points at the SAME object&#10;q.x = 99&#10;print(p.x)       // 99 — p sees the change too!</code></pre>
      Same code shape, opposite outcome. The struct copy gave <code>b</code> its own independent <code>x</code>. The class assignment handed <code>q</code> a second reference to the one and only <code>Box</code>, so mutating through <code>q</code> is visible through <code>p</code>.

      So, hold this picture in mind for the rest of the episode. Value types are like cash — hand someone a copy and your own wallet is untouched. Reference types are like a shared document link — anyone with the link edits the same page. Almost everything else about Swift's memory model follows from this one idea.

  - id: structs-and-classes
    title: "Structs and classes — what each gives you"
    summary: |
      Both structs and classes let us bundle data and behaviour. They share most of the same syntax: stored properties, computed properties, methods, initialisers, subscripts, protocol conformance. So why two of them? Because they make a deliberate trade between two different sets of powers.
      <br><br>
      A struct is a value type. It gets a <strong>synthesised memberwise initialiser</strong> for free, it has no shared identity, and copying it is cheap and predictable. A class is a reference type. It supports <strong>inheritance</strong>, it is managed by ARC, it can have a <code>deinit</code>, and many references can share one instance.
      <br><br>
      <pre><code>struct Coordinate {&#10;  var latitude: Double&#10;  var longitude: Double&#10;  // No init written — Swift synthesises one:&#10;}&#10;&#10;let here = Coordinate(latitude: 28.6, longitude: 77.2)&#10;&#10;class Vehicle {&#10;  var speed = 0.0&#10;}&#10;&#10;class Bicycle: Vehicle {   // inheritance — classes only&#10;  var hasBasket = false&#10;}</code></pre>
      Notice the struct never declared an initialiser, yet <code>Coordinate(latitude:longitude:)</code> exists. That memberwise initialiser is generated automatically, with one parameter per stored property in declaration order. Classes do not get this — you must write your own.
      <br><br>
      Apple's own guidance is blunt: <strong>prefer structs by default</strong>. Reach for a class only when you genuinely need one of its specific powers — reference identity, inheritance, deinitialisers, or interoperation with Objective-C frameworks that expect reference types. This is a real shift from older object-oriented traditions where the class was the default tool. In SwiftUI this guidance is visible everywhere: your views are structs, and the framework recreates them constantly because copying a value is cheap.
      <br><br>
      A nice piece of history: Swift was announced at Apple's WWDC in June 2014 and made open source in December 2015. Its standard library was designed from day one around value types, partly to escape a whole category of shared-mutable-state bugs that had haunted Objective-C. When you write idiomatic Swift, you are mostly writing values, not objects.

  - id: identity-vs-equality
    title: "Identity vs equality"
    summary: |
      We have two different questions we might ask about two things. "Are these the same object?" and "Do these hold the same value?" Swift keeps them strictly separate, and the second question is the only one that makes sense for structs.
      <br><br>
      <strong>Identity</strong> is tested with <code>===</code> and <code>!==</code>. These ask whether two references point at the exact same instance in memory. Because identity is about a shared instance, <code>===</code> applies <strong>to classes only</strong>. Under the hood it compares the object's <code>ObjectIdentifier</code>, which is essentially its address.
      <br><br>
      <strong>Equality</strong> is tested with <code>==</code> and comes from the <code>Equatable</code> protocol. It asks whether two values are meaningfully equal, regardless of where they live.
      <br><br>
      <pre><code>class Node { var id: Int; init(id: Int) { self.id = id } }&#10;&#10;let n1 = Node(id: 1)&#10;let n2 = Node(id: 1)&#10;let n3 = n1&#10;&#10;print(n1 === n2)   // false — two distinct objects&#10;print(n1 === n3)   // true  — same object&#10;&#10;struct Tag: Equatable { var name: String }&#10;let t1 = Tag(name: "swift")&#10;let t2 = Tag(name: "swift")&#10;print(t1 == t2)    // true  — same value&#10;// print(t1 === t2) // compile error: === needs class types</code></pre>
      Why do structs have no identity? Because there is nothing stable to point at. Every copy of a struct is just as much "the" value as any other — there is no single shared instance whose address would mean anything. Asking whether two <code>Int</code>s are "the same object" is a category error; we can only ask whether they are equal.
      <br><br>
      For your own structs, conforming to <code>Equatable</code> is usually a one-liner. If every stored property is already <code>Equatable</code>, Swift will <strong>synthesise</strong> <code>==</code> for you the moment you declare conformance. No boilerplate, no forgetting a field.

  - id: arc-automatic-reference-counting
    title: "ARC — Automatic Reference Counting"
    summary: |
      Class instances live on the heap, so something has to decide when to free them. That something is <strong>ARC</strong>, Automatic Reference Counting. Every class instance carries a count of how many strong references currently point at it. When that count drops to zero, the instance is deallocated immediately and its <code>deinit</code> runs.
      <br><br>
      Here is the part people most often get wrong: ARC is <strong>not a tracing garbage collector</strong>. There is no background thread periodically scanning the heap looking for unreachable objects, and there are no unpredictable pause times. Instead, the Swift compiler inserts the <code>retain</code> and <code>release</code> calls <strong>at compile time</strong>, right where references are created and destroyed. The bookkeeping is deterministic: an object dies the instant its last reference goes away.

      A little history. This model did not start with Swift. Apple shipped <strong>Objective-C ARC in 2011</strong>, replacing years of error-prone manual <code>retain</code> and <code>release</code> calls that developers wrote by hand. The compiler took over that chore. When Swift arrived in 2014, it simply inherited the same proven reference-counting model, which is why Swift and Objective-C interoperate so smoothly around object lifetimes.
      <br><br>
      And remember: ARC only governs reference types. Structs and enums are not reference counted at all — they are copied, and a copy simply ceases to exist when its scope ends. No retain, no release, no deinit.

  - id: reference-cycles
    title: "Reference cycles — and how to break them"
    summary: |
      ARC is reliable, but it has one blind spot. What if two objects each hold a strong reference to the other? Object A keeps B alive, and B keeps A alive. Neither count can ever reach zero, so neither is freed — even after every outside reference is gone. This is a <strong>strong reference cycle</strong>, and it is the main way you can still leak memory in Swift.
      <br><br>
      <pre><code>class Person {&#10;  let name: String&#10;  var card: Card?&#10;  init(name: String) { self.name = name }&#10;  deinit { print("\(name) freed") }&#10;}&#10;&#10;class Card {&#10;  let number: Int&#10;  var owner: Person?      // strong → cycle!&#10;  init(number: Int) { self.number = number }&#10;  deinit { print("card freed") }&#10;}&#10;&#10;var p: Person? = Person(name: "Ada")&#10;var c: Card? = Card(number: 1)&#10;p?.card = c&#10;c?.owner = p&#10;p = nil; c = nil   // nothing prints — both leaked</code></pre>
      We break the cycle by making one of the two references non-owning. Swift gives us two tools for this.
      <br><br>
      A <strong>weak</strong> reference does not keep its target alive. Because the target can vanish at any moment, a weak reference must be an <code>Optional</code>, and ARC <strong>automatically sets it to nil</strong> when the target is deallocated. An <strong>unowned</strong> reference also does not keep its target alive, but it is non-optional and assumes the target will outlive the reference. Use <code>unowned</code> only when you are certain the relationship can never become dangling; otherwise prefer <code>weak</code>.

  - id: copy-on-write
    title: "Copy-on-write — cheap value semantics"
    summary: |
      Here is a puzzle. <code>Array</code>, <code>String</code>, <code>Dictionary</code> and <code>Set</code> are all value-type structs. By the rules of topic 1, copying one should copy all of its contents. So if you assign a million-element array to a new variable, does Swift really duplicate a million elements every time? That would be ruinously slow.
      <br><br>
      It does not. Swift uses <strong>copy-on-write</strong>, often shortened to COW. When you assign one of these collections to another variable, the two share a single underlying storage buffer. No elements are copied. The copy is deferred — and only happens at the moment a mutation needs to occur on a buffer that more than one variable is using.

      <br><br>
      <pre><code>var a = [1, 2, 3]&#10;var b = a          // shares a's buffer — O(1), no copy&#10;b.append(4)        // b is not unique → fork a private copy now&#10;print(a)           // [1, 2, 3]    — untouched&#10;print(b)           // [1, 2, 3, 4]</code></pre>
      How does the collection know whether it must fork? Before mutating, it checks whether its buffer is uniquely referenced, using the standard-library function <code>isKnownUniquelyReferenced(&amp;_)</code>. If the buffer is shared, it copies first, then mutates the private copy. If the buffer is already unique, it mutates in place with no copy at all.
      <br><br>
      This is the beautiful resolution of the puzzle. You get clean value semantics — assigning <code>a</code> to <code>b</code> never lets one accidentally mutate the other — at the cost of a reference type internally. You pay for the copy only when you actually diverge the data, and never a moment sooner. If you build your own value type wrapping a class, you can use <code>isKnownUniquelyReferenced</code> to give it exactly this behaviour.

  - id: putting-it-together
    title: "Putting it together"
    summary: |
      Let's tie the threads together with the rules you will use every day.
      <br><br>
      To change a stored property from inside a method on a struct, that method must be marked <code>mutating</code>. A value type cannot quietly rewrite itself; the keyword makes the intent explicit and tells the compiler the method needs a writable copy of <code>self</code>. Classes never need this, because mutating a property does not change the reference.
      <br><br>
      <pre><code>struct Counter {&#10;  var count = 0&#10;  mutating func increment() { count += 1 }&#10;}&#10;&#10;var c = Counter()&#10;c.increment()   // fine — c is var&#10;&#10;let d = Counter()&#10;// d.increment() // error: cannot mutate a let value type</code></pre>
      And there is the second rule, hiding in that example. For a value type, <code>let</code> means the entire value is immutable — not just the binding, but every property inside it. A <code>let</code> struct is frozen solid; you cannot call a <code>mutating</code> method on it or reassign any of its fields. This is different from a class, where <code>let</code> only freezes the reference: you may still mutate the object's <code>var</code> properties through that constant reference, because the reference itself never changes.
      <br><br>
      <pre><code>let pt = Point(x: 1)   // struct&#10;// pt.x = 2            // error — whole value is constant&#10;&#10;let box = Box(x: 1)    // class&#10;box.x = 2              // fine — reference is constant, object is not</code></pre>
      Finally, the trivia that ties the whole episode to the language you use. Nearly every type you reach for first in Swift is a value type. <code>Int</code>, <code>Double</code>, <code>Bool</code>, <code>String</code>, <code>Array</code>, <code>Dictionary</code>, <code>Set</code> and even <code>Optional</code> are all defined as structs or enums in the standard library — not as the built-in primitives or reference types you might expect from other languages. An <code>Int</code> is a struct. <code>Optional</code> is an enum with <code>.none</code> and <code>.some</code> cases.
      <br><br>
      That single design decision is why Swift feels so predictable. When you pass a number, a string or an array around, you are passing a copy, and nobody can reach back and mutate your data behind your back. Reference types and ARC are still there for when you need shared identity and lifetimes — but value semantics are the default, and now you know exactly what that means.
