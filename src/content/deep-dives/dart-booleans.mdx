  - id: foundation
    title: "From bit to boolean"
    summary: |
      A transistor is essentially a switch. It can be on or off. We measure that on/off state as a <strong>bit</strong> — the smallest unit of data a computer can store.
      <br><br>
      The simplest data type built directly on top of a bit is the <strong>boolean</strong>. It can take exactly one of two values: <code>true</code> or <code>false</code>.
      <br><br>
      The name comes from <strong>George Boole</strong>, the 19th-century mathematician whose algebra of logic laid the foundation for modern computing. Every <code>if</code> statement, every loop, every yes/no decision our program ever makes ultimately comes down to a boolean.
      <br><br>
      In Dart, we write the type as <code>bool</code> — shortened from <em>boolean</em>.
      <br><br>
      It might seem like the simplest possible type. But like everything in Dart, there is more going on under the hood than meets the eye. Let's take a deep dive.

  - id: declaring
    title: "Declaring booleans — var, final, const"
    summary: "The basic form:<br><br><pre><code>bool tom = false;\nbool jerry = true;</code></pre>Three parts: the type (<code>bool</code>), the name (<code>tom</code>), and the value (<code>false</code>). Pretty standard.<br><br>Writing <code>bool</code> every time is repetitive. Dart can figure out the type from the value on the right, so we can use <code>var</code> instead:<br><br><pre><code>var tom = false;   // Dart infers this is a bool\nvar jerry = true;  // also a bool</code></pre>Both forms produce identical bytecode. Most idiomatic Dart leans on <code>var</code> for local variables and spells out the type only when it adds clarity.<br><br>When we want to lock down a value so it cannot be reassigned, Dart gives us two keywords:<br><br>• <code>final</code> — set once, at runtime. The value can be computed when the program runs.<br>• <code>const</code> — set once, at compile time. The value must be known before the program even starts.<br><br><pre><code>final bool isHuman = true;\nconst bool piIsThree = false;</code></pre>For booleans, <code>const</code> fits naturally because <code>true</code> and <code>false</code> are themselves compile-time constants. A rule of thumb: reach for <code>const</code> if we can, <code>final</code> if we can't, and plain <code>var</code> only when the value genuinely needs to change."

  - id: canonical-instances
    title: "Canonical instances — true and false are singletons"
    summary: "Here is a neat optimization. We might assume that every time we write <code>false</code> in our code, Dart creates a fresh boolean object in memory. But that would be wasteful. These are called <strong>canonical instances</strong>. Some people just call them singletons.<br><br>So when we write <code>bool tom = false;</code>, Dart does <strong>not</strong> build a new boolean object.   - id: comparisons
    title: "Where booleans come from"
    summary: "So far we have been writing <code>true</code> and <code>false</code> by hand. In real code, we rarely do that. Booleans almost always come from <strong>comparison operators</strong>.<br><br><pre><code>int age = 18;\nbool canVote = age &gt;= 18;            // true\nbool isMinor = age &lt; 18;             // false\nbool isExactlyEighteen = age == 18;  // true</code></pre>The comparison operators in Dart:<br><br>• <code>==</code> — equal to<br>• <code>!=</code> — not equal to<br>• <code>&lt;</code> — less than<br>• <code>&gt;</code> — greater than<br>• <code>&lt;=</code> — less than or equal to<br>• <code>&gt;=</code> — greater than or equal to<br><br>Every one of these expressions evaluates to a <code>bool</code>. That is the bridge between the rest of our program — numbers, strings, objects — and the boolean world.<br><br>When we test if a list is empty, when we check if two strings match, when we ask if a value is above some threshold — all of those produce booleans, which then feed into <code>if</code> statements, <code>while</code> loops, and ternary expressions."

  - id: other-operators
    title: "Bitwise gotcha, XOR, ternary, and precedence"
    summary: "Three more operators to know, and the rules about which goes first.<br><br><strong>The bitwise gotcha: &amp; and |</strong><br><br>Dart also has <code>&amp;</code> and <code>|</code> (single-character versions). They work on booleans, but with a critical difference: they do <strong>not</strong> short-circuit. Both sides are always evaluated.<br><br><pre><code>bool a = false &amp; expensiveCheck();   // expensiveCheck() IS called\nbool b = false &amp;&amp; expensiveCheck();  // expensiveCheck() is NOT called</code></pre>In almost every case, we want <code>&amp;&amp;</code> and <code>||</code>, not <code>&amp;</code> and <code>|</code>. Treat the single-character versions as a footgun unless we have a specific reason.<br><br><strong>XOR — exclusive OR</strong><br><br>There is also <code>^</code>, which returns <code>true</code> if <strong>exactly one</strong> of its operands is true:<br><br><pre><code>print(true ^ false);  // true\nprint(true ^ true);   // false\nprint(false ^ false); // false</code></pre>Rare in everyday code, but handy for toggles and parity checks.<br><br><strong>The ternary operator</strong><br><br>Sometimes we want to choose between two values based on a condition. The ternary gives us a one-liner:<br><br><pre><code>// condition ? value if true : value if false\nString status = jerry ? \"Jerry is awake!\" : \"Jerry is sleeping...\";\nprint(status);  // Jerry is awake!</code></pre>We read it as \"if jerry is true, use the first value; otherwise, use the second.\"<br><br><strong>Who goes first?</strong><br><br>When we combine operators without parentheses, Dart evaluates them in a strict order. From highest to lowest, the ones relevant to booleans:<br><br>1. <code>!</code> (NOT)<br>2. <code>&lt;</code>, <code>&gt;</code>, <code>&lt;=</code>, <code>&gt;=</code> (relational)<br>3. <code>==</code>, <code>!=</code> (equality)<br>4. <code>&amp;</code> (bitwise AND)<br>5. <code>^</code> (XOR)<br>6. <code>|</code> (bitwise OR)<br>7. <code>&amp;&amp;</code> (logical AND)<br>8. <code>||</code> (logical OR)<br>9. <code>? :</code> (ternary)<br><br>When in doubt, use parentheses. They override everything and make our code much easier to read."

  - id: truthiness
    title: "Strict truthiness and utility methods"
    summary: "One last thing, and this trips up newcomers from JavaScript, Python, or C constantly.<br><br><strong>Dart requires real booleans.</strong> In Dart, the condition of an <code>if</code> (or a <code>while</code>, or any boolean context) <strong>must</strong> be a real <code>bool</code>. Dart will not auto-convert numbers, strings, or other objects.<br><br><pre><code>if (1) { ... }            // compile error\nif (\"hello\") { ... }      // compile error\nif (someList) { ... }     // compile error\n\nif (someList.isNotEmpty) { ... }  // ok — that's a real bool\nif (count &gt; 0) { ... }            // ok</code></pre>This strictness eliminates a whole class of subtle bugs. If we are coming from somewhere else, the rule is simple. Be explicit. Compare what we mean.<br><br><strong>Two utility methods worth knowing.</strong><br><br><pre><code>// Parse a bool from a string (useful for CLI args, JSON, etc.)\nbool b = bool.parse(\"true\");        // true\nbool? b2 = bool.tryParse(\"nope\");   // null (didn't parse)\n\n// Read a compile-time flag passed via --define\nconst bool inDebugMode = bool.fromEnvironment(\"debug\", defaultValue: false);</code></pre>We won't need these on day one, but it is good to know they exist.<br><br>That's the full tour of <code>bool</code> in Dart: what it is, how Dart stores it, where it comes from, what we can do with it, and the small print that catches newcomers. Master this and we have a solid foundation for everything that follows — because every <code>if</code>, every <code>while</code>, every conditional decision in our program ultimately comes down to a single boolean."
