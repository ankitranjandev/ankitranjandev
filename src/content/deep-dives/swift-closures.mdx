  - id: functions-are-first-class
    title: "Functions and closures are first-class"
    summary: |
      In Swift, a function is not a special, untouchable thing. It is a value. We can store it in a variable, pass it as an argument, and return it from another function. Anything we can do with an <code>Int</code> or a <code>String</code>, we can do with a function.
      <br><br>
      And here is the key idea that runs through this whole episode: a function is just a named closure. A closure is a self-contained block of code that can be passed around and called later. When we write <code>func greet()</code>, we are really giving a name to a closure. The anonymous block <code>{ print("hi") }</code> is the same kind of thing, only without a name.
      <br><br>
      <pre><code>func greet() {&#10;    print("Hello!")&#10;}&#10;&#10;let f = greet      // assign the function to a variable&#10;f()                // call it — prints "Hello!"&#10;&#10;let blocks = [greet, greet, greet]&#10;for block in blocks {&#10;    block()        // prints "Hello!" three times&#10;}</code></pre>
      Because functions are values, they have <strong>types</strong>. A function type is written as the parameter types in parentheses, an arrow, then the return type. <code>(Int) -> Bool</code> is the type of any function that takes an <code>Int</code> and returns a <code>Bool</code>. <code>() -> Void</code> takes nothing and returns nothing. <code>(String, String) -> Bool</code> takes two strings and returns a bool.
      <br><br>
      <pre><code>let isEven: (Int) -> Bool = { n in n % 2 == 0 }&#10;let printer: (String) -> Void = { s in print(s) }&#10;&#10;func apply(_ value: Int, _ test: (Int) -> Bool) -> Bool {&#10;    return test(value)&#10;}&#10;&#10;apply(10, isEven)   // true</code></pre>
      A little history for context. When Swift was unveiled at WWDC in 2014, Chris Lattner and the team made closures central to the language from day one. The <code>{ }</code> syntax with the <code>in</code> keyword was deliberately designed to read well at the call site, because Cocoa was already drowning in Objective-C blocks (which used the famously hard-to-remember <code>^</code> caret syntax). Swift's closures are the spiritual successor to those blocks, but with cleaner syntax and full type inference.

  - id: closure-expression-syntax
    title: "Closure expression syntax — from full to shorthand"
    summary: |
      Swift lets us write the same closure at several levels of brevity. The compiler fills in everything it can infer, so we are free to omit the obvious. Let's take sorting an array of names and peel the ceremony away one layer at a time.
      <br><br>
      We start with the full form: explicit parameter types, an explicit return type, the <code>in</code> keyword separating the signature from the body, and an explicit <code>return</code>.
      <br><br>
      <pre><code>let names = ["Chris", "Ada", "Brian"]&#10;&#10;// 1. Full form&#10;names.sorted(by: { (a: String, b: String) -> Bool in&#10;    return a &lt; b&#10;})</code></pre>
      Because <code>sorted(by:)</code> already knows it wants a <code>(String, String) -> Bool</code>, the types are redundant. And a single-expression closure returns its value implicitly, so <code>return</code> goes too.
      <br><br>
      <pre><code>// 2. Inferred types, implicit return&#10;names.sorted(by: { a, b in a &lt; b })</code></pre>
      Swift gives every closure automatic <strong>shorthand argument names</strong>: <code>$0</code>, <code>$1</code>, <code>$2</code>, and so on. Once we use them, we can drop the parameter list and the <code>in</code> entirely.
      <br><br>
      <pre><code>// 3. Shorthand argument names&#10;names.sorted(by: { $0 &lt; $1 })</code></pre>
      Finally, when a closure is the <em>last</em> argument to a function, we can lift it outside the parentheses as a <strong>trailing closure</strong>. If it is the only argument, the parentheses vanish completely.
      <br><br>
      <pre><code>// 4. Trailing closure&#10;names.sorted { $0 &lt; $1 }</code></pre>

  - id: capturing-the-environment
    title: "Capturing the environment"
    summary: |
      Here is what makes a closure more than a bare block of code. A closure <strong>captures</strong> the variables it references from the surrounding scope. By default it captures them by reference, which means it can both read <em>and</em> write them — and it keeps doing so even after the enclosing function has returned.
      <br><br>
      The classic demonstration is a counter maker. We write a function that returns a closure, and that closure keeps mutating a variable that, on paper, should have stopped existing the moment the outer function returned.
      <br><br>
      <pre><code>func makeCounter() -> () -> Int {&#10;    var count = 0&#10;    return {&#10;        count += 1&#10;        return count&#10;    }&#10;}&#10;&#10;let counter = makeCounter()&#10;print(counter())   // 1&#10;print(counter())   // 2&#10;print(counter())   // 3</code></pre>
      The inner closure captured <code>count</code>. Even though <code>makeCounter()</code> has returned, the closure still holds a live reference to that variable and goes on incrementing it. Each fresh call to <code>makeCounter()</code> creates a new, independent <code>count</code>, so two counters never tread on each other.

      <br><br>
      Capturing by reference is what makes this work, but it is also where surprises hide. Because the closure shares the <em>same</em> variable rather than a snapshot, anything that mutates that variable later is visible inside the closure. That is the difference between capturing a reference and capturing a value, and it is exactly what capture lists let us control.

  - id: capture-lists
    title: "Capture lists — snapshotting and controlling references"
    summary: |
      What if we do not want the live, shared reference? What if we want the closure to remember a value as it was at the moment of creation? That is what a <strong>capture list</strong> is for. We write the names we care about in square brackets right at the start of the closure.
      <br><br>
      <pre><code>var x = 10&#10;&#10;let snapshot = { [x] in print(x) }   // captures a copy of x: 10&#10;let live     = { print(x) }          // captures x by reference&#10;&#10;x = 99&#10;snapshot()   // 10 — frozen at creation time&#10;live()       // 99 — sees the current value</code></pre>
      The <code>[x]</code> capture list says "take the current value of <code>x</code> and stash a copy inside the closure." The closure that captures <code>x</code> implicitly, with no capture list, shares the real variable and so reports its later value.
      <br><br>
      Capture lists matter most for <strong>reference types</strong> — classes. By default a closure captures a class instance strongly, which keeps it alive. We can change that with two keywords inside the capture list:
      <br><br>
      • <code>[weak self]</code> captures <code>self</code> as an <em>optional</em> weak reference. It does not keep <code>self</code> alive, and becomes <code>nil</code> if <code>self</code> has been deallocated.<br>
      • <code>[unowned self]</code> captures <code>self</code> without keeping it alive either, but assumes <code>self</code> will still exist when the closure runs. If it does not, the program crashes — so use it only when you are certain of the lifetime.<br><br>
      <pre><code>final class Downloader {&#10;    var onFinish: (() -> Void)?&#10;&#10;    func start() {&#10;        // weak: self may be gone by the time this runs&#10;        onFinish = { [weak self] in&#10;            self?.cleanUp()&#10;        }&#10;    }&#10;&#10;    func cleanUp() { print("done") }&#10;}</code></pre>
      So we have two distinct tools wearing the same bracket syntax. For value types, a capture list <em>snapshots</em> the value. For reference types, it lets us choose <em>how</em> the reference is held — strongly, weakly, or unowned. Getting this choice right is the whole game when it comes to memory, which is exactly where we are heading.

  - id: escaping-vs-nonescaping
    title: "Escaping vs non-escaping closures"
    summary: |
      Swift draws a sharp line between two kinds of closure parameters, and the distinction is about <em>lifetime</em>: does the closure outlive the function call it was passed into, or not?
      <br><br>
      A <strong>non-escaping</strong> closure is called before the function returns and is never stored anywhere. Think of <code>map</code> or <code>sorted(by:)</code> — they run the closure right there and then throw it away. Since Swift 3 (proposal SE-0103), closures are non-escaping by <em>default</em>. We write nothing special.
      <br><br>
      An <strong>escaping</strong> closure may be stored for later or run after the function has already returned. Completion handlers, stored callbacks, anything pushed onto a queue — these all escape. We mark such a parameter with <code>@escaping</code>, and the compiler insists on it.
      <br><br>
      <pre><code>// Non-escaping: runs now, then gone&#10;func transform(_ x: Int, with f: (Int) -> Int) -> Int {&#10;    return f(x)&#10;}&#10;&#10;// Escaping: stored to run later&#10;var handlers: [() -> Void] = []&#10;func defer(_ work: @escaping () -> Void) {&#10;    handlers.append(work)   // escapes — outlives this call&#10;}</code></pre>

      <br><br>
      Why make non-escaping the default? Two reasons. First, the optimiser. If the compiler knows a closure cannot escape, it can keep its captures on the stack and avoid heap allocation and reference-counting overhead entirely. Second, safety. A non-escaping closure that captures <code>self</code> cannot create a long-lived reference cycle, because it simply does not live long enough — which is why we are never asked to write <code>[weak self]</code> inside a plain <code>map</code>. The danger only appears once a closure can escape, and Swift makes that danger visible by forcing the <code>@escaping</code> annotation.

  - id: retain-cycles
    title: "Retain cycles — the most common real-world Swift memory bug"
    summary: |
      Here it is, the bug that catches every Swift developer eventually. An escaping closure is stored on an object, and that closure captures <code>self</code> strongly. Now <code>self</code> holds the closure, and the closure holds <code>self</code>. Each keeps the other's reference count above zero, ARC sees neither as garbage, and the pair leaks forever.
      <br><br>
      <pre><code>final class ViewModel {&#10;    var onUpdate: (() -> Void)?&#10;&#10;    func configure() {&#10;        // self → onUpdate → self : a strong cycle&#10;        onUpdate = {&#10;            self.refresh()&#10;        }&#10;    }&#10;&#10;    func refresh() { /* ... */ }&#10;    deinit { print("freed") }   // never prints!&#10;}</code></pre>
      The <code>deinit</code> never runs. The object is unreachable from the outside, yet ARC cannot collect it because the closure is still holding a strong reference back to it. This is a textbook <strong>retain cycle</strong>.

      We break the cycle with a capture list. <code>[weak self]</code> turns <code>self</code> into a weak optional inside the closure, so the closure no longer keeps the object alive. Because it is now optional, we unwrap it — usually with the <code>guard let self else { return }</code> dance at the top of the closure, which gives us a strong local <code>self</code> for the rest of the body if the object is still around.
      <br><br>
      <pre><code>onUpdate = { [weak self] in&#10;    guard let self else { return }&#10;    self.refresh()&#10;}</code></pre>
      The alternative is <code>[unowned self]</code>, which avoids the optional and reads more cleanly — but it is a promise to the compiler that <code>self</code> will still be alive when the closure fires. Break that promise and you get a crash, not a leak. Reach for <code>weak</code> when the lifetime is uncertain (network callbacks, timers), and <code>unowned</code> only when <code>self</code> provably outlives the closure. As a rule, if you store an escaping closure on an object and reference <code>self</code> inside it, assume you have a cycle until you have proven otherwise.

  - id: functional-toolkit-and-trivia
    title: "The functional toolkit, and a little trivia"
    summary: |
      Closures are the fuel for Swift's higher-order functions — functions that take other functions as arguments. The standard library leans on them heavily, and once the shorthand syntax is second nature, these read almost like English.
      <br><br>
      <pre><code>let nums = [1, 2, 3, 4, 5]&#10;&#10;nums.map { $0 * 2 }            // [2, 4, 6, 8, 10]&#10;nums.filter { $0 % 2 == 0 }    // [2, 4]&#10;nums.reduce(0) { $0 + $1 }     // 15&#10;&#10;let raw = ["1", "x", "3"]&#10;raw.compactMap { Int($0) }     // [1, 3] — drops nils&#10;&#10;let nested = [[1, 2], [3], [4, 5]]&#10;nested.flatMap { $0 }          // [1, 2, 3, 4, 5] — one level flatter</code></pre>
      Each of these takes a closure and applies it across the collection. <code>map</code> transforms, <code>filter</code> keeps what passes a test, <code>reduce</code> folds everything into a single value, <code>compactMap</code> maps then quietly discards any <code>nil</code> results, and <code>flatMap</code> maps then concatenates one level of nesting away.
      <br><br>
      Two facts worth keeping in your pocket. First, <strong>closures are reference types</strong>. Assign a closure to two variables and they point at the same underlying object, captured state and all — which is precisely why ARC manages closures and why retain cycles are even possible. Second, <strong><code>@autoclosure</code></strong>. It wraps a plain expression in a <code>() -> T</code> closure automatically, so an argument is not evaluated until it is actually needed. This is the trick behind lazy evaluation in the standard library:
      <br><br>
      <pre><code>// The nil-coalescing operator only evaluates the default if needed&#10;func ?? &lt;T&gt;(value: T?, defaultValue: @autoclosure () -> T) -> T {&#10;    if let value { return value } else { return defaultValue() }&#10;}&#10;&#10;let name = optionalName ?? expensiveDefault()   // expensiveDefault() runs only if nil</code></pre>
      The same mechanism powers <code>assert</code>, whose condition is skipped entirely in release builds. To close, a quick contrast with Dart, the subject of our sibling series. Both languages give us first-class closures that capture by reference, and both lean on <code>map</code> and friends. But Dart relies on a tracing garbage collector, so it has no retain cycles and no need for capture lists or <code>weak</code> references at all. Swift trades that convenience for deterministic, immediate deallocation through ARC — and the price of that determinism is the very capture-list discipline this episode has been about.
