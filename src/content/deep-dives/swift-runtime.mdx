  - id: the-journey-overview
    title: "The journey from text to machine code"
    summary: |
      We've seen the headline before: Swift compiles ahead of time to native machine code. But that one sentence hides a remarkable amount of machinery. So let's slow down and trace the whole journey, naming every major stage, because the rest of this episode is just zooming into each one.
      <br><br>
      When we hit build, our <code>.swift</code> text goes through this sequence: it is <strong>lexed</strong> into tokens and <strong>parsed</strong> into an abstract syntax tree; <strong>Sema</strong> type-checks that tree; <strong>SILGen</strong> lowers it into SIL; the <strong>mandatory SIL passes</strong> run the safety diagnostics; the <strong>SIL optimiser</strong> sharpens it; <strong>IRGen</strong> lowers SIL to LLVM IR; LLVM's own optimiser and backend produce machine code; and finally the linker stitches everything into an executable.

      <br><br>
      Here's the part worth noticing. Most languages built on LLVM — think of Clang for C — type-check and then hand off almost directly to LLVM IR. Swift does something unusual: it inserts an entire intermediate representation of its own, <strong>SIL</strong>, the Swift Intermediate Language, in between. Why bother? Because LLVM IR has already forgotten that Swift exists. It knows about integers, pointers, and basic blocks, but it has never heard of a protocol, a generic, or a retain count. By the time we reach LLVM, the high-level facts that make Swift safe and fast are gone.
      <br><br>
      SIL keeps those facts. It still knows this value is a <code>struct</code>, that call goes through a protocol, this generic is parameterised over <code>T</code>, and this reference needs releasing. That knowledge is exactly what Swift needs to prove your code is safe and to perform the optimisations LLVM could never do on its own. SIL is the stage that makes Swift <em>Swift</em>, and we'll spend most of this episode there.

  - id: lexing-and-parsing
    title: "Lexing and parsing — into the tree"
    summary: |
      The first job is the least glamorous and the most universal. The <strong>lexer</strong> scans the raw bytes of our source and groups them into tokens: the keyword <code>func</code>, the identifier <code>balance</code>, the operator <code>+</code>, the literal <code>42</code>. Whitespace and comments mostly fall away. Then the <strong>parser</strong> assembles those tokens into an <strong>abstract syntax tree</strong> — a structured representation where a function declaration has a name, parameters, and a body, and an expression like <code>a + b</code> becomes a node with two children.
      <br><br>
      <pre><code>// This source...
      func double(_ x: Int) -&gt; Int { x * 2 }

      // ...becomes (roughly) this tree:
      FuncDecl "double"
        ParamDecl "x": Int
        ReturnType: Int
        Body:
          BinaryExpr "*"
            DeclRefExpr "x"
            IntegerLiteral 2</code></pre>
      So far this is true of nearly every compiler. What makes Swift's frontend special is a design goal that has nothing to do with producing a binary: it has to power the tools. Xcode's autocomplete, syntax highlighting, refactoring, and inline diagnostics all run through <strong>SourceKit</strong>, which leans on the very same parser.
      <br><br>
      That demand makes the parser <strong>error-resilient</strong>. A traditional batch compiler can give up at the first syntax error. But an editor must keep going — you are, after all, typing half-finished code most of the time. Swift's parser recovers from errors and builds a best-effort tree even when the source is incomplete, so the tools can still offer completions inside a function whose closing brace you haven't typed yet. The same property is why you get squiggly underlines as you type, not only when you build.

  - id: semantic-analysis
    title: "Sema — the constraint-based type checker"
    summary: |
      Now the interesting part. A parsed tree knows the <em>shape</em> of your code but not its <em>meaning</em>. Which <code>+</code> does <code>a + b</code> call? What type is <code>x</code> when you wrote <code>let x = something()</code>? Answering these is the job of <strong>Sema</strong>, semantic analysis, and Swift does it in an unusual way.
      <br><br>
      Most older languages type-check by walking the tree once, computing a type for each node bottom-up. Swift can't, because of type inference and overloading working together. Instead, Sema treats type-checking as a <strong>constraint-satisfaction problem</strong>. For each expression it <strong>generates a system of constraints</strong> — "the result of this call equals the parameter type", "this literal is convertible to some type conforming to <code>ExpressibleByIntegerLiteral</code>" — and then a dedicated <strong>constraint solver</strong> searches for an assignment of types that satisfies all of them at once.

      <br><br>
      The inference is <strong>bidirectional</strong>: information flows both up from sub-expressions and down from the surrounding context. Write <code>let x: Float = 3</code> and the expected type <code>Float</code> flows downward to tell the literal <code>3</code> which type to be. Write <code>let y = [1, 2, 3]</code> and the element type flows upward to make <code>[Int]</code>. The solver reconciles both directions.
      <br><br>
      And here is where the famous error comes from. When an expression mixes many overloaded operators, literals, and possible implicit conversions, the number of candidate solutions can grow <strong>exponentially</strong>. The solver explores this space and, to avoid hanging forever, gives up past a threshold — emitting "the compiler is unable to type-check this expression in reasonable time." The fix is almost always to break the expression up or add an explicit type annotation, which collapses the search space by pinning down part of the solution for the solver.

  - id: silgen-and-raw-sil
    title: "SILGen and raw SIL"
    summary: |
      Once Sema has resolved every type, every overload, and every generic, <strong>SILGen</strong> walks the type-checked tree and lowers it into SIL. This is the moment Swift's high-level syntax becomes an explicit sequence of operations, but crucially, <strong>without losing the Swift-level facts</strong>.
      <br><br>
      SIL is in <strong>SSA form</strong> — static single assignment — meaning every value is assigned exactly once. That property makes data flow trivial to follow, which is what optimisers and verifiers want. But unlike LLVM IR, SIL is <strong>Swift-aware</strong>: it still represents generics with type parameters, protocol calls through witnesses, value-type copies, and ownership of references. You can see it yourself:
      <br><br>
      <pre><code>$ swiftc -emit-silgen -O example.swift

      sil hidden @$s7example6doubleyS2iF : $@convention(thin) (Int) -&gt; Int {
      bb0(%0 : $Int):
        %2 = integer_literal $Builtin.Int64, 2
        %4 = struct_extract %0 : $Int, #Int._value
        %5 = builtin "smul_with_overflow_Int64"(...)
        // ... arithmetic, then:
        return %9 : $Int
      }</code></pre>
      Notice the names. <code>$s7example6double...</code> is the mangled symbol we'll dissect later; <code>$Int</code> and <code>#Int._value</code> are real Swift types and fields, not anonymous LLVM structs. The control flow is split into <strong>basic blocks</strong> (<code>bb0</code>) joined by branches.
      <br><br>
      The output of SILGen is called <strong>raw SIL</strong>. It is a faithful, complete translation of your program — but it is not yet trustworthy. Variables might be used before they're initialised. References might be handled in ways that violate ownership rules. Raw SIL is the unverified first draft. Turning it into <strong>canonical SIL</strong> — the verified, well-formed version — is the job of the next stage, and that stage is mandatory.

  - id: mandatory-sil-passes
    title: "The mandatory passes — how Swift stays safe"
    summary: |
      Here is something many Swift programmers never realise: the checks that make Swift safe do not happen during type-checking. They happen on raw SIL, as <strong>mandatory passes</strong> that run on every build, even with optimisation switched off (<code>-Onone</code>). They transform raw SIL into canonical SIL, and if your program is unsafe, this is where it gets rejected.

      Three of these passes are worth knowing by name. <strong>Definite initialisation</strong> proves that every variable is assigned before it is read. This is why <code>let x: Int; print(x)</code> is a compile error and not a runtime surprise — SIL's SSA form makes it tractable to trace whether a value reached every use. The <strong>exclusivity checker</strong> enforces Swift's memory-access rule: you cannot have two conflicting accesses to the same storage at the same time, which is what makes passing the same variable as two <code>inout</code> arguments illegal. Some of this is caught at compile time; some inserts cheap runtime checks.
      <br><br>
      The third is <strong>ownership verification</strong>. Modern SIL is in <strong>OSSA</strong> form — Ownership SSA — where every value carries an ownership kind (owned, guaranteed, and so on), and the verifier proves that ownership is respected: every owned value is consumed exactly once, nothing is used after it's destroyed. This is the foundation that lets ARC be both correct and, later, heavily optimised.
      <br><br>
      The lesson is that Swift's safety is not a runtime feature bolted on top. It is a property the compiler <em>proves</em> on SIL before your code ever runs. When you see a borrow-checker-style diagnostic, you are reading the output of one of these passes.

  - id: sil-optimisation
    title: "SIL optimisation — where Swift gets fast"
    summary: |
      Canonical SIL is correct, but it is naive. A protocol call is still a dynamic dispatch; a generic function still works through abstract type parameters; every reference assignment still emits a retain and release. The <strong>SIL optimiser</strong> — which runs only when optimisation is enabled — rewrites all of that, and because it still understands Swift, it can do things LLVM never could.
      <br><br>
      <strong>Generic specialisation</strong> is the big one. A generic function like <code>func max&lt;T: Comparable&gt;(_ a: T, _ b: T)</code> is, in its unspecialised form, slow: it operates on opaque values through witness tables. When the optimiser sees it called with a concrete type — <code>max(3, 5)</code> with <code>Int</code> — it stamps out a <strong>specialised copy</strong> that works directly on <code>Int</code>, with no indirection at all. That specialised version can then be inlined and reduced to a single comparison instruction.
      <br><br>
      <strong>Inlining</strong> pulls a callee's body into the caller, eliminating call overhead and, more importantly, exposing the combined code to further optimisation. <strong>Devirtualisation</strong> turns a dynamic call static: if the optimiser can prove the concrete type at a call site, a vtable or witness-table call collapses into a direct call, which can then itself be inlined. And <strong>ARC optimisation</strong> removes redundant retain/release pairs — if a reference is retained and then released within the same scope with nothing in between that could deallocate it, both calls simply vanish.
      <br><br>
      <pre><code>// Before ARC optimisation (conceptually):
      strong_retain %obj
      %r = apply %f(%obj)   // f doesn't outlive this scope
      strong_release %obj   // redundant pair -> removed

      // After: the retain/release disappear entirely.</code></pre>
      This is why a tight Swift loop over a generic collection can compile down to code as lean as hand-written C. The abstraction was real in your source and in SIL — but the optimiser proved it away before the CPU ever sees it.

  - id: irgen-to-machine-code
    title: "IRGen and LLVM — the descent to native code"
    summary: |
      With SIL optimised, Swift has squeezed out everything that needs Swift-level knowledge. Now it can let go. <strong>IRGen</strong> lowers SIL into <strong>LLVM IR</strong>, and from there Swift becomes just another LLVM frontend, handing the rest of the job to the same backend that compiles C, C++, and Rust.
      <br><br>
      The lowering is a genuine loss of information, and that is the point. A SIL protocol-witness call becomes a couple of pointer loads and an indirect call in LLVM IR. A SIL <code>struct</code> becomes a flat sequence of fields or registers. The notion of "protocol" or "generic" is gone — only memory, integers, and control flow remain.
      <br><br>
      <pre><code>$ swiftc -emit-ir example.swift   # LLVM IR (.ll)
      $ swiftc -emit-assembly example.swift   # native asm (.s)

      ; LLVM IR fragment
      define hidden swiftcc i64 @"$s7example6doubleyS2iF"(i64 %0) {
      entry:
        %1 = mul i64 %0, 2
        ret i64 %1
      }</code></pre>
      So what does LLVM add that SIL did not? The classics of low-level optimisation: <strong>register allocation</strong> (deciding which values live in CPU registers versus memory), <strong>instruction selection and scheduling</strong> for the specific target chip, <strong>peephole optimisations</strong>, loop transformations, and finally emitting the actual bytes of x86-64 or ARM64 machine code. LLVM is brilliant at this and entirely target-specific — the same LLVM IR becomes different instructions on an Apple Silicon Mac than on a Linux server.
      <br><br>
      The division of labour is clean. SIL does the optimisations that require understanding Swift; LLVM does the optimisations that require understanding the machine. Each works at the level where it has the knowledge to be effective.

  - id: whole-module-optimisation
    title: "Whole-module and cross-module optimisation"
    summary: |
      There's a catch hiding in everything above. Optimisations like specialisation, inlining, and devirtualisation depend on the compiler being able to <em>see</em> the code it's reasoning about. By default, Swift compiles one file at a time, so a function in <code>A.swift</code> is opaque when compiling <code>B.swift</code>. That limits how much the optimiser can do.
      <br><br>
      <strong>Whole-module optimisation</strong> (WMO) fixes this. With WMO, the compiler treats the entire module as a single unit, so it can inline across files, specialise a generic defined in one file at a call site in another, and devirtualise calls to types it can now prove are never subclassed. Release builds use it by default. The trade-off is build time: change one line and the whole module recompiles, which is exactly why debug builds keep files separate for fast incremental rebuilds.
      <br><br>
      <pre><code>$ swiftc -O -wmo MyLib/*.swift     # whole-module, optimised
      $ swiftc -enable-library-evolution ...   # resilient ABI</code></pre>
      But what about across a <em>module</em> boundary — calling into a framework you don't compile? Normally the optimiser can't inline a function it can't see. The <code>@inlinable</code> attribute opts a function into being exposed in the module's interface so callers <em>can</em> inline it. That power collides with the <strong>library evolution</strong> model (also called resilience): a framework that promises ABI stability must be free to change its internals without breaking compiled clients, which means it must <em>not</em> expose those internals for inlining by default.
      <br><br>
      So there's a genuine tension. Mark something <code>@inlinable</code> and callers get speed, but you've frozen that implementation into your public contract. Keep it opaque and you stay free to evolve, but every call across the boundary pays the price of indirection. The system libraries that ship inside the OS live entirely inside this trade-off, and it's why the resilience model exists at all.

  - id: name-mangling

You can have two functions both called <code>area</code>, one on <code>Circle</code> and one on <code>Square</code>; you can have the same generic function instantiated for a dozen different types. They all need <strong>distinct linker symbols</strong>, and the symbol has to encode enough to be unique. The answer is <strong>name mangling</strong>.

  - id: runtime-type-metadata
    title: "The runtime: type metadata"
    summary: |
      We've reached native code, but Swift is not finished with us — there is a <strong>runtime</strong>, a support library every Swift program links against. It is not a virtual machine and it does not interpret anything; it is a set of data structures and functions that the compiled code calls into for the dynamic parts of the language. At its heart sits <strong>type metadata</strong>.
      <br><br>
      Every type in Swift has a <strong>metadata record</strong>: a runtime data structure describing its size, alignment, how to copy and destroy its values, its name, and — for classes — its vtable and superclass. When you write <code>type(of: value)</code> or <code>T.self</code>, you are getting a pointer to this metadata. These first-class type values are called <strong>metatypes</strong>, and they are how the runtime reasons about types it was handed dynamically.
      <br><br>
      <pre><code>let t = type(of: someValue)   // a metatype
      print(t)                      // uses metadata to print the name
      let array = Array&lt;Int&gt;()       // generic metadata involved
      </code></pre>
      The clever part is <strong>generic metadata</strong>. The compiler cannot pre-build a metadata record for every possible <code>Array&lt;T&gt;</code> — there are infinitely many <code>T</code>. Instead it emits a metadata <strong>pattern</strong>, and the first time the program actually needs <code>Array&lt;Int&gt;</code>, the runtime <strong>instantiates</strong> the concrete metadata on demand and <strong>caches</strong> it, so the second request for <code>Array&lt;Int&gt;</code> is a cheap lookup.
      <br><br>
      This is the runtime counterpart to compile-time specialisation. When the optimiser can specialise a generic, no metadata work is needed at runtime at all. When it can't — because the concrete type isn't known until runtime — the metadata system steps in, building and caching exactly the records the program turns out to need. It is the machinery that makes generics work even when the compiler couldn't see through them.

  - id: modules-stability-and-edges
    title: "Modules, stability, and the edges"
    summary: |
      We'll finish at the boundaries of the system, because that's where the design choices show most clearly. A compiled Swift module can be distributed in two forms. The <strong>.swiftmodule</strong> is a binary serialisation of the module's declarations — fast for the compiler to read, but bound to one compiler version. The <strong>.swiftinterface</strong> is a textual file listing the module's public surface as readable Swift, which a newer compiler can re-process. That textual form is what enables <strong>module stability</strong>.

      <br><br>
      It's worth being precise here, because two stabilities often get muddled. <strong>ABI stability</strong>, delivered in Swift 5 in 2019, is a <em>run-time</em> guarantee: a binary built today keeps working against a future Swift runtime, because data layout and calling conventions are frozen. <strong>Module stability</strong> is a <em>compile-time</em> guarantee: a pre-built binary framework can be imported by a different compiler version. You can have one without the other; they solve different problems.
      <br><br>
      Then there are the edges where Swift's usual machinery is too heavy or too foreign. <strong>Embedded Swift</strong> is a subset designed for microcontrollers and other constrained targets: it drops the runtime metadata and reflection that we've spent this episode admiring, relying entirely on compile-time specialisation so it can produce tiny, runtime-free binaries. No conformance cache, no on-demand metadata — everything resolved before the chip ever runs.
      <br><br>
      And at the opposite edge, Swift has to coexist with decades of existing code. The <strong>Clang importer</strong> reads C and Objective-C headers — and, increasingly, C++ — and synthesises Swift declarations for them on the fly, which is how you can call a UIKit class or a C library directly without any hand-written bindings. So the same compiler that lowers your code through SIL and LLVM also reaches sideways into Clang to make the rest of the world's code look like Swift. From the constraint solver at the top to the Clang importer at the edge, that is how Swift runs.
