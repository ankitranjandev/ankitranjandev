  - id: text-is-hard
    title: "Why text is harder than numbers"
    summary: "Numbers are simple. An <code>int</code> is a fixed number of bits. A <code>double</code> follows IEEE 754. The rules are universal and unambiguous.<br><br>Text is a different beast entirely.<br><br>Humans have invented thousands of writing systems. A single \"character\" might be a Latin letter, a Chinese ideograph, an Arabic ligature, a Korean syllable block, or a family emoji with skin tones. Some characters are combined from multiple pieces. Some look identical but have different underlying codes. Some change shape depending on their neighbors.<br><br>A string library has to make choices — and those choices have trade-offs. In this episode we will see exactly what Dart chose, why it sometimes surprises us, and how to work with text correctly."

  - id: grapheme-clusters
    title: "Grapheme clusters — what users actually see"
    summary: "Unicode allows characters to be <strong>combined</strong>. A family emoji like 👨‍👩‍👧 is actually five code points glued together with Zero Width Joiners (ZWJ, U+200D).<br><br>• 👨 (man) + ZWJ<br>• 👩 (woman) + ZWJ<br>• 👧 (girl)<br><br>The ZWJ tells the renderer to fuse them into one visual unit. The same trick is used for skin tone modifiers (👋🏽 = 👋 + 🏽), flag emoji (🇯🇵 = 🇯 + 🇵), and accented characters (é can be e + combining acute accent).<br><br>What the user sees as a single \"character\" can be one code point, or it can be many code points combined. Unicode calls this visual unit a <strong>grapheme cluster</strong>.<br><br>So we have three levels:<br><br>1. <strong>Code units</strong> — what <code>.length</code> counts (UTF-16 building blocks)<br>2. <strong>Code points</strong> — what <code>.runes</code> counts (Unicode numbers)<br>3. <strong>Grapheme clusters</strong> — what the user perceives as characters"

  - id: characters-package
    title: "The characters package"
    summary: "Dart's core library doesn't have built-in grapheme cluster support. For that, we need the <code>characters</code> package from the Dart team.<br><br><pre><code>import 'package:characters/characters.dart';\n\nString family = '👨‍👩‍👧';\nprint(family.length);              // 8 (code units)\nprint(family.runes.length);        // 5 (code points)\nprint(family.characters.length);   // 1 (grapheme clusters)</code></pre>Now we finally get 1 — the number a human would count.<br><br>The <code>.characters</code> extension also lets us iterate, substring, and manipulate text by grapheme cluster.<br><br><pre><code>String text = 'Hi 👋🏽!';\nfor (var char in text.characters) {\n  print(char);  // H, i, (space), 👋🏽, !\n}\n\n// Safe substring — won't slice through emoji\nprint(text.characters.skip(3).take(1).string);  // 👋🏽</code></pre>For any user-facing text manipulation — limiting input length, truncating for display, counting characters — the <code>characters</code> package is essential."

  - id: string-interning
    title: "String interning and identity"
    summary: "Dart <strong>interns</strong> string literals. When we write the same string literal multiple times in our code, they all point to the same object in memory.<br><br><pre><code>String a = 'hello';\nString b = 'hello';\nprint(identical(a, b));  // true — same object\n\nString c = 'hel' + 'lo';\nprint(identical(a, c));  // true — compile-time constant\n\nString d = ['hel', 'lo'].join();\nprint(identical(a, d));  // false — built at runtime</code></pre>Interning saves memory when the same string appears many times. But it only applies to compile-time constants. Strings built at runtime (from user input, parsing, concatenation with variables) are separate objects.<br><br>For equality, always use <code>==</code>. The <code>identical()</code> function checks object identity, which depends on interning — a detail we usually don't want to rely on."

  - id: common-pitfalls
    title: "Common string pitfalls"
    summary: "<strong>1. Slicing through surrogate pairs.</strong><br><br><pre><code>String emoji = '😀';\nString broken = emoji.substring(0, 1);  // Half an emoji!\nprint(broken);  // Garbage or replacement character</code></pre><strong>Fix —</strong> Use <code>.characters</code> for user-visible text, or check <code>isLowSurrogate</code> / <code>isHighSurrogate</code> if working at the code-unit level.<br><br><strong>2. Comparing strings with different normalization.</strong><br><br>The letter \"é\" can be represented two ways. Precomposed (U+00E9, single code point) or decomposed (U+0065 + U+0301, e + combining acute). They look identical but aren't <code>==</code> to each other. If comparing user input or data from different sources, normalize first.<br><br><strong>3. Counting length for UI constraints.</strong><br><br><pre><code>// Wrong — may slice through emoji\nif (input.length > 50) input = input.substring(0, 50) + '…';\n\n// Right — respects grapheme clusters\nimport 'package:characters/characters.dart';\nif (input.characters.length > 50) {\n  input = input.characters.take(50).string + '…';\n}</code></pre><strong>4. Off-by-one with newlines.</strong><br><br>Different platforms use <code>\\n</code>, <code>\\r\\n</code>, or <code>\\r</code>. When splitting lines or counting them, be aware of the line-ending convention in your input."

  - id: stringbuffer
    title: "StringBuffer for efficient concatenation"
    summary: "Because strings are immutable, repeated concatenation creates many temporary objects.<br><br><pre><code>String result = '';\nfor (var i = 0; i < 1000; i++) {\n  result += 'item $i, ';  // Creates 1000 intermediate strings\n}</code></pre>For building strings in a loop, use <code>StringBuffer</code>.<br><br><pre><code>StringBuffer buffer = StringBuffer();\nfor (var i = 0; i < 1000; i++) {\n  buffer.write('item $i, ');\n}\nString result = buffer.toString();  // One allocation at the end</code></pre><code>StringBuffer</code> grows an internal buffer efficiently, then produces the final string in one allocation.<br><br>For simple cases (a few concatenations), <code>+</code> or string interpolation is fine. The VM optimizes many common patterns. But for loops or large builds, <code>StringBuffer</code> is the right tool.<br><br>That wraps up strings. The key takeaway — <strong>Dart gives us three views of text (code units, code points, and grapheme clusters) and we need to pick the right one for the job.</strong> For most user-facing work, grapheme clusters (via the <code>characters</code> package) are what we actually want."
