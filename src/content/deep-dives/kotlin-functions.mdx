  - id: functions-first-class
    title: "Functions as first-class values"
    summary: |
      In Java before version 8, a function could not exist on its own. To pass behaviour around, you wrapped it in an object — an anonymous inner class implementing some single-method interface. Kotlin treats functions as <strong>first-class values</strong> from the start: a function can be stored in a variable, passed as an argument, and returned from another function, just like an <code>Int</code> or a <code>String</code>.
      <br><br>
      Functions also do not need to live inside a class. Kotlin allows <strong>top-level functions</strong> — declared directly in a file, no surrounding class required. This is why <code>fun main()</code> sits alone, and why the standard library is full of free functions like <code>maxOf</code> and <code>println</code> rather than static methods bolted onto utility classes.
      <br><br>
      Every function has a <strong>function type</strong>, written with an arrow. <code>(Int, Int) -&gt; Int</code> is the type of a function taking two <code>Int</code>s and returning an <code>Int</code>.
      <br><br>
      <pre><code>// A variable holding a function&#10;val add: (Int, Int) -&gt; Int = { a, b -&gt; a + b }&#10;println(add(2, 3))   // 5&#10;&#10;// A reference to an existing function with ::&#10;val printer: (Any?) -&gt; Unit = ::println&#10;printer("hello")</code></pre>
      The <code>::</code> operator creates a <strong>function reference</strong> — a way to grab an existing named function as a value without calling it. These pieces — top-level functions, function types, lambdas, and references — are the vocabulary everything else in this episode is built from.

  - id: higher-order-and-lambdas
    title: "Higher-order functions and lambdas"
    summary: |
      A <strong>higher-order function</strong> is one that takes a function as a parameter or returns one. They are the backbone of Kotlin's collection operations — <code>map</code>, <code>filter</code>, <code>forEach</code> all take a lambda describing what to do with each element.
      <br><br>
      Kotlin has a small syntactic convention that makes these read beautifully. If the last parameter of a function is itself a function, you can move the lambda <em>outside</em> the parentheses — the <strong>trailing lambda</strong>. And if a lambda has a single parameter, you can refer to it with the implicit name <code>it</code> instead of naming it.
      <br><br>
      <pre><code>val numbers = listOf(1, 2, 3, 4, 5)&#10;&#10;// Fully explicit&#10;numbers.filter({ n -&gt; n % 2 == 0 })&#10;&#10;// Trailing lambda (last arg moved out of parens)&#10;numbers.filter { n -&gt; n % 2 == 0 }&#10;&#10;// Implicit single parameter 'it'&#10;numbers.filter { it % 2 == 0 }   // [2, 4]</code></pre>
      That trailing-lambda rule is more far-reaching than it first appears. It is the reason functions like <code>repeat(5) { ... }</code> and, ultimately, entire DSLs read like built-in language constructs rather than function calls. We will lean on it heavily at the end of this episode.

  - id: inline-functions
    title: "Why inline functions exist"
    summary: |
      Lambdas are convenient, but on the JVM they are not free. Each lambda is normally compiled into an object, and capturing variables from the surrounding scope allocates memory. For a function called millions of times — like the lambda you pass to <code>forEach</code> in a hot loop — that overhead adds up.
      <br><br>
      Kotlin's answer is the <code>inline</code> keyword. When a higher-order function is marked <code>inline</code>, the compiler does not generate a function call at all. Instead it copies the function's body — and the body of the lambda you passed — directly into the call site.

      The result is zero call overhead and zero lambda allocation. This is why so much of the standard library's lambda-taking machinery — <code>let</code>, <code>run</code>, <code>apply</code>, <code>filter</code>, <code>map</code> — is inline. Inlining also unlocks two abilities ordinary lambdas lack: <strong>non-local returns</strong> (a <code>return</code> inside the lambda can return from the enclosing function, because there is no separate function to return from) and the <code>reified</code> type parameters covered in the Type System episode.
      <br><br>
      The trade-off: inlining copies code, so over-inlining a large function called in many places bloats the bytecode. Use <code>inline</code> for small higher-order functions, not as a blanket performance fix. The compiler will even warn you when inlining buys nothing.

  - id: extension-functions
    title: "Extension functions"
    summary: |
      This is the feature that most changes how Kotlin <em>feels</em>. An <strong>extension function</strong> lets you add a new method to a class you do not own — without inheriting from it, without modifying its source, without any wrapper.
      <br><br>
      <pre><code>// Add a method to the standard String class&#10;fun String.shout(): String = this.uppercase() + "!"&#10;&#10;"hello".shout()   // "HELLO!"</code></pre>
      Inside the extension, <code>this</code> refers to the instance it was called on (the <strong>receiver</strong>). To the caller, <code>shout()</code> looks and feels exactly like a built-in method of <code>String</code>, complete with autocomplete in the IDE.
      <br><br>
      There is an important subtlety: extensions are <strong>resolved statically</strong>. They are not real members of the class — under the hood, <code>"hello".shout()</code> compiles to a plain static call like <code>shout("hello")</code>. This means an extension cannot access a class's private internals, and it cannot be overridden polymorphically the way a real method can. Which extension is called is decided by the <em>declared</em> type of the variable, not its runtime type.
      <br><br>
      Despite that limitation — or because of it — extensions are everywhere in idiomatic Kotlin. Huge swathes of the standard library are extension functions: nearly all the collection operators (<code>map</code>, <code>filter</code>, <code>groupBy</code>) are extensions on <code>Iterable</code>. They let you keep classes small and focused while adding rich, discoverable functionality from the outside. C# pioneered this idea (its "extension methods" arrived with LINQ in 2007); Kotlin embraced it as a core part of its style.

  - id: scope-functions
    title: "Scope functions: let, run, with, apply, also"
    summary: |
      Kotlin's standard library includes five small, famously-confusing <strong>scope functions</strong>. They all do roughly the same thing — execute a block of code in the context of an object — but they differ on two axes: how the object is referenced inside the block (as <code>this</code> or as <code>it</code>), and what the block returns (the object itself, or the block's result).

      <br><br>
      Two examples make the distinction click. <code>apply</code> returns the receiver, so it is perfect for configuring an object you are creating:
      <br><br>
      <pre><code>val intent = Intent().apply {&#10;    action = "VIEW"      // 'this' is the Intent&#10;    putExtra("id", 42)&#10;}   // returns the configured Intent</code></pre>
      <code>let</code> exposes the object as <code>it</code> and returns the block's result, making it the idiomatic tool for "do something only if non-null":
      <br><br>
      <pre><code>val length: Int? = name?.let {&#10;    println("got $it")&#10;    it.length            // this becomes the result&#10;}</code></pre>
      The choice between them is not arbitrary style — it expresses intent. <code>apply</code>/<code>also</code> say "I am still working with this object." <code>let</code>/<code>run</code>/<code>with</code> say "I am transforming it into something else."

  - id: lambdas-with-receiver-and-dsls
    title: "Lambdas with receiver, and DSLs"
    summary: |
      We now have every ingredient for Kotlin's most powerful trick. Notice that in the <code>apply</code> example, inside the lambda you could write <code>action = "VIEW"</code> as if you were <em>inside</em> the <code>Intent</code> class. That is a <strong>lambda with receiver</strong> — a function type written <code>Intent.() -&gt; Unit</code>, meaning "a lambda whose <code>this</code> is an <code>Intent</code>."
      <br><br>
      Combine a function with a receiver-lambda parameter with the trailing-lambda syntax, and ordinary function calls start to look like a custom mini-language. This is how Kotlin builds <strong>DSLs</strong> (domain-specific languages). The classic example is an HTML builder:
      <br><br>
      <pre><code>val page = html {            // html { } is a function taking an HTML.() -&gt; Unit&#10;    body {&#10;        h1 { +"Welcome" }&#10;        p { +"Built with a Kotlin DSL" }&#10;    }&#10;}</code></pre>
      Every line there is a normal function call with a trailing receiver-lambda. <code>body { }</code> is a method on the <code>html</code> receiver; inside it, <code>this</code> is a <code>Body</code>, so <code>h1 { }</code> resolves against it. The whole thing type-checks at compile time — typos and misplaced tags are errors, not runtime surprises. This same pattern powers Jetpack Compose (<code>Column { Text("hi") }</code>), the Gradle Kotlin build scripts, kotlinx.html, and Ktor's routing.
      <br><br>
      One refinement keeps these DSLs honest: the <code>@DslMarker</code> annotation. Without it, nested builders could accidentally call methods from an outer receiver (writing an <code>h1</code> directly inside <code>html</code> rather than <code>body</code>). <code>@DslMarker</code> tells the compiler to restrict each block to its own receiver, catching those mistakes. The supporting cast — <code>infix</code> functions (write <code>1 to "a"</code> instead of <code>1.to("a")</code>) and <code>operator</code> overloading (define what <code>+</code> means for your type) — adds the final polish that lets a Kotlin DSL read like prose.
