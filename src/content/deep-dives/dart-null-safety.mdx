  - id: billion-dollar-mistake
    title: "The billion-dollar mistake"
    summary: |
      In 1965, Tony Hoare invented the null reference. Decades later, he called it his "billion-dollar mistake" — because null reference errors have caused more crashes, more bugs, and more wasted debugging hours than perhaps any other single language feature.
      <br><br>
      The problem is simple. In most languages, any variable can secretly hold <code>null</code>. We write code assuming a value exists, and then — at runtime — it turns out it does not.
      <br><br>
      <pre><code>// In a language without null safety
      String name = fetchUserName();   // might return null
      print(name.length);              // CRASH at runtime</code></pre>
      The crash comes with no warning. The compiler happily compiled the code. The bug only appears when a user hits the exact path that returns null.

      <br><br>
      Dart 2.12 (released March 2021) introduced <strong>sound null safety</strong>. The idea is simple but powerful: make nullability part of the type. If a variable can be null, the type says so. If it cannot, the compiler enforces it.
      <br><br>
      The result is that null reference errors — the most common crash in programming history — become compile-time errors instead of runtime surprises.

  - id: nullable-vs-non-nullable
    title: "Nullable vs non-nullable types"
    summary: |
      In Dart, types are <strong>non-nullable by default</strong>. A plain <code>String</code> cannot be null. Ever.
      <br><br>
      <pre><code>String name = 'Ankit';   // fine
      name = null;             // compile error — String cannot be null</code></pre>
      To allow null, we add a <code>?</code> to the type. This creates a nullable type.
      <br><br>
      <pre><code>String? name = 'Ankit';  // fine
      name = null;             // also fine — String? allows null</code></pre>
      The <code>?</code> is not decoration. It changes the type entirely. <code>String</code> and <code>String?</code> are different types in Dart's type system.

      The diagram shows how Dart's type hierarchy works. <code>Object</code> is the root of all non-nullable types. <code>Null</code> is its own separate type. A nullable type like <code>String?</code> is actually a union: it can hold a <code>String</code> or <code>Null</code>.

      This means we cannot use a <code>String?</code> where a <code>String</code> is expected — the compiler blocks it.
      <br><br>
      <pre><code>void greet(String name) {
        print('Hello, $name');
      }

      String? maybeName = getUserName();
      greet(maybeName);   // compile error — String? is not String</code></pre>
      The fix is to handle the null case first. Dart gives us several tools to do that.

  - id: null-aware-operators
    title: "Null-aware operators"
    summary: |
      Dart provides four operators specifically designed for working with nullable values. They make null handling concise without sacrificing safety.
      <br><br>
      <strong>The null-aware access operator: <code>?.</code></strong>
      <br><br>
      Use this when the object itself might be null. If it is, the whole expression evaluates to null instead of crashing.
      <br><br>
      <pre><code>String? name = null;
      print(name?.length);     // null (not a crash)
      print(name?.toUpperCase());  // null

      String? name2 = 'Ankit';
      print(name2?.length);    // 5</code></pre>
      <strong>The null-coalescing operator: <code>??</code></strong>
      <br><br>
      Provides a default value when the left side is null.
      <br><br>
      <pre><code>String? name = null;
      String displayName = name ?? 'Guest';
      print(displayName);   // 'Guest'

      String? name2 = 'Ankit';
      String displayName2 = name2 ?? 'Guest';
      print(displayName2);  // 'Ankit'</code></pre>
      <strong>The null-aware assignment operator: <code>??=</code></strong>
      <br><br>
      Assigns a value only if the variable is currently null.
      <br><br>
      <pre><code>String? name;
      name ??= 'Default';   // name is now 'Default'
      name ??= 'Other';     // no change — name is already non-null
      print(name);          // 'Default'</code></pre>
      <strong>The null-aware index operator: <code>?[]</code></strong>
      <br><br>
      Access a list or map element when the collection itself might be null.
      <br><br>
      <pre><code>List&lt;int&gt;? numbers = null;
      print(numbers?[0]);   // null (not a crash)

      List&lt;int&gt;? numbers2 = [1, 2, 3];
      print(numbers2?[0]);  // 1</code></pre>

      <br><br>
      These operators chain naturally. <code>user?.address?.city?.toUpperCase()</code> safely navigates a potentially null chain — if any link is null, the whole expression is null.

  - id: bang-operator
    title: "The bang operator (!)"
    summary: |
      Sometimes we know a value is not null, but the compiler does not. The bang operator <code>!</code> tells Dart: "Trust me, this is not null."
      <br><br>
      <pre><code>String? name = fetchName();   // might be null

      // We've checked elsewhere or have external knowledge
      String definitelyName = name!;   // assert non-null
      print(definitelyName.length);    // safe to use</code></pre>
      <strong>But here is the catch.</strong> If we are wrong — if the value actually is null — the program crashes at runtime. We have just opted out of null safety for that expression.
      <br><br>
      <pre><code>String? name = null;
      String definitelyName = name!;   // CRASH — TypeError at runtime</code></pre>
      <svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 720 220" role="img" aria-label="Bang operator warning" style="max-width: 100%; height: auto;"><style>.ns4-t{font-family:-apple-system,BlinkMacSystemFont,"Segoe UI",Inter,sans-serif;fill:#1e293b}.ns4-title{font-size:16px;font-weight:600}.ns4-mn{font-family:"SF Mono",Menlo,Monaco,monospace;font-size:14px;font-weight:700}.ns4-label{font-size:12px;fill:#64748b}.ns4-warn{font-size:13px;fill:#b45309;font-weight:600}</style><text x="360" y="26" text-anchor="middle" class="ns4-t ns4-title">The bang operator: a sharp tool</text><rect x="80" y="55" width="240" height="90" fill="#d1fae5" stroke="#10b981" stroke-width="2" rx="6"/><text x="200" y="85" text-anchor="middle" class="ns4-t ns4-mn">name!</text><text x="200" y="110" text-anchor="middle" class="ns4-t ns4-label">When you are RIGHT:</text><text x="200" y="130" text-anchor="middle" class="ns4-t ns4-label" fill="#059669">Compiles and runs fine</text><rect x="400" y="55" width="240" height="90" fill="#fee2e2" stroke="#dc2626" stroke-width="2" rx="6"/><text x="520" y="85" text-anchor="middle" class="ns4-t ns4-mn">name!</text><text x="520" y="110" text-anchor="middle" class="ns4-t ns4-label">When you are WRONG:</text><text x="520" y="130" text-anchor="middle" class="ns4-t ns4-label" fill="#dc2626">Runtime crash (TypeError)</text><rect x="120" y="165" width="480" height="40" fill="#fef3c7" stroke="#f59e0b" stroke-width="1.5" rx="4"/><text x="360" y="190" text-anchor="middle" class="ns4-t ns4-warn">Rule: avoid ! unless you have a genuinely good reason. Prefer null checks.</text></svg>
      <br><br>
      <strong>When is <code>!</code> appropriate?</strong>
      <br><br>
      • When we have already checked for null in a way the compiler cannot see (e.g., an <code>assert</code> or a check in a different function).
      <br>
      • In tests, where a crash is acceptable and explicit.
      <br>
      • In initialisation code where we know the value is set before use.
      <br><br>
      <strong>When to avoid <code>!</code>:</strong>
      <br><br>
      • When we are not 100% certain the value is non-null.
      <br>
      • When a safer alternative exists (use <code>??</code>, <code>?.</code>, or a proper null check instead).
      <br><br>
      Think of <code>!</code> as a contract with the compiler: "I promise this is not null. If I'm wrong, let it crash." Use it sparingly.

  - id: late-variables
    title: "Late variables"
    summary: |
      Sometimes we cannot initialise a variable when we declare it, but we know it will be set before we use it. The <code>late</code> keyword tells Dart: "I promise to initialise this later."
      <br><br>
      <pre><code>late String name;

      void init() {
        name = 'Ankit';   // initialised here
      }

      void greet() {
        print('Hello, $name');   // used here — after init()
      }</code></pre>
      Without <code>late</code>, Dart would complain that <code>name</code> is not initialised. With <code>late</code>, the check moves to runtime.
      <br><br>
      <strong>Lazy initialisation.</strong> <code>late</code> also enables lazy evaluation. If we provide an initialiser, it runs only when the variable is first accessed.
      <br><br>
      <pre><code>late String expensiveValue = computeExpensiveValue();

      void main() {
        // expensiveValue has NOT been computed yet
        print('Program started');

        // Now it computes
        print(expensiveValue);   // computeExpensiveValue() runs here
        print(expensiveValue);   // cached — does not recompute
      }</code></pre>
      This is useful for fields that are expensive to compute and might not always be needed.
      <br><br>
      <strong>The danger of <code>late</code>.</strong> If we read a <code>late</code> variable before it has been initialised, we get a runtime error.
      <br><br>
      <pre><code>late String name;

      void main() {
        print(name);   // CRASH — LateInitializationError
      }</code></pre>
      <strong>When to use <code>late</code>:</strong>
      <br><br>
      • Instance variables initialised in a constructor body or <code>initState()</code>.
      <br>
      • Lazy fields that should compute only when accessed.
      <br>
      • Variables set by framework callbacks (Flutter's <code>didChangeDependencies</code>, etc.).
      <br><br>
      <strong>When to avoid <code>late</code>:</strong>
      <br><br>
      • When the variable might genuinely not be set — use nullable instead.
      <br>
      • When we can initialise directly or in the initialiser list.

  - id: required-named
    title: "Required named parameters"
    summary: |
      Named parameters in Dart are optional by default. But sometimes a parameter is both named (for clarity) and required (for correctness). The <code>required</code> keyword combines both.
      <br><br>
      <pre><code>void createUser({
        required String name,
        required String email,
        int age = 0,
      }) {
        print('$name ($email), age $age');
      }

      createUser(name: 'Ankit', email: 'a@b.com');         // works
      createUser(name: 'Ankit');                           // compile error — email required
      createUser(email: 'a@b.com');                        // compile error — name required</code></pre>
      Before null safety, named parameters had to be nullable or have defaults. Now we can have non-nullable named parameters by marking them <code>required</code>.
      <br><br>
      <strong>This is crucial for Flutter widgets.</strong> Most widget constructors use named parameters extensively, and <code>required</code> makes the API self-documenting.
      <br><br>
      <pre><code>class UserCard extends StatelessWidget {
        final String name;
        final String email;
        final int? age;

        const UserCard({
          super.key,
          required this.name,     // must provide
          required this.email,    // must provide
          this.age,               // optional — nullable
        });

        @override
        Widget build(BuildContext context) { ... }
      }</code></pre>
      The <code>required</code> keyword shifts a potential runtime error ("forgot to pass name") into a compile-time error. The IDE shows the error before we even run the code.

  - id: flow-analysis
    title: "Flow analysis and type promotion"
    summary: |
      Dart's compiler is smart. When we check for null, the compiler remembers — and automatically promotes the type from nullable to non-nullable within the safe scope.
      <br><br>
      <pre><code>void greet(String? name) {
        if (name == null) {
          print('Hello, Guest');
          return;
        }

        // Here, Dart KNOWS name is not null
        // Type is promoted from String? to String
        print('Hello, $name');
        print(name.length);   // no error — name is now String
      }</code></pre>
      We did not use <code>!</code>. We did not cast. Dart's <strong>flow analysis</strong> tracked that after the <code>if (name == null) return</code>, name must be non-null in the remaining code.

      <br><br>
      <strong>Promotion works with many patterns:</strong>
      <br><br>
      <pre><code>// Pattern 1: if-return (shown above)
      if (name == null) return;

      // Pattern 2: if-else
      if (name != null) {
        print(name.length);   // promoted inside the block
      }

      // Pattern 3: logical AND
      if (name != null && name.length > 5) {
        // promoted after the && check
      }

      // Pattern 4: throw
      if (name == null) throw ArgumentError('name required');
      print(name.length);   // promoted after the throw</code></pre>
      <strong>When promotion does NOT work:</strong>
      <br><br>
      Instance fields and top-level variables cannot be promoted. Another piece of code might change them between the check and the use. For those, use a local variable.
      <br><br>
      <pre><code>class User {
        String? name;
