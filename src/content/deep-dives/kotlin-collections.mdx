  - id: read-only-vs-mutable
    title: "Read-only and mutable, as separate types"
    summary: |
      Java has one <code>List</code> interface, and it can do everything — read, add, remove, clear. If you hand someone your list, they can mutate it behind your back. Kotlin splits the idea in two at the type level. <code>List&lt;T&gt;</code> is <strong>read-only</strong>: it has <code>size</code>, <code>get</code>, and iteration, but no <code>add</code> or <code>remove</code>. <code>MutableList&lt;T&gt;</code> extends it and adds the modifying operations.

      <br><br>
      The benefit is intent. When a function takes a <code>List&lt;T&gt;</code>, it is promising not to modify it; when it returns one, it is handing you something you are not meant to change. You expose the read-only type at API boundaries and keep the mutable type private.
      <br><br>
      <strong>An important honesty point.</strong> Read-only is <em>not</em> the same as immutable. A <code>List</code> is a read-only <em>view</em> — the object behind it might still be a <code>MutableList</code> that other code can change, and the underlying elements can themselves be mutable. Kotlin's read-only interfaces stop <em>you</em> from mutating through that reference; they do not freeze the data. For true immutability you reach for the kotlinx.collections.immutable library or defensive copies. Understanding this distinction prevents a subtle class of bugs where a "read-only" list changes underneath you.

  - id: creating-collections
    title: "Creating collections"
    summary: |
      Kotlin provides factory functions that read like literals. The read-only builders are the ones you reach for by default; the <code>mutable</code> variants give you something you can modify.
      <br><br>
      <pre><code>val nums = listOf(1, 2, 3)              // List&lt;Int&gt; — read-only&#10;val editable = mutableListOf(1, 2, 3)  // MutableList&lt;Int&gt;&#10;&#10;val unique = setOf("a", "b", "a")      // Set — duplicates dropped → [a, b]&#10;val ages = mapOf("Ann" to 30, "Bo" to 25)   // Map, built with 'to' pairs&#10;&#10;val empty = emptyList&lt;String&gt;()         // shared, allocation-free empty list</code></pre>
      Notice <code>"Ann" to 30</code> — that is the <code>to</code> infix function from the Functions episode, creating a <code>Pair</code>. There is no special map syntax; it is just functions composing.
      <br><br>
      For collections built up with some logic, the <code>build</code> functions give you a temporary mutable scope that returns a read-only result — the best of both worlds:
      <br><br>
      <pre><code>val items = buildList {&#10;    add("first")&#10;    if (includeMore) addAll(extras)&#10;}   // 'items' is a read-only List, built mutably inside</code></pre>
      Under the hood these create the familiar JVM collections (<code>ArrayList</code>, <code>LinkedHashSet</code>, <code>LinkedHashMap</code>), so they interoperate seamlessly with Java code — another payoff of Kotlin's interop-first design.

  - id: the-operator-library
    title: "The operator library: loops you never write"
    summary: |
      The reason idiomatic Kotlin rarely contains a manual <code>for</code> loop is the standard library's vast set of collection operators — all of them extension functions (from the Functions episode) on <code>Iterable</code>. You describe <em>what</em> you want, not the mechanics of looping.
      <br><br>
      <pre><code>data class Person(val name: String, val age: Int, val city: String)&#10;&#10;people&#10;    .filter { it.age &gt;= 18 }                 // keep matching&#10;    .map { it.name }                         // transform each&#10;    .sorted()                                // order&#10;&#10;people.groupBy { it.city }                   // Map&lt;City, List&lt;Person&gt;&gt;&#10;people.associateBy { it.name }               // Map&lt;Name, Person&gt;&#10;people.partition { it.age &gt;= 18 }            // Pair(adults, minors)&#10;people.sumOf { it.age }                      // Int&#10;people.maxByOrNull { it.age }                // Person? (null if empty)</code></pre>
      A few families are worth committing to memory: <code>map</code>/<code>filter</code>/<code>flatMap</code> for transforming, <code>fold</code>/<code>reduce</code> for accumulating a single result, <code>groupBy</code>/<code>associate</code>/<code>partition</code> for restructuring, and the aggregate shortcuts <code>sumOf</code>/<code>count</code>/<code>any</code>/<code>all</code>/<code>none</code>.
      <br><br>
      One naming convention pays off everywhere: operations that might find nothing come in two flavours. <code>first()</code> and <code>max()</code> throw if the collection is empty or no element matches; <code>firstOrNull()</code> and <code>maxOrNull()</code> return <code>null</code> instead. The <code>...OrNull</code> suffix is your cue to handle the empty case with the null-safety tools from the foundations episode, rather than risk an exception.

  - id: eager-vs-lazy-sequences
    title: "Eager Lists vs lazy Sequences"
    summary: |
      Those operators are wonderfully readable, but they hide a cost. On a regular collection, <strong>each operator is eager</strong>: it runs to completion and builds a brand-new intermediate list before the next operator starts. A chain of <code>filter</code> then <code>map</code> over a million items allocates a million-element list just to throw it away.

      <br><br>
      A <strong>Sequence</strong> changes the execution model from eager to <strong>lazy</strong>. Operators on a sequence do not run immediately — they build a pipeline. Nothing happens until a <em>terminal</em> operation (like <code>toList</code>, <code>first</code>, <code>sum</code>) pulls values through. Then each element flows through the entire chain one at a time, and no intermediate collections are created.
      <br><br>
      <pre><code>val firstFive = people.asSequence()&#10;    .filter { it.age &gt;= 18 }&#10;    .map { it.name }&#10;    .take(5)        // pipeline built, nothing run yet&#10;    .toList()       // terminal op — now it runs, stopping after 5</code></pre>
      Because it is lazy, the sequence above stops the moment it has five names — it never examines the rest of the list. Sequences also make <strong>infinite</strong> data possible: <code>generateSequence(1) { it + 1 }</code> is an endless stream of integers you can safely <code>take</code> from.
      <br><br>
      <strong>When to use which.</strong> For small collections and short chains, plain eager operators are simpler and often faster — the laziness has its own per-element overhead. Reach for <code>asSequence()</code> when the collection is large, the chain of operations is long, or you only need part of the result (a <code>find</code> or <code>take</code> that can short-circuit). It is the same eager-vs-lazy trade-off found across many languages — and the sibling, in spirit, of Kotlin's asynchronous <code>Flow</code> from the Concurrency episode.

  - id: maps-in-depth
    title: "Working with maps"
    summary: |
      Maps get their own ergonomic touches. You read with the indexing operator, and iterating gives you entries you can destructure (using the <code>componentN</code> mechanism from the Object Model episode):
      <br><br>
      <pre><code>val ages = mapOf("Ann" to 30, "Bo" to 25)&#10;&#10;val a = ages["Ann"]          // 30, but the type is Int? — key might be absent&#10;val safe = ages["Cy"] ?: 0   // Elvis for a default&#10;&#10;for ((name, age) in ages) {  // destructure each entry&#10;    println("$name is $age")&#10;}</code></pre>
      Indexing a map returns a <em>nullable</em> value, because the key may not be present — the type system surfacing the same null-safety discipline from the foundations episode. For mutable maps, <code>getOrPut</code> is a beloved idiom: return the existing value, or compute, store, and return a new one in a single call.
      <br><br>
      <pre><code>val cache = mutableMapOf&lt;String, Int&gt;()&#10;fun lengthOf(s: String) = cache.getOrPut(s) { s.length }   // memoise</code></pre>
      Transforming maps mirrors the list operators: <code>mapValues</code>, <code>mapKeys</code>, <code>filterKeys</code>, <code>filterValues</code>, and the bridge <code>associate</code>/<code>associateWith</code> that turn a list into a map. Once you internalise that maps, sets, and lists all share the same operator vocabulary, restructuring data becomes a matter of picking the right verb.
