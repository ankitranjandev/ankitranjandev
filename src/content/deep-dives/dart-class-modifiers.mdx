  - id: why-class-modifiers
    title: "Why class modifiers exist"
    summary: |
      Before Dart 3, every class was wide open. Anyone could extend it, implement it, or instantiate it. That sounds flexible, but it creates problems.
      <br><br>
      Consider a library author who writes a <code>DatabaseConnection</code> class. Internally, this class relies on certain methods being called in a specific order. If someone extends it and overrides those methods incorrectly, the whole thing breaks. The library author has no way to prevent this.
      <br><br>
      <pre><code>// Your library&#10;class DatabaseConnection {&#10;  void connect() { /* sets up state */ }&#10;  void query(String sql) { /* uses that state */ }&#10;}&#10;&#10;// Someone else's code&#10;class MyConnection extends DatabaseConnection {&#10;  @override&#10;  void connect() {&#10;    // Oops, forgot to call super.connect()&#10;    // Now query() will crash&#10;  }&#10;}</code></pre>
      Class modifiers give you control. They let you say: "This class is for implementing, not extending." Or: "This class must stay within my library." Or: "This class cannot be subtyped at all."
      <br><br>
      Think of modifiers as <strong>API contracts</strong>. They communicate intent and the compiler enforces it.

  - id: abstract-modifier
    title: "abstract — cannot instantiate"
    summary: |
      The <code>abstract</code> modifier prevents direct instantiation. You cannot call the constructor from outside the class hierarchy.
      <br><br>
      <pre><code>abstract class Shape {&#10;  double get area;   // No implementation&#10;  void draw();       // Must be overridden&#10;}&#10;&#10;class Circle extends Shape {&#10;  final double radius;&#10;  Circle(this.radius);&#10;&#10;  @override&#10;  double get area => 3.14159 * radius * radius;&#10;&#10;  @override&#10;  void draw() => print('Drawing circle');&#10;}&#10;&#10;// var s = Shape();     // Error: abstract class&#10;var c = Circle(5);      // OK</code></pre>
      Abstract classes can have both abstract members (no implementation) and concrete members (with implementation). Subclasses must implement all abstract members.
      <br><br>
      Use <code>abstract</code> when you want to define a template. The class describes <em>what</em> something should do, but subclasses decide <em>how</em>.

  - id: base-modifier
    title: "base — extend only, no implementing"
    summary: |
      The <code>base</code> modifier allows extending but blocks implementing. This forces subclasses to inherit your implementation, not just your interface.
      <br><br>
      <pre><code>// In your library&#10;base class Vehicle {&#10;  void startEngine() {&#10;    _checkFuel();      // Private method&#10;    _initSystems();    // Must run in order&#10;    print('Engine started');&#10;  }&#10;&#10;  void _checkFuel() { /* ... */ }&#10;  void _initSystems() { /* ... */ }&#10;}&#10;&#10;// Outside your library&#10;class Car extends Vehicle {    // OK&#10;  @override&#10;  void startEngine() {&#10;    super.startEngine();       // Gets the full sequence&#10;    print('Car ready');&#10;  }&#10;}&#10;&#10;// class FakeVehicle implements Vehicle { }  // Error!</code></pre>
      Why block <code>implements</code>? Because implementing creates a completely new class that just happens to have the same method signatures. It doesn't get your actual code. If your class relies on internal state or method call sequences, an implementing class will break those assumptions.

  - id: interface-modifier
    title: "interface — implement only, no extending"
    summary: |
      The <code>interface</code> modifier is the opposite of <code>base</code>. It allows implementing but blocks extending.
      <br><br>
      <pre><code>// In your library&#10;interface class Comparable {&#10;  int compareTo(Object other) => 0;  // Default impl&#10;}&#10;&#10;// Outside your library&#10;class Person implements Comparable {  // OK&#10;  final String name;&#10;  Person(this.name);&#10;&#10;  @override&#10;  int compareTo(Object other) {&#10;    if (other is Person) {&#10;      return name.compareTo(other.name);&#10;    }&#10;    return -1;&#10;  }&#10;}&#10;&#10;// class Employee extends Comparable { }  // Error!</code></pre>
      Use <code>interface</code> when you want to define a contract without forcing users to inherit your implementation. This is useful when:
      <br><br>
      <strong>1.</strong> Your class has implementation details that shouldn't be inherited
      <br>
      <strong>2.</strong> You want to evolve the implementation without breaking subclasses
      <br>
      <strong>3.</strong> The class is meant purely as a type/contract

  - id: final-modifier
    title: "final — complete lockdown"
    summary: |
      The <code>final</code> modifier is the strictest. It blocks both extending and implementing outside the library. The type is frozen.
      <br><br>
      <pre><code>// In your library&#10;final class ApiResponse {&#10;  final int statusCode;&#10;  final String body;&#10;  final Map&lt;String, String&gt; headers;&#10;&#10;  ApiResponse(this.statusCode, this.body, this.headers);&#10;&#10;  bool get isSuccess => statusCode >= 200 && statusCode < 300;&#10;}</code></pre>
      Outside the library, users can create instances and use them normally. But they cannot extend or implement the class.

      <strong>1.</strong> The class represents a complete, self-contained value (like <code>DateTime</code> or <code>Uri</code>)

      <strong>3.</strong> Subclassing would break invariants or security guarantees

  - id: sealed-modifier
    title: "sealed — exhaustive subtypes in one file"
    summary: |
      The <code>sealed</code> modifier restricts subclasses to the same file. This enables exhaustive pattern matching — the compiler knows all possible subtypes.
      <br><br>
      <pre><code>// All in one file: result.dart&#10;sealed class Result&lt;T&gt; {}&#10;&#10;class Success&lt;T&gt; extends Result&lt;T&gt; {&#10;  final T value;&#10;  Success(this.value);&#10;}&#10;&#10;class Failure&lt;T&gt; extends Result&lt;T&gt; {&#10;  final String error;&#10;  Failure(this.error);&#10;}&#10;&#10;// Usage: compiler knows the switch is exhaustive&#10;String describe&lt;T&gt;(Result&lt;T&gt; result) {&#10;  return switch (result) {&#10;    Success(:var value) => 'Got: $value',&#10;    Failure(:var error) => 'Failed: $error',&#10;    // No default needed — compiler knows these are all cases&#10;  };&#10;}</code></pre>
      Sealed classes are implicitly abstract. You cannot instantiate a sealed class directly — only its subtypes.
      <br><br>
      Use <code>sealed</code> for algebraic data types: a fixed set of variants that represent all possible states. Network results, UI states, command types, AST nodes.

  - id: combining-modifiers
    title: "Combining modifiers — valid combinations"
    summary: |
      Modifiers can be combined, but not all combinations are valid. Here are the useful ones:
      <br><br>
      <pre><code>// Abstract + base: extend only, no instantiation&#10;abstract base class Widget { }&#10;&#10;// Abstract + interface: implement only, no instantiation&#10;abstract interface class Serializable { }&#10;&#10;// Abstract + final: can only be used within the library&#10;abstract final class InternalBase { }&#10;&#10;// Base + mixin: extend or mix in, no implementing&#10;base mixin class Disposable { }&#10;&#10;// Interface + mixin: NOT VALID — conflicting constraints</code></pre>
      The logic: <code>abstract</code> removes instantiation. <code>base</code> removes implementing. <code>interface</code> removes extending. <code>final</code> removes both. <code>sealed</code> restricts subtypes to the same file.
      <br><br>
      When choosing, ask yourself:
      <br><br>
      <strong>Should users extend this?</strong> If no, use <code>interface</code> or <code>final</code>.
      <br>
      <strong>Should users implement this?</strong> If no, use <code>base</code> or <code>final</code>.
      <br>
      <strong>Should users instantiate this?</strong> If no, use <code>abstract</code>.
      <br>
      <strong>Are all subtypes known?</strong> If yes, use <code>sealed</code>.
