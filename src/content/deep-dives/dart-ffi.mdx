  - id: why-ffi-exists
    title: "Why FFI exists — when Dart isn't enough"
    summary: |
      Dart runs inside a virtual machine. The VM handles memory, checks types, and keeps us safe from the kind of bugs that crash airplanes. That's good. But sometimes we need to escape.
      <br><br>
      <strong>The four reasons to reach for FFI:</strong>
      <br><br>
      <strong>1. Existing C libraries.</strong> Decades of battle-tested code exists in C — compression (zlib), encryption (OpenSSL), image processing (libpng), databases (SQLite). Rewriting them in Dart would take years and introduce bugs. FFI lets us call them directly.
      <br><br>
      <strong>2. Performance-critical code.</strong> The Dart VM is fast, but C is faster. For tight loops processing millions of data points — audio DSP, physics simulations, machine learning inference — the overhead of garbage collection and bounds checking matters. Native code skips all of it.
      <br><br>
      <strong>3. Platform APIs.</strong> Operating systems expose their functionality through C. Want to read battery level on iOS, enumerate Bluetooth devices on Android, or call Windows system APIs? The OS speaks C. FFI translates.
      <br><br>
      <strong>4. Hardware access.</strong> Sensors, GPUs, USB devices — they all have C drivers. To talk to a barcode scanner or control a robotic arm, we go through native code.

      <br><br>
      <strong>The trade-off.</strong> FFI gives us power, but it takes away safety. Inside the Dart VM, null pointer access throws an exception. In native code, it crashes the process. Memory leaks are impossible in pure Dart — the garbage collector handles everything. In FFI, we allocate and free manually. One mistake and we leak gigabytes.
      <br><br>
      <pre><code>// Pure Dart — safe, managed, automatic
      var list = [1, 2, 3];   // GC will clean this up

      // FFI — manual memory management, no safety net
      final ptr = calloc&lt;Int32&gt;(3);   // we own this memory
      // ... use it ...
      calloc.free(ptr);                // forget this = memory leak</code></pre>
      That's the deal. FFI lets us escape the sandbox, but we become responsible for everything the sandbox was protecting us from.

      <br><br>
      <strong>Loading a library at runtime.</strong>
      <br><br>
      <pre><code>import 'dart:ffi';
      import 'dart:io' show Platform;

      DynamicLibrary loadLibrary() {
        if (Platform.isMacOS || Platform.isIOS) {
          return DynamicLibrary.open('libsqlite3.dylib');
        } else if (Platform.isAndroid || Platform.isLinux) {
          return DynamicLibrary.open('libsqlite3.so');
        } else if (Platform.isWindows) {
          return DynamicLibrary.open('sqlite3.dll');
        }
        throw UnsupportedError('Unsupported platform');
      }

      void main() {
        final lib = loadLibrary();
        print('Library loaded: $lib');
      }</code></pre>
      <strong>DynamicLibrary.process()</strong> gives access to symbols already loaded in the current process — useful for system libraries that are linked at compile time.
      <br><br>
      <strong>DynamicLibrary.executable()</strong> returns symbols from the main executable itself — used when native code is statically linked into the app.
      <br><br>
      <strong>Bundling with Flutter.</strong> For Flutter apps, native libraries go in specific directories per platform:
      <br><br>
      <pre><code>// iOS: ios/Frameworks/libfoo.framework
      // Android: android/app/src/main/jniLibs/&lt;arch&gt;/libfoo.so
      // macOS: macos/Frameworks/libfoo.dylib
      // Linux: linux/libs/libfoo.so
      // Windows: windows/libs/foo.dll</code></pre>
      <strong>Common errors.</strong> The most frequent mistake is a path issue. <code>DynamicLibrary.open()</code> throws <code>ArgumentError</code> if the library isn't found. On Android, the path must be just the filename — Android's linker searches the right directories. On desktop, we often need an absolute path or to place the library in the working directory.
      <br><br>
      <pre><code>// Wrong on Android — full paths don't work
      DynamicLibrary.open('/data/app/libfoo.so');  // fails

      // Right on Android — just the filename
      DynamicLibrary.open('libfoo.so');  // linker finds it

      // Desktop — relative to working directory, or absolute path
      DynamicLibrary.open('./libs/libfoo.so');</code></pre>

      <strong>Pointer&lt;T&gt;</strong> is the core type. The type parameter tells us what's at that address.
      <br><br>
      <pre><code>import 'dart:ffi';
      import 'package:ffi/ffi.dart';  // for calloc/malloc

      void main() {
        // Allocate space for one Int32
        final ptr = calloc&lt;Int32&gt;();
        ptr.value = 42;
        print(ptr.value);  // 42
        print(ptr.address); // something like 140732920829952

        // Allocate an array of 10 Int32s
        final array = calloc&lt;Int32&gt;(10);
        array[0] = 100;
        array[9] = 999;

        // Don't forget to free!
        calloc.free(ptr);
        calloc.free(array);
      }</code></pre>
      <strong>calloc vs malloc.</strong> Both allocate memory. <code>calloc</code> zeroes the memory first; <code>malloc</code> leaves it uninitialised (faster, but we must write before reading). Both come from <code>package:ffi</code>.
      <br><br>
      <strong>nullptr — the null pointer.</strong> In C, null is just the address 0. Dart FFI represents this with <code>nullptr</code>. Dereferencing it crashes.
      <br><br>
      <pre><code>Pointer&lt;Int32&gt; ptr = nullptr;
      print(ptr.address);  // 0
      // ptr.value;  // CRASH — null pointer dereference</code></pre>
      <strong>Pointer arithmetic.</strong> We can move a pointer forward or backward with <code>elementAt()</code> or array indexing.
      <br><br>
      <pre><code>final array = calloc&lt;Int32&gt;(5);
      final third = array.elementAt(2);  // pointer to array[2]
      third.value = 777;
      print(array[2]);  // 777</code></pre>
      <strong>The Arena pattern — automatic cleanup.</strong> Manual free calls are error-prone. The <code>Arena</code> allocator collects all allocations and frees them at once when we're done.
      <br><br>
      <pre><code>import 'package:ffi/ffi.dart';

      void processData() {
        using((Arena arena) {
          final buffer = arena&lt;Uint8&gt;(1024);
          final name = arena&lt;Utf8&gt;();
          // ... use buffer and name ...
          // No need to free — Arena handles it when the block exits
        });
      }</code></pre>

      <strong>The key insight:</strong> <code>NativeType</code> classes (<code>Int32</code>, <code>Double</code>, etc.) are compile-time markers. We don't create instances of them. We use them as type parameters: <code>Pointer&lt;Int32&gt;</code>, <code>Pointer&lt;Double&gt;</code>.
      <br><br>
      <strong>Strings require conversion.</strong> Dart strings are UTF-16 internally. C strings are typically UTF-8, null-terminated. The <code>package:ffi</code> helpers handle conversion.
      <br><br>
      <pre><code>import 'package:ffi/ffi.dart';

      // Dart String → Native UTF-8
      final nativeString = 'Hello, FFI!'.toNativeUtf8();
      print(nativeString.address);  // raw pointer address

      // Native UTF-8 → Dart String
      String dartString = nativeString.toDartString();
      print(dartString);  // 'Hello, FFI!'

      // Don't forget to free the allocated memory
      calloc.free(nativeString);</code></pre>
      <strong>Arrays.</strong> C arrays are just contiguous memory. We access them through pointer indexing.
      <br><br>
      <pre><code>final arr = calloc&lt;Float&gt;(100);
      for (var i = 0; i &lt; 100; i++) {
        arr[i] = i * 1.5;
      }
      print(arr[50]);  // 75.0
      calloc.free(arr);</code></pre>
      <strong>Unsigned types.</strong> For unsigned integers, use <code>Uint8</code>, <code>Uint16</code>, <code>Uint32</code>, <code>Uint64</code>. Dart's <code>int</code> is signed, so values get reinterpreted as unsigned when crossing the boundary.

  - id: calling-c-functions
    title: "Calling C functions — lookupFunction and signatures"
    summary: |
      We've loaded the library and understand the types. Now we call functions. The key method is <code>lookupFunction</code>, and it has two type parameters that often confuse newcomers.
      <br><br>
      <pre><code>// C function we want to call:
      // int add(int a, int b);

      // In Dart:
      import 'dart:ffi';

      // 1. Native signature — how C sees it
      typedef AddNative = Int32 Function(Int32 a, Int32 b);

      // 2. Dart signature — how Dart sees it
      typedef AddDart = int Function(int a, int b);

      void main() {
        final lib = DynamicLibrary.open('libmath.so');

        // Look up the function, casting between signatures
        final add = lib.lookupFunction&lt;AddNative, AddDart&gt;('add');

        // Now call it like a normal Dart function
        print(add(40, 2));  // 42
      }</code></pre>
      <strong>Why two type parameters?</strong> The native signature uses FFI types (<code>Int32</code>, <code>Float</code>, <code>Pointer</code>) — these describe the C ABI. The Dart signature uses Dart types (<code>int</code>, <code>double</code>, <code>Pointer</code>) — these are what we actually work with in Dart code. The two must match in structure, but use different type vocabularies.

      <strong>Calling functions that take pointers.</strong> Many C APIs pass data through pointers. We allocate memory, pass the pointer, and read the result.
      <br><br>
      <pre><code>// C: void get_version(int* major, int* minor);
      typedef GetVersionNative = Void Function(Pointer&lt;Int32&gt;, Pointer&lt;Int32&gt;);
      typedef GetVersionDart = void Function(Pointer&lt;Int32&gt;, Pointer&lt;Int32&gt;);

      void main() {
        final lib = DynamicLibrary.open('libfoo.so');
        final getVersion = lib.lookupFunction&lt;GetVersionNative, GetVersionDart&gt;('get_version');

        // Allocate two integers to receive the output
        final major = calloc&lt;Int32&gt;();
        final minor = calloc&lt;Int32&gt;();

        getVersion(major, minor);
        print('Version: ${major.value}.${minor.value}');

        calloc.free(major);
        calloc.free(minor);
      }</code></pre>
      <strong>Error handling from C.</strong> C functions often return error codes. We check them and throw Dart exceptions.
      <br><br>
      <pre><code>// C: int open_file(const char* path);  // returns -1 on error
      final result = openFile(path);
      if (result &lt; 0) {
        throw FileSystemException('Failed to open file');
      }</code></pre>
      <strong>Variadic functions.</strong> C's variadic functions (<code>printf</code>, etc.) require special handling — they need <code>@FfiNative</code> annotations or generated bindings. Manual FFI can't handle them directly because the ABI varies by argument count.

      <strong>NativeCallable</strong> creates a native function pointer that, when called from C, invokes a Dart function.
      <br><br>
      <pre><code>import 'dart:ffi';

      // The C function type for our callback
      typedef ProgressCallback = Void Function(Int32 percent);

      // The Dart function we want C to call
      void onProgress(int percent) {
        print('Progress: $percent%');
      }

      void main() {
        // Wrap the Dart function as a native callable
        final callback = NativeCallable&lt;ProgressCallback&gt;.isolateLocal(
          onProgress,
        );

        // Get the pointer to pass to C
        final ptr = callback.nativeFunction;

        // Pass ptr to the C library...
        // When C calls ptr(50), onProgress(50) runs in Dart

        // When done, close the callable to free resources
        callback.close();
      }</code></pre>
      <strong>isolateLocal vs listener.</strong> <code>NativeCallable.isolateLocal</code> creates a callback that must be called from the same isolate. <code>NativeCallable.listener</code> creates one that can be called from any thread — the call gets queued to the isolate's event loop.
      <br><br>
      <strong>The closure limitation.</strong> Callbacks passed to C cannot be closures that capture local variables. The underlying mechanism doesn't support carrying extra state. If we need state, we must use a global or static variable, or pass user data through a <code>void*</code> parameter that C passes back to us.
      <br><br>
      <pre><code>// This won't work — captures 'prefix'
      // void logWithPrefix(String prefix) {
      //   final callback = NativeCallable&lt;...&gt;.isolateLocal(
      //     (int n) => print('$prefix: $n'),  // closure!
      //   );
      // }

      // Instead, use a top-level or static function
      void onEvent(int code) {
        print('Event: $code');
      }

      final callback = NativeCallable&lt;...&gt;.isolateLocal(onEvent);</code></pre>
      <strong>Pointer.fromFunction — the older approach.</strong> Before <code>NativeCallable</code>, we used <code>Pointer.fromFunction</code>. It still works but has limitations: the function must be a top-level or static function (not a lambda), and we must provide an <code>exceptionalReturn</code> value for error cases.
      <br><br>
      <pre><code>static int compare(int a, int b) => a - b;

      void main() {
        // exceptionalReturn is returned if the Dart code throws
        final ptr = Pointer.fromFunction&lt;Int32 Function(Int32, Int32)&gt;(
          compare,
          0,  // exceptionalReturn value
        );
      }</code></pre>

      <br>• Fields must be <code>external</code> — they're backed by native memory
      <br>• Each field needs a type annotation: <code>@Int32()</code>, <code>@Double()</code>, <code>@Pointer()</code>, etc.

        external Point topLeft;

        rect.ref.topLeft.y = 0;
        rect.ref.bottomRight.x = 100;
        rect.ref.bottomRight.y = 50;

        calloc.free(rect);
      }</code></pre>
      <strong>Passing structs by value vs by pointer.</strong> Some C functions take structs by value (the whole struct is copied onto the stack). Others take pointers. FFI supports both.
      <br><br>

      typedef DistanceNative = Double Function(Point a, Point b);
      typedef DistanceDart = double Function(Point a, Point b);

      // C: void scale(Point* p, double factor);  // by pointer
      typedef ScaleNative = Void Function(Pointer&lt;Point&gt;, Double);
      typedef ScaleDart = void Function(Pointer&lt;Point&gt;, double);</code></pre>
      <strong>Arrays in structs.</strong> Fixed-size arrays use the <code>@Array</code> annotation.
      <br><br>
      <pre><code>final class Matrix3x3 extends Struct {
        @Array(9)
        external Array&lt;Float&gt; values;
      }

      void main() {
        final m = calloc&lt;Matrix3x3&gt;();
        m.ref.values[0] = 1.0;  // identity diagonal
