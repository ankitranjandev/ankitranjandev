  - id: the-problem-sealed-classes-solve
    title: "The problem — switches that lie"
    summary: |
      Consider a network request. It can succeed, fail, or still be loading. We model this with inheritance:
      <br><br>
      <pre><code>abstract class NetworkState {}

      class Loading extends NetworkState {}
      class Success extends NetworkState {
        final String data;
        Success(this.data);
      }
      class Error extends NetworkState {
        final String message;
        Error(this.message);
      }</code></pre>
      Now we write a switch to handle all cases:
      <br><br>
      <pre><code>String describe(NetworkState state) {
        switch (state) {
          case Loading():
            return 'Loading...';
          case Success(:var data):
            return 'Got: $data';
          case Error(:var message):
            return 'Failed: $message';
        }
      }</code></pre>
      This looks complete. But the compiler doesn't know that. It sees an abstract class with three known subtypes — but someone could add a fourth subtype tomorrow. A new file could declare <code>class Timeout extends NetworkState {}</code> and suddenly our switch is missing a case.
      <br><br>
      The compiler forces us to add a default case, just in case:
      <br><br>
      <pre><code>default:
          throw UnimplementedError();</code></pre>
      That default is a lie. We don't expect it to run. But we have to write it anyway.

  - id: sealed-class-syntax
    title: "Sealed class syntax and rules"
    summary: |
      The fix is one word: <code>sealed</code>.
      <br><br>
      <pre><code>sealed class NetworkState {}

      class Loading extends NetworkState {}
      class Success extends NetworkState {
        final String data;
        Success(this.data);
      }
      class Error extends NetworkState {
        final String message;
        Error(this.message);
      }</code></pre>
      Now the compiler knows the complete set of subtypes. And now our switch needs no default:
      <br><br>
      <pre><code>String describe(NetworkState state) {
        return switch (state) {
          Loading() => 'Loading...',
          Success(:var data) => 'Got: $data',
          Error(:var message) => 'Failed: $message',
        };
      }</code></pre>
      No default. No <code>throw UnimplementedError()</code>. The compiler has verified that every possible case is handled.
      <br><br>
      <strong>The rules of sealed classes:</strong>
      <br><br>
      1. <strong>All subtypes must be in the same library.</strong> The sealed class and all its direct children must be in the same file (or the same library if you use <code>part</code> files). This is how the compiler knows it has seen everything.
      <br><br>
      2. <strong>Sealed classes are implicitly abstract.</strong> You cannot write <code>NetworkState()</code> — only the subtypes can be instantiated.
      <br><br>
      3. <strong>Subtypes can be classes, sealed classes, or final classes.</strong> You have full control over what each subtype allows.
      <br><br>
      <pre><code>sealed class Shape {}

      class Circle extends Shape {
        final double radius;
        Circle(this.radius);
      }

      class Rectangle extends Shape {
        final double width, height;
        Rectangle(this.width, this.height);
      }

      // ERROR: This is in a different file
      class Triangle extends Shape {}</code></pre>
      The last line is a compile error. Triangle cannot extend Shape because Shape is sealed and Triangle is in a different library. The compiler can now guarantee that Shape is either Circle or Rectangle — nothing else.

  - id: exhaustive-switching
    title: "Exhaustive switching — the compiler proves completeness"
    summary: |
      The real power of sealed classes shows up in switches. When the compiler knows all possible subtypes, it can check that we handle every single one.
      <br><br>
      <pre><code>sealed class Result&lt;T&gt; {}

      class Ok&lt;T&gt; extends Result&lt;T&gt; {
        final T value;
        Ok(this.value);
      }

      class Err&lt;T&gt; extends Result&lt;T&gt; {
        final String error;
        Err(this.error);
      }

      // This compiles — both cases handled
      T unwrap&lt;T&gt;(Result&lt;T&gt; result) {
        return switch (result) {
          Ok(:var value) => value,
          Err(:var error) => throw Exception(error),
        };
      }</code></pre>
      But watch what happens if we add a new subtype:
      <br><br>
      <pre><code>class Pending&lt;T&gt; extends Result&lt;T&gt; {}</code></pre>
      Now the switch in <code>unwrap</code> is incomplete. The compiler tells us:
      <br><br>
      <code>Error: The type 'Result&lt;T&gt;' is not exhaustively matched by the switch cases since it doesn't match 'Pending&lt;T&gt;()'.</code>
      <br><br>
      <svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 720 280" role="img" aria-label="Exhaustive switching with sealed classes" style="max-width: 100%; height: auto;"><style>.sc2-t{font-family:-apple-system,BlinkMacSystemFont,"Segoe UI",Inter,sans-serif;fill:#1e293b}.sc2-title{font-size:16px;font-weight:600}.sc2-mn{font-family:"SF Mono",Menlo,Monaco,monospace;font-size:12px}.sc2-label{font-size:11px;fill:#64748b}.sc2-note{font-size:11px;fill:#475569}.sc2-good{font-size:12px;fill:#059669;font-weight:600}</style><text x="360" y="26" text-anchor="middle" class="sc2-t sc2-title">sealed class — the compiler guarantees exhaustiveness</text><rect x="260" y="50" width="200" height="50" fill="#d1fae5" stroke="#10b981" stroke-width="2" rx="6"/><text x="360" y="82" text-anchor="middle" class="sc2-t sc2-mn">sealed class Result</text><line x1="310" y1="100" x2="200" y2="140" stroke="#10b981" stroke-width="2"/><line x1="410" y1="100" x2="520" y2="140" stroke="#10b981" stroke-width="2"/><rect x="100" y="140" width="200" height="45" fill="#dbeafe" stroke="#0175C2" stroke-width="2" rx="6"/><text x="200" y="168" text-anchor="middle" class="sc2-t sc2-mn">Ok</text><rect x="420" y="140" width="200" height="45" fill="#fee2e2" stroke="#dc2626" stroke-width="2" rx="6"/><text x="520" y="168" text-anchor="middle" class="sc2-t sc2-mn">Err</text><rect x="100" y="200" width="520" height="45" fill="#f0fdf4" stroke="#10b981" stroke-width="2" rx="6" stroke-dasharray="0"/><text x="360" y="228" text-anchor="middle" class="sc2-t sc2-mn">switch handles Ok and Err = exhaustive</text><text x="360" y="268" text-anchor="middle" class="sc2-t sc2-good">No default needed.   - id: sealed-vs-abstract-vs-final
    title: "Sealed vs abstract vs final"
    summary: |
      Dart 3 added several class modifiers. Let's see how they compare.
      <br><br>
      <strong>abstract class:</strong> Cannot be instantiated. Can be extended by anyone, anywhere. The compiler doesn't know all subtypes.
      <br><br>
      <strong>sealed class:</strong> Cannot be instantiated. Can only be extended in the same library. The compiler knows all subtypes. Enables exhaustive switching.
      <br><br>
      <strong>final class:</strong> Can be instantiated. Cannot be extended at all. There are no subtypes.
      <br><br>
      <strong>base class:</strong> Can be instantiated and extended. Cannot be implemented (no interfaces from this class).

  - id: modelling-state-with-sealed
    title: "Modelling state with sealed classes"
    summary: |
      Sealed classes are perfect for modelling state machines. Let's build some common patterns.
      <br><br>
      <strong>Pattern 1: Result type</strong>
      <br><br>
      The classic Result pattern encapsulates success or failure without using exceptions for control flow.
      <br><br>
      <pre><code>sealed class Result&lt;T&gt; {
        const Result();
      }

      final class Ok&lt;T&gt; extends Result&lt;T&gt; {
        final T value;
        const Ok(this.value);
      }

      final class Err&lt;T&gt; extends Result&lt;T&gt; {
        final Object error;
        final StackTrace? stackTrace;
        const Err(this.error, [this.stackTrace]);
      }

      // Usage
      Result&lt;int&gt; divide(int a, int b) {
        if (b == 0) return Err('Division by zero');
        return Ok(a ~/ b);
      }

      var result = divide(10, 2);
      var message = switch (result) {
        Ok(:var value) => 'Result: $value',
        Err(:var error) => 'Error: $error',
      };</code></pre>
      <strong>Pattern 2: Option type</strong>
      <br><br>
      Option (also called Maybe) represents a value that might not exist, without using null.
      <br><br>
      <pre><code>sealed class Option&lt;T&gt; {
        const Option();
      }

      final class Some&lt;T&gt; extends Option&lt;T&gt; {
        final T value;
        const Some(this.value);
      }

      final class None&lt;T&gt; extends Option&lt;T&gt; {
        const None();
      }

      // Usage
      Option&lt;User&gt; findUser(String id) {
        var user = database.lookup(id);
        return user != null ? Some(user) : None();
      }

      var greeting = switch (findUser('123')) {
        Some(:var value) => 'Hello, ${value.name}',
        None() => 'User not found',
      };</code></pre>
      <strong>Pattern 3: Async state</strong>
      <br><br>
      <pre><code>sealed class AsyncState&lt;T&gt; {}

      final class Initial&lt;T&gt; extends AsyncState&lt;T&gt; {}

      final class Loading&lt;T&gt; extends AsyncState&lt;T&gt; {}

      final class Loaded&lt;T&gt; extends AsyncState&lt;T&gt; {
        final T data;
        Loaded(this.data);
      }

      final class Failed&lt;T&gt; extends AsyncState&lt;T&gt; {
        final String message;
        Failed(this.message);
      }

      // In a widget
      Widget build(AsyncState&lt;User&gt; state) {
        return switch (state) {
          Initial() => Text('Press button to load'),
          Loading() => CircularProgressIndicator(),
          Loaded(:var data) => Text('Hello, ${data.name}'),
          Failed(:var message) => Text('Error: $message'),
        };
      }</code></pre>
      These patterns are called <strong>algebraic data types</strong> (ADTs). The sealed class defines a sum type — a value that is one of several variants. Each variant can carry different data. The compiler ensures we handle every variant.

  - id: sealed-with-patterns
    title: "Sealed classes with pattern matching"
    summary: |
      Dart 3 introduced both sealed classes and pattern matching. They work beautifully together.
      <br><br>
      <strong>Destructuring in switch cases:</strong>
      <br><br>
      <pre><code>sealed class Shape {}

      final class Circle extends Shape {
        final double radius;
        Circle(this.radius);
      }

      final class Rectangle extends Shape {
        final double width, height;
        Rectangle(this.width, this.height);
      }

      double area(Shape shape) {
        return switch (shape) {
          Circle(:var radius) => 3.14159 * radius * radius,
          Rectangle(:var width, :var height) => width * height,
        };
      }</code></pre>
      The <code>:var radius</code> syntax extracts the field into a local variable. No casting, no temporary variables.
      <br><br>
      <strong>Guards for additional conditions:</strong>
      <br><br>
      <pre><code>String describe(Shape shape) {
        return switch (shape) {
          Circle(radius: var r) when r > 100 => 'Large circle',
          Circle(radius: var r) when r > 10 => 'Medium circle',
          Circle() => 'Small circle',
          Rectangle(width: var w, height: var h) when w == h => 'Square',
          Rectangle() => 'Rectangle',
        };
      }</code></pre>
      The <code>when</code> clause adds a guard. The case only matches if both the type matches and the guard is true.
      <br><br>
      <strong>Nested sealed hierarchies:</strong>
      <br><br>
      <pre><code>sealed class Event {}

      sealed class UserEvent extends Event {}
      final class UserLoggedIn extends UserEvent {
        final String userId;
        UserLoggedIn(this.userId);
      }
      final class UserLoggedOut extends UserEvent {}

      sealed class SystemEvent extends Event {}
      final class SystemStarted extends SystemEvent {}
      final class SystemShutdown extends SystemEvent {}

      void handleEvent(Event event) {
        switch (event) {
          case UserLoggedIn(:var userId):
            print('User $userId logged in');
          case UserLoggedOut():
            print('User logged out');
          case SystemStarted():
            print('System started');
          case SystemShutdown():
            print('System shutting down');
        }
      }</code></pre>
      The switch is exhaustive because it covers all leaf types in the hierarchy.

  - id: real-world-examples
    title: "Real-world examples"
    summary: |
      Let's see sealed classes in action with patterns from real Flutter apps.
      <br><br>
      <strong>Example 1: Navigation events</strong>
      <br><br>
      <pre><code>sealed class NavigationEvent {}

      final class PushRoute extends NavigationEvent {
        final String routeName;
        final Object? arguments;
        PushRoute(this.routeName, [this.arguments]);
      }

      final class PopRoute extends NavigationEvent {}

      final class ReplaceRoute extends NavigationEvent {
        final String routeName;
        ReplaceRoute(this.routeName);
      }

      final class PopUntil extends NavigationEvent {
        final bool Function(Route) predicate;
        PopUntil(this.predicate);
      }

      void handleNavigation(NavigationEvent event, NavigatorState navigator) {
        switch (event) {
          case PushRoute(:var routeName, :var arguments):
            navigator.pushNamed(routeName, arguments: arguments);
          case PopRoute():
            navigator.pop();
          case ReplaceRoute(:var routeName):
            navigator.pushReplacementNamed(routeName);
          case PopUntil(:var predicate):
            navigator.popUntil(predicate);
        }
      }</code></pre>
      <strong>Example 2: Form validation</strong>
      <br><br>
      <pre><code>sealed class ValidationResult {}

      final class Valid extends ValidationResult {}

      final class Invalid extends ValidationResult {
        final List&lt;String&gt; errors;
        Invalid(this.errors);
      }

      ValidationResult validateEmail(String email) {
        var errors = &lt;String&gt;[];
        if (email.isEmpty) errors.add('Email is required');
        if (!email.contains('@')) errors.add('Invalid email format');
        return errors.isEmpty ? Valid() : Invalid(errors);
      }

      Widget buildEmailField(ValidationResult validation) {
        return switch (validation) {
          Valid() => TextField(decoration: InputDecoration(
              border: OutlineInputBorder(),
              suffixIcon: Icon(Icons.check, color: Colors.green),
            )),
          Invalid(:var errors) => TextField(decoration: InputDecoration(
              border: OutlineInputBorder(borderSide: BorderSide(color: Colors.red)),
              errorText: errors.first,
            )),
        };
      }</code></pre>
