  - id: the-problem-adding-one-thing
    title: "The problem — adding one thing without a subclass for every mix"
    summary: |
      You have a piece of text on screen. Now you want some padding around it. Fine — but then you want a background behind it too. And on one screen, a bit of transparency. The text never changes. You're only stacking on extras, one at a time.
      <br><br>
      The first instinct is a subclass per look.
      <br><br>
      <pre><code>class PaddedText extends Text { ... }
      class BorderedText extends Text { ... }
      class PaddedBorderedText extends Text { ... }
      // and one more class every time you mix two features</code></pre>
      It works for the first two. Then someone wants padded-and-transparent, and bordered-and-transparent, and all three at once. Each feature you add doesn't append a class — it <em>doubles</em> the table, because every existing combination now needs a transparent twin.
      <br><br>
      That's the trap: features that should combine freely are welded into a class name. Three independent extras shouldn't cost you eight classes. They should cost you three.
      <br><br>
      The thing you want to vary — the decoration — is tangled into the thing that doesn't, the text. Let's pull them apart, the same way Strategy did for behaviour.

  - id: the-decorator-pattern
    title: "The pattern — wrap an object in one that shares its interface"
    summary: |
      Here's the move. Instead of subclassing the component, you <em>wrap</em> it in another object that has the <strong>same interface</strong>, adds its one thing, then hands the call down to the wrapped child.
      <br><br>
      That sentence carries the whole pattern. Same interface, so the wrapper is a drop-in for the thing it wraps. Adds its one thing. Then delegates — calls the child to do the rest.

      <br><br>
      In code, the component and the decorator both implement one interface. The decorator holds a child of that same type and adds a step around the delegated call.
      <br><br>
      <pre><code>abstract interface class Greeter {
        String greet();
      }

      class PlainGreeter implements Greeter {
        @override
        String greet() =&gt; 'hello';
      }

      // A decorator: same interface, holds a Greeter, adds one thing
      class Shout implements Greeter {
        final Greeter inner;        // the wrapped child
        Shout(this.inner);

        @override
        String greet() =&gt; '\${inner.greet()}!';   // delegate, then add
      }

      class Loud implements Greeter {
        final Greeter inner;
        Loud(this.inner);

        @override
        String greet() =&gt; inner.greet().toUpperCase();
      }</code></pre>
      Because every piece is a <code>Greeter</code>, you can nest them — and because each only adds one thing, the order is yours to choose.
      <br><br>
      <pre><code>Greeter g = Loud(Shout(PlainGreeter()));
      print(g.greet());        // HELLO!</code></pre>
      No subclass of <code>PlainGreeter</code> was written. The base object never knew it was being dressed up.

  - id: flutter-is-decorators
    title: "Flutter is decorators all the way down"
    summary: |
      Here's the part that lands. You already use this pattern constantly — every time you wrap a widget in another widget that takes a single <code>child</code>.
      <br><br>
      Look at the cast. Each of these takes exactly one child and adds exactly one thing.
      <br>• <code>Padding</code> — adds empty space around its child.
      <br>• <code>Center</code> — positions its child in the middle.
      <br>• <code>DecoratedBox</code> — paints a background, border, or gradient behind its child.
      <br>• <code>Opacity</code> — makes its child see-through.
      <br>• <code>SizedBox</code> — pins its child to a fixed width and height.

      None of them subclass <code>Text</code> to gain a feature. They each hold a child and add their one job around it. That's the decorator pattern, named in widget form.
      <br><br>
      <pre><code>const Padding(

        child: DecoratedBox(
          decoration: BoxDecoration(color: Colors.amber),

      A wrapping widget is a decorator with a Flutter accent: it shares the <code>Widget</code> interface, takes a single child, adds one visual or layout concern, and leaves the child untouched. Once you see it, you can't unsee it.

  - id: composing-wrappers
    title: "Composing wrappers — the onion, and why order matters"
    summary: |
      Decorators nest. Wrap a wrapper in a wrapper and you get an onion of widgets, each layer adding its slice. You read it outside-in; it renders inside-out.
      <br><br>
      <pre><code>Padding(                          // outer: space around everything
        padding: const EdgeInsets.all(16),
        child: DecoratedBox(            // middle: background fill
          decoration: const BoxDecoration(color: Colors.amber),
          child: const Text('hello'),  // inner: the content
        ),
      )</code></pre>
      And here's the catch that bites people: <strong>order changes the result.</strong> These two are not the same picture.
      <br><br>
      <pre><code>// padding OUTSIDE the colour → amber box, gap around the box
      Padding(
        padding: const EdgeInsets.all(16),
        child: ColoredBox(color: Colors.amber, child: text),
      )

      // padding INSIDE the colour → amber fills the padding too
      ColoredBox(
        color: Colors.amber,
        child: Padding(
          padding: const EdgeInsets.all(16),
          child: text,
        ),
      )</code></pre>
      In the first, the padding sits outside the colour, so the amber stops at the box edge and there's a transparent gap around it. In the second, the colour wraps the padding, so the amber bleeds right out to fill that 16-pixel border too.
      <br><br>
      Same two decorators, swapped order, different screen. That's not a Flutter quirk — it's the nature of wrapping. Each layer acts on whatever the layer beneath it produced, so the sequence is part of the meaning.
      <br><br>
      <strong>Rule of thumb:</strong> read a wrapped tree from the inside out. The innermost child is your content; each layer outward is one transformation applied to everything so far.

      Decorator asks the composition question instead: what small parts is this dressed in?       <br><br>
      <strong>The trade Strategy and Decorator share:</strong> Strategy swaps the behaviour <em>inside</em> an object; Decorator stacks behaviour <em>around</em> one. Both refuse to model variation with a subclass per case, because that count always multiplies and composition only ever adds.

      <strong>Iterables.</strong> The same story, lazily. <code>list.map(...)</code> and <code>list.where(...)</code> return wrapping iterables that compute nothing until you walk them. Each is a thin <code>Iterable</code> wrapping the one before — a decorator chain you build with dots.
      <br><br>
      <strong>A logging wrapper.</strong> The trick works on your own interfaces too. Say you have a repository and want to log every call without touching the real one. Wrap it.
      <br><br>
      <pre><code>abstract interface class UserRepo {
        Future&lt;User&gt; fetch(int id);
      }

      class LoggingRepo implements UserRepo {
        final UserRepo inner;        // same interface, wrapped
        LoggingRepo(this.inner);

        @override
        Future&lt;User&gt; fetch(int id) async {
          print('fetch(\$id) called');   // add a step
          final user = await inner.fetch(id);  // delegate
          print('fetch(\$id) returned');
          return user;
        }
      }</code></pre>
      The rest of your app still depends on <code>UserRepo</code>. It can't tell whether it's holding the real repo or one dressed in logging — which is exactly the point. You can stack a caching wrapper on top of that, and a retry wrapper on top of <em>that</em>, each unaware of the others.

  - id: when-to-decorate
    title: "When to decorate, when to just configure"
    summary: |
      Decorator is sharp, but it isn't free, and reaching for it reflexively buries you in tree depth.
      <br><br>
      The honest question is: does the framework already give you a property for this? A great deal of what looks like decoration is just configuration. You don't wrap a <code>Text</code> in a colour widget to recolour it — you pass a <code>TextStyle</code>.
      <br><br>
      <pre><code>// configure: one widget, a property
      Text('hi', style: TextStyle(color: Colors.red))

      // decorate: a wrapper that adds a separate concern
      Padding(padding: ..., child: Text('hi'))</code></pre>
      <strong>Rule of thumb:</strong> if the extra is a setting on the object you already have, configure it. If it's a genuinely separate concern that should stack with others and apply to <em>any</em> child, decorate it.
      <br><br>
      Where decoration earns its keep:
      <br>• The concern is independent and reusable — spacing, a background, opacity, logging, caching — and applies to any child, not just this one.
      <br>• You want to combine concerns freely without a class per combination.
      <br>• You want to add the concern without modifying the thing you're wrapping.
      <br><br>
      Where it hurts:
      <br>• <strong>Deep trees.</strong> Six nested wrappers for one button is hard to read, and in Flutter every layer is a real element with a build cost. Reach for a property, a single richer widget like <code>Container</code> (which bundles common decorators for you), or pull the stack into a named widget of your own.
      <br>• <strong>Order confusion.</strong> Because wrapping order changes the result, a tall onion can hide bugs that a flat configuration wouldn't.
      <br><br>
      So you've now met two faces of composition. Strategy swaps the behaviour inside an object; Decorator stacks behaviour around it. Both beat a subclass for every case — and both are already in your hands the moment you wrap one widget in another.
