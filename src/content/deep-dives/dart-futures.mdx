  - id: why-async
    title: "Why we need asynchronous code"
    summary: |
      Some operations take time. A network request might take 500 milliseconds. Reading a large file might take 100 milliseconds. If our code just waited — did nothing else — the entire app would freeze.
      <br><br>
      <pre><code>// Hypothetical synchronous code (don't do this)
      var data = httpGet('https://api.example.com/users');  // blocks for 500ms
      print(data);  // nothing can happen until the above finishes</code></pre>
      In a Flutter app, blocking means no animations, no scrolling, no user input. The screen literally freezes. Users think the app has crashed.
      <br><br>
      <strong>Asynchronous code solves this.</strong> Instead of waiting, we say "start this operation and tell me when it's done." Meanwhile, other code can run — animations continue, buttons respond, the app stays alive.

  - id: what-is-a-future
    title: "What is a Future?"
    summary: |
      A <code>Future&lt;T&gt;</code> is a promise that a value of type T will be available eventually. It's not the value itself — it's a container that will hold the value when the operation completes.
      <br><br>
      <pre><code>Future&lt;String&gt; fetchUsername() {
        // Returns immediately with a Future
        // The actual string comes later
        return Future.delayed(
          Duration(seconds: 2),
          () => 'Ankit',
        );
      }

      void main() {
        var future = fetchUsername();
        print(future);         // Instance of 'Future&lt;String&gt;'
        print(future.runtimeType);  // Future&lt;String&gt;
      }</code></pre>
      A Future can be in one of three states:

  - id: then-catcherror
    title: "Handling Futures with then() and catchError()"
    summary: |
      The classic way to handle a Future is with callbacks. The <code>then()</code> method registers a callback that runs when the Future completes successfully.
      <br><br>
      <pre><code>fetchUsername().then((name) {
        print('Got name: \$name');
      });

      print('This prints first!');

      // Output:
      // This prints first!
      // Got name: Ankit  (after 2 seconds)</code></pre>
      Notice that "This prints first!" appears immediately. The <code>then()</code> callback doesn't block — it schedules code to run later.
      <br><br>
      <strong>Chaining Futures.</strong> If the callback returns a Future, the chain continues.
      <br><br>
      <pre><code>fetchUsername()
          .then((name) => fetchUserDetails(name))
          .then((details) => print(details));</code></pre>
      <strong>Catching errors.</strong> Use <code>catchError()</code> to handle failures.
      <br><br>
      <pre><code>fetchUsername()
          .then((name) => print('Hello, \$name'))
          .catchError((error) => print('Failed: \$error'));</code></pre>
      <strong>The finally equivalent.</strong> Use <code>whenComplete()</code> to run code regardless of success or failure.
      <br><br>
      <pre><code>fetchData()
          .then((data) => process(data))
          .catchError((e) => handleError(e))
          .whenComplete(() => hideLoadingSpinner());</code></pre>
      This callback style works, but it gets messy with complex logic. That's why Dart has async/await.

  - id: async-await
    title: "Async and await — making futures readable"
    summary: |
      The <code>async</code> and <code>await</code> keywords let us write asynchronous code that looks synchronous.
      <br><br>
      <pre><code>// With callbacks
      void loadData() {
        fetchUsername().then((name) {
          fetchDetails(name).then((details) {
            updateUI(details);
          });
        });
      }

      // With async/await
      Future&lt;void&gt; loadData() async {
        var name = await fetchUsername();
        var details = await fetchDetails(name);
        updateUI(details);
      }</code></pre>
      Same logic, but the second version reads top-to-bottom like synchronous code.
      <br><br>
      <strong>Rules of async/await:</strong>
      <br>• A function using <code>await</code> must be marked <code>async</code>
      <br>• An <code>async</code> function always returns a <code>Future</code>
      <br>• <code>await</code> pauses execution until the Future completes
      <br>• Code after <code>await</code> runs when the Future completes
      <br><br>
      <pre><code>Future&lt;int&gt; calculate() async {
        var a = await fetchA();   // pauses here
        var b = await fetchB();   // pauses here
        return a + b;             // runs after both complete
      }

      // Even returning a plain value, it's wrapped in a Future
      Future&lt;int&gt; simple() async {
        return 42;   // actually returns Future&lt;int&gt; that completes with 42
      }</code></pre>
      <strong>Error handling with try/catch.</strong> Errors in async code work just like synchronous code.
      <br><br>
      <pre><code>Future&lt;void&gt; loadData() async {
        try {
          var data = await fetchData();
          process(data);
        } catch (e) {
          print('Error: \$e');
        } finally {
          hideSpinner();
        }
      }</code></pre>

      The answer is the <strong>event loop</strong>. Dart maintains a queue of tasks. When your code awaits a Future, it's not blocking — it's scheduling the rest of the function to run later, when the Future completes.

      <strong>Two queues:</strong> Dart actually has two queues. The <strong>microtask queue</strong> has higher priority — it's drained completely before the event loop picks the next event. Most Future callbacks are microtasks.
      <br><br>
      <pre><code>void main() {
        print('1 - main starts');

        Future(() => print('4 - event queue'));

        Future.microtask(() => print('3 - microtask'));

        print('2 - main ends');
      }

      // Output:
      // 1 - main starts
      // 2 - main ends
      // 3 - microtask
      // 4 - event queue</code></pre>
      <strong>Warning:</strong> Long synchronous code blocks the event loop. No events are processed. The UI freezes. This is why heavy computation should go to isolates (covered in a later episode).

  - id: parallel-futures
    title: "Running Futures in parallel"
    summary: |
      Sequential awaits are easy to read, but sometimes we want operations to run simultaneously.
      <br><br>
      <pre><code>// Sequential — takes 3 seconds total
      var a = await fetchA();   // 1 second
      var b = await fetchB();   // 1 second
      var c = await fetchC();   // 1 second

      // Parallel — takes 1 second total (all run at once)
      var results = await Future.wait([
        fetchA(),
        fetchB(),
        fetchC(),
      ]);
      // results is a List: [resultA, resultB, resultC]</code></pre>
      <code>Future.wait()</code> takes a list of Futures and returns a Future that completes when all of them complete. The result is a list of all values.
      <br><br>
      <strong>Records for typed results.</strong> With Dart 3 records, we can do better:
      <br><br>
      <pre><code>Future&lt;(User, List&lt;Post&gt;, Settings)&gt; loadProfile(int userId) async {
        var (user, posts, settings) = await (
          fetchUser(userId),
          fetchPosts(userId),
          fetchSettings(userId),
        ).wait;
        return (user, posts, settings);
      }</code></pre>
      <strong>Error handling in parallel.</strong> If any Future fails, <code>Future.wait()</code> fails. Use <code>eagerError: false</code> to wait for all Futures regardless.
      <br><br>
      <pre><code>try {
        var results = await Future.wait(
          [fetchA(), fetchB(), fetchC()],
          eagerError: false,
        );
      } catch (e) {
        // At least one failed
      }</code></pre>
      <strong>First to complete.</strong> Use <code>Future.any()</code> when you only need the first result.
      <br><br>
      <pre><code>// Race multiple servers, use first response
      var fastest = await Future.any([
        fetchFromServer1(),
        fetchFromServer2(),
        fetchFromServer3(),
      ]);</code></pre>

  - id: creating-futures
    title: "Creating Futures"
    summary: |
      Most Futures come from libraries — HTTP clients, file I/O, timers. But sometimes we need to create our own.
      <br><br>
      <strong>Future.value()</strong> — a Future that's already complete.
      <br><br>
      <pre><code>Future&lt;int&gt; getCachedOrFetch(String key) {
        var cached = cache[key];
        if (cached != null) {
          return Future.value(cached);   // already have it
        }
        return fetchFromNetwork(key);    // need to fetch
      }</code></pre>
      <strong>Future.error()</strong> — a Future that's already failed.
      <br><br>
      <pre><code>Future&lt;User&gt; getUser(int id) {
        if (id < 0) {
          return Future.error(ArgumentError('ID must be positive'));
        }
        return fetchUser(id);
      }</code></pre>
      <strong>Future.delayed()</strong> — completes after a delay.
      <br><br>
      <pre><code>Future&lt;String&gt; fetchWithDelay() {
        return Future.delayed(
          Duration(seconds: 2),
          () => 'Data loaded',
        );
      }</code></pre>
      <strong>Completer</strong> — manual control over completion. Useful when bridging callback-based APIs to Future-based code.
      <br><br>
      <pre><code>Future&lt;String&gt; readFileCallback() {
        var completer = Completer&lt;String&gt;();

        legacyReadFile(
          'data.txt',
          onSuccess: (content) => completer.complete(content),
          onError: (error) => completer.completeError(error),
        );

        return completer.future;
      }</code></pre>
      <strong>Warning:</strong> A Completer can only be completed once. Calling <code>complete()</code> or <code>completeError()</code> twice throws an error.
