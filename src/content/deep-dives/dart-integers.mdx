  - id: smi
    title: "Smi — when the number lives inside the pointer"
    summary: |
      Every time we write <code>int x = 42</code>, Dart has a choice to make. It can store our number in a fast, lightweight way, or in a heavier, more flexible way — and that choice happens automatically, behind the scenes.
      <br><br>
      We might expect this: Dart allocates an object somewhere in memory holding the value 42, and our variable x holds a reference to that object. But here's the thing — 42 is small. So small that it can fit inside the pointer itself, no allocation needed.
      <br><br>
      This is the <strong>Smi (Small Integer)</strong> trick. A pointer on a 64-bit machine is 64 bits wide. Dart steals one bit to mark whether the pointer is "a real reference to an object" or "actually the number itself". For Smi values, the value lives directly in the pointer slot.

      <br><br>
      When we do arithmetic on a Smi, the CPU operates on the value directly. There's no following a pointer to find the data — the value is the pointer. This is as fast as integer math can be.
      <br><br>
      <strong>How small is "small enough"?</strong> It depends on the build. On 64-bit Dart with compressed pointers (which Flutter uses on mobile), a Smi can hold about ±1 billion. Without compressed pointers (desktop Dart), it can hold up to about ±4.6 quintillion. For most code this is more than enough.
      <br><br>
      The takeaway: Smi values are free. Zero heap allocation, zero indirection, full CPU speed.

  - id: mint
    title: "Mint — when the number outgrows the pointer"
    summary: |
      So what happens when our integer doesn't fit as a Smi anymore?
      <br><br>
      <pre><code>int x = 1 &lt;&lt; 62;  // way too big to be a Smi</code></pre>
      Now Dart has to <strong>box</strong> the value. Boxing means allocating a heap object, storing the 64-bit integer inside it, and putting a real pointer to that object in our variable.
      <br><br>
      We call this a <strong>Mint (Medium Integer)</strong>.

  - id: bigint
    title: "BigInt — when 64 bits aren't enough"
    summary: |
      <pre><code>BigInt huge = BigInt.parse("9999999999999999999999999999999");</code></pre>
      <code>BigInt</code> is a completely separate class. It is not "an int that grew" — we have to opt in by explicitly using <code>BigInt</code>.
      <br><br>
      A BigInt stores its value as an array of digits (typically 32-bit chunks under the hood). Want a number with 1000 digits? BigInt allocates a larger digit array. Every arithmetic operation involves allocations, comparisons, and carry propagation across digits. It is slow, but it is the only way to handle truly massive numbers.
      <br><br>
      <strong>When BigInt is the right tool:</strong> cryptography, scientific computation with arbitrary precision, and financial maths where overflow is unacceptable.
      <br><br>
      <strong>When BigInt is overkill:</strong> counters, IDs, scores, file sizes, timestamps — anything that fits in 64 bits.

  - id: collection-problem
    title: "The collection problem"
    summary: |
      Here is where things get interesting. Boxing is not usually a big deal for one number. It becomes a big deal for <strong>many</strong> numbers.
      <br><br>
      <pre><code>List&lt;int&gt; data = List.filled(1000000, 0);</code></pre>
      A list of one million ints. How is this stored?
      <br><br>
      The list itself is a contiguous block of slots — one per element. Each slot is either a Smi (value inline) or a reference to a Mint. If our values fit as Smis we are fine, but the list is still holding a million pointer-sized slots, when we may only need a million 4-byte or even 1-byte numbers.
      <br><br>
      For numeric work — image processing, audio, networking, game physics — this overhead adds up fast.
      <br><br>
      The solution is <strong>typed data</strong>.

  - id: typed-lists
    title: "Typed lists — packed and unboxed"
    summary: |
      Dart's <code>dart:typed_data</code> library is built for the case where we have lots of numbers and we want them stored efficiently.
      <br><br>
      <pre><code>import 'dart:typed_data';

      Int32List numbers = Int32List(1000000);
      numbers[0] = 42;
      numbers[1] = -100;</code></pre>
      The crucial difference: <code>Int32List</code> stores raw 32-bit integers, packed contiguously, with no boxing. A million of them is exactly 4 MB. There are no pointers, no Mint objects, no per-element overhead. The CPU can stream through them at maximum speed because they live next to each other in memory.

      <br>
      • 16-bit audio samples → <code>Int16List</code>
      <br>
      • Coordinate buffers for graphics → <code>Float32List</code>
      <br>
      • General numeric work where 64-bit is overkill → pick the smallest type that comfortably fits our range

  - id: byte-buffers
    title: "ByteBuffer and ByteData — down to the bytes"
    summary: |
      Sometimes we don't know in advance what types we'll read out of a chunk of bytes. We might be parsing a binary file format where the first 4 bytes are a magic number, the next 2 bytes are a version, the next 4 bytes are a length, and so on.
      <br><br>
      For that, Dart gives us two lower-level tools.
      <br><br>
      <strong>ByteBuffer</strong> is a raw buffer of bytes. We can't read from it directly — we always go through a view.
      <br><br>
      <strong>ByteData</strong> is a view over a ByteBuffer that lets us read or write any numeric type at any byte offset.
      <br><br>
      <pre><code>ByteData data = ByteData(8);
      data.setInt32(0, 42);        // write a 32-bit int at offset 0
      data.setFloat32(4, 3.14);    // write a 32-bit float at offset 4

      int x = data.getInt32(0);    // read it back
      double y = data.getFloat32(4);</code></pre>
      ByteData also lets us specify <strong>endianness</strong> — the order in which the bytes of a multi-byte value are arranged.
      <br><br>
      <pre><code>data.setInt32(0, 42, Endian.big);     // big-endian (network byte order)
      data.setInt32(0, 42, Endian.little);  // little-endian (most CPUs natively)</code></pre>
      This matters when reading binary formats. PNG files use big-endian. Most CPUs use little-endian internally. If we get this wrong, we read garbage.
      <br><br>
      We can also reinterpret the same bytes as a different typed view, with no copying:
      <br><br>
      <pre><code>Uint8List bytes = Uint8List(16);
      ByteBuffer buffer = bytes.buffer;
      Int32List asInts = buffer.asInt32List();  // same bytes, viewed as four 32-bit ints</code></pre>
      Both views look at the same underlying memory. Writing through one is visible through the other. This is incredibly powerful for performance — and a little dangerous, because we're working very close to the metal.

  - id: optimization
    title: "Putting it to work — optimization in practice"
    summary: |
      Let us make this practical. When should we reach for which representation?
      <br><br>
      <strong>For single values:</strong>
      <br>
      • Just use <code>int</code>. Dart will pick Smi or Mint for us.
      <br>
      • Avoid <code>BigInt</code> unless we actually need precision beyond 64 bits.
      <br>
      • In hot loops, try to keep values within Smi range. A counter that stays under a billion runs faster than one that doesn't.
      <br><br>
      <strong>For nullable ints:</strong>
      <br>
      • <code>int?</code> can force boxing because the slot has to also be able to hold null. Avoid <code>int?</code> in tight loops or large collections. Use a sentinel value (<code>-1</code>, <code>0</code>, <code>int.minValue</code>) if we can.
      <br><br>
      <strong>For collections of numbers:</strong>
      <br>
      • Use typed lists (<code>Int32List</code>, <code>Float64List</code>, etc.) instead of <code>List&lt;int&gt;</code> or <code>List&lt;double&gt;</code> whenever the collection is bigger than a few dozen elements.
      <br>
      • Pick the smallest type that fits our range. Don't use <code>Int64List</code> for byte data.
      <br>
      • For very large collections, the difference can be 4-8× memory and a similar factor in speed.
      <br><br>
      <strong>For binary I/O:</strong>
      <br>
      • File bytes, sockets, codecs → <code>Uint8List</code> is the workhorse.
      <br>
      • For parsing structured binary → <code>ByteData</code> with explicit endianness.
      <br><br>
      <strong>For native interop (FFI):</strong>
      <br>
      • Typed lists can be passed to C code without copying. The buffer is a contiguous block of memory that native code can read and write directly. Huge for performance when calling native libraries.
      <br><br>
      <strong>A practical example.</strong> Say we're loading a 1024×1024 grayscale image — one million pixels, each a value 0-255.
      <br><br>
      <pre><code>// The wrong way — boxes everything
      List&lt;int&gt; pixels = List.filled(1024 * 1024, 0);
      // Memory: ~8 MB (pointer per element on 64-bit), plus potential Mint objects

      // The right way — packed and unboxed
      Uint8List pixels = Uint8List(1024 * 1024);
      // Memory: exactly 1 MB</code></pre>
      That's an 8× memory saving, plus much faster iteration because the bytes are contiguous and the CPU's cache loves them. For a real image processing pipeline, the difference between these two choices is the difference between snappy and sluggish.
