  - id: the-duplication-problem
    title: "The same logic, written three times"
    summary: |
      Build a serious app today and you are likely building it more than once. An Android version in Kotlin, an iOS version in Swift, perhaps a web version in TypeScript. Each reimplements the same things: the networking, the data models, the validation rules, the business logic. Three teams, three bugs for every one mistake, three places to keep in sync. This duplication is one of the most expensive facts of modern app development.
      <br><br>
      <strong>Kotlin Multiplatform</strong> (KMP) attacks this with a specific philosophy, and it is worth contrasting it with the alternative you may know from the Dart/Flutter series. Flutter shares <em>everything</em>, including the UI, by shipping its own rendering engine and painting every pixel itself. KMP makes a different bet: share the <strong>logic</strong> — the parts that genuinely should behave identically everywhere — while letting each platform keep its <strong>native UI</strong>, native look, and native performance.
      <br><br>
      The result is that an iOS app built with KMP is still a real iOS app with SwiftUI on top; it just calls into shared Kotlin for its data and rules. You are not asking iOS developers to abandon their platform, only to stop rewriting the network layer. That pragmatism — share what should be shared, keep native what benefits from being native — is the same spirit that runs through the rest of Kotlin's design.

  - id: history-and-stability
    title: "From experiment to stable"
    summary: |
      Multiplatform was a long time coming, and its timeline tells you how seriously JetBrains treats stability. The underlying technology — <strong>Kotlin/Native</strong>, which compiles Kotlin to native binaries via the LLVM toolchain (introduced in the Foundations episode) — was announced around 2017. Multiplatform itself spent years labelled "experimental" and then "alpha," while the team hardened the tooling and the compiler.
      <br><br>
      Kotlin Multiplatform was finally declared <strong>Stable in November 2023</strong>, with Kotlin 1.9.20. That five-plus-year incubation mirrors the patient arc of Kotlin itself — announced in 2011, 1.0 only in 2016. JetBrains would rather ship late and stable than early and shaky.
      <br><br>
      Two developments cemented KMP's momentum. First, <strong>Google embraced it</strong>: at Google I/O 2024, Google announced official support for Kotlin Multiplatform for sharing business logic across Android and iOS, echoing the Android-first endorsement Kotlin received back in 2017. Second, JetBrains shipped <strong>Compose Multiplatform</strong>, extending the Jetpack Compose UI framework beyond Android to desktop, iOS, and the web — so teams that <em>do</em> want to share UI can, while those who prefer native UI still can. KMP stopped being a promising experiment and became a mainstream architecture choice.

  - id: source-sets
    title: "Source sets: common and platform code"
    summary: |
      A multiplatform project organises code into <strong>source sets</strong>. The most important is <code>commonMain</code> — code here is platform-agnostic Kotlin that compiles to <em>every</em> target. Alongside it sit platform-specific source sets like <code>androidMain</code>, <code>iosMain</code>, and <code>jvmMain</code>, which hold code that exists only for that platform.

      <br><br>
      When you build for Android, the compiler combines <code>commonMain</code> with <code>androidMain</code> and produces JVM bytecode packaged as an Android library. When you build for iOS, it combines <code>commonMain</code> with <code>iosMain</code> and produces, via Kotlin/Native, a native framework that an Xcode project can import like any other. The shared module is written once; each platform target compiles it together with its own platform source set.
      <br><br>
      This is why the compilation targets from the Foundations episode matter so much: KMP <em>is</em> those targets working in concert. The same <code>commonMain</code> source flows through Kotlin/JVM for Android, Kotlin/Native for iOS, and Kotlin/JS or Kotlin/Wasm for the web.

  - id: expect-actual
    title: "expect and actual"
    summary: |
      Common code sometimes needs something the platform must provide — the current time, a random-number source, secure storage, a device ID. The shared module cannot implement these itself, but it needs to <em>refer</em> to them. Kotlin's mechanism for this is the <code>expect</code>/<code>actual</code> pair, KMP's signature feature.
      <br><br>
      In <code>commonMain</code> you declare what you <strong>expect</strong> to exist, with no body. In each platform source set you provide the <strong>actual</strong> implementation.

      The compiler enforces the contract: if a target is missing an <code>actual</code> for some <code>expect</code>, the build fails. It works for functions, classes, properties, and more. In practice, modern KMP often prefers plain interfaces and dependency injection over <code>expect</code>/<code>actual</code> for larger abstractions, reserving the mechanism for small, leaf-level platform primitives — but it remains the conceptual heart of how shared code reaches platform capabilities.

  - id: what-to-share
    title: "What to share, and what not to"
    summary: |
      KMP's value depends on drawing the sharing line in the right place. The sweet spot is the <strong>logic that should behave identically everywhere</strong>, and where bugs are most costly to duplicate:
      <br><br>
      • <strong>Data models</strong> — your domain types, serialisation, validation.<br>
      • <strong>Networking</strong> — API clients (Ktor is the multiplatform HTTP client of choice).<br>
      • <strong>Business logic</strong> — rules, calculations, state machines, use cases.<br>
      • <strong>Local persistence</strong> — databases via multiplatform libraries like SQLDelight.<br>
      • <strong>Even ViewModels</strong> — increasingly, presentation logic is shared too.<br><br>
      A healthy ecosystem of multiplatform libraries has grown to support this: <strong>kotlinx.coroutines</strong> (the concurrency from the Concurrency episode runs on every target), <strong>kotlinx.serialization</strong>, <strong>Ktor</strong>, and <strong>SQLDelight</strong>. Because the shared layer is just Kotlin, all the language features from this series — null safety, sealed classes for state, Flow for streams — work uniformly across platforms.
      <br><br>
      What you typically do <em>not</em> share, unless you adopt Compose Multiplatform, is the UI and deeply platform-specific integrations (widgets, permissions, platform design languages). The guiding question is simple: "would I want this to differ per platform?" If the answer is no, it belongs in <code>commonMain</code>.

  - id: compose-multiplatform
    title: "Compose Multiplatform: sharing the UI too"
    summary: |
      For years, "share logic, keep native UI" was KMP's whole story. <strong>Compose Multiplatform</strong>, built by JetBrains on top of Google's Jetpack Compose, changed the menu by making shared UI an <em>option</em> rather than a requirement.
      <br><br>
      Jetpack Compose is Android's modern declarative UI toolkit — you describe the UI as functions of state (recall the lambdas-with-receiver and DSL ideas from the Functions episode; <code>Column { Text("hi") }</code> is exactly that pattern). Compose Multiplatform takes that same Kotlin UI code and renders it on desktop (JVM), iOS, and the web, in addition to Android.
      <br><br>
      <pre><code>// Shared UI — written once, runs on Android, iOS, desktop, web&#10;@Composable&#10;fun Greeting(name: String) {&#10;    Column {&#10;        Text("Hello, $name")&#10;        Button(onClick = { /* ... */ }) { Text("Tap me") }&#10;    }&#10;}</code></pre>
      This puts KMP and Flutter on more comparable footing for teams that want a single UI codebase — but with a crucial difference in heritage: Compose Multiplatform <em>is</em> Compose, the same framework Android developers already use, so an Android team's existing skills and UI carry straight over. The choice is now a spectrum: share only logic and write native UIs; or share logic <em>and</em> UI with Compose. Either way, the shared core is ordinary Kotlin — every concept from this series applies, unchanged, on every platform you ship to.
