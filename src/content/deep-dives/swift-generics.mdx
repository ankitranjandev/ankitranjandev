summary: |
  <strong>Generics</strong> let us write one algorithm that works over many types, with the compiler still checking every use.<br>
  <code>func swap&lt;T&gt;(_ a: inout T, _ b: inout T)</code> replaces a pile of hand-written <code>swapInt</code>, <code>swapString</code>, <code>swapDouble</code> functions.<br>
  Constraints like <code>&lt;T: Comparable&gt;</code> unlock operations on <code>T</code>; <code>where</code> clauses refine them further.<br>
  The standard library is generic to its core — <code>Array&lt;Element&gt;</code>, <code>Dictionary&lt;Key, Value&gt;</code>, <code>Optional&lt;Wrapped&gt;</code>, <code>Result&lt;Success, Failure&gt;</code>.<br>
  <strong>Swift generics are not type-erased.</strong> The compiler emits one shared body that receives type metadata and witness tables at runtime, and may specialise hot paths into concrete copies:<br>
  <pre><code>func max&lt;T: Comparable&gt;(_ a: T, _ b: T) -&gt; T {&#10;    a &gt; b ? a : b&#10;}&#10;// One body. Type of T arrives at runtime as metadata.&#10;// The optimiser may also stamp out a concrete max-for-Int.</code></pre>
topics:
  - id: the-duplication-problem
    title: "The duplication problem — one algorithm, many types"
    summary: |
      Let's start with a tiny task: swapping two values. In Swift, before generics enter the picture, we might reach for something like this.
      <br><br>
      <pre><code>func swapInts(_ a: inout Int, _ b: inout Int) {&#10;    let temp = a&#10;    a = b&#10;    b = temp&#10;}&#10;&#10;func swapStrings(_ a: inout String, _ b: inout String) {&#10;    let temp = a&#10;    a = b&#10;    b = temp&#10;}&#10;&#10;func swapDoubles(_ a: inout Double, _ b: inout Double) {&#10;    let temp = a&#10;    a = b&#10;    b = temp&#10;}</code></pre>
      Look at those three functions. The bodies are identical. The only thing that changes is the type name in the signature. Every new type we want to swap means another copy-paste, another place to fix a bug, another function to keep in sync. That's the duplication problem in miniature.
      <br><br>
      So what's the algorithm here, really? "Take two storage locations of the same type, exchange their contents." Nothing about that depends on whether the values are integers or strings. The shape of the code is the same for every type. We're just lacking a way to say "any type" without giving up the compiler's checks.
      <br><br>
      Generics are that way. We write the body once, with a placeholder standing in for the concrete type, and the caller supplies the real type at each call site.
      <br><br>
      <pre><code>func swap&lt;T&gt;(_ a: inout T, _ b: inout T) {&#10;    let temp = a&#10;    a = b&#10;    b = temp&#10;}&#10;&#10;var x = 1, y = 2&#10;swap(&amp;x, &amp;y)         // T is Int&#10;&#10;var s = "a", t = "b"&#10;swap(&amp;s, &amp;t)         // T is String</code></pre>

  - id: generic-functions-and-type-parameters
    title: "Generic functions and type parameters"
    summary: |
      That <code>&lt;T&gt;</code> after the function name is a <strong>type parameter</strong>. It's a placeholder, a stand-in for a real type that the caller will decide. The letter <code>T</code> is just a convention (it stands for "Type"); we could call it anything, and for clarity we often do — <code>Element</code>, <code>Key</code>, <code>Value</code>.
      <br><br>
      Here's a slightly meatier example: finding the index of the first value in an array that matches a target.
      <br><br>
      <pre><code>func firstIndex&lt;T: Equatable&gt;(of value: T, in array: [T]) -&gt; Int? {&#10;    for (index, element) in array.enumerated() {&#10;        if element == value {&#10;            return index&#10;        }&#10;    }&#10;    return nil&#10;}&#10;&#10;let names = ["Anna", "Bo", "Cleo"]&#10;firstIndex(of: "Bo", in: names)   // Optional(1), T is String&#10;firstIndex(of: 7, in: [3, 7, 9])  // Optional(1), T is Int</code></pre>
      Notice we never wrote <code>firstIndex&lt;String&gt;(...)</code> at the call site. We just passed a <code>String</code> and an array of strings, and Swift worked out that <code>T</code> must be <code>String</code>. This is <strong>type inference</strong> filling in the type parameter for us. The compiler looks at the arguments, sees what types they are, and binds <code>T</code> accordingly — once, consistently, for that whole call.
      <br><br>
      That consistency matters. Because both parameters use the same <code>T</code>, the compiler insists they agree. We cannot pass a <code>String</code> as the value and an array of <code>Int</code> as the haystack. Reusing the same type parameter is how we say "these must be the same type", and the compiler holds us to it.
      <br><br>
      And here's the question that leads us straight into the next idea. How did <code>element == value</code> even compile? Comparing two arbitrary values of an unknown type <code>T</code> with <code>==</code> shouldn't be allowed — not every type supports equality. The answer is that little <code>: Equatable</code> hanging off the type parameter. Without it, the function wouldn't build. Let's look at what that is.

  - id: constraints
    title: "Constraints — telling the compiler what T can do"
    summary: |
      Here's the catch with an unconstrained type parameter. If all the compiler knows is "some type <code>T</code>", then it cannot let us do much with a value of that type. We can move it around, store it, return it. But we cannot add it, compare it, or call methods on it, because <code>T</code> might be a type that doesn't support any of that.
      <br><br>
      <pre><code>func largest&lt;T&gt;(_ values: [T]) -&gt; T? {&#10;    var result = values.first&#10;    for value in values {&#10;        if value &gt; result! {  // error: T might not support &gt;&#10;            result = value&#10;        }&#10;    }&#10;    return result&#10;}</code></pre>
      The compiler is right to complain. What if <code>T</code> is some type with no notion of order? A <strong>constraint</strong> is how we promise the compiler that <code>T</code> supports a particular capability — and, in exchange, the compiler lets us use that capability inside the function.
      <br><br>
      <pre><code>func largest&lt;T: Comparable&gt;(_ values: [T]) -&gt; T? {&#10;    var result = values.first&#10;    for value in values {&#10;        if value &gt; result! {  // fine: Comparable guarantees &gt;&#10;            result = value&#10;        }&#10;    }&#10;    return result&#10;}&#10;&#10;largest([3, 1, 4, 1, 5])              // Optional(5)&#10;largest(["pear", "fig", "kiwi"])      // Optional("pear")</code></pre>
      Think of a constraint as a gate. Only types that satisfy it are allowed through to fill in <code>T</code>, and once inside, the constraint unlocks the operations we promised.

      <br><br>
      We can require more than one capability at once by combining constraints with <code>&amp;</code>.
      <br><br>
      <pre><code>func describe&lt;T: Comparable &amp; CustomStringConvertible&gt;(_ a: T, _ b: T) {&#10;    let bigger = a &gt; b ? a : b&#10;    print("The larger is \(bigger.description)")&#10;}</code></pre>
      And for richer requirements — especially ones that talk about the <em>associated</em> types we'll meet shortly — we reach for a <code>where</code> clause. A <code>where</code> clause lets us add conditions that a bare <code>: SomeProtocol</code> can't express.
      <br><br>
      <pre><code>func sum&lt;S: Sequence&gt;(_ values: S) -&gt; Int&#10;    where S.Element: Numeric, S.Element == Int {&#10;    values.reduce(0, +)&#10;}&#10;&#10;sum([1, 2, 3])          // 6&#10;sum(1...10)             // 55, works on a Range too</code></pre>
      The shorthand <code>&lt;T: Comparable&gt;</code> and the longer <code>where T: Comparable</code> mean the same thing; the <code>where</code> form simply has room for conditions about associated types and equality relationships. The rule of thumb: simple "this type conforms to that protocol" goes inline, anything about <code>Element</code> or type equality goes in a <code>where</code> clause.

  - id: generic-types
    title: "Generic types — building Stack, and a generic standard library"
    summary: |
      Functions aren't the only things that can be generic. Types can be too. Let's build the classic example, a stack — a last-in, first-out collection — that holds elements of any single type.
      <br><br>
      <pre><code>struct Stack&lt;Element&gt; {&#10;    private var items: [Element] = []&#10;&#10;    mutating func push(_ item: Element) {&#10;        items.append(item)&#10;    }&#10;&#10;    mutating func pop() -&gt; Element? {&#10;        items.popLast()&#10;    }&#10;&#10;    var top: Element? { items.last }&#10;    var count: Int { items.count }&#10;}&#10;&#10;var numbers = Stack&lt;Int&gt;()&#10;numbers.push(1)&#10;numbers.push(2)&#10;numbers.pop()             // Optional(2)&#10;&#10;var words = Stack&lt;String&gt;()&#10;words.push("hi")</code></pre>
      The type parameter <code>Element</code> sits right after the type name, and from then on it's a real type we can use for stored properties, parameters, and return types. A <code>Stack&lt;Int&gt;</code> and a <code>Stack&lt;String&gt;</code> are genuinely different types — you cannot accidentally push a string onto an integer stack, and the compiler proves it.
      <br><br>
      Here's the part that surprises people coming from older languages. Once you notice generic types, you start seeing them everywhere in Swift — because the standard library is generic right down to its foundations. The everyday types you use without thinking are all generic.
      <br><br>
      <pre><code>// Every one of these is a generic type:&#10;Array&lt;Element&gt;            //  [Int]   is sugar for Array&lt;Int&gt;&#10;Dictionary&lt;Key, Value&gt;    //  [K: V]  is sugar for Dictionary&lt;K, V&gt;&#10;Set&lt;Element&gt;&#10;Optional&lt;Wrapped&gt;         //  Int?    is sugar for Optional&lt;Int&gt;&#10;Result&lt;Success, Failure&gt;</code></pre>
      That little <code>?</code> for optionals? It's syntactic sugar for <code>Optional&lt;Wrapped&gt;</code>, an ordinary generic enum. The square brackets for arrays and dictionaries? Sugar for <code>Array&lt;Element&gt;</code> and <code>Dictionary&lt;Key, Value&gt;</code>. There is no special magic compiler type for "list" the way some languages have. Swift's collections are written in Swift, as generic types, using the very same tools available to us. When we write our own <code>Stack&lt;Element&gt;</code>, we are building exactly the kind of thing the standard library is built from.

  - id: associated-types
    title: "Associated types — the input on the other side"
    summary: |
      Generic types put their type parameter in angle brackets up front: <code>Stack&lt;Element&gt;</code>. Protocols play the same game, but from the other direction. A protocol can't take a type parameter in angle brackets. Instead it declares an <strong>associated type</strong> — a placeholder that each conforming type must fill in.
      <br><br>
      The canonical example is <code>Sequence</code>, the protocol behind every for-in loop in Swift.
      <br><br>
      <pre><code>protocol Sequence {&#10;    associatedtype Element&#10;    func makeIterator() -&gt; ...&#10;}&#10;&#10;extension Array: Sequence {&#10;    // Element is inferred from how Array uses it&#10;}</code></pre>
      The difference is one of direction. With a generic struct, <em>the caller</em> chooses the type — we write <code>Stack&lt;Int&gt;</code> and pick <code>Int</code> ourselves. With an associated type, <em>the conforming type</em> chooses. When <code>Array&lt;Int&gt;</code> conforms to <code>Sequence</code>, it fixes <code>Element</code> to <code>Int</code>; when <code>String</code> conforms, its <code>Element</code> is <code>Character</code>. The protocol just says "there will be an <code>Element</code>"; each adopter decides what it is.
      <br><br>
      That's why you write <code>some Sequence</code> rather than <code>Sequence&lt;Int&gt;</code> in older Swift — the protocol had no slot to put the <code>Int</code> in. You constrained the associated type indirectly through a <code>where</code> clause.
      <br><br>
      <pre><code>func total&lt;S&gt;(_ seq: S) -&gt; Int where S: Sequence, S.Element == Int {&#10;    seq.reduce(0, +)&#10;}</code></pre>
      That's verbose. So Swift 5.7 introduced <strong>primary associated types</strong> (SE-0346), which let a protocol promote one or more associated types into angle brackets, just like a generic type's parameter.
      <br><br>
      <pre><code>// The declaration now reads:&#10;protocol Sequence&lt;Element&gt; { associatedtype Element; ... }&#10;&#10;// Which gives us this lovely shorthand:&#10;func total(_ seq: some Sequence&lt;Int&gt;) -&gt; Int {&#10;    seq.reduce(0, +)&#10;}&#10;&#10;let xs: some Collection&lt;Int&gt; = [1, 2, 3]</code></pre>
      So <code>Collection&lt;Int&gt;</code> finally reads the way you'd hope — "a collection of <code>Int</code>" — even though under the hood it's still an associated type, not a type parameter. It's a piece of syntax that closed a long-standing readability gap between protocols and generic types.

  - id: the-runtime-model
    title: "The runtime model — metadata, witness tables, and specialisation"
    summary: |
      Now for the part that makes Swift's generics genuinely distinctive. The question every generics system has to answer is: when the program actually runs, what happens to the type? There are roughly three schools of thought, and Swift picks a fourth path between them.
      <br><br>
      <strong>Java erases.</strong> A <code>List&lt;String&gt;</code> and a <code>List&lt;Integer&gt;</code> compile down to the same raw <code>List</code>; the type argument is thrown away after the compiler has finished checking. At runtime there is no <code>String</code> in there at all, which is why you cannot ask <code>list instanceof List&lt;String&gt;</code> in Java.
      <br><br>
      <strong>C++ stamps out copies.</strong> A template is a recipe, and the compiler bakes a separate, fully concrete function or class for every type you ever use it with. <code>std::vector&lt;int&gt;</code> and <code>std::vector&lt;double&gt;</code> become two distinct compiled bodies. Fast, but it can lead to serious code bloat when many types are involved.
      <br><br>
      <strong>Swift keeps the type, and shares the code.</strong> By default the compiler emits <em>one</em> shared implementation of a generic function. It works for any type because the type's information — its size, how to copy it, how it conforms to a protocol — is passed in <em>at runtime</em> as <strong>type metadata</strong> and <strong>protocol witness tables</strong>. The witness table is essentially a lookup of "here is how this concrete type satisfies that protocol's requirements". So there's no erasure: the type really is present at runtime.

      <br><br>
      But passing metadata around has a cost. So Swift adds an escape hatch: <strong>specialisation</strong>. When the optimiser can see exactly which type a generic is being used with — and it often can, especially with <code>@inlinable</code> code — it stamps out a concrete, C++-style copy with the type baked in.   - id: power-features-and-trivia
    title: "Power features — conditional conformance and variadic generics"
    summary: |
      With the core in place, let's look at two of the more advanced corners, both of which you rely on constantly even if you never write them by hand.
      <br><br>
      <strong>Conditional conformance</strong> (SE-0143, Swift 4.1) lets a generic type conform to a protocol only when its type parameter does. The textbook example: an array is itself <code>Equatable</code> exactly when its elements are.
      <br><br>
      <pre><code>extension Array: Equatable where Element: Equatable {&#10;    // == compares element by element&#10;}&#10;&#10;[1, 2, 3] == [1, 2, 3]            // true — Int is Equatable&#10;&#10;// [SomeNonEquatable] == [...]    // won't compile — and rightly so</code></pre>
      This is why <code>[Int]</code> can be compared with <code>==</code> but an array of a type that isn't itself comparable cannot. The conformance is conditional on the element. The same trick gives you <code>[T]</code> being <code>Hashable</code> when <code>T</code> is, <code>Optional&lt;T&gt;</code> being <code>Equatable</code> when <code>T</code> is, and so on, all the way through the standard library.
      <br><br>
      <strong>Variadic generics</strong> (SE-0393, Swift 5.9) tackle a problem that plagued Swift for years: how do you write a function generic over an <em>arbitrary number</em> of type parameters? Before 5.9, the standard library was littered with hand-written overloads — a <code>zip</code> for two sequences, the tuple <code>==</code> defined separately for arity 2, 3, 4, 5, 6 — because there was no way to say "a tuple of any number of types".
      <br><br>
      <pre><code>// A parameter pack: zero or more type parameters at once&#10;func tie&lt;each T&gt;(_ values: repeat each T) -&gt; (repeat each T) {&#10;    (repeat each values)&#10;}&#10;&#10;tie(1, "two", 3.0)        // (Int, String, Double)&#10;tie(true)                 // (Bool)</code></pre>
      The <code>each T</code> declares a <em>parameter pack</em> — a placeholder for a whole sequence of type parameters — and <code>repeat</code> expands over it. One function now covers every arity, replacing what used to be a stack of copy-pasted overloads.
      <br><br>
      Step back and it all connects. SwiftUI's <code>View</code> hierarchy is generic from top to bottom — a <code>VStack</code> is generic over its <code>Content</code>, and the result builder that assembles your view tree leans on these exact features. The standard library you reach for every day — <code>Array</code>, <code>Dictionary</code>, <code>Optional</code>, <code>Result</code> — is generic to its core. Generics aren't a niche feature you opt into. In Swift, they are the material the language itself is made from.
