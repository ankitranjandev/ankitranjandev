    summary: |

      <br><br>
      Here's the part newcomers from Java or C# often miss: all three are <strong>generic structs</strong>, not classes. Their full names are <code>Array&lt;Element&gt;</code>, <code>Set&lt;Element&gt;</code>, and <code>Dictionary&lt;Key, Value&gt;</code>. The square-bracket syntax we write every day is just sugar — <code>[Int]</code> means <code>Array&lt;Int&gt;</code>, <code>[String: Int]</code> means <code>Dictionary&lt;String, Int&gt;</code>.
      <br><br>
      <pre><code>let langs: [String] = ["swift", "kotlin", "dart", "swift"]&#10;let unique: Set&lt;String&gt; = ["swift", "kotlin", "dart"]&#10;let released: [String: Int] = ["swift": 2014, "go": 2009]</code></pre>
      Because they are structs, they have <strong>value semantics</strong>. Assigning one collection to another, or passing it into a function, gives the recipient its own independent value — not a shared reference. That single design choice ripples through everything else in this episode, so hold onto it. It is the opposite of how arrays and maps behave in most object-oriented languages, where you pass a reference and a callee can mutate your data behind your back.

  - id: array-internals
    title: "Array internals — a contiguous buffer"
    summary: |
      An <code>Array</code> stores its elements in a single <strong>contiguous buffer</strong> in memory — one block, elements laid out back to back. That layout is why indexing is O(1): the machine multiplies the index by the element size, adds it to the buffer's base address, and reads. No searching involved.
      <br><br>
      So what happens when we append and the buffer is full? Swift allocates a new buffer, copies the old elements across, and frees the old one. To keep that from happening on every single append, it grows the buffer geometrically — roughly <strong>doubling the capacity</strong> each time it runs out of room.

      <br><br>
      Most appends just write into a spare slot, which is genuinely O(1). Once in a while an append triggers a doubling, which is O(n) because everything must be copied. Spread across all the appends, the average cost stays constant — that is what <strong>amortised O(1)</strong> means. Swift exposes two numbers that make this visible: <code>count</code> is how many elements we have, and <code>capacity</code> is how many the current buffer can hold before it must grow.
      <br><br>
      <pre><code>var nums = [Int]()&#10;print(nums.count, nums.capacity)   // 0 0&#10;nums.append(1)&#10;print(nums.count, nums.capacity)   // 1 1, then grows: 1, 3, 6, 12...&#10;&#10;// If we know the size, skip the doublings entirely:&#10;var ready = [Int]()&#10;ready.reserveCapacity(10_000)      // one allocation, no copies</code></pre>
      Two operations deserve a warning. <code>insert(_:at:)</code> and <code>remove(at:)</code> near the <strong>front</strong> of an array are O(n), because every element after the insertion point has to shuffle along to make room or close the gap. Appending and removing at the <strong>end</strong> are the cheap operations. If you find yourself inserting at index 0 in a loop, that loop is quietly O(n squared).

  - id: set-dictionary-internals
    title: "Set and Dictionary — hash tables under the hood"
    summary: |
      Array's <code>contains</code> has to walk the buffer element by element. Set and Dictionary do something cleverer: they are <strong>hash tables</strong>. Instead of searching, they compute where a value should live and jump straight to it.
      <br><br>
      When we look up a key, Swift feeds it to a <code>Hasher</code>, which produces an integer hash value. That integer, reduced modulo the number of buckets, points at exactly one bucket. We then check just that bucket. No scan of the whole collection.

      <br><br>
      Sometimes two different keys hash to the same bucket. That is a <strong>collision</strong>, and it is normal — the table just keeps both in that bucket and compares keys with <code>==</code> to tell them apart. As long as the buckets stay short, lookup, insertion, and <code>contains</code> all average out to O(1).
      <br><br>
      For any of this to work, the element type of a Set, or the key type of a Dictionary, must conform to the <code>Hashable</code> protocol. Swift's standard types — <code>Int</code>, <code>String</code>, <code>Bool</code>, and so on — already conform. For our own types, the compiler can synthesise the conformance automatically when every stored property is itself <code>Hashable</code>.
      <br><br>
      <pre><code>struct Point: Hashable {&#10;    let x: Int&#10;    let y: Int&#10;    // == and hash(into:) are synthesised for free&#10;}&#10;&#10;var seen: Set&lt;Point&gt; = []&#10;seen.insert(Point(x: 1, y: 2))   // works, because Point is Hashable</code></pre>
      One rule binds <code>Hashable</code> and <code>Equatable</code> together: if two values are equal, they <strong>must</strong> produce the same hash. Break that contract — for instance by hashing only some of the properties you compare in <code>==</code> — and the table will quietly fail to find values it definitely contains.

      <br><br>
      The trap that catches people is <code>Array.contains</code>. On a handful of elements it is perfectly fine — the constant factors are tiny and the code reads naturally. The cost only bites when the array is large and we call <code>contains</code> repeatedly, perhaps inside a loop. That pattern is secretly O(n times m), and it is the classic reason a feature feels instant on test data and crawls in production.
      <br><br>
      <pre><code>// Slow when ids is large and we check many times: O(n) per lookup&#10;let ids: [Int] = loadThousands()&#10;for query in queries where ids.contains(query) { ... }&#10;&#10;// Fast: build a Set once, then each check is O(1)&#10;let idSet = Set(ids)&#10;for query in queries where idSet.contains(query) { ... }</code></pre>
      The rule of thumb: if you are repeatedly asking "is this thing in here?", a Set will almost always beat an Array. If you mostly care about order and indexed access, stay with the Array.

  - id: value-semantics-cow
    title: "Value semantics and copy-on-write"
    summary: |
      Because Array, Set, and Dictionary are structs, copying one ought to copy all its elements. For a million-element array that would be ruinous. Swift dodges the cost with a trick called <strong>copy-on-write</strong>, or COW.
      <br><br>
      When we assign a collection to a second variable, Swift does <strong>not</strong> copy the buffer. Both variables point at the same underlying storage, and the storage keeps a reference count. The expensive copy is deferred until someone actually mutates a buffer that is shared. At that moment, and only then, Swift makes a private copy so the mutation can't be seen by the other holder.

      The payoff is the best of both worlds: collections behave like independent values, yet passing them around stays cheap. We only pay for a copy when we change a buffer that someone else is still holding.
      <br><br>
      There is one more consequence of value semantics worth burning in. A collection declared with <code>let</code> is <strong>immutable in its entirety</strong> — not just the variable binding, but the contents. Calling <code>append</code> on a <code>let</code> array is a compile error, full stop. This is unlike a class reference, where <code>let</code> only freezes the reference and you can still mutate the object it points to. With Swift's value-type collections, <code>let</code> means genuinely constant.
      <br><br>
      <pre><code>let fixed = [1, 2, 3]&#10;// fixed.append(4)   // error: cannot use mutating member on a 'let' constant&#10;var growable = [1, 2, 3]&#10;growable.append(4)   // fine</code></pre>

  - id: order-not-guaranteed
    title: "Why iteration order is NOT guaranteed"
    summary: |
      Run this twice and you may get two different printouts:
      <br><br>
      <pre><code>let s: Set&lt;Int&gt; = [1, 2, 3, 4, 5]&#10;for value in s { print(value) }&#10;// Run 1:  3 1 5 2 4&#10;// Run 2:  2 4 1 3 5   — a different order, same program</code></pre>
      This is not a bug. Swift <strong>deliberately</strong> refuses to promise an iteration order for Set and Dictionary. Since Swift 4.2 (the proposal was SE-0206), <code>Hashable</code> hashing uses the <strong>SipHash</strong> algorithm seeded with a <strong>per-process random seed</strong>. Every time the program launches, the seed changes, so the bucket positions change, so the iteration order changes.
      <br><br>
      Why on earth would a language make its output non-deterministic on purpose? Two reasons. First, security. If an attacker can predict which bucket a key lands in, they can craft thousands of inputs that all collide into one bucket, degrading every lookup to O(n) and grinding a server to a halt — a <strong>hash-flooding</strong> denial-of-service attack. A secret, per-process seed makes those collisions impossible to predict. Second, discipline. By scrambling the order, Swift stops us from accidentally writing code that depends on an order the type never promised. The bug surfaces immediately on your machine instead of mysteriously in production.
      <br><br>
      This is the sharpest contrast with <strong>Dart</strong>, and it is worth calling out directly. Dart's default <code>Set</code> and <code>Map</code> literals are backed by <code>LinkedHashSet</code> and <code>LinkedHashMap</code>, which <strong>preserve insertion order</strong>. Iterate a Dart map and you get the keys back in the order you inserted them, every run, guaranteed. Swift makes the opposite choice: its Set and Dictionary give you no order at all, and the order you do observe is randomised between runs.
      <br><br>
      <pre><code>// Swift — order is unspecified and varies between runs&#10;let d = ["a": 1, "b": 2, "c": 3]&#10;for (k, v) in d { print(k, v) }    // any order, changes each launch&#10;&#10;// Need a stable order? Sort explicitly:&#10;for k in d.keys.sorted() { print(k, d[k]!) }   // a, b, c — every time</code></pre>
      So the practical rule is simple. Never rely on the order of a Swift Set or Dictionary. If you need order, either keep the data in an Array, or sort the keys at the point where order actually matters. Arrays, of course, always preserve their order — that promise is the whole point of an Array.

  - id: beyond-the-big-three
    title: "Beyond the big three, and a little trivia"
    summary: |
      The three collections cover almost everything, but Swift's standard library has a few specialised siblings worth knowing.
      <br><br>
      <strong>ArraySlice</strong> is what you get when you take a range of an array, like <code>array[2..&lt;5]</code>. Crucially, a slice does <strong>not</strong> copy — it shares the parent's storage and just remembers a start and end. That makes slicing cheap, but it has a sharp edge: a slice keeps the whole parent buffer alive in memory, and its indices are inherited from the parent (a slice's first index may not be 0). When you want a standalone array, wrap it: <code>Array(slice)</code>.
      <br><br>
      <pre><code>let full = [10, 20, 30, 40, 50]&#10;let slice = full[2...]         // ArraySlice, shares storage&#10;print(slice.startIndex)        // 2, not 0&#10;let standalone = Array(slice)  // now a real, independent [Int]</code></pre>
      <strong>ContiguousArray</strong> is a variant that guarantees a contiguous C-style buffer even for class element types, where a plain <code>Array</code> might use a bridged Objective-C storage on Apple platforms. For pure-Swift numeric work it can shave off a little overhead.
      <br><br>
      <strong>Lazy sequences</strong> let us chain <code>map</code> and <code>filter</code> without building an intermediate array at each step. Writing <code>array.lazy.map(...).filter(...)</code> defers the work until something actually consumes the result, which can save a lot of allocation on long pipelines.
      <br><br>
      <pre><code>let result = (1...1_000_000).lazy&#10;    .map { $0 * 2 }&#10;    .filter { $0 % 3 == 0 }&#10;    .prefix(5)&#10;// Only computes enough to produce 5 elements&#10;print(Array(result))   // [6, 12, 18, 24, 30]</code></pre>
      All of this rests on a protocol hierarchy. <code>Sequence</code> is the base — anything you can iterate once with a <code>for</code> loop. <code>Collection</code> refines it with indices and multi-pass traversal, and Array, Set, and Dictionary all conform. That shared protocol is exactly why <code>map</code>, <code>filter</code>, and <code>reduce</code> work uniformly across all three: they are defined once on <code>Sequence</code>, and every collection inherits them.
      <br><br>
      A closing piece of history. Swift was announced at WWDC in June 2014 and open-sourced in December 2015, and these collection types have been part of the standard library since version 1.0. The randomised hashing that scrambles Set and Dictionary order, though, arrived later in Swift 4.2 in 2018 — a reminder that even a value type as ordinary-looking as a dictionary carries a deliberate security decision inside it.
