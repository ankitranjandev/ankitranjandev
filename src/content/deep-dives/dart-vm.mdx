  - id: dart-runtime-landscape
    title: "The Dart runtime landscape"
    summary: |
      Most programmers think of Dart as a language. But Dart is really a <strong>platform</strong> — a language, a compiler, a runtime, and a set of tools that work together.
      <br><br>
      When we write Dart code, something has to run it. In JavaScript, that's V8 or SpiderMonkey. In Java, that's the JVM. In Dart, it's the <strong>Dart VM</strong> — a virtual machine that executes Dart code and manages memory.
      <br><br>
      But here's what makes Dart unusual: it doesn't pick just one execution model. Dart has <strong>three</strong> different ways to run code, and each one is optimised for a different purpose.

      <br><br>
      <strong>JIT (Just-In-Time)</strong> compiles code while the program runs. This is what we use during development — it enables hot reload, fast iteration, and rich debugging. The trade-off is startup time and memory: the compiler itself has to be loaded.
      <br><br>
      <strong>AOT (Ahead-Of-Time)</strong> compiles everything to native machine code before the program runs. This is what ships in Flutter release builds. No compiler overhead at runtime, instant startup, smaller memory footprint. The trade-off is that hot reload doesn't work — the code is frozen at compile time.
      <br><br>
      <strong>Kernel</strong> is an intermediate representation — a platform-independent binary AST. Tools like the analyser and formatter work at this level. Most developers never touch it directly, but it's the glue between the frontend (parsing) and the backends (JIT/AOT).
      <br><br>
      This three-mode architecture is why Dart can be both a rapid-iteration development language and a high-performance production runtime. Let's see how each mode actually works.

    summary: |
      When we run <code>dart run app.dart</code> or <code>flutter run</code> in debug mode, we're using JIT compilation. The Dart VM loads our source code, compiles it to machine code on the fly, and starts executing.
      <br><br>
      But JIT isn't just "compile and run." The VM is constantly watching how our code behaves, and it uses that information to make the code faster.

      <br><br>
      <strong>How the feedback loop works:</strong>
      <br><br>
      1. The VM starts by compiling functions to <strong>unoptimised machine code</strong>. This is fast to generate but not especially fast to run.
      <br><br>
      2. As the code runs, the VM collects <strong>profile data</strong>: which functions are called often (hot), what types are actually passed to them, which branches are taken.
      <br><br>
      3. When a function becomes hot enough, the <strong>optimising compiler</strong> kicks in. It uses the profile data to generate much faster code — inlining function calls, eliminating type checks when the type is always the same, unrolling loops.
      <br><br>
      4. The optimised code <strong>replaces</strong> the old code. Future calls to that function run the faster version.
      <br><br>
      <strong>Deoptimisation — when assumptions fail.</strong> The optimising compiler makes assumptions based on what it has seen so far. "This function always receives an int." But what if, later, someone passes a String?
      <br><br>
      The VM handles this gracefully: it <strong>deoptimises</strong>. It throws away the optimised code and falls back to the unoptimised version, which can handle any type. The profile data updates, and the optimiser might try again with new assumptions.
      <br><br>
      <pre><code>void process(dynamic value) {
        // First 1000 calls: value is always int
        // Optimiser assumes int, generates fast int-specific code
        // Call 1001: value is String
        // Deoptimise! Fall back to generic code
      }</code></pre>
      <strong>Hot reload — the JIT superpower.</strong> Because the JIT keeps the VM running and can recompile individual functions, Dart can replace code without restarting the app. This is hot reload — we change a line, save the file, and see the result instantly. The VM patches the running code in place, preserving app state. No other compiled language offers this during development.

      <br><br>
      <strong>What AOT gives up:</strong>
      <br>- No hot reload — code is frozen at compile time
      <br>- No runtime reflection (in full AOT mode) — everything must be known statically
      <br>- Longer build times — compilation happens once, upfront
      <br><br>
      <strong>What AOT gains:</strong>
      <br>- Instant startup — no parsing, no JIT compilation
      <br>- Smaller memory footprint — no compiler in memory
      <br>- Consistent performance — no warm-up jitter
      <br>- Smaller binaries — tree shaking removes unused code
      <br><br>
      <pre><code># Compile a standalone executable
      dart compile exe app.dart -o myapp

      # Flutter release build (uses AOT)
      flutter build apk --release
      flutter build ios --release</code></pre>
      This is why the same Dart code feels different in debug vs release mode. Debug mode is JIT — slower to start, but you get hot reload. Release mode is AOT — instant and smooth from the first frame.

  - id: tree-shaking
    title: "Tree shaking — dead code elimination"
    summary: |
      When we import a package, we rarely use everything in it. Maybe we import all of Flutter, but our app only uses a handful of widgets. In a naive compiler, all that unused code would ship in the final binary.
      <br><br>
      Tree shaking solves this. The compiler starts from <code>main()</code> and traces every function, class, and variable that is actually reachable. Anything not in that trace gets removed.

      <br><br>
      <strong>Why "tree shaking"?</strong> Think of the program as a tree. <code>main()</code> is the root, function calls are branches. We shake the tree; whatever isn't connected falls off.
      <br><br>
      <strong>What survives tree shaking:</strong>
      <br>- Any code reachable from <code>main()</code>
      <br>- Anything called via <code>Function.apply</code> or <code>noSuchMethod</code>
      <br>- Code preserved by <code>@pragma('vm:entry-point')</code>
      <br><br>
      <strong>What gets removed:</strong>
      <br>- Unused classes, functions, methods
      <br>- Unused enum values
      <br>- Unreachable branches (if the condition is provably constant)
      <br><br>
      <pre><code>// This entire class gets removed if never instantiated
      class DebugHelper {
        static void logState() { ... }
        static void dumpWidgetTree() { ... }
      }

      // If this is never called, the whole class is gone
      // DebugHelper.logState();</code></pre>
      <strong>The flip side — what breaks tree shaking:</strong>
      <br><br>
      Dynamic features like <code>dart:mirrors</code> (reflection) defeat tree shaking because the compiler cannot prove what's unused. This is why reflection is disabled in AOT mode. If we use <code>Function.apply</code> or <code>dynamic</code> heavily, the compiler must keep more code "just in case."
      <br><br>
      For maximum tree shaking, keep types static and avoid <code>dynamic</code> where possible.

      <br><br>
      <strong>The old generation</strong> holds objects that survived multiple young-gen collections. It's larger and collected less often. The collector uses <strong>mark-sweep</strong> or <strong>mark-compact</strong>: mark everything reachable from the roots, then sweep away (or compact) the rest.
      <br><br>
      <strong>Write barriers.</strong> When old-gen objects point to young-gen objects, the GC needs to know — otherwise it might collect a young object that's still reachable. Write barriers are small checks inserted into every pointer write. When we assign a reference, the barrier records it so the GC doesn't miss it.

Marking happens on a background thread; only the final "stop-the-world" phase pauses the app, and it's kept short.       <strong>Why no shared memory?</strong> Because it eliminates whole classes of bugs. No data races, no deadlocks from lock contention, no need for synchronisation primitives. The trade-off is that communication has a cost — messages must be copied or serialised.
      <br><br>
      <pre><code>// Spawn an isolate for heavy computation
      void main() async {
        final result = await Isolate.run(() {
          // This runs in a separate isolate
          // with its own heap
          return computeHeavyTask();
        });
        print(result);
      }</code></pre>
      <strong>What can be sent between isolates:</strong>
      <br>- Primitives (int, double, bool, null)
      <br>- Strings
      <br>- Lists and Maps of sendable types
      <br>- SendPort and ReceivePort
      <br>- Transferable objects (TransferableTypedData)
      <br>- Custom objects (via serialisation)
      <br><br>
      <strong>What cannot be sent:</strong>
      <br>- Closures that capture non-sendable state
      <br>- Arbitrary objects with references to the heap
      <br><br>
      For CPU-intensive work — image processing, JSON parsing, cryptography — isolates let us use multiple cores without thread-safety headaches. Flutter's <code>compute()</code> function is a wrapper around this pattern.
      <br><br>
      <pre><code>// Flutter's compute() helper
      final result = await compute(parseJson, rawJson);

      // Equivalent to spawning an isolate manually
      // but with automatic serialisation handling</code></pre>
      The isolate model is more restrictive than threads, but that restriction is the point. We trade raw flexibility for safety and simplicity. For most app development, this is the right trade.
