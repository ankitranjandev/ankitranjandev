  - id: everything-is-an-object
    title: "Everything is an object"
    summary: |
      Java has a split personality. It has <strong>primitives</strong> — <code>int</code>, <code>double</code>, <code>boolean</code> — which are fast but not objects, and it has their boxed <strong>object</strong> counterparts — <code>Integer</code>, <code>Double</code>, <code>Boolean</code> — which can be null and live on the heap. Every Java developer has been bitten by the difference (autoboxing surprises, <code>NullPointerException</code> on an <code>Integer</code>, <code>==</code> comparing references instead of values).
      <br><br>
      Kotlin erases that split, at least from the programmer's point of view. In Kotlin there are no primitive types. <code>Int</code>, <code>Double</code>, and <code>Boolean</code> are ordinary classes with methods, and they sit in a single, unified type hierarchy. You can call <code>42.toString()</code> or <code>3.coerceAtLeast(5)</code> because <code>Int</code> is a real type.
      <br><br>
      Does that mean Kotlin is slow, boxing every number? No — and this is the clever part. The compiler quietly compiles <code>Int</code> down to a JVM primitive <code>int</code> wherever it safely can, and only falls back to the boxed <code>Integer</code> when it must (for example, a nullable <code>Int?</code>, or an <code>Int</code> used as a generic type argument). You get the clean mental model of "everything is an object" with most of the performance of primitives.
      <br><br>
      At the very top of this unified hierarchy sits <code>Any</code> — the root of all non-nullable types, Kotlin's equivalent of Java's <code>Object</code>. Every class you write implicitly extends <code>Any</code>, which is where <code>equals()</code>, <code>hashCode()</code>, and <code>toString()</code> come from. But <code>Any</code> is not quite the top. Above it sits something stranger.

  - id: the-type-hierarchy
    title: "Top and bottom: Any?, Unit, and Nothing"
    summary: |
      A well-designed type system has a <strong>top type</strong> (something every value is) and a <strong>bottom type</strong> (a subtype of everything). Kotlin has both, plus a curious little type in between that replaces Java's <code>void</code>.

      <br><br>
      <strong><code>Any?</code></strong> is the true top type: every possible value, including <code>null</code>, is an <code>Any?</code>. Its non-nullable sibling <code>Any</code> is the root of all non-nullable types.
      <br><br>
      <strong><code>Unit</code></strong> is Kotlin's replacement for <code>void</code>. The difference is subtle but important: <code>void</code> means "no return type at all," whereas <code>Unit</code> is a real type that has exactly <em>one</em> value, also written <code>Unit</code>. The name comes from type theory and functional languages, where the "unit type" is the canonical type with a single inhabitant. Because <code>Unit</code> is a real value, it can be used as a generic type argument — which matters a lot when you write generic code like <code>Function&lt;Unit&gt;</code>. A function that "returns nothing useful" actually returns <code>Unit</code>; you just never write it.
      <br><br>
      <strong><code>Nothing</code></strong> is the bottom type, and it is genuinely mind-bending the first time you meet it. It has <em>no instances at all</em> — you can never hold a value of type <code>Nothing</code>. It is a subtype of every other type. What could such a thing possibly be for?

  - id: the-power-of-nothing
    title: "The surprising power of Nothing"
    summary: |
      <code>Nothing</code> is the type of an expression that <strong>never produces a value</strong> — because it always throws, or loops forever. A function declared to return <code>Nothing</code> is promising the compiler "control will never return from here normally."
      <br><br>
      <pre><code>fun fail(message: String): Nothing {&#10;    throw IllegalStateException(message)&#10;}</code></pre>
      Why is that useful? Because <code>Nothing</code> is a subtype of everything, the compiler can slot a <code>Nothing</code>-returning call into <em>any</em> type context, and it understands that the code after it is unreachable. This is what makes the Elvis-with-throw idiom type-check so cleanly:
      <br><br>
      <pre><code>val name: String = user.name ?: fail("name is required")&#10;// If user.name is null, fail() throws — so after this line,&#10;// the compiler KNOWS name is a non-null String.</code></pre>
      The standard library's <code>TODO()</code> function returns <code>Nothing</code> for the same reason — you can drop <code>TODO()</code> into a function of any return type while you stub it out, and it compiles.
      <br><br>
      <code>Nothing</code> also rescues type inference. What is the element type of an empty list? The library says <code>emptyList()</code> returns <code>List&lt;Nothing&gt;</code>, and because <code>Nothing</code> is a subtype of everything, that empty list is assignable to <code>List&lt;String&gt;</code>, <code>List&lt;Int&gt;</code>, anything. Likewise <code>null</code>'s own type is <code>Nothing?</code>. These oddities — <code>Unit</code> with one value, <code>Nothing</code> with none — are not academic trivia; they are load-bearing parts of how everyday Kotlin code infers types correctly.

  - id: smart-casts
    title: "Smart casts and exhaustive when"
    summary: |
      In Java, checking a type and then using it means writing the type twice: <code>if (x instanceof String) { String s = (String) x; ... }</code>. Kotlin finds this redundant. Once you check a type, the compiler <strong>smart casts</strong> the value for you.
      <br><br>
      <pre><code>fun describe(x: Any): String {&#10;    if (x is String) {&#10;        // x is smart-cast to String here — no cast needed&#10;        return "string of length ${x.length}"&#10;    }&#10;    return "not a string"&#10;}</code></pre>
      The same flow analysis powers null handling (after <code>if (x != null)</code>, <code>x</code> is non-null) and pairs beautifully with <code>when</code>, Kotlin's vastly more powerful <code>switch</code>. A <code>when</code> can branch on types, ranges, and arbitrary conditions, smart-casting in each branch:
      <br><br>
      <pre><code>fun area(shape: Shape): Double = when (shape) {&#10;    is Circle -&gt; PI * shape.radius * shape.radius   // smart-cast to Circle&#10;    is Rectangle -&gt; shape.width * shape.height       // smart-cast to Rectangle&#10;}</code></pre>
      When <code>when</code> is used as an <em>expression</em> (its result is assigned or returned), the compiler insists it be <strong>exhaustive</strong> — every possible case must be covered, or you must add an <code>else</code>. Combined with sealed classes (covered in the Object Model episode), this gives you compile-time proof that you have handled every variant of a type. The same caveat from null safety applies: smart casts only work on values the compiler knows cannot change between the check and the use — local <code>val</code>s, not mutable properties that another thread could alter.

  - id: generics-and-erasure
    title: "Generics and type erasure"
    summary: |
      Generics let you write one <code>Box</code> that works for any type, with full type safety:
      <br><br>
      <pre><code>class Box&lt;T&gt;(val value: T)&#10;val intBox = Box(42)        // T inferred as Int&#10;val strBox = Box("hello")   // T inferred as String</code></pre>
      Kotlin runs on the JVM, and it inherits one of the JVM's most consequential design decisions: <strong>type erasure</strong>. At runtime, the type argument <code>T</code> is <em>erased</em> — a <code>Box&lt;Int&gt;</code> and a <code>Box&lt;String&gt;</code> are both just <code>Box</code>. The compiler uses the type information to check your code, then throws it away so the bytecode stays compatible with older Java.
      <br><br>
      This was a deliberate choice when generics were added to Java 5 in 2004: erasure preserved backward compatibility with pre-generics code, at the cost of losing type info at runtime. Kotlin lives with the consequence. You cannot, in ordinary code, ask "is this a <code>List&lt;String&gt;</code>?" at runtime — the <code>&lt;String&gt;</code> is gone.
      <br><br>
      <pre><code>fun &lt;T&gt; isType(value: Any): Boolean {&#10;    // return value is T   // ERROR: cannot check erased type&#10;    return false&#10;}</code></pre>
      Kotlin offers an escape hatch for this — <code>reified</code> type parameters — which we will reach shortly. But first, the most important thing generics force you to think about: <strong>variance</strong>.

  - id: variance-in-and-out
    title: "Variance: in, out, and PECS"
    summary: |
      Here is a question that has confused programmers for decades. A <code>Cat</code> is an <code>Animal</code>. So is a <code>List&lt;Cat&gt;</code> a <code>List&lt;Animal&gt;</code>? It seems like it should be — but allowing it naively is unsafe, because if you could treat a list of cats as a list of animals, you could then add a <code>Dog</code> to it. This is the <strong>variance</strong> problem.
      <br><br>
      Kotlin solves it with two keywords you declare on the type parameter itself — <strong>declaration-site variance</strong>, which is cleaner than Java's wildcards (<code>? extends</code> / <code>? super</code>) scattered at every use.
      <br><br>
      <svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 720 250" role="img" aria-label="Covariance with out for producers and contravariance with in for consumers" style="max-width: 100%; height: auto;"><style>.va-t{font-family:-apple-system,BlinkMacSystemFont,"Segoe UI",Inter,sans-serif;fill:#1e293b}.va-title{font-size:16px;font-weight:600}.va-kw{font-family:"SF Mono",Menlo,Monaco,monospace;font-size:20px;font-weight:700}.va-name{font-size:13px;font-weight:700;fill:#1e293b}.va-desc{font-size:11px;fill:#64748b}.va-code{font-family:"SF Mono",Menlo,Monaco,monospace;font-size:11px;fill:#475569}</style><text x="360" y="24" text-anchor="middle" class="va-t va-title">Producer-out, Consumer-in (PECS)</text><rect x="30" y="46" width="320" height="180" rx="8" fill="#ecfdf5" stroke="#10b981" stroke-width="1.5"/><text x="190" y="78" text-anchor="middle" class="va-t va-kw" fill="#10b981">out T</text><text x="190" y="102" text-anchor="middle" class="va-t va-name">covariant — a PRODUCER</text><text x="190" y="128" text-anchor="middle" class="va-t va-desc">only returns T, never accepts it</text><text x="190" y="152" text-anchor="middle" class="va-t va-code">interface Source&lt;out T&gt;</text><text x="190" y="172" text-anchor="middle" class="va-t va-code">{ fun next(): T }</text><text x="190" y="202" text-anchor="middle" class="va-t va-desc">Source&lt;Cat&gt;       <br><br>
      <strong><code>out T</code></strong> (covariance) means the type only ever <em>produces</em> <code>T</code> — it appears in return positions, never as a parameter. A producer of cats is safely a producer of animals, so <code>Source&lt;Cat&gt;</code> is a subtype of <code>Source&lt;Animal&gt;</code>. This is why Kotlin's read-only <code>List&lt;out E&gt;</code> is covariant: you can pass a <code>List&lt;Cat&gt;</code> where a <code>List&lt;Animal&gt;</code> is expected, because you can only read from it.
      <br><br>
      <strong><code>in T</code></strong> (contravariance) means the type only ever <em>consumes</em> <code>T</code> — it appears in parameter positions. A thing that can accept any animal can certainly accept a cat, so <code>Sink&lt;Animal&gt;</code> is a subtype of <code>Sink&lt;Cat&gt;</code>. This is the direction <code>Comparable&lt;in T&gt;</code> goes.
      <br><br>
      The mnemonic from Joshua Bloch's <em>Effective Java</em> is <strong>PECS</strong>: "Producer Extends, Consumer Super" — in Kotlin terms, "Producer <code>out</code>, Consumer <code>in</code>." When a class neither only-produces nor only-consumes (like <code>MutableList</code>, which does both), it must be invariant: <code>MutableList&lt;Cat&gt;</code> is <em>not</em> a <code>MutableList&lt;Animal&gt;</code>, and that is exactly the safety the compiler is protecting. When you do not know or care about the argument, the <strong>star projection</strong> <code>List&lt;*&gt;</code> says "a list of something."

  - id: reified-type-parameters
    title: "Reified type parameters"
    summary: |
      Type erasure normally means a generic function cannot know its type argument at runtime. Kotlin punches a hole through this with a feature Java does not have: <code>reified</code> type parameters on <code>inline</code> functions.
      <br><br>
      Recall from the Functions episode that an <code>inline</code> function has its body copied into each call site at compile time. Because the body is inlined, the compiler knows the concrete type argument at every call — so it can substitute the real type directly into the generated code. Marking the type parameter <code>reified</code> unlocks that:
      <br><br>
      <pre><code>inline fun &lt;reified T&gt; Any.isA(): Boolean = this is T&#10;&#10;println("hi".isA&lt;String&gt;())   // true&#10;println(42.isA&lt;String&gt;())     // false&#10;&#10;// The classic use: type-safe JSON parsing with no Class&lt;T&gt; argument&#10;inline fun &lt;reified T&gt; String.fromJson(): T =&#10;    gson.fromJson(this, T::class.java)&#10;&#10;val user: User = jsonString.fromJson()   // T inferred, no boilerplate</code></pre>
      Inside a <code>reified</code> function you can use <code>T</code> as if it were a real type: do <code>is T</code> checks, write <code>T::class</code>, create reflection references. This is why so many Kotlin libraries have such clean APIs — instead of forcing you to pass a <code>Class&lt;T&gt;</code> token around (as Java libraries must), they hide it behind a reified inline function. The limitation is the flip side of how it works: <code>reified</code> requires <code>inline</code>, so it cannot be used on regular functions or stored for later — the type is known only because the call was inlined.
