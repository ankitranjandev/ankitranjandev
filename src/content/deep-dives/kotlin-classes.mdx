  - id: classes-and-constructors
    title: "Classes and the primary constructor"
    summary: |
      A Java class that just holds a name and an age needs a field declaration, a constructor that assigns each field, and usually a pair of accessors — a dozen lines of mechanical code. Kotlin collapses all of that into the class header itself, using the <strong>primary constructor</strong>.
      <br><br>
      <pre><code>class Person(val name: String, var age: Int)</code></pre>
      That single line declares a class, a constructor taking two arguments, and two properties — <code>name</code> (read-only, because <code>val</code>) and <code>age</code> (mutable, because <code>var</code>). There is no body because none is needed. When you do need setup logic that runs at construction, an <code>init</code> block handles it, and it can see the constructor parameters:
      <br><br>
      <pre><code>class Person(val name: String, var age: Int) {&#10;    init {&#10;        require(age &gt;= 0) { "age cannot be negative" }&#10;    }&#10;}</code></pre>
      A class can also declare <strong>secondary constructors</strong> with the <code>constructor</code> keyword for alternative ways to build it, but in practice Kotlin's <strong>default and named arguments</strong> usually make them unnecessary — instead of three overloaded constructors, you write one primary constructor with default values. This is the same conciseness pillar from the flagship episode, applied to object construction: say what the class <em>is</em>, and let the compiler write the ceremony.

  - id: properties-not-fields
    title: "Properties, not fields"
    summary: |
      A subtle but important difference from Java: in Kotlin you declare <strong>properties</strong>, never raw fields. <code>val name: String</code> is not a field — it is a property with a generated getter (and, for <code>var</code>, a setter). You always interact with a property through its accessors, even when you write what looks like direct access.
      <br><br>
      This means you can give a property custom accessor logic without changing how callers use it — no need to refactor a public field into a getter later, the Java tax that gave the world endless boilerplate getters "just in case."
      <br><br>
      <pre><code>class Rectangle(val width: Int, val height: Int) {&#10;    // A computed property — no stored value, just a getter&#10;    val area: Int&#10;        get() = width * height&#10;&#10;    var label: String = ""&#10;        set(value) {&#10;            field = value.trim()   // 'field' is the backing field&#10;        }&#10;}&#10;&#10;val r = Rectangle(3, 4)&#10;println(r.area)    // 12 — looks like a field, runs the getter</code></pre>
      The special identifier <code>field</code> inside an accessor refers to the <strong>backing field</strong> — the actual storage behind the property. Kotlin only generates a backing field when a property needs one (when an accessor actually references <code>field</code>); a purely computed property like <code>area</code> stores nothing at all. This "everything is a property" model is what lets Kotlin classes stay so small while remaining flexible.

  - id: data-classes
    title: "Data classes"
    summary: |
      Many classes exist only to hold data — a request, a response, a row, a coordinate. For these, writing <code>equals()</code>, <code>hashCode()</code>, and <code>toString()</code> by hand is tedious and a frequent source of bugs (forget one field in <code>equals</code> and you get baffling behaviour). The <code>data</code> modifier tells the compiler to generate them all from the primary-constructor properties.

      <br><br>
      <pre><code>data class Point(val x: Int, val y: Int)&#10;&#10;val a = Point(1, 2)&#10;val b = Point(1, 2)&#10;println(a == b)            // true — value equality, not reference&#10;println(a)                 // Point(x=1, y=2)&#10;val c = a.copy(y = 99)     // Point(x=1, y=99) — clone with one change&#10;val (x, y) = a             // destructuring via componentN()</code></pre>
      Two generated members deserve a note. <code>copy()</code> is the workhorse of immutable data: rather than mutating a <code>Point</code>, you make a near-identical copy with a few fields changed — the foundation of immutable state in modern Kotlin UIs. And <code>componentN()</code> functions enable <strong>destructuring</strong>, letting you unpack an object into several variables at once.
      <br><br>
      A few rules keep data classes honest: they must have at least one primary-constructor parameter, and the generated <code>equals</code>/<code>hashCode</code>/<code>toString</code> consider <em>only</em> the properties declared in the primary constructor — anything declared in the class body is ignored by them. They also cannot be <code>abstract</code> or <code>sealed</code>. They are a tool for plain data, and they are superb at it.

  - id: objects-and-companions
    title: "object and companion object — Kotlin has no static"
    summary: |
      Kotlin removed a keyword every Java programmer knows: there is no <code>static</code>. The reasoning is that <code>static</code> members sit awkwardly outside the object model — they are not part of any instance, cannot implement interfaces cleanly, and complicate inheritance. Kotlin replaces <code>static</code> with something more principled: the <code>object</code> keyword.
      <br><br>
      An <strong>object declaration</strong> defines a class <em>and</em> its single instance in one go — a <strong>singleton</strong>, baked into the language so you never write the tricky thread-safe singleton boilerplate again.
      <br><br>
      <pre><code>object AppConfig {&#10;    val version = "1.0.0"&#10;    fun describe() = "App v$version"&#10;}&#10;&#10;AppConfig.describe()   // access the one instance directly</code></pre>
      When you want members tied to a <em>class</em> rather than to instances — the role <code>static</code> played — you nest a <strong>companion object</strong> inside the class. Members of the companion are accessed through the class name, just like static members, but the companion is a real object: it can implement interfaces, hold state, and be passed around.
      <br><br>
      <pre><code>class User private constructor(val name: String) {&#10;    companion object {&#10;        // a factory function — the idiomatic replacement for static methods&#10;        fun create(name: String): User = User(name.trim())&#10;    }&#10;}&#10;&#10;val u = User.create("  Ankit  ")   // looks static, but it's a companion call</code></pre>
      A common pattern combines the two ideas: a <code>private</code> constructor plus a companion <strong>factory function</strong>, giving you full control over how instances are made. Objects also appear as <strong>anonymous objects</strong> (<code>object : Listener { ... }</code>), Kotlin's replacement for Java's anonymous inner classes.

  - id: sealed-classes
    title: "Sealed classes: closed hierarchies"
    summary: |
      Often a value is one of a fixed, known set of shapes. A network result is <em>Loading</em>, <em>Success</em>, or <em>Error</em>. A UI event is one of a handful of kinds. An <code>enum</code> can model a fixed set of <em>constants</em>, but what if each case needs to carry different data — a success holds a payload, an error holds a message? That is what a <strong>sealed class</strong> (or sealed interface) is for.
      <br><br>
      "Sealed" means the set of direct subclasses is <strong>known and closed</strong> at compile time — they must all be declared in the same module. The compiler therefore knows every possible case, which it uses to make <code>when</code> <strong>exhaustive</strong> without an <code>else</code>.

      The real payoff is maintenance. If you later add a fourth case — say <code>Empty</code> — every <code>when</code> that handled <code>Result</code> without an <code>else</code> immediately fails to compile, pointing you to exactly the places that must handle the new state. This turns a whole class of "forgot to handle that case" runtime bugs into compile errors. (Notice <code>Nothing</code> from the Type System episode doing real work here, and the <code>out T</code> variance making <code>Loading</code> usable as any <code>Result</code>.)

  - id: delegation
    title: "Delegation: enums, interfaces, and by"
    summary: |
      Two more pieces complete Kotlin's object model. First, <strong>enum classes</strong> — like Java's, but able to carry properties and methods on each constant:
      <br><br>
      <pre><code>enum class Planet(val gravity: Double) {&#10;    EARTH(9.81), MARS(3.71), MOON(1.62);&#10;    fun weight(mass: Double) = mass * gravity&#10;}</code></pre>
      Second, and more distinctively, Kotlin makes <strong>delegation</strong> a first-class language feature through the <code>by</code> keyword — a direct nod to the design principle "favour composition over inheritance." <strong>Class delegation</strong> lets one class implement an interface by forwarding the work to another object, with zero boilerplate:
      <br><br>
      <pre><code>interface Logger { fun log(msg: String) }&#10;&#10;class ConsoleLogger : Logger {&#10;    override fun log(msg: String) = println(msg)&#10;}&#10;&#10;// Service IS-A Logger, but delegates all Logger methods to 'impl'&#10;class Service(impl: Logger) : Logger by impl</code></pre>
      <strong>Property delegation</strong> applies the same idea to a single property: the property's get/set are handled by a delegate object. The standard library ships several, and they are everywhere in real code — <code>by lazy { }</code> computes a value once on first access, <code>by Delegates.observable { }</code> fires a callback on every change, and <code>by map</code> backs a property with a map entry:
      <br><br>
      <pre><code>val config: Config by lazy { loadConfig() }   // computed once, on first use&#10;var name: String by Delegates.observable("") { _, old, new -&gt;&#10;    println("name: $old -&gt; $new")&#10;}</code></pre>
      Finally, a design decision worth knowing: in Kotlin, classes and members are <strong>final by default</strong>. To allow inheritance or overriding you must explicitly mark them <code>open</code>. This directly follows Joshua Bloch's advice in <em>Effective Java</em> — "design and document for inheritance, or else prohibit it" — making the safe choice the default and forcing you to opt in to the fragile one. It is the same philosophy as null-safety and immutability: Kotlin nudges you toward the robust option and asks you to be deliberate about the risky one.
