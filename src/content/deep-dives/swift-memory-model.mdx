  - id: two-worlds
    title: "Two worlds — value vs reference"
    summary: |
      Every question we will ask in this episode reduces to one fork in the road. When a name is bound to a value, does that name <strong>own an independent copy</strong>, or does it hold a <strong>pointer to something shared</strong>? Swift splits the world cleanly. Structs, enums and tuples are value types: copied on assignment, copied on argument passing, with no shared identity. Classes, actors and closures are reference types: shared, heap-resident, and tracked by ARC.
      <br><br>
      The introductory episode showed the behaviour. Here we go under it. The behaviour falls out of a single physical fact — where the bytes live, and who else can reach them. Hold this in mind: a value type <em>is</em> its bytes; a reference type is a pointer to its bytes.

      <br><br>
      This is also where <strong>identity</strong> and <strong>equivalence</strong> part ways. Two values can be <em>equivalent</em> (<code>==</code>, the <code>Equatable</code> protocol) without being the same storage. Only reference types have <em>identity</em> (<code>===</code>, comparing <code>ObjectIdentifier</code>, essentially the address). Asking whether two <code>Int</code>s are "the same object" is meaningless — there is no single object, only a value that may exist in many places at once. Keep these two questions separate and most of Swift's surprises evaporate.

  - id: where-values-live
    title: "Where values live — stack, heap, inline"
    summary: |
      Let's get concrete about storage. A local value type with a known size lives on the <strong>stack</strong> — a contiguous region the function owns, where allocation is just moving a pointer and deallocation is moving it back. A class instance lives on the <strong>heap</strong>, reached through a pointer; allocating it is a real call into the allocator. Value types can also live <strong>inline</strong> inside whatever contains them: a struct field inside another struct is laid out flat, not boxed.
      <br><br>
      <code>MemoryLayout</code> lets us inspect this directly, and the three numbers it reports are not interchangeable. <code>size</code> is the bytes the value actually uses. <code>stride</code> is the bytes one element occupies inside an <code>Array</code> — size rounded up to satisfy alignment, so it includes trailing padding. <code>alignment</code> is the address boundary the value must start on.
      <br><br>
      <pre><code>struct Pair { var a: Int; var b: Int }&#10;MemoryLayout&lt;Pair&gt;.size       // 16&#10;MemoryLayout&lt;Pair&gt;.stride     // 16&#10;MemoryLayout&lt;Pair&gt;.alignment  // 8&#10;&#10;struct Mixed { var flag: Bool; var n: Int }&#10;MemoryLayout&lt;Mixed&gt;.size       // 9  (1 used + 8)&#10;MemoryLayout&lt;Mixed&gt;.stride     // 16 (padded up to alignment)&#10;MemoryLayout&lt;Mixed&gt;.alignment  // 8</code></pre>
      Two <code>Int</code>s make a 16-byte struct because each <code>Int</code> is 8 bytes on a 64-bit platform, and that whole 16 bytes sits on the stack — no allocation, no pointer chase. The <code>Mixed</code> example is the more revealing one. Its <code>size</code> is 9, but its <code>stride</code> is 16: a <code>Bool</code> followed by an <code>Int</code> needs 7 bytes of padding so the next element in an array starts on an 8-byte boundary. This is why <code>size</code> and <code>stride</code> diverge, and why you should always reason about array memory in terms of <code>stride</code>.
      <br><br>
      A class is different. <code>MemoryLayout&lt;SomeClass&gt;.size</code> is always 8 — the size of the pointer, not the object. The object itself lives on the heap and carries overhead the struct never pays. That overhead is the subject of the next topic.

  - id: class-instance-anatomy
    title: "Anatomy of a class instance"
    summary: |
      What does a heap-allocated class instance actually look like in memory? It is not just your stored properties. Every Swift class instance is a <strong>HeapObject</strong>, and it begins with a fixed header before any of your fields appear.
      <br><br>
      The header is two machine words. The first is the <strong>type-metadata pointer</strong>, historically called the <code>isa</code> pointer after its Objective-C ancestor. It points at the class's metadata record — the runtime structure describing the type, including its vtable for dynamic dispatch, its superclass, and its size. The second word holds the <strong>inline reference counts</strong>: a packed bitfield containing the strong count and the unowned count, plus a flag indicating whether a side table has been spun up for this object.

      So a class wrapping a single <code>Int</code> is not 8 bytes on the heap — it is 16 bytes of header plus 8 bytes of payload, reached through an 8-byte pointer on the stack. This is the concrete cost of reference semantics, and it is exactly why Apple's guidance is to prefer structs. A struct of the same shape pays none of it: no header, no pointer, no allocation.
      <br><br>
      The metadata pointer is also what makes dynamic features work. When you call a method that might be overridden, the runtime reads the metadata to find the right implementation. When you ask for an instance's type with <code>type(of:)</code>, it reads the same pointer. The header is not bookkeeping bolted on the side — it is how a reference type knows what it is.

  - id: arc-mechanics
    title: "ARC mechanics — retain, release, deinit"
    summary: |
      Now the engine. Each strong reference to a class instance contributes one to the strong count in that header. The Swift compiler inserts a call to <code>swift_retain</code> when a new strong reference is formed, and a call to <code>swift_release</code> when one goes away. When <code>swift_release</code> drops the count to zero, the object is destroyed <strong>immediately</strong>: its <code>deinit</code> runs, its stored properties are released in turn, and the memory is handed back to the allocator.
      <br><br>
      The word that matters is <em>immediately</em>. ARC is not a tracing garbage collector. There is no background thread scanning the heap for unreachable objects, and there are no stop-the-world pauses. Destruction is deterministic — it happens at a knowable point in your program, at the instruction where the last reference is released. That determinism is why <code>deinit</code> is a reliable place to close a file or release a lock.
      <br><br>
      <pre><code>final class Resource {&#10;  let id: Int&#10;  init(_ id: Int) { self.id = id; print("open \(id)") }&#10;  deinit { print("close \(id)") }&#10;}&#10;&#10;func scope() {&#10;  let a = Resource(1)   // swift_retain not needed; init returns +1&#10;  let b = a             // swift_retain(a)  → count 2&#10;  _ = b&#10;}                       // swift_release(b), then swift_release(a)&#10;                        // count 0 → deinit → "close 1"</code></pre>
      Deinit ordering follows reference structure, not declaration order. When an object dies, ARC releases each of its stored reference-typed properties; if one of those drops to zero, <em>its</em> deinit runs before control returns, and so on down the graph. The result is a depth-first teardown rooted at the object that died. There is no guaranteed order between unrelated objects going out of scope, only that each is destroyed when its own count hits zero.
      <br><br>
      And to repeat the boundary precisely: ARC governs reference types only. A struct or enum is never retained or released. It is copied, and a copy simply ceases to exist when its scope ends — though if a struct <em>contains</em> a class reference, copying that struct retains the class, and dropping the copy releases it. We will return to that hybrid cost at the end.

  - id: arc-at-sil
    title: "ARC at the SIL level — +1, +0, and OSSA"
    summary: |
      Why is ARC traffic a real performance concern, and how does the compiler make it cheap? The answer lives in <strong>SIL</strong>, Swift Intermediate Language — the representation the compiler uses between the type-checked AST and LLVM IR. You can see it with <code>swiftc -emit-sil</code>. SIL is where Swift runs its own high-level optimisations, including ARC optimisation, before LLVM ever sees the code.
      <br><br>
      SIL describes how values cross function boundaries using <strong>ownership conventions</strong>. A value passed at <strong>+1</strong> is <em>owned</em>: the callee receives a reference it is responsible for releasing. A value passed at <strong>+0</strong> is <em>guaranteed</em> (also called borrowed): the caller promises to keep it alive for the duration of the call, so the callee neither retains nor releases it. Most parameters are +0, which is why a well-optimised Swift program emits far less ARC traffic than a naive count of references would suggest.

      <br><br>
      Modern SIL carries ownership explicitly in a form called <strong>OSSA</strong> — Ownership SSA. In OSSA every value has a defined owner, and operations like <code>copy_value</code>, <code>destroy_value</code> and <code>begin_borrow</code> make the lifetime visible to the optimiser. The ARC optimiser then performs <strong>redundant retain/release elimination</strong>: if a value is retained and then released with no operation in between that could observe the count, both can be deleted. A retain that is immediately followed by a release across a borrowed call is a classic pair the optimiser removes outright.
      <br><br>
      <pre><code>// Conceptual SIL after lowering (simplified)&#10;%a = apply @Obj.init()          // owned, +1&#10;%b = copy_value %a              // retain → count 2&#10;apply @use(%a)                  // guaranteed, +0 — borrows, no ARC&#10;destroy_value %b               // release → count 1&#10;destroy_value %a               // release → count 0 → deinit</code></pre>
      Why does any of this matter to you? Because every retain and release is an atomic increment or decrement on that header word, and atomics are not free, especially under contention. The optimiser's whole job here is to prove that ARC operations are unnecessary and delete them. When you mark a class <code>final</code>, avoid escaping closures, or pass values <code>borrowing</code> (a later topic), you are giving the optimiser the information it needs to win these eliminations.

  - id: weak-and-unowned
    title: "weak and unowned — side tables and traps"
    summary: |
      A strong reference keeps its target alive. Sometimes we want a reference that does not. Swift offers two, and they are implemented very differently under the hood.
      <br><br>
      A <strong>weak</strong> reference must be <code>Optional</code>, and ARC automatically zeroes it to <code>nil</code> the moment its target is deallocated. To make that possible, a weak reference does not point straight at the object. It points at an entry in a <strong>side table</strong> — a separate runtime structure that holds the weak count and a slot the object knows to clear during deallocation. The first time an object gains a weak reference, the runtime allocates a side table for it and stores a pointer to it in the header. When the object dies, the runtime walks the side table and nils every weak reference, so no dangling pointer can survive.

      <br><br>
      An <strong>unowned</strong> reference is cheaper and sharper. The common form, <code>unowned(safe)</code>, does not keep its target alive and is non-optional. It does not require a side table for zeroing; instead it relies on the unowned count in the header, and accessing it after the object's strong count has reached zero <strong>traps</strong> — a deliberate crash rather than reading freed memory. There is also <code>unowned(unsafe)</code>, which omits even that check and behaves like a raw pointer; use it only when you can prove the lifetime yourself.
      <br><br>
      <pre><code>protocol DataSource: AnyObject { func rows() -&gt; Int }&#10;&#10;final class TableView {&#10;  weak var delegate: DataSource?   // weak: delegate may outlive nothing&#10;}&#10;&#10;final class Customer {&#10;  let card: CreditCard&#10;  init() { card = CreditCard(owner: self) }&#10;}&#10;final class CreditCard {&#10;  unowned let owner: Customer       // a card never outlives its owner&#10;  init(owner: Customer) { self.owner = owner }&#10;}</code></pre>
      The rule of thumb: use <code>weak</code> when the reference can legitimately become <code>nil</code> — the delegate pattern is the canonical case, where the view must not keep its controller alive. Use <code>unowned</code> only when the relationship guarantees the target outlives the reference, as a credit card cannot outlive its customer. When in doubt, reach for <code>weak</code>; a stray <code>nil</code> is recoverable, a trap is not.

  - id: reference-cycles-in-depth
    title: "Reference cycles in depth"
    summary: |
      ARC has exactly one blind spot, and it follows directly from how it counts. If object A holds a strong reference to B, and B holds a strong reference back to A, then each keeps the other's count above zero forever. Remove every outside reference and the pair still cannot die. This is a <strong>strong reference cycle</strong>, and it is the only way to leak memory in ordinary Swift.
      <br><br>
      The classic shape is a parent and a child that both point at each other. The fix is to decide which direction is <em>owning</em> and which is merely <em>referencing</em>. A parent usually owns its children strongly; each child should refer back to its parent <code>weak</code> or <code>unowned</code>, so the back-edge does not contribute to the count.
      <br><br>
      <pre><code>final class Parent {&#10;  var children: [Child] = []        // strong, owning&#10;}&#10;final class Child {&#10;  weak var parent: Parent?          // weak, non-owning back-edge&#10;}</code></pre>
      The subtler trap is closures. A closure is a reference type, and when it captures <code>self</code> it holds a <em>strong</em> reference to it by default. Store that closure on <code>self</code> — as a completion handler, a Combine sink, a timer callback — and you have built a cycle between the object and its own closure. Nothing outside is wrong; the object simply owns a closure that owns the object.
      <br><br>
      <pre><code>final class Loader {&#10;  var onDone: (() -&gt; Void)?&#10;  func start() {&#10;    onDone = { [weak self] in     // capture list breaks the cycle&#10;      guard let self else { return }&#10;      self.finish()&#10;    }&#10;  }&#10;  func finish() { }&#10;}</code></pre>
      The capture list <code>[weak self]</code> is the same fix as a weak property, applied to the reference a closure holds implicitly. Use <code>[weak self]</code> when the closure may outlive <code>self</code> and a missing <code>self</code> is acceptable; use <code>[unowned self]</code> only when the closure is guaranteed to run during <code>self</code>'s lifetime. To reason about any suspected leak, draw the strong edges as arrows and look for a loop you can follow back to where you started. Every such loop is a cycle, and every cycle is broken by turning one edge non-owning.

  - id: copy-on-write-by-hand
    title: "Copy-on-write, by hand"
    summary: |
      Here is the puzzle that ties values and references together. <code>Array</code>, <code>String</code>, <code>Dictionary</code> and <code>Set</code> are all value-type structs, so by the rules of topic one, assigning one should copy its contents. Yet assigning a million-element array is instant. How?
      <br><br>
      The answer is <strong>copy-on-write</strong>, and it is not a language feature — it is a pattern the standard library implements <em>by hand</em>. Each of these structs is a thin wrapper holding a single reference to a heap <strong>buffer</strong>, which is a class. Assigning the struct copies only that one pointer and retains the buffer; both variables now share it. The expensive copy is deferred until someone mutates, and even then it happens only if the buffer is shared. The decision is made with one standard-library call: <code>isKnownUniquelyReferenced(&amp;storage)</code>.

      <pre><code>final class Box&lt;T&gt; { var value: T; init(_ v: T) { value = v } }&#10;&#10;struct COW&lt;T&gt; {&#10;  private var storage: Box&lt;T&gt;&#10;  init(_ v: T) { storage = Box(v) }&#10;&#10;  var value: T { storage.value }&#10;&#10;  mutating func mutate(_ body: (inout T) -&gt; Void) {&#10;    if !isKnownUniquelyReferenced(&amp;storage) {&#10;      storage = Box(storage.value)   // shared → fork first&#10;    }&#10;    body(&amp;storage.value)             // now unique → mutate in place&#10;  }&#10;}</code></pre>

This is value semantics with the cost profile of references, and it is exactly how every Swift collection behaves.   - id: law-of-exclusivity
    title: "The Law of Exclusivity (SE-0176)"
    summary: |
      With copy-on-write relying on a buffer being uniquely referenced, Swift needs a guarantee that a value is not being read and written through two paths at once. That guarantee is the <strong>Law of Exclusivity</strong>, introduced as SE-0176. The rule: a variable may not be accessed in conflicting ways at the same time. A read can overlap another read, but a write must have <strong>exclusive</strong> access — no other read or write to the same storage may overlap it.
      <br><br>
      Most violations are caught <strong>statically</strong> at compile time. But some accesses can only be checked at run time — notably when an <code>inout</code> argument aliases a global or escapes through a closure. For those, the compiler inserts a runtime <strong>exclusivity check</strong> that <strong>traps</strong> if it detects an overlapping access. In SIL these appear as <code>begin_access</code> and <code>end_access</code> markers bracketing each access; the runtime tracks active accesses and crashes on a conflict rather than letting you corrupt data.

  - id: inout-in-depth
    title: "inout in depth — copy-in, copy-out"
    summary: |
      We reach for <code>inout</code> when a function must modify its caller's variable. It is tempting to think of it as pass-by-reference, but the model Swift actually defines is <strong>copy-in / copy-out</strong>, also called writeback. The semantics are precise: at the call, the argument's current value is copied <em>in</em> to the parameter; the function works on that local copy; and when the function returns, the final value is copied <em>out</em>, overwriting the original.
      <br><br>
      <pre><code>func bump(_ n: inout Int) { n += 1 }&#10;&#10;var score = 41&#10;bump(&amp;score)   // 1. copy score (41) in&#10;               // 2. body sets the copy to 42&#10;               // 3. copy 42 back out to score&#10;print(score)   // 42</code></pre>
      Why define it as copy-in/copy-out rather than as a raw address? Because the source of an <code>inout</code> argument is not always plain storage. It might be a computed property with a getter and setter, or a subscript, or a property with observers. The writeback model gives all of these one consistent meaning: call the getter to copy in, run the body, then call the setter to copy out. A computed property has no address to point at, yet <code>inout</code> works on it perfectly because the model is defined in terms of get and set, not pointers.
      <br><br>
      Here is where exclusivity and <code>inout</code> meet. Because <code>inout</code> demands exclusive access for the whole call, the compiler may safely apply an <strong>as-if pass-by-reference</strong> optimisation: when the argument is ordinary stack storage with no observers, it can skip the copies and let the function mutate the original directly, because exclusivity guarantees nobody else can observe the storage mid-call. The observable behaviour is identical to copy-in/copy-out; the optimiser simply elides the copies when it can prove they make no difference. So the <em>model</em> is writeback, and the <em>implementation</em> is often a direct address — and exclusivity is the proof that lets one stand in for the other.

  - id: ownership-frontier
    title: "The ownership frontier — ~Copyable"
    summary: |
      Everything so far assumed values can be freely copied. But some things should not be copied — a file descriptor, a lock, a unique hardware handle. Copying them invites double-free bugs and aliasing. Swift 5.9 opened this frontier with <strong>non-copyable types</strong>, written <code>~Copyable</code> (SE-0390). Suppress the implicit <code>Copyable</code> conformance and the compiler enforces that the value has exactly one owner at a time. There is no copy, so there is nothing to alias.
      <br><br>
      To move such values around safely, SE-0377 added two parameter ownership modifiers. A <strong>consuming</strong> parameter takes ownership: the value is <em>moved</em> in, and the caller's binding becomes invalid afterwards — using it again is a compile error. A <strong>borrowing</strong> parameter lends temporary read access and leaves ownership with the caller, so the caller may keep using the value after the call returns.

      <pre><code>struct FileDescriptor: ~Copyable {&#10;  let fd: Int32&#10;  init(_ fd: Int32) { self.fd = fd }&#10;  consuming func close() { /* close(fd) */ }&#10;  borrowing func peek() -&gt; Int32 { fd }&#10;  deinit { /* close(fd) if not already closed */ }&#10;}&#10;&#10;func use(_ f: borrowing FileDescriptor) { _ = f.peek() }&#10;&#10;var file = FileDescriptor(3)&#10;use(file)        // borrowed — file still valid afterwards&#10;file.close()     // consumed — file is now invalid&#10;// use(file)     // compile error: used after consume</code></pre>
      Why is this such a leap? Because a non-copyable type gives us a <strong>resource handle that the compiler proves you cannot misuse</strong> — no double-close, no use-after-close, no accidental shared mutation — without any heap allocation and without ARC. The guarantees you previously got only from a reference type, with all its header and reference-counting overhead, you can now get from a flat value on the stack. This is the move-only model, and it is how Swift reaches into systems-programming territory while keeping its safety promises.

  - id: putting-it-to-work
    title: "Putting it to work"
    summary: |
      Let's land all of this where it touches real code. The most common surprise in practice is the <strong>hybrid value type</strong>: a struct that contains a class reference. Copying the struct is cheap on its fields, but each contained reference triggers a <code>swift_retain</code>, and dropping the copy triggers a <code>swift_release</code>. So a "value" type can quietly carry ARC traffic proportional to how many references it holds.
      <br><br>
      <pre><code>struct Cache {            // looks like a value type...&#10;  var id: Int            // free to copy&#10;  var box: Buffer        // class — copying Cache retains box&#10;}&#10;final class Buffer { var data = [UInt8]() }</code></pre>
      Copy a <code>Cache</code> in a tight loop and you pay an atomic retain and release every iteration, even though no data is duplicated. The fix is usually to borrow rather than copy, to mark the class <code>final</code> so the optimiser can reason about it, or to hand the value across an API boundary with the ownership modifiers from the previous topic. Accidental copies also hide in closures that capture a value, in <code>defer</code> blocks, and in passing a value where the callee takes it <code>owning</code>.
      <br><br>
      How do you actually see any of this? Two tools. The first is the SIL itself — compile with <code>swiftc -emit-sil file.swift</code> and read the <code>copy_value</code>, <code>destroy_value</code>, <code>retain</code> and <code>release</code> operations the compiler emitted. If you see retains and releases bracketing a hot loop, that is your ARC cost made visible. The second is <strong>Instruments</strong>: the Allocations instrument shows heap allocations and the live-object count, and the Leaks instrument flags reference cycles that were never freed.
      <br><br>
      <pre><code># See exactly what ARC the compiler kept&#10;swiftc -O -emit-sil Cache.swift | grep -E "retain|release|copy_value"&#10;&#10;# Or watch allocations live&#10;# Product ▸ Profile ▸ Allocations / Leaks in Instruments</code></pre>
      Pull the threads together and the whole model is one coherent design. Values live inline and copy cheaply; references live on the heap behind a header and are freed deterministically by ARC the instant their count hits zero. Copy-on-write gives collections value semantics at reference cost. Exclusivity keeps every mutation sound. And non-copyable types extend the same safety to resources that must never be copied at all. Once you can see where a value lives and who is allowed to touch it, Swift stops surprising you — because you are now reading the same map the compiler is.
