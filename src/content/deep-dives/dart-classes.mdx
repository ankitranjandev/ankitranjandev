  - id: why-classes
    title: "Why classes exist"
    summary: |
      We have functions. We have data. Why do we need another concept?
      <br><br>
      The answer is <strong>coupling</strong>. Some data and some functions belong together. A user has a name and an email. A user can update their profile. A user can log out. The data and the behaviour are inseparable.
      <br><br>
      <pre><code>// Without classes — data and functions are separate
      var userName = 'Ankit';
      var userEmail = 'ankit@example.com';

      void updateEmail(String newEmail) {
        userEmail = newEmail;   // which user? unclear
      }

      // With a class — data and behaviour travel together
      class User {
        String name;
        String email;

        User(this.name, this.email);

        void updateEmail(String newEmail) {
          email = newEmail;   // obviously this user's email
        }
      }</code></pre>
      A class is a blueprint. It describes what data an object will hold (fields) and what it can do (methods). An object is a specific instance of that blueprint — a real user with a real name.

  - id: constructors
    title: "Constructors — the many ways to build objects"
    summary: |
      A constructor is a special function that creates an object. Dart gives us several flavours, each solving a different problem.
      <br><br>
      <strong>The generative constructor</strong> — the default. It creates a new instance every time.
      <br><br>
      <pre><code>class Point {
        final double x;
        final double y;

        // Long form
        Point(double x, double y) : x = x, y = y;

        // Short form — same thing
        Point(this.x, this.y);
      }

      var p = Point(3, 4);</code></pre>
      The <code>this.x</code> syntax is Dart's shorthand for "assign the parameter to the field of the same name." It's so common that the long form is rarely seen.
      <br><br>
      <strong>Named constructors</strong> — when one constructor isn't enough.
      <br><br>
      <pre><code>class Point {
        final double x;
        final double y;

        Point(this.x, this.y);

        Point.origin() : x = 0, y = 0;

        Point.fromJson(Map&lt;String, double&gt; json)
            : x = json['x']!,
              y = json['y']!;
      }

      var p1 = Point(3, 4);
      var p2 = Point.origin();
      var p3 = Point.fromJson({'x': 1, 'y': 2});</code></pre>
      <strong>Factory constructors</strong> — when you need control over what gets returned.
      <br><br>
      <pre><code>class Logger {
        static final Logger _instance = Logger._internal();

        factory Logger() {
          return _instance;   // always returns the same object
        }

        Logger._internal();   // private constructor
      }

      var a = Logger();
      var b = Logger();
      print(identical(a, b));   // true — same object</code></pre>
      A factory constructor can return an existing instance, a subtype, or even null (if the return type is nullable). Regular constructors always create a new instance of exactly that class.

      <br><br>
      <strong>Const constructors</strong> — compile-time canonicalisation.
      <br><br>
      <pre><code>class Point {
        final double x;
        final double y;

        const Point(this.x, this.y);
      }

      const p1 = Point(0, 0);
      const p2 = Point(0, 0);
      print(identical(p1, p2));   // true — same canonical instance</code></pre>
      When all fields are final and the constructor is marked <code>const</code>, Dart can create the object at compile time. Two <code>const Point(0, 0)</code> expressions anywhere in the program will be the exact same object in memory.

  - id: initializer-list
    title: "Initializer lists and assertions"
    summary: |
      Sometimes we need to do work before the constructor body runs — setting final fields, calling superclass constructors, or validating inputs. That's what the initializer list is for.
      <br><br>
      <pre><code>class Rectangle {
        final double width;
        final double height;
        final double area;

        Rectangle(this.width, this.height)
            : area = width * height,           // computed field
              assert(width > 0),               // validation
              assert(height > 0);
      }</code></pre>
      The initializer list runs <strong>before</strong> the constructor body. This matters for final fields — they must be set before the body runs, so computed values go in the initializer list.
      <br><br>
      <strong>Order matters:</strong>
      <br>1. Initializer list (left to right)
      <br>2. Superclass constructor
      <br>3. Constructor body
      <br><br>
      <pre><code>class Animal {
        final String name;
        Animal(this.name) {
          print('Animal constructor');
        }
      }

      class Dog extends Animal {
        final String breed;

        Dog(String name, this.breed)
            : super(name) {    // calls Animal(name)
          print('Dog constructor');
        }
      }

      var d = Dog('Buddy', 'Labrador');
      // prints: Animal constructor
      // prints: Dog constructor</code></pre>
      <strong>Assertions in initializer lists.</strong> The <code>assert</code> statements only run in debug mode. They're stripped out of release builds, so they're free in production. Use them liberally to catch bugs early.
      <br><br>
      <pre><code>class Percentage {
        final double value;

        Percentage(this.value)
            : assert(value >= 0 && value <= 100,
                  'Percentage must be 0-100, got $value');</code></pre>

  - id: inheritance
    title: "Inheritance — extending behaviour"
    summary: |
      Inheritance lets us build new classes on top of existing ones. The child class gets all the parent's fields and methods, and can add or override them.
      <br><br>
      <pre><code>class Animal {
        final String name;
        Animal(this.name);

        void speak() => print('...');
      }

      class Dog extends Animal {
        Dog(String name) : super(name);

        @override
        void speak() => print('$name says woof!');
      }

      class Cat extends Animal {
        Cat(String name) : super(name);

        @override
        void speak() => print('$name says meow!');
      }

      void main() {
        List&lt;Animal&gt; pets = [Dog('Buddy'), Cat('Whiskers')];
        for (var pet in pets) {
          pet.speak();   // polymorphism — right method called
        }
      }
      // Buddy says woof!
      // Whiskers says meow!</code></pre>

  - id: abstract-classes
    title: "Abstract classes and interfaces"
    summary: |
      Sometimes we want to define a contract without providing the implementation. That's what abstract classes are for.
      <br><br>
      <pre><code>abstract class Shape {
        double get area;           // no body — must be implemented
        double get perimeter;

        void describe() {          // has a body — inherited as-is
          print('Area: $area, Perimeter: $perimeter');
        }
      }

      class Circle extends Shape {
        final double radius;
        Circle(this.radius);

        @override
        double get area => 3.14159 * radius * radius;

        @override
        double get perimeter => 2 * 3.14159 * radius;
      }</code></pre>
      An abstract class cannot be instantiated directly. <code>Shape()</code> is a compile error. But we can extend it, and then create instances of the concrete subclass.
      <br><br>
      <strong>Interfaces — implicit in Dart.</strong> Every class automatically defines an interface. Any class can implement any other class's interface.
      <br><br>
      <pre><code>class Printable {
        void printSelf() => print(toString());
      }

      class Report implements Printable {
        final String title;
        Report(this.title);

        @override
        void printSelf() => print('Report: $title');

        @override
        String toString() => 'Report($title)';
      }</code></pre>
      <strong>extends vs implements:</strong>
      <br>• <code>extends</code> — inherit implementation (fields, method bodies)
      <br>• <code>implements</code> — promise to provide the interface (must override everything)
      <br><br>
      A class can extend one class but implement many interfaces. This gives us single inheritance with multiple interface conformance.
      <br><br>
      <pre><code>abstract class Flyable {
        void fly();
      }

      abstract class Swimmable {
        void swim();
      }

      class Duck extends Animal implements Flyable, Swimmable {
        Duck(String name) : super(name);

        @override
        void fly() => print('$name is flying');

        @override
        void swim() => print('$name is swimming');
      }</code></pre>

  - id: equality
    title: "Object equality — == vs identical()"
    summary: |
      When are two objects "the same"? Dart gives us two different answers.
      <br><br>
      <strong>identical()</strong> — are they the exact same object in memory?
      <br><br>
      <pre><code>var a = Point(3, 4);
      var b = Point(3, 4);
      var c = a;

      print(identical(a, b));   // false — different objects
      print(identical(a, c));   // true — same object</code></pre>
      <strong>== operator</strong> — are they meaningfully equal?
      <br><br>
      By default, <code>==</code> does the same thing as <code>identical()</code>. But we can override it to define our own equality.
      <br><br>
      <pre><code>class Point {
        final double x;
        final double y;

        Point(this.x, this.y);

        @override
        bool operator ==(Object other) {
          if (identical(this, other)) return true;
          return other is Point && other.x == x && other.y == y;
        }

        @override
        int get hashCode => Object.hash(x, y);
      }

      var a = Point(3, 4);
      var b = Point(3, 4);
      print(a == b);            // true — same values
      print(identical(a, b));   // false — different objects</code></pre>
      <svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 720 240" role="img" aria-label="Identity vs equality" style="max-width: 100%; height: auto;"><style>.c4-t{font-family:-apple-system,BlinkMacSystemFont,"Segoe UI",Inter,sans-serif;fill:#1e293b}.c4-title{font-size:16px;font-weight:600}.c4-mn{font-family:"SF Mono",Menlo,Monaco,monospace;font-size:12px}.c4-label{font-size:12px;fill:#64748b}.c4-note{font-size:11px;fill:#475569}</style><text x="360" y="26" text-anchor="middle" class="c4-t c4-title">Identity vs Equality</text><rect x="40" y="55" width="300" height="150" fill="#fee2e2" stroke="#dc2626" stroke-width="2" rx="6"/><text x="190" y="82" text-anchor="middle" class="c4-t c4-mn" font-weight="700">identical(a, b)</text><text x="60" y="110" class="c4-t c4-note">Same object in memory?</text><text x="60" y="135" class="c4-t c4-note">• Checks memory address</text><text x="60" y="155" class="c4-t c4-note">• Cannot be overridden</text><text x="60" y="175" class="c4-t c4-note">• Always O(1)</text><text x="60" y="195" class="c4-t c4-label">Use for: caching, singletons</text><rect x="380" y="55" width="300" height="150" fill="#dcfce7" stroke="#10b981" stroke-width="2" rx="6"/><text x="530" y="82" text-anchor="middle" class="c4-t c4-mn" font-weight="700">a == b</text><text x="400" y="110" class="c4-t c4-note">Meaningfully equal?</text><text x="400" y="135" class="c4-t c4-note">• Checks logical equality</text><text x="400" y="155" class="c4-t c4-note">• Override operator ==</text><text x="400" y="175" class="c4-t c4-note">• Must also override hashCode</text><text x="400" y="195" class="c4-t c4-label">Use for: value comparison</text></svg>
      <br><br>
      <strong>The hashCode contract.</strong> If <code>a == b</code>, then <code>a.hashCode == b.hashCode</code>. This is required for Sets and Maps to work correctly. If two objects are equal, they must have the same hash code. The reverse isn't required — different objects can have the same hash code (that's a collision, and it's fine).

  - id: getters-setters
    title: "Getters, setters, and encapsulation"
    summary: |
      Fields expose data directly. Sometimes we want more control — validation, computation, or logging. That's where getters and setters come in.
      <br><br>
      <pre><code>class Temperature {
        double _celsius;   // private field (underscore prefix)

        Temperature(this._celsius);

        // Getter — computed property
        double get fahrenheit => _celsius * 9 / 5 + 32;

        // Setter — with validation
        set celsius(double value) {
          if (value < -273.15) {
            throw ArgumentError('Below absolute zero!');
          }
          _celsius = value;
        }

        double get celsius => _celsius;
      }

      var temp = Temperature(20);
      print(temp.fahrenheit);   // 68.0
      temp.celsius = 25;        // uses the setter
      temp.celsius = -300;      // throws ArgumentError</code></pre>
      <strong>The underscore convention.</strong> In Dart, a leading underscore makes a name <strong>library-private</strong>. It's not accessible from other files. This is how we hide implementation details.
      <br><br>
      <pre><code>// In user.dart
      class User {
        String _password;   // private — other files can't access

        User(this._password);

        bool checkPassword(String attempt) {
          return attempt == _password;
        }
      }

      // In main.dart
      var user = User('secret123');
      print(user._password);        // ERROR — can't access
      print(user.checkPassword('secret123'));   // true</code></pre>
      <strong>Final fields vs getters.</strong> For simple read-only data, a final field is simpler. Use a getter when the value is computed or when you might need to add logic later.
      <br><br>
      <pre><code>class Circle {
        final double radius;     // simple read-only field

        Circle(this.radius);

        double get area => 3.14159 * radius * radius;   // computed
        double get circumference => 2 * 3.14159 * radius;
      }</code></pre>
      The caller doesn't know (or care) whether <code>area</code> is a stored field or a computed getter. That's encapsulation — we can change the implementation without changing the interface.
