  - id: the-problem
    title: "The problem extensions solve"
    summary: |
      We've all been here. You're working with a <code>String</code> and you need a method that doesn't exist. Maybe you want to check if it's a valid email. Maybe you want to capitalise the first letter. Maybe you want to parse it as JSON.
      <br><br>
      <pre><code>// What we want to write
      var email = 'test@example.com';
      if (email.isValidEmail) { ... }

      // What we have to write without extensions
      if (isValidEmail(email)) { ... }</code></pre>
      The second version works, but it feels backwards. The data comes first, then the operation. With methods, we write <code>email.isValidEmail</code> — the noun, then the verb. It reads naturally.
      <br><br>
      But we can't modify the <code>String</code> class. It's part of Dart's core library. We don't own it.
      <br><br>
      <strong>Extension methods solve this.</strong> They let us add methods to any type — even types we don't control — without modifying the original class or creating a wrapper.

  - id: syntax
    title: "Extension syntax"
    summary: |
      An extension is a named block that adds members to an existing type.
      <br><br>
      <pre><code>extension StringValidation on String {
        bool get isValidEmail {
          return contains('@') && contains('.');
        }

        bool get isNumeric {
          return double.tryParse(this) != null;
        }
      }

      void main() {
        print('test@example.com'.isValidEmail);  // true
        print('42'.isNumeric);                    // true
        print('hello'.isNumeric);                 // false
      }</code></pre>
      The structure is: <code>extension Name on Type { members }</code>. Inside the extension, <code>this</code> refers to the instance — the string we're calling the method on.
      <br><br>
      <strong>Extension methods</strong> — work just like regular methods.
      <br><br>
      <pre><code>extension StringHelpers on String {
        String truncate(int maxLength) {
          if (length <= maxLength) return this;
          return '\${substring(0, maxLength)}...';
        }

        String repeat(int times) {
          return List.filled(times, this).join();
        }
      }

      'Hello World'.truncate(5);   // 'Hello...'
      'ha'.repeat(3);              // 'hahaha'</code></pre>
      <strong>Extension getters and setters</strong> — add computed properties.
      <br><br>
      <pre><code>extension StringCasing on String {
        String get capitalised {
          if (isEmpty) return this;
          return '\${this[0].toUpperCase()}\${substring(1)}';
        }

        String get reversed {
          return split('').reversed.join();
        }
      }

      'hello'.capitalised;   // 'Hello'
      'dart'.reversed;       // 'trad'</code></pre>
      <strong>Extension operators</strong> — even operators can be extended.
      <br><br>
      <pre><code>extension NumericString on String {
        String operator *(int times) {
          return List.filled(times, this).join();
        }
      }

      'ha' * 3;   // 'hahaha'</code></pre>

  - id: generic-extensions
    title: "Generic extensions"
    summary: |
      Extensions can be generic, adding methods to parameterised types like <code>List&lt;T&gt;</code> or <code>Future&lt;T&gt;</code>.
      <br><br>
      <pre><code>extension ListHelpers&lt;T&gt; on List&lt;T&gt; {
        T? get firstOrNull => isEmpty ? null : first;

        T? get lastOrNull => isEmpty ? null : last;

        List&lt;T&gt; takeWhileInclusive(bool Function(T) test) {
          var result = &lt;T&gt;[];
          for (var item in this) {
            result.add(item);
            if (!test(item)) break;
          }
          return result;
        }
      }

      var numbers = [1, 2, 3, 4, 5];
      numbers.firstOrNull;                           // 1
      &lt;int&gt;[].firstOrNull;                           // null
      numbers.takeWhileInclusive((n) => n < 3);     // [1, 2, 3]</code></pre>
      <strong>Constrained generics</strong> — require the type parameter to implement something.
      <br><br>
      <pre><code>extension ComparableList&lt;T extends Comparable&lt;T&gt;&gt; on List&lt;T&gt; {
        T? get maxOrNull {
          if (isEmpty) return null;
          return reduce((a, b) => a.compareTo(b) > 0 ? a : b);
        }

        T? get minOrNull {
          if (isEmpty) return null;
          return reduce((a, b) => a.compareTo(b) < 0 ? a : b);
        }
      }

      [3, 1, 4, 1, 5].maxOrNull;   // 5
      ['z', 'a', 'm'].minOrNull;   // 'a'</code></pre>
      The constraint <code>T extends Comparable&lt;T&gt;</code> ensures <code>compareTo</code> is available. Without it, the extension couldn't call comparison methods.

  - id: resolution
    title: "How extension methods are resolved"
    summary: |
      Extension methods are resolved statically — at compile time, based on the declared type, not the runtime type.
      <br><br>
      <pre><code>extension on int {
        String get description => 'an integer';
      }

      extension on num {
        String get description => 'a number';
      }

      int x = 42;
      num y = 42;

      print(x.description);   // 'an integer'
      print(y.description);   // 'a number' — even though it's really an int</code></pre>
      The variable <code>y</code> is declared as <code>num</code>, so it uses the <code>num</code> extension — even though the actual value is an <code>int</code>.
      <br><br>
      <svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 720 200" role="img" aria-label="Static vs dynamic dispatch" style="max-width: 100%; height: auto;"><style>.e2-t{font-family:-apple-system,BlinkMacSystemFont,"Segoe UI",Inter,sans-serif;fill:#1e293b}.e2-title{font-size:16px;font-weight:600}.e2-mn{font-family:"SF Mono",Menlo,Monaco,monospace;font-size:11px}.e2-label{font-size:11px;fill:#64748b}.e2-note{font-size:11px;fill:#475569}</style><text x="360" y="26" text-anchor="middle" class="e2-t e2-title">Extensions use static dispatch (declared type matters)</text><rect x="40" y="55" width="300" height="120" fill="#fee2e2" stroke="#dc2626" stroke-width="1.5" rx="6"/><text x="190" y="80" text-anchor="middle" class="e2-t e2-mn" font-weight="600">Instance methods</text><text x="60" y="105" class="e2-t e2-note">• Dynamic dispatch (runtime type)</text><text x="60" y="125" class="e2-t e2-note">• Can be overridden in subclasses</text><text x="60" y="145" class="e2-t e2-note">• Polymorphism works</text><text x="60" y="165" class="e2-t e2-mn">Animal a = Dog(); a.speak(); // woof</text><rect x="380" y="55" width="300" height="120" fill="#dbeafe" stroke="#0175C2" stroke-width="1.5" rx="6"/><text x="530" y="80" text-anchor="middle" class="e2-t e2-mn" font-weight="600">Extension methods</text><text x="400" y="105" class="e2-t e2-note">• Static dispatch (declared type)</text><text x="400" y="125" class="e2-t e2-note">• Cannot be overridden</text><text x="400" y="145" class="e2-t e2-note">• No polymorphism</text><text x="400" y="165" class="e2-t e2-mn">num n = 42; n.ext(); // uses num ext</text></svg>
      <br><br>
      <strong>This means extension methods cannot be overridden.</strong> If you need polymorphic behaviour, use instance methods or mixins instead.
      <br><br>
      <strong>Explicit extension application.</strong> When multiple extensions define the same member, you can disambiguate:
      <br><br>
      <pre><code>extension A on String {
        String get decorated => '*** \$this ***';
      }

      extension B on String {
        String get decorated => '--- \$this ---';
      }

      // Ambiguous — won't compile
      // 'hello'.decorated;

      // Explicit — works
      A('hello').decorated;   // '*** hello ***'
      B('hello').decorated;   // '--- hello ---'</code></pre>

  - id: unnamed-extensions
    title: "Unnamed and private extensions"
    summary: |
      Extensions can be unnamed. This is useful for one-off helpers that don't need to be referenced elsewhere.
      <br><br>
      <pre><code>extension on String {
        bool get isBlank => trim().isEmpty;
      }

      '   '.isBlank;   // true</code></pre>
      Unnamed extensions cannot be explicitly applied — there's no name to use. If there's a conflict, you can't disambiguate.
      <br><br>
      <strong>Private extensions</strong> — prefix the name with underscore.
      <br><br>
      <pre><code>extension _InternalHelpers on String {
        String get _processed => trim().toLowerCase();
      }

      // Only usable within this file
      var clean = input._processed;</code></pre>
      Private extensions are useful for internal implementation helpers that shouldn't leak into the public API.
      <br><br>
      <strong>Exporting extensions.</strong> Named extensions can be exported from libraries, making them available to consumers.
      <br><br>
      <pre><code>// In string_extensions.dart
      extension StringExtensions on String {
        String get capitalised => ...;
      }

      // In another file
      import 'string_extensions.dart';

      'hello'.capitalised;   // works because extension is imported</code></pre>
      Extensions are only available when they're in scope. Import the file, get the methods. Don't import it, they don't exist.

  - id: practical-patterns
    title: "Practical extension patterns"
    summary: |
      Extensions shine for specific use cases. Here are patterns that work well.
      <br><br>
      <strong>Pattern 1: Fluent APIs on primitives</strong>
      <br><br>
      <pre><code>extension DurationHelpers on int {
        Duration get milliseconds => Duration(milliseconds: this);
        Duration get seconds => Duration(seconds: this);
        Duration get minutes => Duration(minutes: this);
        Duration get hours => Duration(hours: this);
      }

      await Future.delayed(500.milliseconds);
      var timeout = 30.seconds;
      var workday = 8.hours;</code></pre>
      <strong>Pattern 2: Safe parsing</strong>
      <br><br>
      <pre><code>extension SafeParsing on String {
        int? toIntOrNull() => int.tryParse(this);
        double? toDoubleOrNull() => double.tryParse(this);
        DateTime? toDateTimeOrNull() => DateTime.tryParse(this);
      }

      var age = '25'.toIntOrNull() ?? 0;
      var price = 'invalid'.toDoubleOrNull();   // null</code></pre>
      <strong>Pattern 3: Collection conveniences</strong>
      <br><br>
      <pre><code>extension IterableHelpers&lt;T&gt; on Iterable&lt;T&gt; {
        Map&lt;K, List&lt;T&gt;&gt; groupBy&lt;K&gt;(K Function(T) keyFn) {
          var result = &lt;K, List&lt;T&gt;&gt;{};
          for (var item in this) {
            var key = keyFn(item);
            (result[key] ??= []).add(item);
          }
          return result;
        }

        T? firstWhereOrNull(bool Function(T) test) {
          for (var item in this) {
            if (test(item)) return item;
          }
          return null;
        }
      }

      var users = [User('Alice', 25), User('Bob', 25), User('Carol', 30)];
      var byAge = users.groupBy((u) => u.age);
      // {25: [Alice, Bob], 30: [Carol]}</code></pre>
      <strong>Pattern 4: Null-safe operations</strong>
      <br><br>
      <pre><code>extension NullableString on String? {
        bool get isNullOrEmpty => this == null || this!.isEmpty;
        bool get isNullOrBlank => this == null || this!.trim().isEmpty;
        String orEmpty() => this ?? '';
      }

      String? name;
      if (name.isNullOrBlank) {
        name = 'Anonymous';
      }</code></pre>

      <br><br>
      <strong>Don't use extensions when:</strong>
      <br><br>
      <strong>1. You need state.</strong> Extensions can't add fields. If you need to store data per instance, use a mixin or wrapper class.
      <br><br>
      <pre><code>// Won't work — extensions can't have instance fields
      extension on User {
        int loginCount = 0;   // ERROR
      }</code></pre>
      <strong>2. You need polymorphism.</strong> Extension methods use static dispatch. If you need different behaviour based on runtime type, use instance methods.
      <br><br>
      <pre><code>extension on Animal {
        void feed() => print('feeding animal');
      }

      extension on Dog {
        void feed() => print('feeding dog');
      }

      Animal pet = Dog();
      pet.feed();   // 'feeding animal' — not what we wanted</code></pre>
      <strong>3. The method is core to the type's identity.</strong> If every user of the type needs this method, it probably belongs on the class itself, not an extension.
      <br><br>
      <strong>4. You're adding too many.</strong> Twenty extension methods on String is a code smell. Consider whether a dedicated class would be clearer.
